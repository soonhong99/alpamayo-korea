## 1. 문제 정의 — 왜 이 연구를 시작했는가

### 현재 상황

Alpamayo 1.5는 자율주행 차량이 앞으로 어떻게 달릴지를 예측하는 10B 파라미터 AI 모델이다.  
이 모델이 경로 1개를 계산하는 데 걸리는 시간은 **약 5~6초**다.

```
현재:  ~5,009ms (약 5초)
목표:  ≤ 99ms
격차:  약 50배
```

자율주행 차량은 끊임없이 빠른 판단이 필요하다.  
5초에 한 번 결정한다면 시속 60km 차량이 **83미터를 무판단 상태로 달리게 된다**.

### 하드웨어 환경

우리는 NVIDIA Jetson AGX Thor 보드에서 모델을 실행한다.  
이 보드의 핵심 특징은 **CPU와 GPU가 메모리를 물리적으로 공유한다**는 것이다.  
이를 **iGPU (integrated GPU)** 또는 **통합 메모리 아키텍처**라고 부른다.

```
[일반 PC / 서버 dGPU 구조]
CPU ─── 시스템 RAM (CPU 전용)
GPU ─── VRAM (GPU 전용)
CPU ↔ GPU 데이터 전달: PCIe 버스 경유 (느림, ~16 GB/s)

[Jetson AGX Thor iGPU 구조]
CPU ─┐
     ├── 공유 DRAM 128GB (CPU+GPU 동시 사용)
GPU ─┘
CPU ↔ GPU 데이터 전달: 메모리 주소 공유 (빠름, 복사 없음)
```

iGPU는 PCIe 병목이 없다는 장점이 있지만,  
**CPU와 GPU가 같은 메모리 버스를 나눠 쓴다**는 구조적 차이가 있다.

## 2. 논문 아이디어 — 무엇을 배웠나

### 논문 1: RT-Swap (RTAS 2024)

**제목**: *RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference*  
**핵심 문제**: GPU 메모리(VRAM)가 부족한 embedded GPU에서 여러 DNN을 동시에 실행할 때 속도가 느려지는 문제

**해결 아이디어 — 비동기 스왑 파이프라인**

AI 모델은 수십 개의 레이어로 구성된다. 각 레이어를 실행하려면:
1. 해당 레이어의 가중치(weight)를 메모리에서 읽어온다 (로딩)
2. 읽어온 가중치로 행렬 연산을 수행한다 (계산)

기존 방식은 이 두 단계를 순서대로 실행한다:

```
기존 순차 방식:
[레이어1 로딩] → [레이어1 계산] → [레이어2 로딩] → [레이어2 계산] → ...
←────── 로딩 + 계산 ──────→ ←────── 로딩 + 계산 ──────→

총 시간 = Σ(로딩 시간 + 계산 시간)
```

RT-Swap의 핵심 아이디어:
레이어1을 **계산하는 동안** 레이어2를 **미리 로딩**하면, 두 작업이 겹쳐서 총 시간이 줄어든다.

```
RT-Swap 비동기 방식:
스트림 A (로딩): [레이어1 로딩][레이어2 로딩][레이어3 로딩]...
스트림 B (계산):     ↑겹침↑   [레이어1 계산][레이어2 계산]...

총 시간 ≈ Σ max(다음 레이어 로딩 시간, 현재 레이어 계산 시간)
       < Σ(로딩 + 계산)
```

수식으로:
$$T_{sequential} = \sum_{i} (t_{load,i} + t_{compute,i})$$
$$T_{async} = \sum_{i} \max(t_{load,i+1},\ t_{compute,i})$$

계산 시간이 로딩 시간보다 길면, 로딩이 계산 뒤에 숨어서 **로딩 시간이 공짜**가 된다.

**이것이 작동하는 전제 조건**:  
dGPU에는 **DMA 엔진(Direct Memory Access Engine)**이 GPU 연산 코어(SM)와 **물리적으로 분리**되어 있어서, 메모리 로딩과 행렬 연산이 진짜로 동시에 실행될 수 있다.

### 논문 2: Demand Layering (RTSS 2022)

**제목**: *Demand Layering for Real-Time DNN Inference with Minimized Memory Usage*  
**핵심 문제**: GPU 메모리가 부족해 모델 전체를 올릴 수 없을 때도 실시간 추론이 가능한가?

**해결 아이디어 — 운영체제의 Virtual Memory에서 영감**

운영체제는 RAM이 부족할 때 하드디스크를 마치 RAM처럼 쓰는 "가상 메모리" 기법을 쓴다.  
필요한 페이지가 없으면 그때 디스크에서 불러오는 것이 "Demand Paging"이다.

Demand Layering은 이 개념을 DNN에 적용했다:

```
운영체제 비유                    Demand Layering 적용
─────────────────────────────────────────────────────
가상 메모리 (전체 주소 공간)  ↔  전체 모델 가중치 (DRAM)
물리 메모리 (실제 RAM)        ↔  GPU 메모리 (제한된 공간)
페이지 = RAM 단위              ↔  레이어 = 모델 실행 단위
Page Fault → 디스크에서 로드  ↔  Layer Miss → DRAM에서 로드
```

핵심: 현재 실행 중인 레이어와 바로 다음 레이어 **2개만** 메모리에 올려두면 충분하다.  
나머지 레이어는 필요할 때 불러온다 → 메모리 사용량 최소화 + prefetch로 속도 유지

---

### 두 논문의 결합 — 우리의 시도

두 논문의 아이디어를 합치면 다음과 같은 설계가 나온다:

```
[우리가 구현하려 했던 것]

prefetch 스트림 → [레이어 i+1 가중치를 L2 캐시로 미리 이동]
compute 스트림  →                [레이어 i 행렬 연산]
                  ←────── 두 작업을 동시에 실행 ──────→
```

CUDA에는 "Stream"이라는 개념이 있어서, 서로 다른 작업을 다른 Stream에 배치하면 동시에 실행할 수 있다.

## 3. 검증 실험 — 이론이 맞는지 먼저 확인했다

### 3.1 레이어별 계산 시간 측정 (Phase 0-B)

파이프라인이 효과를 내려면 한 가지 조건이 필요하다:  
**레이어 계산 시간(t_compute) > 다음 레이어 로딩 시간(t_prefetch)**  
그래야 로딩이 계산 시간 뒤에 완전히 숨겨진다.

각 레이어의 계산 시간을 CUDA Event로 직접 측정한 결과:

| 측정 항목 | 결과 |
|-----------|------|
| 레이어당 평균 계산 시간 | **2.62 ms** |
| MLP 가중치 이론 로딩 시간 (DRAM 기준) | **0.44 ms** |
| Attention 가중치 이론 로딩 시간 | **0.11 ms** |
| 파이프라인 조건 성립 여부 | **✅ 전 36개 레이어 전부 성립** |

수치만 보면 **이론적으로 파이프라인이 완전히 가능**했다.  
계산이 로딩보다 2.62 / 0.44 = **6배** 오래 걸리니, 로딩이 계산에 충분히 숨겨질 것으로 예상했다.

### 3.2 대역폭 측정 (Phase 0-A)

| 측정 항목 | 결과 |
|-----------|------|
| GPU L2 캐시 대역폭 | **1,126 GB/s** |
| GPU DRAM 대역폭 | **227 GB/s** |

L2 캐시는 DRAM보다 5배 빠르다.  
"가중치를 미리 L2에 올려두면 실제 계산 때 L2에서 읽어 빠르게 처리된다"는 가설을 세웠다.

## 4. 실제 실험 — 결과 ❌

### 구현 방식

`tensor.sum()`을 prefetch 스트림에서 호출하는 방식으로 가중치를 L2에 "워밍(warming)"한 후, compute 스트림에서 실제 레이어 연산을 수행했다.

```python
# prefetch stream에서: 다음 레이어 가중치를 L2로 올리기
with torch.cuda.stream(prefetch_stream):
    next_layer_weights.view(-1).sum()   # L2 캐시로 데이터를 끌어올리는 트릭

# compute stream에서: 현재 레이어 계산
with torch.cuda.stream(compute_stream):
    output = current_layer(hidden_state)

torch.cuda.synchronize()
```

### 측정 결과

| 모드 | ms/step | 65 step 합산 | 비교 |
|------|---------|-------------|------|
| Baseline (파이프라인 없음) | 95.96 ms | 6,237 ms | 기준 |
| Async Pipeline 적용 | 147.82 ms | 9,609 ms | **0.65× — 54% 느려짐 ❌** |

파이프라인이 빠르게 만들 것이라 예상했지만, **오히려 3.4초 더 느려졌다**.

## 5. 실패 원인 분석 — 왜 이렇게 됐나

### 원인 1 (치명적): `tensor.sum()`은 "읽기를 2번" 시킨다

이것이 가장 중요한 발견이다.

**우리가 기대한 동작**:
```
sum() → 데이터를 L2에 올린다 → 이후 GEMM이 L2에서 빠르게 읽는다
```

**실제 동작**:
```
sum() → 386 MB 전체를 DRAM에서 L2로 읽어온다
         └─ 하지만 L2는 32 MB밖에 없다
         └─ 386 MB 중 마지막 32 MB만 L2에 남는다 (나머지는 이미 쫓겨남)
GEMM  → 386 MB 처음부터 다시 DRAM에서 읽는다 (앞부분은 이미 L2에 없음)
결과: DRAM을 2번 읽게 된다
```

문제의 핵심: **GPU L2 캐시 = 32 MB, 레이어 가중치 = 386 MB**  
레이어 하나가 L2 전체보다 **12배 크다**. L2에 레이어 전체를 담을 수가 없다.

추가된 DRAM 트래픽 계산:
```
36 레이어 × 386 MB 추가 읽기 = 13.9 GB 추가
13.9 GB ÷ 227 GB/s = 약 61 ms 추가 지연
실측 overhead: 147.82 − 95.96 = 51.86 ms  ← 이론값과 거의 일치
```

---

### 원인 2 (구조적): iGPU에는 독립적인 DMA 엔진이 없다

RT-Swap이 dGPU에서 작동하는 이유는 하드웨어 구조 때문이다:

```
dGPU 내부 구조:
┌──────────────────────────────────┐
│  DMA 엔진 (VRAM ↔ System RAM)    │  ← 메모리 복사 전담 (SM과 독립)
│  SM 연산 코어 (행렬 연산)         │  ← 계산 전담
│  위 두 개가 물리적으로 분리됨!    │
└──────────────────────────────────┘
→ DMA와 SM이 진짜로 동시에 실행 가능
```

```
iGPU (Thor) 구조:
┌──────────────────────────────────┐
│  SM 연산 코어                    │  ← 메모리 읽기 + 계산 모두 SM이 담당
│  공유 DRAM (CPU+GPU 함께 사용)   │
│  별도 DMA 엔진 없음              │
└──────────────────────────────────┘
→ "prefetch"를 SM이 수행 → GEMM과 SM 자원 경쟁 발생
```

iGPU에서는 prefetch 작업 자체가 SM 연산 코어를 사용한다.  
계산(GEMM)과 prefetch(sum)가 모두 SM 코어를 쓰기 때문에 진짜 병렬 실행이 안 된다.

---

### 원인 3 (환경적): 통합 메모리에서 CPU-GPU 동시 실행 시 대역폭 경쟁

추가 측정 실험에서 발견된 사실:

| 실행 조건 | GPU 대역폭 | CPU 대역폭 | 전체 대역폭 |
|-----------|-----------|-----------|-----------|
| GPU 단독 | 196 GB/s | — | 196 GB/s |
| GPU + CPU 12코어 동시 | **114 GB/s (-42%)** | 147 GB/s | 261 GB/s |

CPU가 메모리를 같이 사용하면 **GPU 대역폭이 42% 줄어든다**.  
두 프로세서가 같은 물리 메모리 버스를 나눠 쓰기 때문이다.

이는 파이프라이닝을 위해 CPU를 동시에 활용하는 전략도 GPU 속도를 희생한다는 의미다.

## 5.4 추가 탐구: "그러면 32 MB씩 잘라서 올리면 되지 않나?" (교수님 제안 분석)

> 교수님께서 직접 제안하신 아이디어다:  
> "레이어 하나(386 MB) 전체를 L2에 올리려 하니까 실패하는 거잖아.  
> 그럼 L2 크기(32 MB) 안에 들어가는 조각으로 잘라서, 조각씩 올리면서 계산하면 되지 않나?"

이 아이디어는 방향이 정확히 맞다. 그런데 실제로 적용했을 때 효과가 있는지는 "무엇을 잘라서 올리는가"에 따라 결과가 완전히 달라진다. 아래에서 경우별로 분석한다.

---

### 케이스 A: 가중치(Weight) 타일링 — cuBLAS가 이미 하고 있다

행렬 연산(GEMM)을 수행하는 NVIDIA의 라이브러리인 **cuBLAS**는 이미 내부적으로 타일링을 한다.

```
cuBLAS의 자동 타일링:
─────────────────────────────────────────────────
[386MB 가중치 행렬]
  ┌──┬──┬──┬──┐
  │T0│T1│T2│T3│   각 타일 = 약 96 KB
  ├──┼──┼──┼──┤   → L1 캐시(Shared Memory, 128 KB)에 들어가는 크기
  │T4│T5│T6│T7│
  └──┴──┴──┴──┘
```

GPU 하드웨어가 이미 이 방식으로 동작하기 때문에, 우리가 소프트웨어 수준에서  
"32 MB씩 잘라서 처리하자"고 추가로 구현해도 이미 일어나고 있는 일을 다시 하는 것이다.

---

### 케이스 B: "32 MB씩 올리면 SM이 L2(1,126 GB/s)에서 읽으니 빠르지 않나?" — 핵심 반론 분석

이 질문이 가장 정교하고 중요하다. 명확히 짚어보자.

제안의 논리:
```
[현재]  DRAM ──(231 GB/s)──→ SM      (너무 느림)
[제안]  DRAM ──(231 GB/s)──→ L2 ──(1,126 GB/s)──→ SM
        32 MB씩 잘라서 L2에 채운 뒤, SM은 L2에서 5배 빠른 속도로 읽는다
```

직관은 완벽히 올바르다. 그런데 한 가지 함정이 있다.

**L2에 데이터를 채우는 속도 자체가 DRAM 속도로 제한된다.**

```
타일 1개(32 MB) 처리 순서:

  ① DRAM → L2 채우기:   32 MB / 231 GB/s  = 0.14 ms  ← 느린 구간 (병목)
  ② L2   → SM 읽기:     32 MB / 1,126 GB/s = 0.028 ms ← 빠른 구간

  타일 12개(386 MB) 전체:
  최적 파이프라인(①과 ② 완전히 겹침):
    총 시간 = max(0.14, 0.028) ms × 12개 = 0.14 ms × 12 = 1.68 ms
```

비교:
```
  직접 DRAM → SM 읽기: 386 MB / 231 GB/s = 1.67 ms
  L2 타일링:                               1.68 ms  ← 사실상 동일
```

결국 전체 처리 시간은 **가장 느린 구간(DRAM→L2)에 의해 결정**된다.  
L2→SM 구간이 5배 빠르더라도, L2를 채우는 속도가 느리면 SM은 그냥 기다리게 된다.

```
수도꼭지 비유:

  수도꼭지(DRAM, 231 GB/s) → 작은 양동이(L2, 32 MB) → 강력 펌프(SM, 1,126 GB/s)

  결과: 양동이는 채워지자마자 순식간에 비워지고 (0.028 ms),
        다음 양동이가 채워질 때까지 (0.14 ms) 펌프가 놀게 된다.
        총 물 처리량은 수도꼭지 압력(DRAM 속도)으로 결정된다.
```

L2 고속 대역폭(1,126 GB/s)이 실제 이득을 주는 경우는 딱 하나다:  
**DRAM을 거치지 않고, L2에 이미 있는 데이터를 재사용할 때.**

```
  weight: 매 step마다 DRAM에서 새로 올려야 함 → L2 속도 이득 없음 ❌
  KV Cache: 이전 step에서 L2에 쓰인 것을 다음 step에서 그대로 재사용 → L2 속도 이득 있음 ✅
```

수치로 확인:

```
  레이어 1개 weight  = 386 MB  >>  L2 = 32 MB → 항상 DRAM에서 새로 읽음
  KV Cache 전체      = 12 MB   <   L2 = 32 MB → L2에 상주, DRAM 접근 없음
```

이것이 **KV Temporal Reuse(KV 시간적 재사용)**의 물리적 근거다.  
다음 프레임에서 이전 프레임의 KV Cache를 재사용하면, 그 12 MB는 DRAM을 거치지 않고  
1,126 GB/s 속도로 바로 읽혀서 prefill 시간을 대폭 단축할 수 있다.

---

### 케이스 C: prefetch + 타일링 조합 — 계산이 너무 빠른 것이 걸림돌

"32 MB 타일을 prefetch하면서 이전 타일을 compute하면 되지 않나?"는 아이디어도 자연스럽게 나온다.  
실제로 이 방향이 올바른데, 숫자를 보면 여전히 막힌다.

```
타일 1개(32 MB) 기준 시간 비교:

  prefetch 시간:  32 MB ÷ 231 GB/s = 0.14 ms
  compute  시간:  [1, 3584] × [3584, 2048] = ~0.01 ms

타임라인:
  prefetch tile_0 ━━━━━━━━━━━━━━ (0.14 ms)
  compute  tile_0 ━ (0.01 ms)
                     ↑ 계산이 끝났는데 다음 tile prefetch가 아직 진행 중
```

**계산이 prefetch보다 14배 빠르다.** 계산이 너무 빨리 끝나서 GPU가 다음 타일을 기다리며 놀게 된다.  
레이어 전체가 너무 작은 입력(토큰 1개) 대비 너무 크기 때문에 생기는 구조적 문제다.

*참고: 입력이 토큰 수백 개인 prefill 단계에서는 이 비율이 역전되어 타일링이 효과를 낸다.  
Flash Attention이 바로 이 원리로 prefill 단계의 attention 연산을 가속한다.*

---

### 케이스 D: 실제로 "덜 읽는" 방법 — Activation Sparsity (조건부 가능)

타일링 아이디어의 핵심 직관 — "전부 읽을 필요 없이 필요한 것만 읽으면 된다" — 을  
실제로 구현한 연구 방향이 있다. **활성화 희소성(Activation Sparsity) 이용**이다.

FFN의 계산 구조:
```python
# Qwen2 (Alpamayo 백본) FFN 구조
hidden = silu(gate_proj(x)) * up_proj(x)   # 중간 뉴런: 18944개
output = down_proj(hidden)
```

만약 `hidden` 벡터의 80%가 0에 가깝다면, down_proj의 해당 열(column)을 읽을 필요가 없다.

```
down_proj 열 선택:
  전체:         18944열 × 7 KB/열 = 128 MB
  20%만 읽기:   3789열  × 7 KB/열 = 25 MB  ← L2에 들어간다!
```

이 방향의 선행 연구:

| 논문 | 방법 | 결과 |
|------|------|------|
| **DejaVu** (NeurIPS 2023) | ReLU 모델에서 활성 뉴런 사전 예측 | 2× speedup |
| **PowerInfer** (SOSP 2023) | 자주 쓰이는 뉴런 GPU, 나머지 CPU | iGPU 친화적 구조 |

**우리 상황에서의 장벽**: Alpamayo(Qwen2 기반)는 **SiLU 활성화 함수**를 쓴다.  
ReLU(출력이 0 또는 양수)와 달리 SiLU는 연속 함수로 출력이 진짜 0이 되지 않는다.  
따라서 임계값 기반 근사가 필요하고, 이는 정확도 손실로 이어질 수 있다.  
**"시스템 최적화만으로 해결"이라는 제약 조건 내에서는 적용이 어렵다.**

---

### 정리: 교수님 아이디어의 핵심 통찰이 적용되는 진짜 위치

교수님 제안의 직관 — "L2에 들어가는 크기만 다루면 된다" — 은 완전히 올바르다.  
다만 그 혜택이 발생하는 위치가 **가중치(Weight)가 아닌 KV Cache**다.

```
가중치:
  레이어 1개 = 386 MB  >> L2 = 32 MB  → L2 재사용 불가, 항상 DRAM에서 읽음

KV Cache (이미 L2에 들어감! ✅):
  36 layers × 8 heads × 128 dim × 19 steps × 2 bytes ≈ 12 MB  < L2 = 32 MB
```

KV Cache는 이미 L2 크기 안에 들어간다.  
**이것이 KV Cache Temporal Reuse(시간적 재사용)의 물리적 근거**다:  
인접 프레임 간에 KV Cache를 재활용하면, 그 데이터는 L2에 남아 있으므로  
다음 프레임 prefill에서 새로 읽어올 필요가 없어 큰 시간 절약이 기대된다.

---

### 요약 표

| 방법 | decode 효과 | 이유 |
|------|-----------|------|
| 가중치 32 MB 타일링 | ❌ 없음 | cuBLAS가 이미 수행, 총 DRAM 읽기량 불변 |
| Prefetch + 타일링 | ❌ 없음 | compute가 prefetch보다 14배 빠름 |
| Activation sparsity (ReLU) | ✅ 최대 2× | 실제 0값 skip, DRAM 읽기 감소 |
| Activation sparsity (SiLU) | 🟡 조건부 | 근사 필요, 정확도 손실 위험 |
| **KV Cache 재활용** | **✅ 유망** | **KV = 12 MB < L2 = 32 MB, 실제 재사용 가능** |

## 6. 핵심 교훈 — 이 실험에서 얻은 것

### 교훈 1: iGPU ≠ dGPU, 논문 아이디어의 전제 조건을 먼저 검증해야 한다

RT-Swap은 dGPU의 독립적인 DMA 엔진을 전제로 설계됐다.  
Thor iGPU에는 이 엔진이 없다. 전제 조건이 다르면 같은 기법이 오히려 역효과를 낸다.

> **교훈**: 논문 아이디어를 새로운 하드웨어에 이식할 때는, 그 아이디어가 작동하는 하드웨어 전제 조건을 먼저 확인해야 한다.

---

### 교훈 2: Alpamayo decode는 DRAM bandwidth-bound다

이 실험을 통해 Alpamayo decode의 병목이 명확히 확인됐다.

```
seq=1 decode 1 step에서:
  계산(GEMM) 시간: ~0.005 ms (매우 짧음)
  가중치 로딩(DRAM 읽기): ~86 ms (거의 전부)

→ "DRAM에서 얼마나 빨리 가중치를 읽어오는가"가 속도를 결정한다
→ 물리적 하한: 22 GB ÷ 231 GB/s = 86 ms/step (DRAM 대역폭 한계)
```

L2 캐시(32 MB)가 너무 작아서 레이어 가중치(386 MB)를 담지 못한다.  
결국 매 step마다 DRAM에서 22 GB의 가중치를 다시 읽어야 한다.

---

### 교훈 3: "음수 결과(Negative Result)"도 논문의 기여가 된다

우리가 실패한 것은 단순한 실패가 아니다.  
"iGPU 통합 메모리 환경에서 기존 layer prefetch 기법이 왜 작동하지 않는가"를  
실측 데이터로 증명했다는 것이 논문의 motivation이 된다.

> **논문 서사 (Narrative)**:  
> "기존 RT-Swap/Demand Layering 아이디어를 iGPU에 그대로 적용하면 실패한다.  
> (우리가 실증함) 따라서 iGPU 특성에 맞는 새로운 시스템 설계가 필요하다."

---

### 교훈 4: iGPU에서 작동하는 최적화 방향은 따로 있다

이번 실험 결과를 바탕으로, Thor iGPU에서 실제로 효과적인 방향을 좁혔다:

| 방향 | 원리 | 상태 |
|------|------|------|
| KV Cache 사전 할당 (KV 가방) | 동적 메모리 재할당 제거 | ✅ 완성, 검증됨 |
| 조기 종료 (Early Exit) | 불필요한 decode step 제거 | ✅ 완성 (14,700ms→2,067ms) |
| KV Cache 시간적 재사용 | 인접 프레임 공통 KV 재활용 | 🔜 다음 실험 |
| Speculative Decoding | 작은 모델이 초안, 큰 모델이 검증 | 🔜 탐색 예정 |
| torch.compile 가속 | JIT 컴파일로 Python 오버헤드 제거 | 🔜 즉시 시도 가능 |

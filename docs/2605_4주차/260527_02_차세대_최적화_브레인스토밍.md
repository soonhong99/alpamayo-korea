## 한 줄 요약

> Alpamayo decode는 **memory-bandwidth-bound** 이다.  
> **대전제: 모델 가중치·구조를 변경하지 않는다. 시스템 레벨 파이프라이닝만.**  
> 최적화 방향:  
> (1) **컴포넌트를 시간 축으로 겹친다** (Vision Encoder ∥ Decode, Action Expert 파이프라인)  
> (2) **GPU-CPU 동기화 비용을 제거한다** (Static KV Cache + CUDAGraph)  
> (3) **iGPU 통합 메모리 특성을 활용한다** (zero-copy, unified memory, L2 residency)

## 현재 파악된 사실 (Context)

```
Alpamayo 1.5 추론 타임라인 (1 inference):
  Vision Encoder  →  VLM Prefill  →  VLM Decode × 65  →  Action Expert
     ~0.1 s           ~0.2 s              6.13 s             ~0.1 s
     (빠름)            (빠름)              ← 95% 병목 →        (빠름)
```

**VLM Decode 병목의 정확한 정체**:

| 항목 | 수치 |
|------|------|
| 레이어당 가중치 | 385.9 MB |
| 레이어당 실행 시간 | 2.62 ms |
| 유효 DRAM 대역폭 | 147 GB/s (피크 227의 65%) |
| 순수 compute 시간 (FLOPs 기반) | ~0.005 ms |
| DRAM 읽기 비율 | **99%** |
| 레이어 수 | 36 |
| 1 step 총 시간 | 94 ms |

**핵심 결론**: seq=1 decode는 **DRAM 읽기 대기**가 전부다.  
여러 작업을 동시에 실행하거나, GPU-CPU 불필요한 동기화를 없애는 방향으로 설계한다.

## A. 컴포넌트 간 파이프라인

### A1. Vision Encoder ∥ VLM Decode ★★★

**아이디어 출처**: 클라우드 LLM 서빙의 Continuous Batching + Pipeline Parallelism  
**핵심 관찰**: Vision Encoder 실행과 VLM Decode 사이에 **데이터 의존성이 없다**

```
현재 (순차 실행):
  [Inference N]
  Vision Enc(N) → Prefill(N) → Decode(N) × 65 → Action(N)
                                  6.13 s

제안 (파이프라인):
  [Inference N]                  [Inference N+1]
  Vision Enc(N) → Prefill(N) → Decode(N) × 65 → Action(N)
                                     ↕ N의 decode 6초 동안:
                               Vision Enc(N+1) 미리 실행
                               → N+1 시작 지연 최소화
```

**iGPU에서 특히 유리한 이유**:
- Vision Encoder는 상대적으로 작은 모델 (compute-bound, SM 사용)
- VLM Decode는 DRAM BW-bound (SM은 거의 유휴) → **서로 다른 자원을 주로 사용**
- 통합 메모리 → 데이터 복사 없이 결과 즉시 공유 (zero-copy)

**구현 방향**:
- CUDA stream 분리: Vision Encoder는 별도 stream, Decode는 default stream
- Python `threading` 또는 `asyncio`로 다음 inference의 Vision Enc 사전 실행
- `alpamayo1_5.py`의 추론 루프에서 inference 간 dependency 끊기

**기대 효과**: 연속 추론 시 pipeline latency **~1.5× 개선**

---

### A2. Action Expert 조기 시작 ★☆☆

**아이디어 출처**: Early Exit + Speculative Execution (클라우드 서빙 연구)  
**핵심 관찰**: CoT 추론의 초반 토큰이 어느 정도 완성되면 Action Expert를 partial하게 시작할 수 있을 가능성

```
현재:
  Decode 1~65 완료 → Action Expert 시작

제안:
  Decode 1~32 완료 → Action Expert "예비 실행" (rough trajectory)
  Decode 33~65 완료 → Action Expert 최종 실행 (refined trajectory)
  → rough trajectory를 fallback으로 사용 (tight latency constraint 시)
```

**난이도**: 높음 (모델 내부 의존성 파악 필요)  
**기대 효과**: latency 유연성 확보 (coarse-to-fine 출력)

## B. Decode 구조 개선 (seq=1 탈출)

### B1. 다중 샘플 배치 ★★★ (즉시 실험 가능)

**아이디어 출처**: Batched Inference (표준 ML 기법)  
**핵심 관찰**: Alpamayo는 이미 `num_traj_samples` 파라미터로 다중 경로 샘플링을 지원

```python
# 현재 (num_traj_samples=1):
pred_xyz, pred_rot, extra = model.sample_trajectories_from_data_with_vlm_rollout(
    data=model_inputs,
    num_traj_samples=1,    # ← seq=1, memory-bound
    ...
)

# 제안 (num_traj_samples=8):
pred_xyz, pred_rot, extra = model.sample_trajectories_from_data_with_vlm_rollout(
    data=model_inputs,
    num_traj_samples=8,    # ← seq=8, compute-bound으로 전환!
    ...
)
```

**왜 빠른가?**:
```
가중치 읽기: 386 MB (샘플 수에 무관, 항상 1번만 읽음)
연산량: 8배 증가 (8개 샘플 동시 처리)
→ 샘플당 유효 BW: 386 MB ÷ 8 = 48 MB
→ seq=8 → compute가 증가 → memory-bound 탈출
→ 샘플당 추론 시간: 크게 감소
```

**이론적 speedup 계산**:

| num_samples | 예상 total 시간 | 샘플당 시간 | 비고 |
|-------------|-----------------|-------------|------|
| 1 (현재) | 6.13 s | 6.13 s | memory-bound |
| 4 | ~2.0 s | 0.50 s/sample | 부분 compute-bound |
| 8 | ~1.5 s | 0.19 s/sample | compute-bound |

**자율주행에서의 의미**: num_traj_samples ↑ → 더 많은 경로 후보 → minADE 향상 → **정확도와 속도 동시 개선** 가능

**실험 방법**: `test_inference.py`의 `num_traj_samples` 변경 후 시간 측정  
**기대 효과**: **2~4× speedup per sample**

### B2. Speculative Decoding ★★☆

**아이디어 출처**: *Speculative Decoding* (Leviathan et al., 2023), 클라우드 LLM 서빙의 핵심 기법

**핵심 아이디어**:
```
[기존]
Alpamayo (10B): 토큰 1개씩 × 65번 = 65 forward pass (각각 seq=1)

[Speculative Decoding]
Draft 모델 (1B이하): 토큰 4개 한번에 예측 (빠름)
Verifier = Alpamayo: 4개를 병렬 검증 (seq=4, compute-bound!)
  - 모두 맞으면: 4 토큰 한번에 확정 → 4× 빠름
  - k번째부터 틀리면: k번째 토큰부터 재생성
→ 평균 2~4× speedup
```

**iGPU에서 유망한 이유**:
- Verification (seq=4~8): compute-bound → iGPU SM 효율적으로 활용
- Draft 모델: 작아서 빠름, 통합 메모리로 즉시 접근
- Draft와 Verify를 번갈아 실행하는 구조는 stream 분리 불필요

**구현 난이도**: 높음 (Draft 모델 선정 + 검증 로직 구현)  
**기대 효과**: **2~4× speedup** (acceptance rate에 의존)

## C. iGPU 특화 시스템 최적화

### C1. CPU-GPU 통합 메모리 zero-copy 파이프라인 ★★☆

**아이디어 출처**: Unified Memory Architecture (NVIDIA iGPU/SoC 특성)  
**Thor 특화 아이디어**: CPU와 GPU가 물리적으로 같은 DRAM 사용 → PCIe 없음

```
[기존 (dGPU 방식으로 생각)]
카메라 프레임 → CPU DRAM → PCIe 전송 → GPU VRAM → 인코딩

[Thor iGPU 최적 방식]
카메라 프레임 → CPU DRAM = GPU DRAM (동일!)
→ zero-copy: GPU가 CPU 전처리 결과를 즉시 접근 가능
```

**구체적 활용**:
- CPU ARM cores (Thor는 12-core ARM): 이미지 resize, normalize, 포스트프로세싱 담당
- GPU: VLM decode 집중
- 두 작업 동시 실행 (CPU 작업이 GPU decode 6.13s와 overlap)

**실험 방법**: `torch.utils.data.DataLoader(pin_memory=False)` + unified memory 직접 할당 확인

---

### C2. Static KV Cache + CUDAGraph ★★★ (KV Cache 가방)

**아이디어 출처**:
- CUDAGraph (NVIDIA, TensorRT-LLM에서 표준화)
- Static KV Cache (HuggingFace `transformers.StaticCache`, vLLM PagedAttention의 사전 예약 개념)
- **Nsight 관찰**: 65 decode step × `cudaStreamSynchronize` → 불필요한 CPU-GPU 동기화가 핵심 overhead

---

#### 문제: 매 step마다 CPU-GPU 동기화

Nsight 프로파일에서 관찰한 패턴:

```
Prefill → [cudaStreamSynchronize] → step_000 → [cudaStreamSynchronize]
        → step_001 → [cudaStreamSynchronize] → ... × 65
```

**왜 이게 발생하는가?**  
HuggingFace의 `generate()` 루프는 매 step 종료 후 생성된 토큰이 EOS인지 CPU에서 확인한다:

```python
# HuggingFace generate() 내부 (단순화)
for step in range(max_new_tokens):
    outputs = model(**inputs)
    next_token = outputs.logits[:, -1, :].argmax(dim=-1)
    next_token_id = next_token.item()   # ← GPU→CPU 동기화! 매 step마다
    if next_token_id == eos_token_id:
        break
    inputs = update_inputs(inputs, next_token)
```

`.item()` 한 번이 `cudaStreamSynchronize`를 발생시킨다.  
65 step = **65번 GPU 멈춤**. 이것이 decode step 사이의 연두색 블록이다.

---

#### 아이디어: KV Cache 가방 — 미리 공간을 사다 놓자

**핵심 발상**: 자전거 여행을 떠나기 전, 최대한 담을 수 있는 가방(배낭)을 미리 준비해 놓는다.  
KV cache도 마찬가지 — 최대 decode step 수만큼의 공간을 처음부터 pre-allocate 해 두면:

1. 매 step마다 KV cache 크기가 바뀌지 않는다 → **텐서 shape 고정**
2. Shape가 고정되면 CUDAGraph가 capture 가능하다 → **kernel launch overhead 제거**
3. EOS 검사를 GPU 루프 밖으로 빼낼 수 있다 → **동기화 65번 → 1번**

---

#### 수학: KV Cache 크기는 얼마인가?

Alpamayo 1.5의 LM 부분 (Qwen3VL 기반) 아키텍처:
- 레이어 수: $L = 36$
- KV head 수 (GQA): $H_{kv} = 8$
- Head dimension: $d_{head} = 128$
- 데이터 타입: BF16 (2 bytes)
- K와 V 각각 저장: 2

**토큰 1개당 전체 레이어 KV cache 크기**:

$$\text{KV per token} = L \times 2 \times H_{kv} \times d_{head} \times 2 \text{ bytes}$$
$$= 36 \times 2 \times 8 \times 128 \times 2 = \textbf{147,456 bytes} \approx \textbf{144 KB}$$

**실제 시나리오별 크기**:

| 구간 | 토큰 수 | KV cache 크기 |
|------|---------|---------------|
| Prefill (visual + system prompt) | ~400 토큰 (추정) | 400 × 144 KB ≈ **56 MB** |
| Decode (최대 65 step) | 65 토큰 | 65 × 144 KB ≈ **9.1 MB** |
| **Static 전체 pre-allocation** | **465 토큰** | **~65 MB** |

**65 MB를 미리 잡아 두는 것** — 128 GB iGPU DRAM의 0.05%.  
이 공간을 고정해 두면 65번의 `cudaStreamSynchronize`를 **1번**으로 줄일 수 있다.

---

#### 구현: HuggingFace StaticCache + CUDAGraph

```python
from transformers import StaticCache

# 1. KV cache를 미리 pre-allocate (465 토큰 × 65 MB)
static_cache = StaticCache(
    config=lm_model.config,
    batch_size=1,
    max_cache_len=465,          # prefill_len + max_decode
    device=device,
    dtype=torch.bfloat16,
)

# 2. CUDAGraph capture (딱 한 번)
g = torch.cuda.CUDAGraph()
static_input_ids = torch.zeros(1, 1, dtype=torch.long, device=device)
static_cache_position = torch.zeros(1, dtype=torch.long, device=device)

with torch.cuda.graph(g):
    logits = lm_model(
        input_ids=static_input_ids,
        past_key_values=static_cache,
        cache_position=static_cache_position,
        use_cache=True,
    ).logits

# 3. Decode loop — CPU 동기화 없음
output_tokens = []
for step in range(max_decode):
    static_input_ids.copy_(current_token)
    static_cache_position.fill_(prefill_len + step)

    g.replay()                              # GPU 안에서만 실행, CPU 멈춤 없음
    # next_token = logits.argmax() — GPU 위에서 계산 후 버퍼에 저장

# 4. 루프 끝나면 단 1번만 CPU로 가져옴
all_tokens = output_token_buffer.cpu()      # ← 동기화 1회
eos_pos = (all_tokens == eos_id).nonzero()
```

---

#### Alpamayo에 특히 유리한 이유

Alpamayo의 decode가 **65 step으로 고정**되어 있다는 점이 결정적이다:

```
일반 LLM: 언제 EOS가 나올지 모름 → 매 step EOS 확인 필수
Alpamayo: 65 waypoint → decode step 수가 거의 고정
  → 애초에 EOS 확인 자체가 필요 없을 수도 있다!
  → 그냥 65 step 돌고 결과 한 번에 가져오면 됨
```

---

#### 기대 효과 정량화

```
현재 overhead (추정):
  cudaStreamSynchronize: ~0.1~0.5 ms/회 × 65 step = 6.5~32.5 ms
  kernel launch overhead: ~7 μs × 300 kernels/step × 65 step = 136 ms
  합계: 최소 140 ms / step (실제 decode 6.13s의 ~2.3%)

CUDAGraph 적용 후:
  kernel launch overhead: ~0 (graph replay = 1 launch)
  sync overhead: 1회만
  → 최소 140 ms 이상 절감

  + KV cache shape 고정으로 내부 메모리 재할당도 제거
```

**주의사항**:
- CUDAGraph는 **동적 control flow (if/else, while loop)가 있으면 capture 불가**
- Alpamayo의 decode step 내부에 동적 분기가 있는지 확인 필요
- `past_key_values`가 tuple로 반환되는 구조에서 StaticCache로 교체 시 API 호환성 확인 필요

### C3. cudaMemPrefetchAsync (진정한 unified memory prefetch) ★☆☆

**문제**: 기존 `tensor.sum()` prefetch의 실패 원인 해소  
**핵심**: CUDA Unified Memory로 할당된 텐서에만 `cudaMemPrefetchAsync` 사용 가능

```python
import ctypes
cuda = ctypes.CDLL('libcuda.so')

# 모델 가중치를 unified memory로 할당 (모델 로드 방식 변경 필요)
# cudaMemPrefetchAsync(ptr, size, device_id, stream)
# → DMA-like 힌트, SM 사용 없음 → 진정한 비동기 prefetch
```

**왜 tensor.sum()이 실패했나**:
- `tensor.sum()` = SM이 DRAM에서 읽어서 계산 → SM 점유 (GEMM과 충돌)
- `cudaMemPrefetchAsync` = unified memory 페이지 마이그레이션 힌트 → SM 비점유

**난이도**: 매우 높음 (HuggingFace 모델 로드 방식 수정 필요)  
**기대 효과**: RT-Swap의 올바른 iGPU 구현 → 이론적으로 ~1.2~1.5× speedup

```
Step 2. StaticCache 교체 → EOS sync 제거
   → Nsight로 sync 횟수 65 → 1 확인
   → overhead 정확히 측정 (현재 추정 140 ms 이상)

Step 3. CUDAGraph 캡처 (Step 2 이후)
   → decode step 내 동적 분기 여부 먼저 확인
   → 성공 시 kernel launch overhead 0으로 만듦

Step 4. 위 결과를 바탕으로 교수님과
        Vision Encoder ∥ Decode 파이프라인 방향 결정
```

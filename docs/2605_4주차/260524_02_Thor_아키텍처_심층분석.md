## 2. 질문 1: 96 Tensor Cores + 20 SMs 구조

### 2.1 NVIDIA GPU SM 내부 구조 (Blackwell 기준)

NVIDIA GPU에서 **Tensor Core(TC)** 는 SM 내부에 있다. Blackwell(SM 11.0) SM 하나의 내부 구조:

```
SM (Streaming Multiprocessor)
├── Processing Block 0
│   ├── 32× FP32 CUDA Cores
│   ├── 16× FP64 Cores
│   ├── 1× 4th-gen Tensor Core  ← BF16/FP16/FP8 등 지원
│   └── 1× Warp Scheduler
├── Processing Block 1
│   ├── 32× FP32 CUDA Cores
│   ├── 1× 4th-gen Tensor Core
│   └── 1× Warp Scheduler
├── Processing Block 2
│   └── ... (동일 구조)
├── Processing Block 3
│   └── ... (동일 구조)
├── 228 KB L1/Shared Memory (SM당)
└── Register File
```

일반적으로 Blackwell SM = 4개 Processing Block → **SM당 4 Tensor Cores**.

### 2.2 Thor의 수치 해석

| 스펙 | 값 | 계산 |
|------|----|------|
| CUDA Core 총합 | 2,560 | 2,560 / 20 SM = **128 CUDA Cores/SM** ✅ (4 PB × 32) |
| Tensor Core 총합 | 96 | 96 / 20 SM = **4.8 TC/SM** ❌ (정수 아님) |
| SM 수 | 20 | 실측 (`props.multi_processor_count`) |

→ **96/20 = 4.8이 되는 이유**는 Thor가 완전한 GPU 다이의 일부만 활성화했기 때문으로 추정.

### 2.3 해석: Partial Die (추론)

NVIDIA는 고급 GPU 다이를 edge 제품에 사용할 때 일부 SM을 비활성화한다:
- GB202 풀 다이: 다수의 SM, 많은 TC
- Thor용 부분 활성화: 20 SM, 96 TC

96 TC / 20 SM = 4.8 → 가능한 해석:
1. **실제로는 SM 24개** 중 4개 비활성화 → 24 × 4 = 96 TC → 활성 20 SM, 남은 4 SM에 TC는 카운트되지만 SM은 비활성
2. **일부 SM이 TC 5개, 나머지가 4개** → (4 × 5) + (16 × 4) = 20 + 64 = 84 ❌ → 계산 안 맞음
3. **TC 개수 카운팅 방식 차이**: 다이어그램의 "96 Tensor Cores"가 sub-unit 단위 카운팅

→ 가장 유력한 해석: **풀 다이에 24 SM(96 TC)이 있고, 4개 SM이 비활성화되어 GPU 연산엔 참여 안 하지만 TC 카운트에는 포함** (추론, NVIDIA 미공개)

### 2.4 연구 의미

Alpamayo 추론에서:
- **실제 사용 가능한 TC**: 20 SM × 4 TC = 80개
- GEMM(행렬 곱) 연산 → Tensor Core로 가속
- 65 decode step은 batch_size=1 → Tensor Core 활용률 낮음 (Tensor Core는 큰 행렬에 효율적)
- **이것이 DRAM bottleneck의 근본 원인**: compute가 빨라도 weights 읽기가 병목

## 3. 질문 2: 16 MB System Cache의 정체

### 3.1 세 가지 캐시의 차이

| 캐시 | 위치 | 크기 | 관리 | 접근 주체 |
|------|------|------|------|-----------|
| **CPU L1** | CPU die 내, 코어별 | 64 KB/core | HW 자동 | CPU 코어만 |
| **CPU L2** | CPU die 내, 코어별 | 1 MB/core | HW 자동 | CPU 코어만 |
| **CPU L3** | CPU die 내, 공유 | 16 MB | HW 자동 | CPU 코어 전체 |
| **GPU L2** | GPU die 내 | **32 MB** | HW 자동 | GPU SM 전체 |
| **16 MB System Cache** | SoC 인터커넥트 | **16 MB** | HW 자동 | **CPU + GPU + 기타** |

→ **CPU L3 ≠ System Cache**: 둘 다 16 MB이지만 서로 다른 물리적 캐시

### 3.2 System Cache의 역할

```
[CPU 메모리 접근 경로]
CPU Core → L1 → L2 → L3 → System Cache → Memory Control Fabric → DRAM

[GPU 메모리 접근 경로]  
GPU SM → GPU L2 (32MB) → System Cache → Memory Control Fabric → DRAM
```

System Cache는 **AXI/CHI 인터커넥트 상의 LLC(Last Level Cache)**:
1. **CPU-GPU 데이터 공유 가속**: CPU가 쓴 데이터를 GPU가 읽을 때 DRAM까지 안 가도 됨
2. **캐시 코히런시 포인트**: CPU와 GPU 메모리 뷰 일관성 유지
3. **다른 마스터 공유**: ISP(카메라), Video Encoder/Decoder, PVA 등도 사용

### 3.3 System Cache의 Alpamayo 적용 가능성

**현재 상황 (최악)**:
- Alpamayo 가중치 20.64 GB >> System Cache 16 MB
- GPU가 레이어 순서대로 가중치 접근 → System Cache miss 연속
- CPU도 동시에 다른 작업 → System Cache 추가 pressure

**기회 (향후 연구)**:
- KV Cache(7-28 MB) → 16 MB System Cache와 비슷한 크기
- GPU가 KV Cache 접근 전 CPU가 미리 prefetch → System Cache에 적재
- GPU가 weights 접근 시 `STREAMING` 힌트 → System Cache bypass
- KV Cache만 System Cache에 고정 → attention 연산에서 DRAM 접근 줄임

```
이상적인 상태 (연구 목표):
GPU L2 (32MB):   [KV Cache 일부 pin] 
System Cache (16MB): [KV Cache 일부]
→ Attention: L2/System Cache hit → 4.76× faster
→ MLP weights: DRAM streaming (bypass cache)
```

→ 구현 방법: CUDA L2 Persistent Residency (SM 8.0+ 지원, Thor SM 11.0 ✅)

## 5. 질문 4: CPU 멀티코어 대역폭

### 5.1 이론

```
단일 코어: 33 GB/s (실측, perf_counter, 순수 memcpy)

14코어 이론적 최대:
- 단순 선형 확장 시: 33 × 14 = 462 GB/s
- 실제: DRAM Controller가 병목 → 273 GB/s 이론 한계
- 실제 달성 가능 추정: 150-220 GB/s (DRAM 포화)
```

### 5.2 중요성: CPU Prefetching을 위한 대역폭

Async Pipeline에서 CPU 역할:
- GPU가 Layer N을 compute하는 동안 CPU가 Layer N+1을 prefetch
- CPU prefetch 속도 >= GPU compute 속도여야 파이프라인이 지연 없음
- GPU compute Layer N 시간 vs CPU prefetch Layer N+1 (96 MB) 시간 비교 필요

```
요구 CPU prefetch 속도:
- MLP layer = 96 MB
- GPU compute time for MLP (추정): 2-5 ms
- 필요 CPU BW: 96 MB / 3 ms = 32 GB/s → 단일 코어도 충분할 수도
```

→ 하지만 iGPU에서 CPU와 GPU가 같은 DRAM을 공유하므로 CPU prefetch = GPU가 쓸 DRAM BW 뺏는 것

**핵심 딜레마**:
- CPU prefetch로 layer를 System Cache/GPU L2에 올려놓으면 이후 GPU 접근이 빠름
- 하지만 CPU가 DRAM 접근할 때 GPU의 DRAM BW를 일부 빼앗김
- **net gain 여부는 실험으로만 확인 가능**

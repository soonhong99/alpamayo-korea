**논문**: "RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference"  
**학회**: RTAS 2024 (Real-Time and Embedded Technology and Applications Symposium)  
**분석일**: 2026-05-24  
**중요도**: ★★★★★ — 교수님이 "가장 충격적"이라고 표현한 핵심 논문

---

## 1. 논문 개요

### 한 줄 요약
Multi-DNN 실시간 추론에서 GPU 메모리 부족 문제를 **비동기 스왑 파이프라인**으로 해결하고, real-time guarantee를 제공하는 시스템.

### 핵심 문제
- 자율주행, 로봇 등 edge 시스템에서 **여러 DNN을 동시에 실행**해야 함
- GPU 메모리(특히 embedded GPU)는 작음 → 모든 모델 weights를 동시에 적재 불가
- 기존 방법: weight swapping (DRAM ↔ GPU mem)은 I/O와 compute가 순차 실행 → 느림
- **교수님이 "충격적"이라고 한 부분**: I/O와 compute를 **동시에(async)** 실행하면 DRAM bandwidth와 compute가 중첩됨

### 해당 시스템 조건
- Embedded/edge GPU (iGPU) — DRAM이 CPU/GPU 공유
- Real-time deadline 보장 필요 (WCET 분석 포함)
- Jetson 계열 (Xavier, Orin 등)에서 실험

## 2. 핵심 기여 (Contributions)

### 2.1 Async Swap Pipeline ★★★★★

**기존 방법 (Synchronous Swap)**:
```
Timeline:
[Swap In Layer 1] → [Compute Layer 1] → [Swap In Layer 2] → [Compute Layer 2] → ...
 I/O와 compute가 겹치지 않음 → 총 시간 = Σ(swap_i + compute_i)
```

**RT-Swap의 방법 (Asynchronous Swap)**:
```
Timeline:
CUDA Stream 1 (DMA):   [Swap In L1][Swap In L2][Swap In L3]...
CUDA Stream 2 (Compute):      [Compute L1][Compute L2][Compute L3]...
                        ▲
                  DMA와 compute 중첩!
총 시간 ≈ max(Σswap_i, Σcompute_i) << Σ(swap_i + compute_i)
```

**충격 포인트**: GPU hardware가 **DMA engine과 compute engine을 물리적으로 분리**하여 동시 실행 가능. 이것을 CUDA Stream으로 활용.

### 2.2 Real-Time WCET Analysis

- Async swap이 있어도 worst-case execution time을 수학적으로 보장
- WCET(i) = max(swap_time_next, compute_time_current) 형태로 bound
- Deadline miss 없이 real-time 보장

### 2.3 Multi-DNN 스케줄링

- 여러 DNN이 동시에 실행될 때 swap 순서 최적화
- 메모리 부족 시 어떤 레이어를 evict할지 결정하는 정책

## 3. 수학적 구조

### 3.1 기본 파이프라인 시간 모델

**Synchronous (기존)**:
$$T_{sync} = \sum_{i=1}^{N} (t_{swap,i} + t_{compute,i})$$

**Asynchronous (RT-Swap)**:
$$T_{async} = \sum_{i=1}^{N} \max(t_{swap,i+1},\ t_{compute,i})$$

여기서:
- $N$: 레이어 수
- $t_{swap,i}$: 레이어 $i$ 가중치를 DRAM → GPU mem으로 복사하는 시간
- $t_{compute,i}$: 레이어 $i$ 연산 시간

**Speedup 조건**: 각 레이어에서 swap과 compute가 충분히 중첩될 때

$$\text{Speedup} = \frac{T_{sync}}{T_{async}} = \frac{\sum_i (t_{s,i} + t_{c,i})}{\sum_i \max(t_{s,i+1}, t_{c,i})}$$

compute가 swap보다 오래 걸리면 (compute-bound): speedup ≈ 2×  
swap이 compute보다 오래 걸리면 (memory-bound): speedup < 2×  

### 3.2 WCET Bound (Real-Time 보장)

레이어 $i$의 실행 시간 (async 파이프라인에서):
$$T_i^{WCET} = \max\left(t_{swap,i+1}^{WCET},\ t_{compute,i}^{WCET}\right) + t_{sync}$$

전체 inference WCET:
$$T_{total}^{WCET} = t_{swap,1}^{WCET} + \sum_{i=1}^{N-1} T_i^{WCET} + t_{compute,N}^{WCET}$$

## 4. 시스템 구현

### 4.1 CUDA Stream 구조

```python
# RT-Swap 스타일 구현 (의사 코드)
stream_compute = cuda.Stream()
stream_dma     = cuda.Stream()

# Layer 0: 먼저 swap in
cuda.memcpy_async(layer[0].gpu_buf, layer[0].cpu_ptr, 
                  layer[0].size, stream_dma)
stream_dma.synchronize()

for i in range(N):
    # Layer i+1 prefetch (DMA stream)
    if i + 1 < N:
        cuda.memcpy_async(layer[i+1].gpu_buf, layer[i+1].cpu_ptr,
                          layer[i+1].size, stream_dma)  # 비동기
    
    # Layer i compute (compute stream) — DMA와 동시!
    layer[i].forward(input, stream_compute)
    
    # 둘 다 끝날 때까지 대기
    stream_compute.synchronize()
    stream_dma.synchronize()
    
    # Layer i 메모리 해제 (선택)
    layer[i].free_gpu_buf()
```

### 4.2 Double Buffering

더 advanced한 버전: 두 개의 GPU buffer를 번갈아 사용
```
Buffer A: [현재 계산 중인 레이어 가중치]
Buffer B: [다음 레이어 가중치 prefetch 중]

Step 1: A = Layer0, B = prefetch Layer1
Step 2: compute(A=Layer0), prefetch(B=Layer1→Layer2)
Step 3: compute(B=Layer1), prefetch(A=Layer2)
...
```

## 6. Alpamayo에 적용 가능성 분석

### 6.1 Alpamayo와의 차이점

| 항목 | RT-Swap 대상 | Alpamayo |
|------|-------------|---------|
| 모델 종류 | ResNet, YOLO 등 classification/detection | VLA (Vision-Language-Action), autoregressive |
| 실행 방식 | 단일 forward pass | 65 step autoregressive decode |
| 메모리 | embedded GPU limited | 128 GB iGPU (충분) |
| 레이어 구조 | 일반 CNN 레이어 | 28 Transformer layers + Action Expert |

### 6.2 주요 차이: 메모리 여유

RT-Swap은 GPU 메모리 **부족** 문제를 다룸.  
Thor는 128 GB 통합 메모리가 있어 **메모리 부족은 없음**.

→ **RT-Swap의 "swap" 개념은 직접 적용 불가**  
→ **하지만 "Async Pipeline" 개념은 그대로 적용 가능**

### 6.3 Thor에서의 재해석

Thor에서 "swap" = weights가 DRAM에 항상 있지만 **L2 캐시로의 prefetch**:

```
[Thor 버전 Async Pipeline]
CUDA Stream 1 (DMA):   [Prefetch L1 → GPU L2][Prefetch L2 → GPU L2]...
CUDA Stream 2 (Compute):     [Compute with L1][Compute with L2]...

효과: L2 prefetch 시간 + compute 시간 중첩
     → L2 hit rate 향상 → 효과적 BW = L2 BW(1100 GB/s) 일부 활용
```

### 6.4 Thor에서 예상 Speedup

Alpamayo per-step (20.64 GB total):
- MLP layer (96 MB): t_compute ≈ ?, t_prefetch ≈ 96MB/1100GB/s = 0.087ms (L2 경우)
- Q/O layer (32 MB): t_compute ≈ ?, t_prefetch ≈ 32MB/1100GB/s = 0.029ms (L2 경우)

실제 speedup은 compute time 프로파일링 후 계산 가능.

## 7. 핵심 한계 (분석자 평가)

### 7.1 논문이 인정한 한계
- Swap overhead가 크면 파이프라인 이득 감소
- 레이어 간 dependency가 강하면 중첩 어려움

### 7.2 분석자가 발견한 한계 (iGPU 특수성)
1. **통합 메모리 환경**: DMA copy = CPU mem → GPU mem이 아니라 같은 DRAM 내 이동 → DMA가 "공짜"가 아님, 오히려 추가 BW 소비
2. **Bandwidth contention**: CPU prefetch + GPU compute가 같은 DRAM BW를 나눔
3. **Alpamayo는 compute-bound가 아닐 수 있음**: batch_size=1 decode → Tensor Core utilization 낮음 → compute time << swap time → async 효과 제한

## 8. 우리 연구에서 취할 아이디어

1. **CUDA Stream 이중화**: Compute와 prefetch를 다른 stream에 배치
2. **`cudaMemPrefetchAsync`**: iGPU에서 메모리 prefetch 힌트
3. **Layer granularity 관리**: 레이어 단위로 메모리 생명주기 관리
4. **WCET 분석 방법론**: Real-time 논문을 위한 worst-case 분석 framework
5. **Double buffering 패턴**: 두 레이어 버퍼 교대 사용

```python
# Thor용 적용 핵심 코드 패턴
# weights는 이미 DRAM에 있음 (128GB 여유)
# prefetch = DRAM → GPU L2 캐시로 올리는 것

with torch.cuda.stream(prefetch_stream):
    # 다음 레이어 weights를 L2로 "warm up"
    _ = next_layer_weights.sum()  # L2에 올리는 trick

with torch.cuda.stream(compute_stream):
    # 현재 레이어 compute
    output = current_layer(input)

torch.cuda.synchronize()
```

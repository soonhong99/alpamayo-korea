## 왜 이 실험들을 하는가

### 교수님 연구 방향 (2026-05-24 확정)

> *"모델 수정 없이 Async Pipeline + iGPU 메모리 최적화로 Alpamayo throughput 극대화"*  
> *"cudaMemPrefetchAsync + CUDA Stream 이중화로 layer prefetch와 compute 중첩"*  
> *"양자화는 하면 안 됨. 시스템적 파이프라이닝과 스케줄링으로 해결."*

### 5종 실험이 필요한 이유

1. **Decode는 이미 물리 한계**: 22GB ÷ 231 GB/s = 95ms 이론 최솟값, 실측 107ms = 88% 효율. decode 자체를 GPU에서 더 빠르게 할 방법은 없다.
2. **CPU 대역폭이 낭비되고 있다**: GPU decode 동안 CPU는 유휴 상태. iGPU sweet spot(2코어, 37 GB/s 획득 + GPU 7.5% 손실)을 활용하면 병렬 처리 가능.
3. **4,839ms 중 GPU가 독점하는 모든 구간이 타겟**: VE(728ms) + Prefill(1,423ms) + Decode(1,818ms) + Flow(870ms) — 구간들을 중첩하거나 CPU로 분산하면 됨.
4. **KV Temporal Reuse(실험 C) 결과가 scheduling 여지를 만들었다**: keyframe/reuse 프레임 구분으로 VE(728ms)와 Prefill(1,423ms)을 건너뛰는 프레임이 생김 → 이 여유 시간을 활용할 수 있다.

```
→ Decode 1 step은 더 이상 빠르게 할 수 없다.
  CPU가 도울 수 있는 건 오직 "다른 일을 병렬로 하는 것"뿐.
```

### 실험 D: CUDA Stream 이중화 — Decode(t) ∥ VE(t+1)

#### 한 줄 요약
> "Frame t의 Decode(1,818ms) 창 안에서 Frame t+1의 VE(728ms)를 별도 CUDA Stream으로 동시 실행한다."

#### 왜 하는가
Decode가 1,818ms 동안 GPU를 독점하지만, **GPU SM(Stream Multiprocessor)에는 여유 컴퓨팅 리소스가 있을 수 있다.** VE는 ViT 기반으로, Decode(weight-heavy, memory-bound)와 다른 연산 특성을 가질 수 있다. CUDA Stream 이중화는 교수님이 직접 요청한 기법이다.

#### 이 실험을 통해 얻을 수 있는 것
- **성공 시**: 정상 상태(steady-state)에서 Frame당 VE 728ms 은닉 → 파이프라인 throughput 향상
- **실패 시**: VE와 Decode가 DRAM BW를 두고 경합한다는 실증 데이터 → 다음 최적화 방향 결정
- **어느 쪽이든**: iGPU에서 두 GPU 커널의 BW 경합 동작 방식을 정량화 (논문 기여)

#### 핵심 코드 구조

```python
stream_decode = torch.cuda.Stream()
stream_ve     = torch.cuda.Stream()

# Frame t: decode 시작
with torch.cuda.stream(stream_decode):
    for step in range(MAX_DECODE_STEPS):
        t_step_start = time.perf_counter()
        # decode 1 step (baseline: 107ms/step)
        output = model.vlm(input_ids=cur_token, past_key_values=kv, ...)
        torch.cuda.synchronize(stream_decode)
        step_ms = (time.perf_counter() - t_step_start) * 1000

# Frame t+1: VE 동시 실행 (decode 시작과 동시에)
with torch.cuda.stream(stream_ve):
    t_ve_start = time.perf_counter()
    # pixel_values 준비는 CPU에서 미리 완료
    visual_features = model.vlm.visual(pixel_values_t1, image_grid_thw_t1)
    torch.cuda.synchronize(stream_ve)
    ve_ms = (time.perf_counter() - t_ve_start) * 1000

# 결과 동기화
torch.cuda.synchronize()
```

#### 측정 지표
| 측정 항목 | 목표값 | 의미 |
|---------|-------|------|
| Decode step time (overlap 시) | ≤ 115ms (+7% 이내) | 2코어 sweet spot 수준의 BW 경합 |
| VE 완료 시간 (overlap 시) | ≤ 1,818ms | Decode 창 안에 끝나야 함 |
| 전체 파이프라인 절약 | ≥ 500ms | 실질적 throughput 향상 |

#### 예상 시나리오

| 시나리오 | 조건 | 결과 |
|---------|------|------|
| ✅ 성공 | VE가 compute-bound → BW 경합 적음 | Decode +7~15%, VE Decode 창 안에 완료 → 유효 |
| ⚠️ 부분 성공 | BW 경합 존재하나 관리 가능 | Decode +20~30%, VE 늦게 끝남 → 조율 필요 |
| ❌ 실패 | VE도 memory-bound → DRAM BW 반반 나눔 | Decode 107ms → 200ms+, 총 손해 |

#### 교수님 방향 정합성
★★★★★ — CUDA Stream 이중화 = 교수님이 명시적으로 요청한 기법의 직접 적용

### 실험 E: CPU 2코어 병렬 전처리 파이프라인

#### 한 줄 요약
> "GPU가 Decode+Flow(2,688ms) 돌리는 동안, CPU 2코어(sweet spot)로 다음 프레임의 카메라 전처리 + ego 토큰 계산 + tokenize를 미리 완료한다."

#### 왜 하는가
현재 inference 루프에서 추론이 끝나면 다음 프레임 데이터 준비가 **GPU 추론과 순차적으로** 진행된다. CPU는 GPU decode 동안 완전히 유휴 상태다. 파레토 분석에서 CPU 2코어는 GPU decode에 7.5% 손실만 주면서 37 GB/s를 획득할 수 있다는 것이 증명됐다.

**전처리 파이프라인이 없으면** 아무리 inference를 빠르게 해도 프레임 간 setup 시간이 dead time으로 남는다.

#### 이 실험을 통해 얻을 수 있는 것
- **먼저**: 현재 setup time이 얼마나 되는지 실측 (지금 모름)
- **구현 후**: setup time을 GPU inference와 완전히 중첩 → 프레임 간 dead time 제거
- **파레토 분석 실증**: "CPU 2코어 affinity binding"의 실제 GPU 영향을 측정

#### 핵심 구현 (Python threading + CPU affinity)

```python
import os, threading, queue

class PreprocessWorker:
    def __init__(self, n_cores=2):
        self.n_cores = n_cores
        self.input_queue  = queue.Queue(maxsize=2)
        self.output_queue = queue.Queue(maxsize=2)
        self.thread = threading.Thread(target=self._run, daemon=True)
        self.thread.start()
    
    def _run(self):
        # ★ CPU affinity: 정확히 2코어만 사용 (sweet spot)
        os.sched_setaffinity(0, {4, 5})  # core 4, 5번 고정
        
        while True:
            clip_id, t_us = self.input_queue.get()
            
            t0 = time.perf_counter()
            data = load_physical_aiavdataset(clip_id, t0_us=t_us)
            messages = helper.create_message(...)
            input_ids, tok_data = prepare_inputs(messages, ...)
            setup_ms = (time.perf_counter() - t0) * 1000
            
            self.output_queue.put((input_ids, tok_data, setup_ms))

# 메인 루프 (추론 중 전처리 overlap)
worker = PreprocessWorker(n_cores=2)
worker.input_queue.put((CLIP_ID, T1_US))  # GPU inference 시작과 동시에 t1 전처리

# GPU: t0 inference 실행
result_t0 = run_inference(model, input_ids_t0, tok_data_t0)

# t1 전처리 결과 가져오기 (이미 완료됐어야 함)
input_ids_t1, tok_data_t1, setup_ms = worker.output_queue.get()
print(f"setup 완료: {setup_ms:.0f}ms (GPU inference와 overlap됨)")
```

#### 측정 지표
| 측정 항목 | 성공 기준 |
|---------|---------|
| 현재 setup time (순차) | 측정값 자체가 기여 |
| Decode step time w/ 2코어 active | ≤ 115ms (baseline 107ms + 7.5%) |
| Setup time이 Decode(1,818ms) 내 완료 여부 | ✅ → 완전 hiding |

#### 예상 결과
- Setup time이 수백 ms 이상일 경우 → 완전 은닉으로 directly 임팩트
- Setup time이 수십 ms 수준일 경우 → 직접 임팩트는 작지만 스케줄러 프레임워크로서의 가치
- Decode step time 실측: 2코어 affinity가 107ms → 115ms 이내인지 확인 (파레토 이론 검증)

#### 교수님 방향 정합성
★★★★☆ — CPU sweet spot 활용 + task-level scheduling = 파레토 분석 결론의 직접 실증

### 실험 F: CPU Flow + GPU VE 동시 실행 (태스크 분리)

#### 한 줄 요약
> "Decode가 끝나면 Flow를 CPU에 넘기고, GPU는 즉시 다음 프레임 VE를 시작. Flow(870ms)를 VE(728ms) 안에 은닉."

#### 왜 하는가
현재 Flow(870ms)와 VE(728ms)는 **완전히 순차적**으로 GPU에서 실행된다. Flow는 Decode 결과물(17개 trajectory 토큰)을 64 waypoint로 변환하는 후처리다. **이 연산은 다음 프레임 VE와 아무런 데이터 의존성이 없다.** 동시에 실행할 구조적 근거가 있다.

iGPU 통합 메모리 환경에서 CPU ↔ GPU 데이터 전송은 물리적으로 zero-copy(같은 DRAM)이므로, Flow 결과를 CPU에서 계산하더라도 transfer overhead가 없다.

#### 이 실험을 통해 얻을 수 있는 것
- **성공 시**: Flow 870ms → CPU에서 X ms. GPU는 그 동안 VE(728ms) 실행 → 870ms 은닉 or VE 은닉
- **CPU Flow 시간 측정**: CPU에서 Flow가 얼마나 걸리는지 자체가 미지수 → 실측 필요
- **파이프라인 구조 검증**: CPU/GPU 태스크 분리가 iGPU 통합 메모리에서 유효한지 확인

#### 핵심 실험 절차

```python
# Step 1: Flow 연산이 GPU인지 CPU인지 확인
print(next(model.flow_decoder.parameters()).device)

# Step 2: CPU에서 Flow 실행 시간 측정
flow_cpu = model.flow_decoder.cpu()
traj_tokens_cpu = traj_tokens.cpu()  # unified memory → zero-copy (iGPU)

t0 = time.perf_counter()
with torch.no_grad():
    waypoints_cpu = flow_cpu(traj_tokens_cpu)
flow_cpu_ms = (time.perf_counter() - t0) * 1000
print(f"Flow on CPU: {flow_cpu_ms:.0f}ms  (GPU baseline: 870ms)")

# Step 3: GPU VE + CPU Flow 동시 실행
thread_flow = threading.Thread(
    target=run_flow_on_cpu, args=(flow_cpu, traj_tokens_cpu,)
)

t_parallel_start = time.perf_counter()
thread_flow.start()

# GPU: 다음 프레임 VE 즉시 시작
with torch.cuda.stream(stream_ve):
    visual_features_next = model.vlm.visual(pixel_values_next, ...)

thread_flow.join()  # Flow 완료 대기
torch.cuda.synchronize(stream_ve)
parallel_ms = (time.perf_counter() - t_parallel_start) * 1000

print(f"병렬 실행: {parallel_ms:.0f}ms  (순차: 870+728=1598ms)")
```

#### 예상 시나리오

| CPU Flow 시간 | 결과 |
|-------------|------|
| < 728ms | ✅ Flow가 VE 안에 완전 은닉 → 870ms 공짜 |
| 728~870ms | ⚠️ VE보다 오래 걸리지만 870ms보다 절약 → 부분 이득 |
| > 870ms | ❌ CPU Flow가 병목 → GPU가 기다려야 함 → 손해 |

Flow decoder의 파라미터 수/연산량이 작을 경우 CPU가 충분히 빠를 가능성 있음.

#### 교수님 방향 정합성
★★★★☆ — CPU-GPU 태스크 분리 + iGPU zero-copy 활용 = 시스템 스케줄링의 핵심

### 실험 G: 3-Stage Producer-Consumer 스트리밍 파이프라인

#### 한 줄 요약
> "전체 추론을 3단계 파이프라인으로 재설계. 각 단계가 병렬 실행되는 steady-state 구조를 구현."

#### 왜 하는가
실험 D~F는 개별 구간의 overlap을 테스트한다. 실험 G는 이 모든 것을 **하나의 일관된 파이프라인 아키텍처**로 통합하는 것이다.

현재 구조: **Stage들이 순차적 → 앞 Stage가 끝나야 다음 시작**  
목표 구조: **Stage들이 동시 실행 → 병목(가장 느린 Stage)이 전체 throughput 결정**

```
현재 (순차, single frame):
CPU: [setup]
GPU: [VE][Prefill/Suffix][Decode][Flow]
총: 4,839ms per frame

목표 (pipelined, steady state):
Stage 1 (CPU 2코어): [setup t0][setup t1][setup t2][setup t3]...
Stage 2 (GPU A):          [VE+Pre t0]   [VE+Pre t1]  [VE+Pre t2]...
Stage 3 (GPU B):                   [Dec+Flow t0] [Dec+Flow t1]...

Steady-state 병목: max(Stage1, Stage2, Stage3)
  Stage1: ~200ms (추정)
  Stage2: VE(728) + Prefill(1,423) = 2,151ms  [또는 Suffix만(175ms)]
  Stage3: Decode(1,818) + Flow(870) = 2,688ms  ← 병목

Steady-state throughput: 1 frame per 2,688ms
현재: 1 frame per 4,839ms
→ Throughput 1.80× 향상
```

#### 이 실험을 통해 얻을 수 있는 것
- **논문 핵심 기여**: "Alpamayo 추론을 3-stage async pipeline으로 재설계" = 시스템 스케줄링 최적화의 완전한 형태
- **KV reuse 통합**: Keyframe에서 Stage2 = 2,151ms, Reuse frame에서 Stage2 = 175ms → Stage3(2,688ms)이 항상 병목 → Keyframe 비용 숨김
- **교수님 방향의 완성**: CUDA stream + CPU scheduling + async pipeline을 모두 활용

#### 핵심 구현 구조

```python
import queue, threading, torch

class AlpamayoAsyncPipeline:
    """3-Stage Producer-Consumer 파이프라인"""
    
    def __init__(self, model):
        self.model = model
        # Stage 간 데이터 버퍼 (깊이 2 = double buffering)
        self.s1_to_s2 = queue.Queue(maxsize=2)  # CPU→GPU VE
        self.s2_to_s3 = queue.Queue(maxsize=2)  # VE→Decode
        
        # CUDA Stream 이중화
        self.stream_prefill = torch.cuda.Stream()
        self.stream_decode  = torch.cuda.Stream()
        
        # Stage별 통계
        self.stage_times = {"s1": [], "s2": [], "s3": []}
    
    def stage1_worker(self, frame_iter):
        """Stage 1: CPU 2코어 - 전처리"""
        os.sched_setaffinity(0, {4, 5})  # CPU affinity: 2코어 고정
        
        for t_us in frame_iter:
            t0 = time.perf_counter()
            data = load_and_preprocess(t_us)   # 카메라, ego, tokenize
            self.stage_times["s1"].append((time.perf_counter()-t0)*1000)
            self.s1_to_s2.put(data)             # Stage 2로 전달
    
    def stage2_worker(self, keyframe_policy):
        """Stage 2: GPU Stream A - VE + Prefill/Suffix"""
        while True:
            data = self.s1_to_s2.get()
            t0 = time.perf_counter()
            
            with torch.cuda.stream(self.stream_prefill):
                if keyframe_policy.is_keyframe(data):
                    kv, logits = full_prefill(self.model, data)
                else:
                    kv, logits = suffix_prefill(self.model, data, keyframe_policy.get_kv())
            
            torch.cuda.synchronize(self.stream_prefill)
            self.stage_times["s2"].append((time.perf_counter()-t0)*1000)
            self.s2_to_s3.put((kv, logits, data["meta"]))
    
    def stage3_worker(self):
        """Stage 3: GPU Stream B - Decode + Flow"""
        while True:
            kv, logits, meta = self.s2_to_s3.get()
            t0 = time.perf_counter()
            
            with torch.cuda.stream(self.stream_decode):
                trajectory = decode_and_flow(self.model, kv, logits)
            
            torch.cuda.synchronize(self.stream_decode)
            self.stage_times["s3"].append((time.perf_counter()-t0)*1000)
            output_queue.put((trajectory, meta))
    
    def get_pipeline_stats(self):
        return {
            "stage1_avg_ms": mean(self.stage_times["s1"]),
            "stage2_avg_ms": mean(self.stage_times["s2"]),
            "stage3_avg_ms": mean(self.stage_times["s3"]),  # 이것이 throughput을 결정
            "bottleneck":    "stage3" if mean(self.stage_times["s3"]) > mean(self.stage_times["s2"]) else "stage2",
        }
```

| 항목 | 목표 |
|------|------|
| Steady-state throughput | ≥ 1 frame / 2,688ms (현재 대비 1.8×) |
| 첫 번째 프레임 latency (cold start) | 4,839ms (변화 없음, 허용) |
| 파이프라인 stall 없음 | queue overflow 없이 stable |
| Stage3 시간 | 2,688ms (Decode + Flow, 변화 없어야 함) |

#### 교수님 방향 정합성
★★★★★ — Async pipeline + CUDA stream + CPU scheduling의 완전한 통합. 논문의 system-level contribution 핵심

### 실험 H: cudaMemPrefetchAsync를 이용한 KV Cache L2 Prefetch

#### 한 줄 요약
> "Decode step N 계산 중, CPU 스레드가 step N+1에서 필요한 KV cache를 GPU L2에 미리 올리도록 cudaMemPrefetchAsync를 호출한다."

#### 왜 하는가
교수님이 직접 언급하신 `cudaMemPrefetchAsync` 기법이다. Decode는 매 step마다:
1. 22GB 모델 가중치 읽기 (DRAM→GPU SM)
2. 444MB KV cache 읽기 (DRAM→GPU SM)

KV cache(444MB)는 GPU L2(32MB)보다 크지만, **각 transformer layer가 접근하는 KV 슬라이스는 12MB 수준** (444MB ÷ 36 layer). 이 슬라이스 단위로 미리 L2에 prefetch할 수 있다.

#### 이 실험을 통해 얻을 수 있는 것
- `cudaMemPrefetchAsync`가 실제로 decode step latency를 줄이는지 측정
- iGPU unified memory에서 prefetch 힌트가 동작하는 방식 파악
- HW prefetcher가 이미 optimal한지, 또는 software hint가 도움이 되는지 확인

#### 핵심 구현

```python
import ctypes
libcuda = ctypes.CDLL("libcuda.so")

def prefetch_kv_layer(kv_cache, layer_idx, device_id=0):
    """다음 layer의 KV를 GPU L2에 미리 올림"""
    if layer_idx >= len(kv_cache.key_cache):
        return
    k = kv_cache.key_cache[layer_idx]
    v = kv_cache.value_cache[layer_idx]
    
    # cudaMemPrefetchAsync: unified memory 어드레스를 GPU 페이지로 migrate
    libcuda.cuMemPrefetchAsync(
        k.data_ptr(), k.nbytes, device_id, ctypes.c_void_p(0)
    )
    libcuda.cuMemPrefetchAsync(
        v.data_ptr(), v.nbytes, device_id, ctypes.c_void_p(0)
    )

# Decode loop에서 사용
prefetch_thread = None
for step in range(MAX_DECODE_STEPS):
    # step N 계산
    output = model.vlm(input_ids=cur, past_key_values=kv, ...)
    
    # step N+1을 위한 KV prefetch를 백그라운드로 트리거
    if prefetch_thread:
        prefetch_thread.join()
    prefetch_thread = threading.Thread(
        target=prefetch_kv_layer,
        args=(kv, step + 1,)  # 다음 step에서 attention할 layer
    )
    prefetch_thread.start()
```

#### 예상 결과
- **낙관적**: KV cache L2 hit → step latency 107ms → 95ms (12ms/step, 17steps = 204ms 절약)
- **현실적**: iGPU hardware prefetcher가 이미 sequential access를 최적화하고 있음 → 효과 미미 (≤5ms/step)
- **어떤 결과든 가치**: cudaMemPrefetchAsync 동작 방식 실증 → 논문 실험 근거

#### 교수님 방향 정합성
★★★★★ — cudaMemPrefetchAsync = 교수님이 명시적으로 언급하신 API의 직접 구현

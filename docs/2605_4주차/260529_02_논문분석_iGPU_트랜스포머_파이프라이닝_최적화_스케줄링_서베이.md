# iGPU Transformer 추론 파이프라이닝·최적화·스케줄링 종합 서베이

**작성일**: 2026-05-29  
**대상 시스템**: Jetson AGX Thor (SM 11.0 / CUDA 13.0 / 128GB LPDDR5X / 231 GB/s)  
**대상 모델**: Alpamayo 1.5 (10B VLA: Cosmos Reason2 8.2B + Action Expert 2.3B)  
**현재 베이스라인**: VE(728ms) + Prefill(1,423ms) + Decode(1,818ms, 17step×107ms) + Flow(870ms) = **4,839ms**

---

## 0. 서론 — 왜 iGPU는 다른가

Alpamayo 1.5를 Jetson AGX Thor에서 최적화하는 것은 단순히 "LLM inference를 빠르게 하는" 문제가 아니다. Thor의 iGPU 환경은 일반적인 데이터센터 GPU(dGPU)와 근본적으로 다른 메모리 아키텍처를 가지기 때문에, 기존 LLM 최적화 기법의 상당수가 그대로 적용될 수 없다.

본 서베이에서 가장 먼저 짚어야 할 실험 사실이 있다. **Thor 보드에서 GPU decode 실행 중 CPU 12코어가 동시에 메모리를 접근하면 GPU 대역폭이 196 GB/s에서 114 GB/s로 41.6% 감소한다** (실험 Exp D, 2026-05-28). 이것은 dGPU에서는 일어나지 않는 현상이다. dGPU에서는 GPU가 전용 GDDR6/HBM 인터페이스를 가지므로 CPU의 메모리 접근이 GPU 대역폭에 영향을 주지 않는다.

이 사실 하나로 다음 결론이 도출된다:
- **CUDA Stream 병렬화(Vision Encoder + Decode 동시 실행)는 단순히 켜면 되는 것이 아니다.** 오버랩으로 얻는 시간 이득이 대역폭 경합에 의한 Decode 페널티보다 커야 의미가 있다.
- **RT-Swap (RTAS'24)과 Demand Layering (RTSS'22)이 dGPU를 전제로 설계된 부분은 iGPU에서 재검토가 필요하다.**

본 서베이는 이러한 이해를 토대로, iGPU에서 실제로 적용 가능한 기법과 그렇지 않은 기법을 정확히 분류하고, Alpamayo에 대한 구체적인 실험 로드맵을 제시한다.

## 1. iGPU 아키텍처 심층 분석

### 1.1 메모리 시스템 구조

**Jetson AGX Thor의 메모리 사양:**
- 용량: 128GB LPDDR5X @ 4266 MHz, 256-bit bus
- 이론 최대 대역폭: **273 GB/s** (공식 스펙), 실측 GPU 단독 최대 **196 GB/s** (Exp D)
- CPU 12코어 단독 최대 대역폭: **169 GB/s** (Exp A)
- CPU+GPU 동시 실행 시 총 대역폭: **261 GB/s** (메모리 채널 분할 점유)

LPDDR5X는 원래 스마트폰 AP용으로 설계된 저전력 DRAM이다. GDDR6와 비교하면:
- **GDDR6** (예: RTX 4090): 이론 대역폭 1,008 GB/s (384-bit bus @ 21 Gbps)
- **HBM2e** (예: A100 SXM): 이론 대역폭 2,000 GB/s
- **LPDDR5X** (Thor): 이론 대역폭 273 GB/s — dGPU 대비 4~8배 낮음

이 차이는 매우 중요하다. Alpamayo 모델은 BF16 기준 22GB이다. decode 1 step에서 모든 가중치를 한 번 읽어야 하므로 BF16 물리 하한은 `22GB ÷ 231GB/s ≈ 95ms`다 (실측 107ms, 88% bandwidth efficiency). 같은 모델을 A100에서 실행하면 이론 하한 `22GB ÷ 2000GB/s ≈ 11ms`가 되어 약 9배의 차이가 난다. **이 대역폭 격차가 Thor에서 모든 최적화 논의의 출발점이다.**

### 1.2 단일 메모리 컨트롤러의 함의

dGPU에서 CPU-GPU 간 통신은 PCIe 인터페이스를 통해 이루어지며, GPU는 자체 VRAM을 위한 전용 메모리 컨트롤러를 가진다. 반면 iGPU는 CPU와 GPU가 동일한 DRAM 칩에 동일한 메모리 버스를 공유한다.

**이로 인한 결과:**
1. **CPU-GPU zero-copy**: PCIe 없이 CPU가 할당한 메모리를 GPU가 직접 읽을 수 있다 (CUDA 13.0 UVM full coherence 지원)
2. **대역폭 경합 (Bandwidth Contention)**: CPU와 GPU가 동시에 메모리를 접근하면 서로의 대역폭이 줄어든다. 실험에서 GPU 대역폭이 41.6% 감소했다.
3. **DMA copy engine도 같은 버스를 사용**: H2D(Host-to-Device) DMA 전송 역시 LPDDR5X 버스를 사용하므로, DMA transfer와 GPU compute가 "진정한 병렬"로 실행되더라도 메모리 대역폭은 여전히 공유된다.

**Heterogeneous SoC 연구에서의 공식 입증**: Pagani et al. (2022), "Evaluating Controlled Memory Request Injection for Efficient Bandwidth Utilization and Predictable Execution in Heterogeneous SoCs," *ACM Transactions on Embedded Computing Systems*는 NVIDIA Tegra TX2에서 CPU, GPU, DMA 간 메모리 대역폭 경합을 정량적으로 분석했다. 이 논문은 "heterogeneous SoC에서 한 처리기의 메모리 접근이 다른 처리기의 지연시간을 예측 불가능하게 증가시킨다"고 결론지었다. Thor 환경의 Exp D 실험 결과는 이 분석과 정확히 일치한다.

### 1.3 캐시 계층 구조

Thor(SM 11.0, Blackwell) GPU 캐시 계층:
- **GPU L1/L2 (on-die SRAM)**: 매우 빠름 (수 TB/s 수준), 그러나 용량 제한 (GPU L2 = 약 32MB)
- **LPDDR5X (shared DRAM)**: 273 GB/s — L2 대비 50~100배 느림
- **모델 대비 L2 크기**: Alpamayo 22GB 모델 vs. L2 32MB → 비율 = 687× → **L2 cache는 가중치를 저장하기에 너무 작다**

CPU 캐시 계층:
- **CPU L2 (per-core)**: ~1MB/core (Cortex-A78AE)
- **CPU L3 (shared)**: ~16MB (공식 스펙)
- CPU 캐시도 모델 22GB 대비 압도적으로 작다

**이 캐시 크기 분석에서 도출되는 핵심 결론**: L2 캐시에 KV cache(수십 MB ~ GB 수준)를 "pinning"하는 아이디어는 L2가 32MB에 불과한 Thor에서 매우 제한적으로만 적용될 수 있다. KV cache의 크기가 L2를 초과하는 순간 eviction이 발생한다.

### 1.4 DMA 엔진 vs Shader Core 동시 실행 가능성

Jetson AGX Orin의 기기 정보(`deviceQuery`)에 의하면 "Concurrent copy and kernel execution: Yes with 2 copy engine(s)"이다. Thor(차세대)도 동일한 구조를 가진다. 즉, **DMA engine(Copy Engine)과 Shader Core(Execution Engine)는 물리적으로 분리된 하드웨어이므로 동시 실행이 가능하다.**

그러나 중요한 제약이 있다:
1. **H2D(Host to Device)와 D2H(Device to Host)를 동시에 실행**: 2개의 Copy Engine이 있으므로, H2D transfer와 D2H transfer는 동시에 가능하다.
2. **2개의 H2D transfer를 동시에 실행**: 불가능. "physical memory link is shared like a water pipe"로, 두 H2D 스트림이 있어도 실제로는 순차 실행된다.
3. **DMA와 compute 동시 실행 시 대역폭 공유**: DMA engine이 DRAM에서 데이터를 읽는 것과 GPU shader core가 DRAM에서 가중치를 읽는 것은 동일한 LPDDR5X 버스를 사용한다. 따라서 두 작업이 동시에 실행되면 각각의 유효 대역폭은 감소한다. 이것이 RT-Swap이 iGPU에서 완전히 기대한 이득을 내지 못하는 근본 원인이다.

**결론**: iGPU에서 "DMA와 compute 중첩"은 기술적으로 가능하지만, dGPU에서처럼 "완전 병렬(full overlap)"이 아니라 "부분 병렬(partial overlap with bandwidth sharing)"이다. 이득은 있을 수 있으나 dGPU에 비해 훨씬 작다.

## 2. 핵심 논문 분석

### 2.1 RT-Swap (RTAS 2024)

**전체 제목**: "RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference"  
**저자**: Woosung Kang, Jinkyu Lee, Youngmoon Lee, Sangeun Oh, Kilho Lee, Hoon Sung Chwa  
**학회**: IEEE Real-Time and Embedded Technology and Applications Symposium (RTAS), 2024, pp. 373-385  
**DOI**: 10.1109/RTAS61025.2024.00037

#### 핵심 기여

RT-Swap은 **GPU 메모리 용량 부족** 문제를 해결하기 위한 실시간 메모리 관리 프레임워크다. 여러 DNN 태스크가 동시에 실행될 때 GPU 메모리가 모든 모델 가중치를 담지 못하면, CPU 메모리를 확장 저장소로 활용하는 **비동기 스왑 파이프라인(Async Swap Pipeline)** 을 구현한다.

**스왑 방식 비교:**
```
[기존 동기식 스왑]:
 |— Swap In Layer 1 —| — Compute Layer 1 — |— Swap In L2 —|— Compute L2 —|
 총 시간 = Σ(swap_i + compute_i)

[RT-Swap 비동기 파이프라인]:
 Stream 1 (DMA):    |—Swap L1—|—Swap L2—|—Swap L3—|...
 Stream 2 (Compute):         |—Compute L1—|—Compute L2—|...
 총 시간 ≈ max(Σswap_i, Σcompute_i)
```

**성능 결과**: GPU 물리 메모리 용량의 최대 96.2% 이상을 요구하는 작업 집합에서도, 기존 접근 방식 대비 최소 72% 더 많은 DNN 태스크 집합을 schedulable하게 만들었다.

**Real-Time WCET 분석**: 각 레이어의 `WCET(i) ≈ max(swap_time_{i+1}, compute_time_i)`로 모델링하여 deadline miss 없는 실시간 보장을 제공한다.

**구현**: CUDA 10.2 이상의 Low-Level GPU VMM(Virtual Memory Management) API 기반. PyTorch와 Darknet 모두 구현.

#### iGPU(Thor) 적용 가능성 분석

RT-Swap은 dGPU를 전제로 설계되었으나, 그 아이디어는 iGPU에도 부분적으로 적용된다:

| 측면 | dGPU에서 RT-Swap | iGPU(Thor)에서의 현실 |
|------|----------|----------|
| DMA+Compute 병렬성 | PCIe 전용 채널 사용 → 완전 병렬 | 공유 LPDDR5X 버스 → 부분 병렬 (Exp D: GPU BW 41.6% 감소) |
| 스왑 대상 | CPU DRAM → GPU VRAM | CPU/GPU 공유 DRAM (물리적으로 동일) → 스왑 개념 자체가 다름 |
| 실제 이득 | 이론상 `max(swap, compute)` | 대역폭 경합으로 인해 이득 감소 |

**핵심 교훈**: Decode 단계(r=12.1, 극단적 DMA-intensive)에서 async prefetch를 시도해도, iGPU에서는 DMA transfer 자체가 compute에게서 대역폭을 빼앗아 decode가 느려진다. **이 트레이드오프의 손익분기점 찾기가 연구의 핵심이다.**

### 2.2 Demand Layering (RTSS 2022)

**전체 제목**: "Demand Layering for Real-Time DNN Inference with Minimized Memory Usage"  
**학회**: IEEE Real-Time Systems Symposium (RTSS), 2022  
**arXiv**: 2210.04024  
**코드**: github.com/AveesLab/demand-layering

#### 핵심 기여

Demand Layering은 GPU 메모리를 단일 레이어 크기로 최소화하는 on-demand loading 방식이다. DNN을 layer-by-layer로 실행하면서, 다음 레이어의 파라미터를 SSD → 호스트 메모리 → GPU 메모리로 미리 읽어들이는 3단 파이프라인을 구성한다:

```
Read Engine (SSD→pageable): [L0][  L1  ][  L2  ]...
Copy Engine (Host→GPU):       [L0][  L1  ]...
Exec Engine (GPU compute):      [L0][  L1  ]...
```

**성능 결과**: 
- **메모리 절감**: 96.5% (모델 전체 크기 대비 단일 레이어 크기로 축소)
- **지연 오버헤드**: 평균 14.8% 증가 (파이프라이닝으로 대부분의 I/O 지연을 숨김)

#### OOM-Free Alpamayo와의 연결

2025년 발표된 "OOM-Free Alpamayo via CPU-GPU Memory Swapping for Vision-Language-Action Models" (arXiv:2605.11678)은 Demand Layering의 아이디어를 Alpamayo 1.5에 직접 적용했다. Alpamayo 각 모듈의 DMA-to-EXE 비율(r = C_DMA / C_EXE)을 측정한 결과:

| 모듈 | r 값 | 분류 |
|------|------|------|
| Vision Encoder (ViT) | 0.11 | **EXE-intensive** (DMA 완전 은닉 가능) |
| VLM Prefill | 0.66 | **EXE-intensive** (DMA 완전 은닉 가능) |
| VLM Decode | **12.1** | **DMA-intensive** (파이프라이닝만으로 불충분) |
| Diffusion Expert | 3.5 | **DMA-intensive** (파이프라이닝만으로 불충분) |

이 비율이 말하는 것은 명확하다: **Decode는 compute 시간보다 전송 시간이 12배 더 길다**. 따라서 Demand Layering으로 Decode를 최적화하려면 파이프라이닝만으로는 부족하고, 반드시 일부 레이어를 GPU 메모리에 상주(resident)시켜야 한다.

**Double Flat Buffer (DFB) 설계**: 2개의 GPU 버퍼를 교대로 사용하여 malloc/free 오버헤드를 제거하고, CUDA 이벤트로 DMA-done과 Compute-done을 동기화한다. RTX 5070 Ti(PCIe Gen5)에서:
- Sequential demand layering: 1.14×
- Pipelined demand layering: 1.33×
- 28개 VLM 레이어 GPU 상주 포함: **3.55× 개선**

#### iGPU(Thor) 적용 가능성

Demand Layering이 전제하는 3단 파이프라인은 SSD → CPU DRAM → GPU VRAM의 계층적 메모리를 활용한다. **Thor에서는 CPU DRAM과 GPU VRAM이 물리적으로 동일하므로 이 파이프라인이 근본적으로 다르게 동작한다:**
- SSD → DRAM 단계: 유효 (Thor에도 NVMe SSD가 존재)
- DRAM → GPU 단계: "이동" 자체가 의미 없음 (같은 물리 주소)
- 따라서 iGPU에서의 "demand layering"은 **SSD → unified DRAM → GPU cache** 형태로 재설계해야 한다.

### 2.3 FlashAttention 계열

**FlashAttention (NeurIPS 2022)**: "FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness"  
**저자**: Tri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra, Christopher Ré  
**arXiv**: 2205.14135

FlashAttention의 핵심은 **Attention 연산을 HBM read/write 횟수를 최소화하도록 tile-wise로 재구성**하는 것이다. 기존 Attention은 N×N 크기의 중간 행렬(score matrix)을 HBM에 쓰고 다시 읽으므로 O(N²) HBM I/O가 발생한다. FlashAttention은 SRAM 내에서 tile-wise softmax를 수행하여 HBM I/O를 O(N)으로 줄인다.

**성능 결과**:
- 10~20× 메모리 사용량 감소
- A100에서 2~4× 속도 향상 (HBM I/O 절감 효과)

**Thor에서의 의미**: Thor의 GPU SRAM(L1)에 전체 attention tile을 올릴 수 있다면 FlashAttention의 이득이 적용된다. 그러나 Alpamayo의 현재 구현은 이미 `sdpa`(CUDA built-in scaled dot product attention) 기본값을 사용하고 있으며, 이것이 FlashAttention 백엔드를 자동으로 선택한다 (attn_mask=None 또는 bool mask일 때). **2026-05-28 확정 사항에 따르면 이 조합이 이미 최선이다.**

**Flash-Decoding (2023)**: 장문(long-context) 생성에서 decode 단계의 병렬성을 향상시킨 변형. Alpamayo의 decode sequence length가 짧으므로 직접적 이득은 제한적이다.

### 2.4 Speculative Decoding

#### 기본 원리

Speculative Decoding (Leviathan et al., 2023; Chen et al., 2023)은 소형 draft model이 여러 토큰을 빠르게 생성하고, 대형 target model이 이를 병렬로 검증(verify)하는 기법이다. Draft model의 토큰이 target model의 예측과 일치하면 해당 토큰을 수용하여 여러 토큰을 한 번의 forward pass로 처리한다.

**이론적 이득**: 단일 스텝에서 k개 토큰을 수용하면 `k×1` 대신 `k × batch_compute = 1step`으로 처리되므로, memory-bandwidth-bound decode 구간에서 `k×` 가속이 가능하다.

#### Spec-VLA — VLA 전용 Speculative Decoding

"Spec-VLA: Speculative Decoding for Vision-Language-Action Models with Relaxed Acceptance" (arXiv:2507.22424, 2025)는 VLA 모델에 특화된 speculative decoding 프레임워크다.

**핵심 특징**:
- Draft model: Llama decoder layer 1개 (추가 fine-tuning 불필요)
- **모델 아키텍처 변경 없음**: backbone VLA 모델 자체는 수정하지 않는다
- **Relaxed Acceptance**: VLA 토큰은 bin-ID 기반이므로 절대 차이로 유사도를 측정하여 수용 기준을 완화
- 수용 기준 완화 시 acceptance length 26~44% 향상

**성능 결과** (LIBERO 벤치마크):
- LIBERO-Goal: 1.42× 가속
- LIBERO-Object: 1.38× 가속
- LIBERO-Spatial: 1.28× 가속

**Alpamayo 적용 조건**: Alpamayo의 reasoning trace(decode 17step, ~17 토큰)는 구조화된 패턴을 따를 가능성이 높다. 자율주행 도메인에서 연속 프레임의 reasoning 출력은 유사한 어휘 패턴을 반복할 가능성이 있어 acceptance rate가 높을 수 있다.

**제약**: Draft model의 forward pass 비용이 추가된다. iGPU에서는 대역폭이 추가 비용으로 작용하므로 실제 이득이 이론보다 낮을 수 있다.

#### Self-Speculative Decoding

"Draft & Verify: Lossless Large Language Model Acceleration via Self-Speculative Decoding" (arXiv:2309.08168, 2023)은 별도의 draft model 없이 동일 모델의 중간 레이어에서 early exit로 draft를 생성하는 방법이다. **별도 모델이 필요 없어** 메모리 오버헤드가 없다는 장점이 있다.

### 2.5 KV Cache 재사용 (Prefix Caching / Radix Attention)

#### Prefix Caching (vLLM)

Kwon et al. (SOSP 2023), "Efficient Memory Management for Large Language Model Serving with PagedAttention"에서 제안된 **PagedAttention**은 KV cache를 고정 크기 블록으로 나누어 비연속 메모리 할당을 허용한다. 이를 바탕으로 vLLM은 자동 prefix caching을 구현하여, 동일한 prefix를 가진 요청들이 KV cache를 공유한다.

**Alpamayo 관련성**: 자율주행에서 연속 프레임 간 system prompt(시나리오 설명, 좌표계 등)는 동일하다. 이 공통 prefix의 KV cache를 재사용하면 prefill 계산을 줄일 수 있다.

#### RadixAttention (SGLang)

Zheng et al. (2024), "Fast and Expressive LLM Inference with RadixAttention and SGLang"은 KV cache를 Radix Tree 자료구조로 관리하여 토큰 수준의 prefix matching을 실현한다. vLLM의 블록 수준 해싱 대비 더 세밀한 재사용이 가능하다.

#### FlashDrive — Alpamayo 직접 적용 사례

"FlashDrive: Flash Vision-Language-Action Inference For Autonomous Driving" (Z-Lab, 2025)은 Alpamayo 1.5에 KV cache temporal reuse를 직접 적용한 가장 중요한 논문이다.

**핵심 아이디어**: 자율주행 VLA는 연속 멀티카메라 비디오 스트림을 처리한다. 4프레임 × 4뷰 슬라이딩 윈도우에서 연속 타임스텝 간 75%의 프레임이 동일하다. 새로운 프레임만 처리하고 나머지 3개 프레임의 KV cache를 재사용하면 vision computation을 75% 줄일 수 있다.

**기술적 구현**:
- Pre-RoPE key caching: 프레임이 evict/insert 될 때 rotary positional embedding을 실시간으로 업데이트
- Custom streaming attention mask: view-major 토큰 순서를 수용하는 커스텀 causal mask

**성능 결과**:
- RTX PRO 6000: 716ms → 159ms (**4.5× 가속**)
- Jetson Thor 포함 다양한 플랫폼: **4.0×~5.7× 가속**
- 정확도 손실: minADE6@6.4s 기준 0.07m (무시 가능)

**Alpamayo 적용 가능성**: 이 논문은 Alpamayo 1.5 자체를 대상으로 하므로 직접 적용 가능성이 매우 높다. 다만 streaming fine-tuning이 필요하다고 언급되어 있어, 완전히 training-free인지 확인이 필요하다.

### 2.6 기타 관련 논문

#### VLA-Cache (NeurIPS 2025)

"VLA-Cache: Efficient Vision-Language-Action Manipulation via Adaptive Token Caching" (arXiv:2502.02175)은 training-free inference 가속 방법이다.

**알고리즘**:
1. **Static Token Selection**: 연속 프레임 간 cosine similarity로 변화 없는 패치 식별 (threshold τ 기반)
2. **Task-Relevant Token Filtering**: language decoder의 cross-attention score로 중요 토큰은 강제 재계산
3. **Layer-Adaptive Reusing**: attention entropy 측정으로 레이어별 재사용 비율 동적 조절

**성능**: CUDA latency 1.7× 가속, control frequency 15% 향상, task success rate 손실 무시 가능

**Training-free** — 모델 아키텍처 변경 없음 ★

#### ActionFlow (arXiv:2512.20276)

"ActionFlow: A Pipelined Action Acceleration for Vision Language Models on Edge"는 Cross-Request Pipeline 전략을 제안한다.

**핵심 아이디어**: VLA 추론을 macro-pipeline의 micro-request로 재정의. 현재 요청의 memory-bound Decode와 다음 요청의 compute-bound Prefill을 동시에 실행.

**기술**:
- **Unified KV Ring Buffer**: FusedRoPEAndWriteKV, VarlenAttention, InPlaceKVShift 3개 커스텀 kernel fusion
- **Cross-Request Pipeline**: 이전 decode tokens가 병렬로 처리되는 동안 다음 prefill 실행

**성능** (OpenVLA-7B):
- Jetson AGX Orin: 1.25 FPS → 3.20 FPS (**2.56× 가속**)
- K=32에서 최대 4.36×

**iGPU에서의 주의점**: Decode(memory-bound)와 Prefill(compute-bound)를 동시에 실행하면 메모리 대역폭이 공유된다. iGPU에서는 이 overlap의 이득이 dGPU보다 작을 수 있다.

#### OOM-Free Alpamayo (arXiv:2605.11678)

앞서 2.2절에서 자세히 분석했다. dGPU 환경에서 Alpamayo의 on-demand layer loading을 구현하여 3.55× 가속을 달성했다. Thor(iGPU)에서는 PCIe가 없으므로 SSD → unified DRAM 경로만 유효하다.

#### MindVLA-U1 (arXiv:2605.12624)

Li Auto의 MindVLA-U1은 단일 VLM backbone에서 reasoning과 action을 통합 처리하는 architecture다. **Bounded FIFO memory channel**로 프레임 간 memory token을 전파하여 multi-frame 재처리를 피한다. 성능: 9.7 FPS (action-only) ~ 15.9 FPS. 이것은 모델 수준 변경이므로 Alpamayo에 직접 적용은 불가하지만, 설계 원리는 참고할 만하다.

#### Ghidorah (arXiv:2505.23219, 2025)

Sun Yat-sen University 연구팀의 Ghidorah는 **통합 메모리 edge 기기**를 위한 LLM inference 시스템이다. Speculative decoding + Hetero-Core Model Parallelism (HCMP) 결합으로 Jetson Xavier NX에서 7.6× 가속을 달성했다.

**통합 메모리 관련 기여**:
- Zero-copy mechanism을 활용하여 CPU-GPU 간 데이터 이동 최소화
- Attention의 sparse portion(새 토큰)은 CPU, dense portion(KV cache multiplication)은 GPU에 배분
- ARM CPU의 SpMM 최적화: 3.49× 향상

**한계**: 별도의 draft model(Medusa head) 필요 → 추가 메모리 사용

#### Orca / Continuous Batching (OSDI 2022)

Yu et al. (OSDI 2022), "Orca: A Distributed Serving System for Transformer-Based Generative Models"는 iteration-level 스케줄링으로 **36.9× throughput 향상**을 달성했다. vLLM이 이 아이디어를 오픈소스로 구현했다.

**Alpamayo 관련성**: Thor에서 단일 요청 latency를 최소화하는 것이 목표이므로, continuous batching의 직접 이득은 제한적이다. 그러나 비동기 파이프라인 구조 설계에 참고할 수 있다.

## 3. CUDA 시스템 레벨 기법

### 3.1 cudaMemPrefetchAsync — Layer Prefetching

`cudaMemPrefetchAsync(ptr, size, device, stream)`는 unified memory 포인터를 비동기적으로 특정 장치로 마이그레이션하는 API다. CUDA 13.0에서 Thor는 `concurrentManagedAccess = 1`과 `pageableMemoryAccessUsesHostPageTables = 1`을 지원하여, system-allocated 메모리(`malloc`, `mmap`)를 직접 GPU에서 접근할 수 있다.

**핵심 동작**: 지정한 stream의 이전 연산이 완료된 후 migration이 시작되고, 이후 연산이 시작되기 전에 완료된다. 즉, stream-ordered async prefetch다.

**iGPU에서의 특수성**: 
- Thor에서는 CPU와 GPU가 동일한 물리 DRAM을 사용하므로, `cudaMemPrefetchAsync`는 데이터 복사가 아니라 **GPU TLB entry 업데이트 및 cache line invalidation** 정도만 수행된다.
- 실제 데이터 이동이 없으므로, SSD → DRAM 단계는 여전히 별도로 처리해야 한다.
- CUDA 13.0 문서에 의하면 "Managed allocations aren't GPU-cached in CUDA 13.0" — managed memory가 GPU L2에 자동으로 캐시되지 않는다는 점에 주의.

**이중 스트림 prefetch 패턴**:
```python
# Stream 1: 다음 레이어 가중치 prefetch
cudaMemPrefetchAsync(next_layer_weights, size, GPU_DEVICE, stream1)

# Stream 2: 현재 레이어 compute
launch_kernel(current_layer, stream2)

# 동기화
cudaStreamWaitEvent(stream2, stream1_done_event, 0)
```

이 패턴에서 iGPU의 핵심 질문은: stream1의 prefetch(DRAM read)와 stream2의 compute(도 DRAM read)가 동시에 실행될 때 DRAM 대역폭을 어떻게 분배하는가?

### 3.2 CUDA Graphs

CUDA Graphs는 GPU 연산 그래프를 한 번 capture한 뒤 반복적으로 replay하는 메커니즘이다. 주요 이점: 커널 런치 오버헤드(~5-10μs/kernel) 제거, CPU-GPU sync 감소.

**Thor에서의 제약** (2026-05-28 확정):
- `reduce-overhead` 모드: Triton 3.7.0 ↔ PyTorch 2.8.0 API 불일치 (aarch64용 Triton 없음)
- `cudagraphs` 모드: Qwen3VL의 `_deepstack_process`에서 dynamic boolean indexing(`visual_pos_masks`)이 CUDA Graph capture와 근본 비호환
- **결론**: 현재 Alpamayo 모델 아키텍처에서 torch.compile 및 CUDA Graphs는 적용 불가.

향후 custom C++/CUDA 커널을 직접 작성하여 CUDA Graph를 수동으로 구성하는 방법이 있으나, 이는 상당한 engineering 비용이 필요하다.

### 3.3 L2 Cache Persistence API

NVIDIA는 compute capability 8.0 이상 (A100, Ampere+)부터 `cudaStreamAttrValue`와 `accessPolicyWindow`를 통해 L2 cache에 데이터를 "persisting" 속성으로 pin할 수 있는 API를 제공한다.

```c
cudaStreamAttrValue attr;
attr.accessPolicyWindow.base_ptr    = kv_cache_ptr;
attr.accessPolicyWindow.num_bytes   = kv_cache_size;
attr.accessPolicyWindow.hitRatio    = 1.0f;  // 100% 캐시 시도
attr.accessPolicyWindow.hitProp     = cudaAccessPropertyPersisting;
attr.accessPolicyWindow.missProp    = cudaAccessPropertyStreaming;
cudaStreamSetAttribute(stream, cudaStreamAttributeAccessPolicyWindow, &attr);
```

**Thor(SM 11.0)에서의 현실적 한계**:
- GPU L2 캐시 크기: ~32MB
- KV cache 크기 (17 decode steps, seq_len ~3086): 상당히 클 수 있음 (수십 MB ~ 수백 MB)
- **KV cache가 L2보다 크면 자동으로 evict됨** — 32MB L2에 전체 KV cache를 pin하는 것은 불가능

**현실적 활용 방안**: 자주 접근하는 KV cache의 "핫" 블록(예: 시스템 프롬프트, 첫 N 레이어)만 persisting으로 설정하고, 나머지는 streaming으로 처리. 그러나 이 분할 최적화의 효과를 정량화하려면 실험이 필요하다.

**PyTorch와의 호환성 문제**: PyTorch의 memory allocator가 L2 persistence 설정을 덮어쓰는 알려진 issue가 있다 (pytorch/pytorch #101489). Custom CUDA extension으로 우회가 필요할 수 있다.

### 3.4 Double Buffering Pattern

Double Buffering은 2개의 GPU 버퍼를 교대로 사용하여 다음 레이어의 가중치를 prefetch하는 동시에 현재 레이어를 compute하는 패턴이다.

**OOM-Free Alpamayo의 DFB (Double Flat Buffer)**:
- 버퍼 구조: 슬롯 0, 슬롯 1 (각각 `max_layer_size`)
- 동기화: 2개의 CUDA 이벤트 (DMA-done, Compute-done)
- CPU 핀드 메모리: `pin_memory()`로 H2D 전송 속도 극대화

**iGPU에서의 변형**: SSD에서 데이터를 읽어 GPU L2 캐시에 올리는 형태의 double buffering을 고려할 수 있다. 단, iGPU에서는 "H2D" 개념이 없으므로 실제 이동 없이 GPU 측 TLB 준비와 prefetch hint 제공으로 구현한다.

**Decode(r=12.1) 에서의 의미**: Decode는 DMA-to-EXE 비율이 12.1이므로, 각 레이어의 compute 시간이 가중치 전송 시간보다 훨씬 짧다. Double buffering으로 transfer를 overlap해도 "남는" transfer 시간(r=12.1 → 11.1배의 초과 전송 시간)은 숨길 수 없다. **이것이 Decode에서 파이프라이닝 단독으로 이득이 제한되는 수학적 이유다.**

### 3.5 메모리 풀링

동적 KV cache 할당에서 매번 `cudaMalloc`/`cudaFree`를 호출하면 수십 μs의 오버헤드가 발생한다. 메모리 풀링(memory pool)은 미리 대용량 메모리를 할당한 뒤 내부에서 slice하여 사용하는 패턴이다.

**CUDA 11.2+ Memory Pool API**: `cudaMallocAsync`/`cudaFreeAsync` + `cudaMemPool`로 stream-ordered 메모리 재사용이 가능하다. 이미 vLLM 등이 활용하고 있다.

**iGPU 이점**: iGPU의 128GB 통합 메모리는 dGPU의 VRAM 한계가 없으므로, 필요한 KV cache를 미리 대용량 풀로 할당해 두는 것이 더 자유롭다. OOM 없이 풀 크기를 크게 설정 가능.

## 4. Transformer 특화 최적화 (양자화 없음)

### 4.1 Layer-Level Pipeline

**기본 개념**: Transformer의 레이어 단위 실행을 파이프라인으로 분해하여, 레이어 N의 compute와 레이어 N+1의 가중치 prefetch를 중첩시킨다.

**Alpamayo VLM Decode에서의 적용 계획** (RTAS'24 + Demand Layering 아이디어 결합):
```
CUDA Stream A (DMA/Prefetch):   [L1 prefetch][L2 prefetch][L3 prefetch]...
CUDA Stream B (Compute):                [L1 compute][L2 compute][L3 compute]...
CUDA Event:                              ↑          ↑          ↑
                                    (wait DMA)  (wait DMA)  (wait DMA)
```

**iGPU에서의 예상 효과 계산**:
- Decode 총 시간: 17 steps × 107ms = 1,818ms
- 각 step의 레이어별 compute: 107ms / (num_layers) 수준
- DMA와 compute 중첩 시 이론 이득: `(1 - 1/r)×100%` ≈ `(1 - 1/12.1)×100% ≈ 91.7%` → **하지만 iGPU 대역폭 경합으로 실제 이득은 훨씬 작음**

따라서 layer-level pipeline의 실효 이득은 실험으로 측정해야 한다. 현재 `260529_01_CPU_GPU_비동기_파이프라이닝_5종_실험계획.md`의 Exp 2~3 (Double Buffering, Prefetch-only)이 이를 목표로 한다.

### 4.2 KV Cache 재사용 전략

자율주행 연속 추론 맥락에서 두 가지 차원의 KV cache 재사용이 가능하다:

**① 프레임 내 재사용 (Intra-frame / Prefix Caching)**:
- 각 inference step의 system prompt(자율주행 맥락 설명, 좌표계 등)는 불변
- 이 prefix의 KV cache를 매 step마다 재계산하지 않고 재사용
- 적용 기법: vLLM prefix caching, SGLang RadixAttention

**② 프레임 간 재사용 (Inter-frame / Temporal KV Reuse)**:
- 연속 타임스텝에서 75% 프레임이 동일 → 해당 vision token의 KV cache 재사용
- 적용 기법: FlashDrive (4.5× 가속), VLA-Cache (1.7× 가속)
- 이미 `260528_claude_03_KV_Temporal_Reuse_실험계획_및_검증결과.md`에서 실험 계획 수립됨

**③ 장거리 재사용 (Long-range / Embodied Memory)**:
- KEEP (arXiv:2602.23592, 2025)처럼 수십~수백 프레임에 걸친 환경 상태를 KV cache로 압축
- Alpamayo는 단발성 trajectory prediction이므로 장거리 재사용 구조가 아님 → 적용 제한적

### 4.3 Attention 메모리 접근 패턴 최적화

**Flash Decoding (2023)**: 긴 context에서 attention을 parallel segment로 나누어 처리. Alpamayo decode의 context length가 길어질수록 (접두사 토큰 포함) 효과 증가.

**Paged Attention (vLLM)**: KV cache를 page 단위로 분산 저장하여 fragmentation 해소. iGPU의 넉넉한 통합 메모리 환경에서는 fragmentation보다 대역폭이 병목이므로 직접 이득은 제한적. 그러나 메모리 관리 복잡도 감소는 유의미.

**Async KV Cache Prefetching** (Dong et al., arXiv:2504.06319, 2025): 다음 decode step의 KV cache를 현재 step의 matrix multiplication 실행 중에 비동기로 prefetch. 성능 결과: Qwen 2.5 72B에서 1.6×, Llama 2 70B에서 1.4× throughput 향상. iGPU에서는 KV cache가 이미 같은 DRAM에 있으므로 "prefetch" 보다 L2 cache warm-up에 가까운 효과를 낸다.

### 4.4 Weight Streaming

**SSD에서 가중치 스트리밍**: Apple의 "LLM in a Flash" 연구는 NVMe SSD에서 직접 LLM 가중치를 스트리밍한다. iPhone 17 Pro에서 12GB RAM으로 400B 모델을 실행하는 사례가 보고되었다.

**Thor에서의 적용**: Thor는 128GB unified DRAM으로 22GB Alpamayo를 충분히 담을 수 있다. 따라서 SSD → DRAM streaming이 Thor에서는 필요하지 않다. **그러나 DRAM → GPU L2 cache에 대한 "weight prefetching"은 여전히 유의미하다**: 각 decode step에서 22GB를 한 번씩 L2를 거쳐 읽으므로, L2 cache의 효과적인 활용이 중요하다.

**FlashMem (ASPLOS 2026)**: 모바일 GPU를 위한 메모리 계층 최적화. 정적 weight loading 스케줄과 동적 texture memory 활용을 결합. Jetson 계열 iGPU와 직접 연관된 최신 연구로 추후 세부 분석이 필요하다.

## 5. 실시간 시스템 + DNN 스케줄링

### 5.1 실시간 DNN 스케줄링 개요

자율주행 inference는 **10 Hz** 기준으로 100ms 이내에 처리해야 한다. 120 km/h에서 100ms 지연은 3.3m 이동에 해당한다. Alpamayo의 현재 4,839ms는 이 요구사항을 크게 초과하지만, **Rolling Trajectory 연속 스트리밍 파이프라인**으로 재정의하면 단일 inference latency가 아닌 "다음 waypoint 제공까지의 지연"이 실질적 요구사항이 된다.

**실시간 DNN 연구의 흐름**:
- DART (RTSS 2019): CPU/GPU 파이프라인 병렬 DNN inference
- REEF (OSDI 2022): 마이크로초 단위 reset-based preemption
- GCAPS (ECRTS 2024): GPU context-aware 선점형 우선순위 스케줄링

### 5.2 GPU 태스크 선점·전환

**REEF** (Han et al., OSDI 2022): "Microsecond-scale Preemption for Concurrent GPU-accelerated DNN Inferences"

DNN 추론 커널은 대부분 **idempotent**(재실행해도 같은 결과)하다는 관찰에 기반하여, 우선순위가 높은 RT 태스크가 도착하면 현재 실행 중인 BE(Best-Effort) 태스크의 커널을 즉시 kill하고 RT 커널을 실행하는 **Reset-based Preemption**을 구현한다.

**성능**: RT 태스크 end-to-end latency 오버헤드 < 2%, 전체 throughput 최대 7.7× 향상.

**Alpamayo 적용 가능성**: VE → Prefill → Decode → Flow의 파이프라인에서, Decode 단계를 BE로 설정하고 새로운 센서 프레임 처리(VE)를 RT로 설정하면 선점형 스케줄링을 적용할 수 있다. 그러나 이는 현재 단일 inference를 완료시키는 전략과 상충하므로, rolling pipeline에서만 의미 있다.

### 5.3 우선순위 기반 스케줄링

**GCAPS** (Wang et al., ECRTS 2024): "GPU Context-Aware Preemptive Priority-based Scheduling for Real-Time Tasks"

GPU device driver 레벨에서 GPU context 스케줄링을 제어하여, 일반 GPU 접근 코드에 "one-line macro"만 추가하면 선점형 우선순위 스케줄링을 활성화한다.

**Alpamayo 관련성**: 현재 Alpamayo는 단일 DNN이므로 multi-tenant 스케줄링이 필요하지 않다. 그러나 미래에 perception DNN, planning DNN, Alpamayo를 병렬로 실행해야 하는 시나리오에서는 GCAPS 같은 스케줄링이 필요할 것이다.

**DARIS** (arXiv:2504.08795): DNN inference 태스크의 deadline을 인식하여 throughput을 극대화하는 scheduler. Periodic soft real-time 태스크에 최적화.

**Efficient CUDA Stream Management for Multi-DNN Real-Time Inference on Embedded GPUs** (ScienceDirect, 2023): **embedded GPU에서 여러 DNN의 CUDA stream을 효율적으로 관리하는 방법론**을 직접 다룬다. Jetson Xavier/Orin 계열에서 여러 DNN이 동시 실행될 때 발생하는 bandwidth contention과 우선순위 관리를 연구했다.

## 6. Alpamayo 적용 가능성 분석

### 6.1 직접 적용 가능 기법 (우선순위 순)

#### Priority 1: KV Temporal Reuse (FlashDrive 방식)

**근거**: FlashDrive는 Alpamayo 1.5를 대상으로 4.0~5.7× 가속을 보고했다. 기법 자체는 inference 엔지니어링 수준에서 구현 가능하다.

**기대 효과**: VE(728ms) + Prefill(1,423ms) 구간에서 vision token의 75%를 skip하면 이론상 encode+prefill이 ~1/4로 단축된다. 4,839ms → 약 2,300~2,500ms 수준.

**구현 요구사항**:
- 연속 프레임 간 KV cache를 보존하는 버퍼 관리
- Pre-RoPE key caching 구현
- Custom streaming attention mask
- **주의**: FlashDrive는 streaming fine-tuning이 필요하다고 언급했으므로, 완전한 training-free 구현 가능 여부 확인 필요

**iGPU 이점**: 128GB 통합 메모리로 KV cache buffer를 충분히 할당 가능 (OOM 없음).

#### Priority 2: VLA-Cache (Training-Free Token Caching)

**근거**: Training-free이므로 즉시 적용 가능. 모델 아키텍처 변경 없음.

**기대 효과**: CUDA latency 1.7× 가속 (단, 로봇 manipulation 기준; 자율주행은 다를 수 있음).

**구현 요구사항**:
- Cosine similarity로 static/dynamic vision token 분류
- Cross-attention score 기반 task-relevant token filtering
- Layer-adaptive reuse ratio 조정

#### Priority 3: Async KV Cache Prefetching

**근거**: Dong et al. (2025) 결과에서 1.4~1.8× throughput 향상. iGPU 환경에서도 "next step의 KV cache를 현재 step compute 중에 GPU L2로 warm up"하는 효과가 있다.

**기대 효과**: Decode 17 step × 107ms = 1,818ms에서 ~1.3~1.5× 개선 → ~1,200~1,400ms

**구현 요구사항**:
- CUDA stream 분리: compute stream + prefetch stream
- `cudaMemPrefetchAsync` 또는 explicit memory access hint

**iGPU 특이사항**: prefetch가 실제 이동이 아닌 GPU L2 warm-up에 해당. 효과 측정이 중요.

#### Priority 4: ActionFlow Cross-Request Pipeline

**근거**: Jetson AGX Orin에서 2.56× 가속 (같은 계열 embedded GPU). Decode와 Prefill의 cross-request overlap.

**기대 효과**: Rolling inference 맥락에서 throughput 2~3× 향상. 단발 latency는 동일하지만 연속 추론 FPS 개선.

**구현 요구사항**:
- Unified KV Ring Buffer 구현
- FusedRoPEAndWriteKV, VarlenAttention, InPlaceKVShift CUDA kernel
- Multi-request 상태 관리

**iGPU 주의**: Decode(memory-bound)와 Prefill(compute-bound) 동시 실행 시 대역폭 경합 발생. Exp D에서 41.6% GPU BW 감소가 확인되었으므로, overlap 이득이 이 페널티를 상쇄하는지 반드시 측정해야 한다.

### 6.2 조건부 적용 가능 기법

#### Spec-VLA / Speculative Decoding

**조건**: 소형 draft model 추가 필요 (Spec-VLA: Llama decoder layer 1개). 추가 GPU 메모리 소요.

**잠재적 이득**: Decode 17step에서 acceptance length 1.5~2.0배 증가 시 decode time ~30~50% 감소 → Decode 1,818ms → ~900~1,200ms

**iGPU 제약**: draft model도 같은 LPDDR5X 대역폭을 소비한다. 대역폭이 제한 요인인 환경에서 두 모델이 경쟁하면 실효 이득이 줄어든다.

**연구 가치**: 자율주행 reasoning trace는 구조화된 패턴을 반복하므로 acceptance rate가 높을 수 있다. Alpamayo 전용 draft model을 dstil로 학습하여 acceptance rate를 측정하는 것이 의미 있는 실험이다.

#### L2 Cache Persistence (Partial)

**조건**: 전체 KV cache가 아닌 "핫" 블록(system prompt KV, 초기 레이어)만 persisting 설정.

**잠재적 이득**: 자주 읽히는 KV cache 블록의 L2 hit rate 향상 → attention kernel 실행 시간 단축

**iGPU 제약**: L2 = 32MB, KV cache는 수십 MB 이상. 극히 일부만 상주 가능.

**구현 복잡도**: PyTorch L2 persistence API 호환성 문제 해결 필요 (pytorch #101489).

#### Double Buffering for Layer Prefetch

**조건**: SSD(Thor NVMe)에서 가중치를 streaming하는 경우에만 유효.

**현재 Thor 상황**: 22GB 모델이 이미 128GB DRAM에 올라가 있으므로, SSD streaming은 필요하지 않다. **DRAM → GPU L2** 수준의 double buffering만 고려 가능.

**잠재적 이득**: Decode r=12.1에서는 pipelining 이득이 이론적으로 `1 - 1/r ≈ 92%`지만, iGPU에서 DMA와 compute가 대역폭을 공유하므로 실제 이득은 실험으로만 알 수 있다.

### 6.3 iGPU 제약으로 불가능한 기법

#### RT-Swap 원형 그대로 적용

- **이유**: RT-Swap은 CPU DRAM → GPU VRAM의 PCIe 전송이 compute와 독립 채널로 실행되는 dGPU를 전제한다. iGPU에서는 이 전제가 성립하지 않는다. 대역폭 경합이 발생하여 예상 이득이 실현되지 않는다.
- **교훈 (2026-05-28)**: RT-Swap + Demand Layering의 아이디어를 iGPU에 적용하려면, "대역폭 경합이 있음에도 불구하고 pipeline이 유익한 조건"을 수학적으로 분석하는 것이 본 연구의 contribution이 되어야 한다.

#### torch.compile / CUDA Graphs (Automatic)

- **이유**: 2026-05-28 확정 실험 결과: Triton aarch64 미지원, Qwen3VL dynamic boolean indexing과 CUDA Graph 비호환.
- **대안**: 수동 CUDA kernel 작성으로 제한된 범위에서만 가능.

#### Demand Layering 원형 그대로 (SSD 기반)

- **이유**: Thor에서 22GB 모델이 이미 128GB DRAM에 상주하므로, SSD → DRAM 파이프라인이 불필요하다.
- **iGPU 재해석**: "DRAM → GPU L2" 수준의 on-demand fetch + prefetch 패턴으로 재설계해야 한다.

#### Continuous Batching (vLLM 방식)

- **이유**: 단일 모델, 단일 요청 latency 최적화가 목표이므로, 다중 요청 throughput 극대화 기법인 continuous batching은 현재 단계에서 직접적 이득이 없다.
- **미래 활용**: rolling pipeline에서 여러 timestep을 batch로 처리하는 경우에 고려 가능.

#### HBM 기반 L2 캐시 전략 (전체 KV cache pinning)

- **이유**: A100(HBM, 2TB/s) 등에서 "KV cache를 on-chip SRAM에 pinning"하는 전략은 SRAM 크기가 수십 MB인 경우 작은 모델에서만 유효하다. Alpamayo의 KV cache는 L2 32MB를 크게 초과한다.

### 6.4 권장 실험 로드맵

아래 로드맵은 예상 이득 / 구현 난이도 / iGPU 적합성을 종합적으로 고려한 우선순위 순서다.

#### Phase 1 (단기, ~2주): 시스템 수준 측정 및 baseline 완성

| 실험 | 목표 | 예상 측정값 |
|------|------|------------|
| Exp 1: Baseline decode BW 측정 | 현재 decode의 실제 BW 효율 | 88% (현재 확인됨) |
| Exp 2: Double Buffering 순수 overhead 측정 | DFB 도입 시 decode latency 변화 | -5% ~ +10% |
| Exp 3: Prefetch hint 효과 측정 | cudaMemPrefetchAsync 유무 비교 | ±0% ~ -15% |
| **Exp 4: KV Temporal Reuse 구현** | 연속 2프레임 KV cache 재사용 | **VE+Prefill 50~75% 감소** |

#### Phase 2 (중기, ~4주): KV Temporal Reuse 완성

| 구현 항목 | 기술 | 예상 이득 |
|----------|------|----------|
| Pre-RoPE key cache 버퍼 | Python level 구현 | VE+Prefill 3~4× 감소 |
| Custom streaming attention mask | HF attention mask override | 정확도 유지 확인 |
| Multi-frame KV reuse (4프레임) | 75% vision 재사용 | 전체 pipeline 2× |

#### Phase 3 (장기, ~8주): Cross-Request Pipeline

| 구현 항목 | 기술 | 예상 이득 |
|----------|------|----------|
| Decode ↔ Prefill overlap | 2 CUDA stream + 대역폭 균형 분석 | FPS 2× (rolling) |
| 대역폭 손익분기점 계산 | Exp D 결과 기반 이론 모델 | 논문 contribution |
| 실시간 파이프라인 데모 | 5Hz rolling trajectory | 실용화 증명 |

## 7. 결론 — 연구 방향 재정렬

### 7.1 본 서베이의 핵심 발견

1. **iGPU의 대역폭 경합은 근본적 제약이다**: 실험으로 확인된 41.6% GPU 대역폭 감소는 기존 dGPU 논문들의 가정을 뒤집는다. 이 사실 자체가 새로운 연구 발견(novel finding)이며, iGPU에서 DNN 최적화 논문의 citation 대상이 될 수 있다.

2. **VLA 특수성이 최적화 기회를 만든다**: 자율주행 VLA는 연속 프레임에서 75% 중복성이 있다. 이 temporal redundancy를 활용한 KV cache 재사용은 이론적으로 Prefill을 4× 줄일 수 있다. FlashDrive가 이미 Alpamayo에서 4.5× 가속을 보고했으나, **iGPU(Thor)에서의 효과는 별도 측정이 필요하다.**

3. **RT-Swap과 Demand Layering의 이데아는 유효하나 iGPU 재해석이 필요하다**: 두 논문의 핵심 아이디어(비동기 prefetch, layer-level pipeline)는 iGPU에서도 적용 가능하다. 그러나 dGPU 전제(PCIe 독립 채널)가 없으므로, **"iGPU에서의 최적 overlap 조건"을 수학적으로 도출하는 것이 본 연구의 theoretical contribution**이다.

4. **Decode phase가 가장 어렵고 가장 중요하다**: r=12.1로 DMA-intensive한 Decode는 단순 파이프라이닝으로 개선이 제한적이다. Speculative decoding이나 KV cache 재사용으로 Decode step 수 자체를 줄이는 접근이 더 효과적이다.

### 7.2 최종 기대 개선 경로

현재 베이스라인 4,839ms에서 단계적 개선:

| 단계 | 적용 기법 | 예상 latency | 누적 개선 |
|------|----------|------------|----------|
| 현재 | 베이스라인 | **4,839ms** | — |
| Phase 1 | KV Temporal Reuse (FlashDrive 방식) | ~2,500ms | ~1.9× |
| Phase 2 | VLA-Cache + Async KV Prefetch | ~1,800ms | ~2.7× |
| Phase 3 | Cross-Request Pipeline (ActionFlow) | ~1,200ms* | ~4.0× |
| 이론 하한 | BF16 물리 대역폭 한계 | **~1,462ms** (decode only) + α | — |

*Phase 3는 단발 latency가 아닌 rolling FPS 기준

### 7.3 논문 contribution 방향

본 연구의 차별점은 다음 세 가지로 정의할 수 있다:

1. **iGPU unified memory에서의 VLA inference 최적화**: dGPU 전제의 기존 논문과 달리, 단일 메모리 컨트롤러 환경에서의 정량적 분석
2. **Temporal KV cache reuse for autonomous driving VLA**: FlashDrive의 아이디어를 Jetson Thor iGPU 환경에서 실측 검증하고 iGPU 특화 최적화를 추가
3. **Bandwidth contention model**: CPU-GPU 동시 실행 시 대역폭 분배의 수학적 모델 및 파이프라이닝 손익분기점 공식 도출

이 세 가지를 결합하면 "edge iGPU에서의 실시간 VLA inference를 위한 시스템 최적화" 논문으로 RTSS 2027 또는 IROS/ICRA 2027 워크숍 수준의 contribution이 될 수 있다.

## 0. 현재 상태 정리

### 실측 수치 (sdpa 기본값, BF16, N=1) ← 2026-05-28 확정

```
완전한 추론 파이프라인 (260528_calibrate_seqlen.py, sdpa):
  Vision Encoder :   728ms  (15%)
  LM Prefill     : 1,423ms  (29%)  ← seq=3,086 tokens, Flash Attention 경로
  Decode         : 1,818ms  (38%)  ← 17 steps × 107ms/step, 90% MBU
  Flow           :   870ms  (18%)
  ─────────────────────────────
  합계           : 4,838ms         ← 현재 baseline
```

> ⚠️ 이전 기록 "6,554ms (prefill 4,487ms + decode 2,067ms)"는 **eager+StaticCache 커스텀 경로만** 측정한 값 (VE·Flow 제외).  
> 정확한 전체 파이프라인 baseline은 **4,838ms**.

### attn_implementation 최종 정리 (2026-05-28)

| 조합 | LM+Decode만 | 전체 파이프라인 | 문제 |
|------|------------|----------------|------|
| eager + StaticCache | 6,554ms | ~8,152ms | eager = 2.6× 느림 |
| sdpa + StaticCache | 5,076ms | ~6,674ms | StaticCache → Flash Attn 비활성화, 2× 느림 |
| **sdpa + DynamicCache** | **3,241ms** | **4,838ms** | ✅ 현재 최선 |

→ **결론**: attn_implementation 미지정(sdpa)+DynamicCache(기본값)만 사용. eager/StaticCache 사용 금지.

### 물리적 하한선 (BF16 변경 불가)

| 항목 | 값 | 의미 |
|------|-----|------|
| Thor DRAM 대역폭 | 231 GB/s | 측정값 |
| 모델 가중치 | 22 GB | BF16 |
| Decode 1 step 최솟값 | **86ms** | 22GB ÷ 231GB/s |
| 17 step 최솟값 | **1,462ms** | decode만 |
| BF16 전체 최솟값 | **~4,400ms** | 모든 SW 최적화 완벽 가정 |

> **결론**: BF16 + 시스템 최적화만으로 99ms 전체 추론은 물리적으로 불가능하다.  
> 1단계 목표 (~3,500ms)는 torch.compile + KV Temporal Reuse 조합으로 접근한다.  
> 최종 목표 (99ms)는 단일 추론이 아닌 **연속 스트리밍 파이프라인 구조**로 재정의해야 한다.

## 2. 실험 1: torch.compile 적용 테스트

### 왜 이 실험이 필요한가

CUDA Graph는 HuggingFace 내부의 Python 제어 흐름(D2H sync) 때문에 캡처에 실패했다.  
`torch.compile(mode="reduce-overhead")`는 JIT 컴파일러가 Python 오버헤드를 자동으로 분석해  
CUDA Graph와 유사한 효과를 얻을 수 있다.

```
현재 Decode loop 문제:
  매 step: Python → kernel launch → GPU 실행 → Python 복귀 → 다음 kernel ...
           ↑────── Python dispatch overhead ──────↑

torch.compile 적용 후:
  첫 step: 트레이싱 + 컴파일 (느림)
  이후 step: 컴파일된 경로 재사용 → Python overhead 제거
```

이전 실측: CPU serial bottleneck = **419ms/run** (pytorch_trace 분석 기반)  
이 중 상당 부분이 kernel launch dispatch overhead이며, `torch.compile`이 제거 가능한 타겟이다.

### 무엇을 측정할 것인가

```python
# 적용 방법 (코드 1줄) — sdpa+DynamicCache 기본값과 조합
model.vlm = torch.compile(model.vlm, mode="reduce-overhead")
```

| 측정 항목 | 현재 (sdpa baseline) | 목표 |
|-----------|-------------------|----|
| Decode ms/step | **107ms** | ≤ 90ms |
| Decode 전체 | **1,818ms** (17steps) | ≤ 1,530ms |
| Prefill | **1,423ms** | ≤ 1,200ms |
| 첫 번째 실행 오버헤드 | — | 트레이싱 시간 측정 |
| 추가 GPU 메모리 | — | compile 캐시 크기 측정 |

> ⚠️ 이전 기록 "109ms/step, 2,067ms" 는 eager+StaticCache 기준. sdpa 기준으로 갱신됨.

### 왜 model.vlm 만 컴파일하는가

```
외부 decode loop (Python): tracker.update() → .cpu() D2H sync 포함
  → CUDA Graph 직접 캡처 불가 (이전 실험에서 실패 확인)
  → torch.compile 적용 범위에서도 제외

model.vlm 내부 (순수 forward pass): attention → FFN → LayerNorm
  → D2H sync 없음 → torch.compile이 tracing 가능
  → mode="reduce-overhead": 내부적으로 CUDA Graph 자동 적용
```

### 주의 사항

- SM 11.0 (Blackwell) + CUDA 13.0 + aarch64에서 Triton 지원 여부 확인 필요
  - PyTorch 2.8.0 소스 빌드에 Triton이 포함됐는지 확인: `python -c "import triton; print(triton.__version__)"`
  - Triton 없으면 `mode="default"` (Inductor만 사용) 으로 폴백
- `attn_implementation` 미지정(sdpa) + 이 테스트 조합 → eager 사용 절대 금지
- 실패 시 단계: `reduce-overhead` → `default` → `none` (포기)
- 스크립트: `scripts/inference/260528_torch_compile_test.py --mode reduce-overhead`

### 예상 소요 시간 / 난이도

- 구현: 30분 (코드 1줄 추가)
- 검증: 반나절 (Thor에서 실행 + 결과 분석)
- 난이도: 낮음

## 3. 실험 2: KV Cache Temporal Reuse (시간적 재사용)

### 왜 이 실험이 가장 중요한가

현재 추론의 가장 큰 낭비는 **같은 장면을 반복해서 처음부터 다시 처리한다는 것**이다.

```
자율주행 카메라가 10Hz로 촬영한다면:
  t=0.0s: 교차로 진입 프레임
  t=0.1s: 교차로 프레임 (0.0s와 95% 동일)
  t=0.2s: 교차로 프레임 (0.1s와 90% 동일)

현재 시스템:
  t=0.0s: Full prefill(1,472ms) — 전체 시각 토큰 처리
  t=0.1s: Full prefill(1,472ms) — 거의 같은 내용 다시 처리 ← 낭비!
  t=0.2s: Full prefill(1,472ms) — 또 처음부터 ← 낭비!
```

KV Cache Temporal Reuse는 이 낭비를 없앤다:
```
t=0.0s: Full prefill(1,472ms) + KV cache 저장
t=0.1s: 변화한 토큰만 증분 처리(~100ms) + KV cache 업데이트
t=0.2s: 변화한 토큰만 증분 처리(~100ms) + KV cache 업데이트
```

이는 **모델 수정이 아니다** — 모델 가중치나 아키텍처는 그대로이고, 어떤 KV를 재사용할지 결정하는 **캐시 관리 정책**이다.

### 이론적 효과

| 항목 | 현재 | KV Reuse 적용 후 |
|------|------|-----------------|
| Prefill (첫 프레임) | 1,472ms | 1,472ms (동일) |
| Prefill (이후 프레임) | 1,472ms | **~100ms** (변화 부분만) |
| 전체 추론 (steady state) | 5,003ms | **~3,200ms** |

**Prefill 병목만 1,372ms 단축** — 전체 추론에서 가장 큰 단일 최적화

### 구현 방향

```
구현 핵심 아이디어:
  1. 이전 프레임의 KV cache를 보존 (KV 가방 재사용)
  2. 새 프레임에서 변화한 visual token만 식별
  3. 변화한 토큰의 KV만 새로 계산, 나머지는 기존 것 재사용
  4. 시스템 cache 관리자가 어떤 KV를 evict/keep할지 결정

참고 개념: FlashDrive (Microsoft Research), Prompt Caching (Anthropic Claude)
```

### 검증 실험 설계

```
실험 A (가능성 검증): 
  동일 프레임 2번 실행 → KV 재사용 시 prefill 시간 측정
  기대: 1,472ms → ~5ms (완전 캐시 히트)

실험 B (점진적 변화): 
  인접 프레임 쌍 (t, t+1)으로 KV 재사용 가능 토큰 비율 측정
  기대: 80~90% 재사용 가능 → 100~300ms

실험 C (실제 시나리오): 
  CLIP_ID 영상에서 연속 프레임 10개 처리 시 평균 prefill 시간 측정
```

### 예상 소요 시간 / 난이도

- 구현: 2~3주
- 검증: 1주
- 난이도: 높음 (visual token 변화 감지 로직 + KV 부분 업데이트 구현)

## 4. 실험 3: Speculative Decoding

### 왜 이 실험이 필요한가

DRAM 대역폭 물리 하한(86ms/step)을 **우회하는 방법**이다.

현재 decode의 문제:
```
seq=1 (토큰 1개씩 생성):
  매 step: 22GB 가중치를 DRAM에서 로딩 → 토큰 1개 생성
  → 22GB 로딩이 지배적, BW wall = 86ms/step
```

Speculative Decoding:
```
1단계 (Draft): 작은 모델(~0.5B)이 토큰 K개를 빠르게 예측
  → 0.5B 모델: 1GB 가중치, BW time = 1GB/231GB/s = 4.3ms
  → K=4 토큰을 4.3ms × 4 = 17ms에 생성

2단계 (Verify): Alpamayo가 K개 토큰을 **한 번에** 검증 (seq=K)
  → seq=K로 한 번 forward → K개 토큰 병렬 검증
  → compute-bound로 전환 → DRAM 읽기 1회로 K토큰 처리

결과: 수락률 α=0.7이면 평균 1 + α×K = 1 + 0.7×4 = 3.8토큰/검증 호출
     → 이론 speedup ≈ 3.8× on decode
```

**핵심 이유**: seq=K 검증이 DRAM 읽기를 1회로 줄여서 대역폭 벽을 우회한다.

### 구현 방향

```
Draft 모델 후보:
  1. Qwen2.5-0.5B (오픈소스, Alpamayo와 같은 토크나이저 계열)
  2. SmolLM-360M (Hugging Face, 경량)
  
중요: 모델 수정 없음 — Draft 모델은 기존 오픈소스 그대로 사용
Alpamayo는 Verifier 역할만 (가중치 수정 없음)
```

### 검증 실험 설계

```
실험 A (기초): Qwen2.5-0.5B Draft → Alpamayo 검증
  측정: acceptance rate α (0 ~ 1)
  기대: α ≥ 0.5 (VLA trajectory 토큰에서의 수락률)
  
실험 B (speedup 측정): Draft K=2,4,8 변화 시 decode 시간
  기대: K=4, α=0.7 → decode 1,926ms → ~600ms
  
실험 C (품질 평가): trajectory 정확도 비교
  기준: standard decode vs speculative decode trajectory 비교
```

### 예상 소요 시간 / 난이도

- 구현: 2~3주
- 검증: 1주
- 난이도: 높음 (Draft 모델 선택, 검증 로직, 토크나이저 호환성)

## 5. 실험 4: Inter-Inference Pipeline (연속 추론 파이프라인)

### 왜 이 실험이 필요한가

단일 추론 latency를 줄이는 것과 별도로,  
**연속된 추론 요청 간의 처리량(throughput)을 높이는 방법**이다.

```
현재 (순차 실행):
  [Inf 0: Vision(715ms) + Prefill(1,472ms) + Decode(1,926ms) + Flow(890ms)]
                                                                              [Inf 1: ...]
  → Inf 0 완전히 끝나야 Inf 1 시작
  → 5초에 1개 결과

파이프라인 적용:
  [Inf 0: Vision + Prefill | Decode + Flow                     ]
                             [Inf 1: Vision + Prefill | Decode  ]
  → Inf 0 Decode 시작과 동시에 Inf 1 Vision+Prefill 시작
  → throughput 향상 (Inf 0의 Decode 시간 동안 Inf 1 준비)
```

**단, 주의사항**: 대역폭 측정 결과 CPU+GPU 동시 실행 시 GPU BW 42% 감소.  
Vision(GPU compute) + Decode(DRAM BW) 동시 실행 시 decode 속도 저하 가능.

### 이 실험의 핵심 질문

> Vision과 Decode를 동시에 실행했을 때의 **BW 경합 페널티**가  
> 파이프라이닝으로 얻는 **시간 단축 이득**보다 작은가?

```
이득: Vision 715ms가 Decode 1,926ms 뒤에 숨음 → throughput 향상
손실: Decode BW 42% 감소 → decode 1,926ms → ~2,700ms 지연 가능

실험 없이는 어느 쪽이 더 큰지 알 수 없다.
이것이 실험이 필요한 이유.
```

### 검증 실험 설계

```
실험 A: Vision과 Decode를 동시에 실행했을 때 decode 속도 측정
  기대: decode가 느려지지만, 전체 throughput은 향상

실험 B: 다양한 Vision 크기 (프레임 수)에서 교차점 탐색
  기대: Vision이 가벼울수록 pipeline 이득 > 페널티

실험 C: CPU Vision preprocessing (resize, normalize 등만)으로 GPU BW 경쟁 없애기
  기대: 순수 CPU 전처리는 GPU BW에 영향 없음
```

### 예상 소요 시간 / 난이도

- 구현: 1주
- 검증: 3~4일
- 난이도: 중간

## 6. 최종 목표 달성 경로 — 현실적 로드맵

### 1단계: 4,838ms → ~3,200ms (BF16 시스템 최적화)

```
현재 (sdpa baseline): 4,838ms
  VE:728 + Prefill:1,423 + Decode:1,818 + Flow:870

torch.compile (decode 15% 단축):    -270ms   → ~4,568ms
KV Temporal Reuse (steady state):  -1,285ms  → ~3,283ms
  └ prefill 1,423ms → ~138ms (3,086→300 tokens)
Early Exit: 이미 적용 (7× decode 단축, 베이스라인에 포함)
──────────────────────────────────────────────
1단계 예상 도달: ~3,200ms
```

### 2단계: ~3,200ms → ~1,500ms (Speculative Decoding 추가)

```
3,200ms에서:
Speculative Decoding (decode ~3× 단축):
  현재 Decode: 1,818ms → ~640ms (α=0.7, K=4)
  나머지: VE:728 + Prefill:138 + Flow:870 + Decode:640
────────────────────────────
예상 도달: ~2,376ms → Vision-Decode Pipeline 추가 시 ~1,700ms
```

### 3단계: 99ms — 연속 스트리밍 파이프라인으로 재정의

BF16 전체 추론 99ms는 물리적으로 불가능하다 (decode floor: 1,634ms).  
하지만 다음 방식으로 **차량 제어 주기 99ms**는 달성 가능하다:

```
Rolling Trajectory 전략:
  t=0s:     Full inference 시작 (~1,500ms with all optimizations)
  t=0.1s:   waypoint[0]로 차량 제어 (10Hz, 100ms 주기)
  t=0.2s:   waypoint[1]로 차량 제어
  ...
  t=1.5s:   추론 완료 → 새 궤적 생성됨
  t=1.5s:   다음 추론 즉시 시작 (파이프라인)

→ 차량 제어 루프: 100ms마다 waypoint 실행 ✅
→ 궤적이 항상 준비된 상태 유지 ✅
→ 추론 결과 지연 (latency): ~1,500ms이지만 throughput은 실시간 ✅
```

## 7. 실험 우선순위 및 일정

| 우선순위 | 실험 | 예상 기간 | 기대 효과 | 리스크 |
|---------|------|----------|----------|--------|
| **1** | torch.compile 테스트 | 반나절 | decode ~15~20% 단축 | 낮음 (코드 1줄) |
| **2** | KV Temporal Reuse PoC | 2~3주 | prefill 1,472ms → 100ms | 높음 (구현 복잡) |
| **3** | Speculative Decoding | 2~3주 | decode 2~3× 단축 | 중간 (수락률 미지수) |
| **4** | Inter-Inference Pipeline | 1주 | throughput 향상 측정 | 중간 (BW 경합 변수) |

### 권장 시작 순서

```
즉시 시작 (이번 주):
  → torch.compile 테스트 (반나절, 리스크 낮음)
    결과 분석 → 효과 있으면 기본 적용

다음 단계 (2주 후 시작):
  → KV Temporal Reuse — 가장 큰 효과, 시스템 레벨 핵심 기여
    구현 중 교수님과 방향 중간 점검

병렬 탐색 (탐색적):
  → Speculative Decoding draft 모델 후보 조사 및 acceptance rate 예비 측정
```

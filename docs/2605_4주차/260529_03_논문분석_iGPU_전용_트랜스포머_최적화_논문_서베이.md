# iGPU 전용 Transformer·LLM 최적화 논문 집중 서베이

**작성일**: 2026-05-29  
**작성 목적**: dGPU 최적화 논문(RT-Swap, Demand Layering, FlashAttention, vLLM 등)이 아닌, **iGPU 통합 메모리 구조**를 주된 설계 고려사항으로 삼은 Transformer/LLM 추론 최적화 논문을 집중 탐색하고 연구 공백(Research Gap)을 진단한다.

---

## 핵심 질문

> **"iGPU의 고유 특성(통합 메모리, 단일 메모리 컨트롤러, CPU·GPU 대역폭 경쟁)을 주된 설계 동인(primary design consideration)으로 삼은 Transformer 추론 최적화 논문이 존재하는가?"**

본 서베이는 15개 이상의 검색 쿼리를 WebSearch로 실행하여 가능한 한 광범위하게 탐색했다.  
결론을 먼저 말하면: **그러한 논문은 극소수이며, Jetson AGX Thor를 포함한 임베디드/엣지 iGPU 전용 파이프라인 설계 논문은 사실상 공백 상태다.**

---

## 1. iGPU 전용 논문 목록 (발견된 모든 논문)

이 절에서는 발견된 논문을 iGPU 관련성 정도에 따라 세 등급으로 분류한다.

### 등급 A — iGPU 통합 메모리가 핵심 설계 동인인 논문

#### A-1. HeteroLLM (HeteroInfer): Characterizing Mobile SoC for Accelerating Heterogeneous LLM Inference
- **저자**: (카이스트·상하이 연구팀)
- **게재**: ACM SOSP 2025 (서울, 2025년 10월)
- **arXiv**: [2501.14794](https://arxiv.org/abs/2501.14794)
- **iGPU 특화 문제**: 모바일 SoC의 통합 메모리 구조에서 GPU·NPU 어느 단일 프로세서도 전체 메모리 대역폭을 포화시키지 못하는 현상. Snapdragon 8 Gen 3 기준 GPU 단독 40~45 GB/s, NPU 단독 40~45 GB/s, **GPU+NPU 병렬 시 ~60 GB/s**(총 68 GB/s의 88%).
- **핵심 기법**: GPU·NPU 이종 병렬 실행(텐서 파티션 수준), 통합 메모리 아키텍처를 활용한 **zero-copy 고속 동기화 메커니즘**(GPU-NPU 동기화 오버헤드 ~400 µs를 통합 메모리로 최소화).
- **성능**: GPU-only 대비 1.34×~6.02× end-to-end 속도 향상.
- **사용 하드웨어**: Qualcomm Snapdragon 8 Gen 3 (모바일 SoC, iGPU+NPU 통합 메모리)
- **Thor 적용 가능성**: **높음.** GPU(SM 11.0)와 잠재적 NPU/DLA 병렬 실행 아이디어를 Thor에 그대로 적용 가능. 단, Thor는 모바일 SoC보다 메모리가 128 GB로 훨씬 크고 대역폭(231 GB/s)도 높다.

#### A-2. Agent.xpu: Efficient Scheduling of Agentic LLM Workloads on Heterogeneous SoC
- **저자**: Xinming Wei, Jiahao Zhang et al.
- **게재**: arXiv (2025년 6월)
- **arXiv**: [2506.24045](https://arxiv.org/abs/2506.24045)
- **iGPU 특화 문제**: **메모리 통합(memory-unified) 이종 SoC에서 iGPU 전용(iGPU-only)** 추론 엔진이 Reactive(저지연 요구) + Proactive(높은 처리량 요구) 에이전트 워크로드를 동시에 처리하지 못하는 문제. CPU·iGPU·NPU가 단일 물리 메모리를 공유하는 구조를 명시적으로 가정.
- **핵심 기법**: 이종 실행 그래프 구성 + 커널 선호도(affinity) 기반 탄력적 가속기 매핑 + **커널 수준 선점(preemption)**으로 Reactive 태스크 응답성 보장.
- **성능**: 산업용 iGPU-only 서빙 엔진 대비 Proactive 처리량 1.2×~4.9× 향상, Reactive 지연 91% 이상 감소.
- **사용 하드웨어**: 소비자 등급 이종 SoC (CPU + iGPU + NPU 통합 메모리 구조)
- **Thor 적용 가능성**: **매우 높음.** "memory-unified heterogeneous SoC"를 명시적 전제로 삼으며, Jetson Thor(CPU + GPU + DLA 통합 메모리)와 구조적으로 동일하다.

#### A-3. Ghidorah: Fast LLM Inference on Edge with Speculative Decoding and Hetero-Core Parallelism
- **저자**: 중산대·북경사범대 연구팀
- **게재**: arXiv (2025년 5월)
- **arXiv**: [2505.23219](https://arxiv.org/abs/2505.23219)
- **iGPU 특화 문제**: 엔드유저 디바이스의 **통합 메모리 설계(unified memory design)** 를 활용하여 추측 디코딩의 Draft-Verify 파이프라인을 CPU·GPU 간 zero-copy로 구성하는 문제.
- **핵심 기법**: 이종 코어 모델 병렬(HCMP, Hetero-Core Model Parallelism) + 아키텍처 인식 프로파일링(ARCA)으로 최적 파티셔닝 결정.
- **성능**: NVIDIA Jetson NX에서 디코딩 단계 **7.6× 속도 향상**.
- **사용 하드웨어**: **NVIDIA Jetson NX** (iGPU, 통합 메모리)
- **Thor 적용 가능성**: **매우 높음.** Jetson 플랫폼 직접 사용, 통합 메모리를 zero-copy 파이프라인의 핵심 자산으로 활용. Thor는 Jetson NX보다 메모리·컴퓨팅 용량이 훨씬 크므로 더 큰 Draft 모델을 쓸 수 있다.

#### A-4. Challenging GPU Dominance: When CPUs Outperform for On-Device LLM Inference
- **저자**: Haolin Zhang, Jeff Huang (Texas A&M University)
- **게재**: arXiv (2025년 5월)
- **arXiv**: [2505.06461](https://arxiv.org/abs/2505.06461)
- **iGPU 특화 문제**: Apple Silicon(iGPU)에서 런타임 메모리 할당 및 메타데이터 동기화 오버헤드가 소규모 배치에서 iGPU를 오히려 CPU보다 느리게 만드는 현상. 통합 메모리임에도 불구하고 GPU 경로에 숨겨진 오버헤드가 존재함을 밝힘.
- **핵심 기법**: iGPU vs CPU 추론 비교 분석, 스레드 과구독(thread oversubscription) 및 양자화 전략의 영향 분석.
- **성능**: iPhone 15 Pro에서 CPU-only 2스레드/F16이 17 tokens/s로 GPU 가속(12.8 tokens/s)을 능가하는 경우 존재.
- **사용 하드웨어**: Apple iPhone 15 Pro (Apple Silicon iGPU, 통합 메모리)
- **Thor 적용 가능성**: **중간.** iGPU 통합 메모리에서의 숨겨진 오버헤드를 경고하는 분석으로 의미 있음. Thor에서도 유사한 현상이 있을 수 있어 프로파일링 전에 GPU 경로를 무조건 신뢰하지 말아야 함을 시사한다.

### 등급 B — Jetson/엣지 iGPU를 **테스트 플랫폼**으로 사용했으나, 설계 동인이 iGPU에 특화되지 않은 논문

#### B-1. Understanding the Performance and Power of LLM Inferencing on Edge Accelerators
- **저자**: Mayank Arya, Yogesh Simmhan
- **게재**: PAISE 2025 (IPDPS 併設 워크숍); 확장판 arXiv [2506.09554](https://arxiv.org/abs/2506.09554)
- **iGPU 관련 내용**: Jetson AGX Orin(64GB LPDDR5, 공유 CPU/GPU)에서 LLaMA3.1, Phi-2, DeepSeek-R1-Qwen 등 2.7B~32.8B 모델 실측. Roofline 분석으로 decode 단계 FFN·어텐션이 대역폭 한계 영역에 있음을 확인.
- **한계**: iGPU 특화 설계 없음. 여러 플랫폼 중 하나로 Jetson을 사용.
- **Thor 적용 가능성**: **높음(데이터 참고용).** 우리 실험 결과와 비교 기준으로 활용 가능.

#### B-2. A Systematic Characterization of LLM Inference on GPUs
- **arXiv**: [2512.01644](https://arxiv.org/pdf/2512.01644)
- **iGPU 관련 내용**: Jetson AGX Orin에서 Llama3-8B를 Roofline 분석. **데이터센터 GPU에서 compute-bound인 FFN 커널이 엣지(Orin)에서는 LPDDR 대역폭 한계로 memory-bound로 전환**됨을 정량적으로 증명. DVFS로 인한 주파수 저하 현상도 관찰.
- **한계**: 설계 제안 없음. 특성 분석에 그침.
- **Thor 적용 가능성**: **높음(분석 참고용).** 우리 Thor 실측 (decode 86ms 물리 하한, 대역폭 한계)과 동일한 현상을 체계적으로 다룬다.

#### B-3. LIME: Accelerating Collaborative Lossless LLM Inference on Memory-Constrained Edge Devices
- **arXiv**: [2512.21835](https://arxiv.org/abs/2512.21835)
- **iGPU 관련 내용**: **이종 Nvidia Jetson 4개 디바이스** 클러스터에서 LLaMA3.3-70B를 파이프라인 병렬로 추론. Interleaved pipeline parallelism + 모델 오프로딩.
- **성능**: 산발적 요청 1.7×, 버스트 요청 3.7× 속도 향상.
- **한계**: 다중 Jetson 클러스터 간 분산 추론에 초점. 단일 iGPU 내부 최적화 아님.
- **Thor 적용 가능성**: **중간.** 파이프라인 설계 아이디어는 참고 가능.

#### B-4. EdgeReasoning: Characterizing Reasoning LLM Deployment on Edge GPUs
- **arXiv**: [2511.01866](https://arxiv.org/abs/2511.01866)
- **iGPU 관련 내용**: 엣지 GPU에서 추론 LLM(CoT 등)의 지연-정확도 트레이드오프를 체계적으로 분석. 지연·전력·에너지 트레이드오프 실측.
- **한계**: iGPU 특화 설계 없음. 특성 분석.

#### B-5. RooflineBench: A Benchmarking Framework for On-Device LLMs via Roofline Analysis
- **arXiv**: [2602.11506](https://arxiv.org/abs/2602.11506); 2026년 3월 발표
- **iGPU 관련 내용**: 엣지 하드웨어(이종 컴퓨트 계층 포함)에서 LLM 추론의 Operational Intensity를 Roofline 모델로 분석. **하드웨어 이종성 유발 효율성 함정(efficiency trap)**을 경고.
- **Thor 적용 가능성**: **높음.** Thor에서 Alpamayo 모델의 Roofline 위치를 확인하는 데 바로 적용 가능한 프레임워크.

#### B-6. LLM Inference at the Edge: Mobile, NPU, and GPU Performance Efficiency Trade-offs Under Sustained Load
- **arXiv**: [2603.23640](https://arxiv.org/abs/2603.23640)
- **iGPU 관련 내용**: 모바일 플랫폼(iPhone 16 Pro, Galaxy S24 Ultra 등)과 전용 NPU(Hailo-10H)의 지속 부하 하 성능·전력 트레이드오프. 열 제한(thermal throttling)이 iGPU 성능을 크게 저하시킴을 발견.
- **Thor 적용 가능성**: **낮음(간접 참고).** Jetson Thor는 모바일 칩과 달리 열 설계 여유가 크다.

### 등급 C — Apple Silicon 통합 메모리를 일부 활용하지만, 이종 가속기 조합에 초점

#### C-1. Native LLM and MLLM Inference at Scale on Apple Silicon (vllm-mlx)
- **arXiv**: [2601.19139](https://arxiv.org/abs/2601.19139); 2026년 1월
- **통합 메모리 활용**: MLX의 zero-copy 텐서 연산, lazy evaluation으로 커널 융합, 멀티모달 모델용 content-based prefix 캐싱.
- **성능**: llama.cpp 대비 21%~87% 처리량 향상 (Qwen3-0.6B~Nemotron-30B), 16 동시 요청 시 4.3× 집계 처리량.
- **사용 하드웨어**: Apple M4 Max, 128GB 통합 메모리.
- **Thor 적용 가능성**: **중간.** Apple-specific MLX 프레임워크이지만, zero-copy 통합 메모리 활용 패턴은 Thor에 conceptually 적용 가능.

#### C-2. Profiling Large Language Model Inference on Apple Silicon: A Quantization Perspective
- **arXiv**: [2508.08531](https://arxiv.org/abs/2508.08531); 2025년 8월
- **통합 메모리 활용**: M2 Ultra·M2 Max·M4 Pro에서 8B~405B 모델, 14개 양자화 방식 체계적 분석. 통합 메모리로 초대형 모델을 NVIDIA GPU보다 비용 효율적으로 운영 가능함을 입증.
- **Thor 적용 가능성**: **중간(데이터 참고용).** 양자화 전략 선택의 참고 자료.

#### C-3. Efficient Mixture-of-Experts LLM Inference with Apple Silicon NPUs (NPUMoE)
- **arXiv**: [2604.18788](https://arxiv.org/abs/2604.18788); 2026년 4월
- **통합 메모리 활용**: Apple M-series의 통합 메모리 풀에서 CPU·GPU·ANE(Apple Neural Engine) 간 zero-copy. MoE 전문가 라우팅의 동적 텐서 형상을 오프라인 보정으로 NPU-friendly하게 변환.
- **성능**: 기존 대비 지연 1.32×~5.55× 감소, 에너지 효율 1.81×~7.37× 향상.
- **Thor 적용 가능성**: **낮음~중간.** Apple ANE 전용 기법이나, DLA(Deep Learning Accelerator)가 있는 Thor에 개념적으로 참고 가능.

#### C-4. Benchmarking and Characterization of LLM Inference on Apple Silicon
- **게재**: ACM Measurement and Analysis of Computing Systems (SIGMETRICS); DOI: 10.1145/3771563
- **통합 메모리 활용**: Apple Silicon M-series에서 LLM 추론의 체계적 특성 분석 (통합 메모리 용량·대역폭과 추론 성능의 관계).
- **Thor 적용 가능성**: **중간(참고용).** 통합 메모리 크기·대역폭과 LLM 성능 관계 데이터 참고.

### 등급 D — CPU/GPU 하이브리드 추론 (통합 메모리 외 PCIe 시스템도 포함)

#### D-1. APEX: Asynchronous Parallel CPU-GPU Execution for Online LLM Inference on Constrained GPUs
- **arXiv**: [2506.03296](https://arxiv.org/abs/2506.03296); 2026년 1월
- **관련성**: KV 캐시 오프로딩을 CPU로 비동기 수행. **decode 단계의 CPU-GPU 비동기 중첩**이 핵심 아이디어 — Alpamayo 파이프라인 설계와 직접적으로 관련.
- **성능**: GPU-only 대비 최대 96% 처리량 향상 (T4 기준 84~96%, A10 기준 11~89%).
- **Thor 적용 가능성**: **높음(개념 이식 가능).** PCIe 대신 통합 메모리를 쓰는 Thor에서는 CPU-GPU 비동기화의 오버헤드가 더 작으므로 이 아이디어가 더 효과적일 수 있다.

#### D-2. KTransformers: Unleashing the Full Potential of CPU/GPU Hybrid Inference for MoE Models
- **게재**: ACM SOSP 2025; DOI: 10.1145/3731569.3764843
- **관련성**: AMX 특화 CPU 커널 + 비동기 CPU-GPU 태스크 스케줄링. MoE 희소성 활용.
- **성능**: 기존 대비 prefill 4.62~19.74×, decode 1.25~4.09× 속도 향상.
- **Thor 적용 가능성**: **낮음~중간.** Alpamayo는 dense 모델이나 파이프라인 스케줄링 설계는 참고 가능.

#### D-3. Fast On-device LLM Inference with NPUs (llm.npu)
- **게재**: ASPLOS 2025; DOI: 10.1145/3669940.3707239; arXiv: [2407.05858](https://arxiv.org/abs/2407.05858)
- **관련성**: 모바일 NPU를 prefill 가속에 활용. CPU·GPU와 병렬 실행으로 이종 SoC 대역폭 분할 활용.
- **성능**: 실제 앱 end-to-end 32.8× 속도 향상, 1B 모델 기준 1,000 tokens/s prefilling 달성.
- **Thor 적용 가능성**: **중간.** Thor의 DLA를 prefill 오프로딩에 활용하는 아이디어로 이어질 수 있음.

#### D-4. Scaling On-Device GPU Inference for Large Generative Models (ML Drift)
- **게재**: CVPRW 2025 (EDGE 워크숍); arXiv: [2505.00232](https://arxiv.org/abs/2505.00232)
- **관련성**: 모바일·데스크탑 iGPU를 포함한 크로스 플랫폼 GPU 추론 엔진. 텐서 가상화로 iGPU의 메모리 레이아웃 유연성 확보.
- **Thor 적용 가능성**: **중간.** 크로스 플랫폼 설계 참고.

### 2.2 Apple Silicon (M-series)

**전용 논문: 발견되지 않음.** Intel Arc / Xe iGPU에서 Transformer 추론을 다룬 학술 논문은 검색 결과에 나타나지 않았다. Intel의 OpenVINO 툴체인 기술 블로그는 존재하나 학술 논문 수준이 아니다.

### 2.5 기타 SoC (AMD MI300A / NVIDIA GH200)

#### Roofline Analysis of Tightly-Coupled CPU-GPU Superchips: A Study on MI300A and GH200
- **게재**: SC Workshops 2025 (P3HPC); ACM DL: 10.1145/3731599.3767497
- **관련성**: MI300A(APU, unified memory)와 GH200(NVLink-C2C)의 Roofline 모델 확장. **단일 메모리 컨트롤러를 공유하는 동시 실행 시의 경쟁(contention)을 Roofline으로 정량화**한 최초 논문 중 하나.
- **Thor 적용 가능성**: **중간~높음.** Thor의 단일 LPDDR5X 컨트롤러에서 CPU·GPU 동시 실행 시 대역폭 경쟁을 분석하는 데 이 방법론을 직접 적용 가능.

## 4. "iGPU 전용 논문이 거의 없다" — 왜 없는가?

검색 결과를 종합하면, iGPU 통합 메모리를 **주된 설계 동인**으로 삼은 Transformer 추론 최적화 논문이 극히 적은 이유는 다음과 같다.

### 4.1 dGPU 중심의 연구 생태계 (구조적 편향)

LLM 연구의 주류는 데이터센터 A100/H100이다. 이 플랫폼은 HBM(~2 TB/s), PCIe 대역폭(~64 GB/s), 분리된 VRAM 등 iGPU와 근본적으로 다른 구조다. 검색에서 반복 등장하는 FlashAttention, vLLM, PagedAttention, Continuous Batching 등의 주요 논문들은 모두 dGPU 전용이며, iGPU에 이식 시 가정이 깨진다.

### 4.2 "Jetson = 테스트 플랫폼" 관행

Jetson은 학계에서 주로 일반 LLM 추론 방법의 벤치마킹 플랫폼으로 사용된다. 실제로 발견된 Jetson 논문들(Understanding Performance, Systematic Characterization)은 설계 기여 없이 특성 분석에 그친다. **Jetson의 통합 메모리 구조를 설계 차별화 요소로 삼은 논문은 Ghidorah 단 1편**이다.

### 4.3 Apple Silicon의 특수성

Apple Silicon 관련 논문은 상대적으로 많지만, Apple MLX 프레임워크와 ANE에 특화되어 있어 NVIDIA CUDA 생태계인 Jetson Thor에 직접 이식하기 어렵다.

### 4.4 Snapdragon/모바일 연구의 NPU 편중

모바일 SoC 연구(HeteroLLM, llm.npu 등)는 NPU와 GPU의 협력에 집중한다. 하지만 Jetson Thor는 NPU보다 GPU 성능이 월등히 높아(SM 11.0, 2,070 TFLOPS FP4), NPU 중심 모바일 연구를 직접 적용하기 어렵다.

### 4.5 단일 메모리 컨트롤러 경쟁 문제의 학문적 무시

가장 큰 공백 중 하나: iGPU에서 CPU와 GPU가 **동일 LPDDR5X 메모리 컨트롤러**를 공유할 때 발생하는 대역폭 경쟁을 정량적으로 모델링하고 이를 스케줄링에 활용한 Transformer 추론 논문은 사실상 없다. Roofline Superchips 논문(SC25)이 이 방향으로 가장 가까운 시도이지만, 대형 서버급 MI300A·GH200을 대상으로 한다. 임베디드 iGPU(LPDDR5X, 231 GB/s) 수준에서의 연구는 미개척이다.

## 5. Alpamayo / Jetson AGX Thor 직접 적용 가능 논문

우선순위 순으로 정리한다.

### 최우선 (직접 적용 가능)

1. **Ghidorah [2505.23219]** — Jetson 플랫폼 직접 사용, 통합 메모리 zero-copy 파이프라인. Alpamayo에 Draft 소형 모델을 도입해 추측 디코딩 + CPU·GPU 이종 파이프라인 적용 가능.

2. **Agent.xpu [2506.24045]** — "memory-unified heterogeneous SoC"에서 iGPU-only 엔진 대비 커널 선점 스케줄링. Alpamayo의 VE(비전 인코더)·LM(언어 모델)·Flow(궤적 생성) 3단계 파이프라인에 커널 수준 스케줄링 개념을 이식 가능.

3. **HeteroLLM [2501.14794]** — GPU+보조가속기(DLA 등) 이종 병렬 실행 + 통합 메모리 동기화 설계. Thor의 DLA(Deep Learning Accelerator)를 decode 단계 일부 연산에 병렬 활용할 때 참고.

4. **APEX [2506.03296]** — Async CPU-GPU decode 오버랩. Thor에서 통합 메모리이므로 PCIe 오버헤드가 없어 APEX보다 더 효과적인 오버랩을 기대할 수 있음.

### 차순위 (분석·참고용)

5. **Systematic Characterization [2512.01644]** — Jetson Orin Roofline 분석. Thor에서도 동일 방법론으로 decode memory-bound 정도를 정량화하는 데 즉시 활용.

6. **RooflineBench [2602.11506]** — 엣지 하드웨어 OI 분석 프레임워크. Alpamayo 각 단계(VE·LM prefill·LM decode·Flow)의 OI를 Thor의 231 GB/s 대역폭과 비교해 병목 진단 가능.

7. **Roofline Analysis Superchips [SC25]** — 단일 메모리 컨트롤러 공유 contention 모델링 방법론. Thor에서 CPU·GPU 동시 실행 시 대역폭 경쟁을 정량화하는 실험 설계에 직접 사용.

8. **Understanding Performance [2506.09554]** — Jetson Orin AGX 실측 데이터. 우리 Thor 실측치(4,838ms)와 비교해 Thor의 우위를 정량화하는 근거 자료.

## 6. 결론 및 연구 공백(Research Gap) 분석

### 6.1 결론 요약

| 질문 | 답변 |
|------|------|
| iGPU 통합 메모리를 주 설계 동인으로 삼은 Transformer 추론 논문이 있는가? | **극소수** (A등급 4편, 대부분 모바일 SoC 또는 Apple Silicon) |
| Jetson 계열 iGPU 전용 설계 논문이 있는가? | **Ghidorah 1편** (Jetson NX, 2025년 5월) |
| Jetson AGX Thor(SM 11.0, 128GB, 231 GB/s)를 전용 타겟으로 삼은 논문이 있는가? | **없음** |
| iGPU의 단일 LPDDR 메모리 컨트롤러 경쟁을 LLM 스케줄링에 활용한 논문이 있는가? | **없음** |
| Layer-level async pipeline을 iGPU 통합 메모리 관점에서 설계한 논문이 있는가? | **없음** |

### 6.2 구체적 Research Gap (미개척 영역)

본 서베이가 확인한 연구 공백을 구체적으로 서술한다.

**Gap 1: iGPU LPDDR5X 단일 메모리 컨트롤러 경쟁 모델링**  
Alpamayo 추론 중 CPU(aarch64 A78AE)와 GPU(SM 11.0)가 동시에 LPDDR5X에 접근할 때 실제 대역폭이 어떻게 분배되는지를 정량적으로 측정하고 스케줄링에 반영한 논문이 없다. 이는 layer prefetch와 compute를 중첩할 때 prefetch의 실제 이득을 결정하는 핵심 변수다.

**Gap 2: 대형 임베디드 iGPU (128GB)에서의 KV Cache L2 Persistent Residency**  
모바일 SoC(8~16GB) 연구들은 메모리 부족 문제에 집중한다. 그러나 Jetson AGX Thor처럼 **128GB 통합 메모리를 가진 대형 임베디드 iGPU**에서는 메모리 용량이 아닌 **대역폭 효율**이 병목이다. GPU L2 Cache(32MB)를 KV Cache의 hot layer에 고정(persistent residency)하는 전략을 iGPU 통합 메모리 구조에서 체계적으로 연구한 논문은 없다.

**Gap 3: VLA 모델의 Multi-stage iGPU 파이프라인**  
VE(비전 인코더) + LM(언어 모델) + Action Expert처럼 이질적인 3단계 구조를 가진 VLA(Vision-Language-Action) 모델의 Async 파이프라인을 iGPU에서 설계한 논문은 전무하다. 기존 LLM 최적화 연구는 단일 decoder-only 모델을 가정한다.

**Gap 4: cudaMemPrefetchAsync + CUDA Stream 이중화의 iGPU 실측 효과**  
dGPU에서 검증된 `cudaMemPrefetchAsync` + 이중 CUDA Stream 기법이 **통합 메모리(PCIe 없음, 단일 컨트롤러)**에서 어느 정도의 실질적 중첩 이득을 주는지 측정한 논문이 없다. iGPU에서는 prefetch가 단순히 동일 컨트롤러로의 read-ahead이므로 dGPU와 다른 특성이 예상되지만 데이터가 없다.

**Gap 5: SM 11.0 / CUDA 13.0 / JetPack 7 생태계에서의 LLM 추론 최적화**  
Jetson AGX Thor의 SM 11.0은 FP4 native 연산(2,070 TFLOPS)을 지원하는 최신 아키텍처다. 이 아키텍처에서 LLM decode의 실제 arithmetic intensity와 memory-boundedness를 측정하고 파이프라인 최적화에 적용한 학술 논문이 없다.

### 6.3 우리 연구의 포지셔닝

위의 Gap 분석을 바탕으로, 본 프로젝트(Alpamayo on Jetson AGX Thor)의 연구 기여는 다음과 같이 정위될 수 있다:

> **"단일 LPDDR5X 메모리 컨트롤러를 공유하는 128GB 임베디드 iGPU(Jetson AGX Thor SM 11.0)에서, VLA 모델의 Multi-stage 구조(VE + LM + Action Expert)에 Layer-level Async Pipeline을 적용해 decode throughput을 극대화한 첫 번째 시스템 연구"**

이 포지셔닝은:
- dGPU 중심 기존 연구(RT-Swap, Demand Layering)와 차별화됨
- 모바일 SoC 연구(HeteroLLM: 8~16GB, 68 GB/s)와 스케일이 다름
- Apple Silicon 연구(MLX: CUDA 비의존)와 플랫폼이 다름
- Ghidorah(Jetson NX)보다 대형 메모리·고성능 플랫폼을 대상으로 함

현재까지 이 조합을 정확히 다룬 선행 연구는 없으며, 이는 연구의 novelty를 강하게 뒷받침한다.

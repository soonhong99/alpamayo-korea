## 소스 코드 분석 요약 (실험 설계 근거)

실험 설계 전 `src_alpamayo1_5/` 전체를 정독한 결과, 다음을 확인했다.

### 1. `tokens_per_future_traj = 64` (default max decode tokens)

```python
# src_alpamayo1_5/models/base_model.py L217
tokens_per_future_traj: int = 64
```

test_inference.py는 `max_generation_length=256`으로 오버라이드한다. 프로파일 결과 실제 생성 토큰은 ~65개.

### 2. `output_logits=True` — 결과 사용 안 함

```python
# src_alpamayo1_5/models/alpamayo1_5.py L266-267
generation_config.output_logits = True
generation_config.return_dict_in_generate = True
```

`vlm_outputs.logits`는 generate() 이후 어디서도 사용되지 않는다. CFG 변형에서는 심지어 즉시 `del vlm_outputs.logits`한다.  
→ **무료 최적화 포인트**: `output_logits=False`로 변경하면 65 step × vocab_size logit 저장 비용 제거.

### 3. `StopAfterEOS.all()` — per-step GPU→CPU sync 원인

```python
# src_alpamayo1_5/models/token_utils.py L199
result = bool(self.eos_found.all())  # .all() → Python bool → GPU-CPU sync!
```

`.all()`이 Python bool로 변환될 때 `cudaStreamSynchronize`가 발생한다.  
→ 65 decode steps = **65번 강제 동기화**.

### 4. `num_return_sequences = num_traj_samples` — 진짜 배치 확장

```python
# src_alpamayo1_5/models/alpamayo1_5.py L266
generation_config.num_return_sequences = num_traj_samples
```

HuggingFace generate()가 내부적으로 배치를 `num_traj_samples`배로 확장한다.  
→ VLM decode에서 seq=N 처리 → memory-bound → compute-bound 전환 가능.

### 5. Action Expert가 VLM KV cache를 재사용

```python
# src_alpamayo1_5/models/alpamayo1_5.py L319-320
prompt_cache = vlm_outputs.past_key_values
...
expert_out_base = self.expert(past_key_values=prompt_cache, ...)
prompt_cache.crop(prefill_seq_len)  # ← StaticCache에서 작동하는지 확인 필요
```

StaticCache의 `crop()` 지원 여부가 Action Expert 호환성의 핵심.

## 실험 1: 배치 확장 (`260527_batch_scaling_exp.py`)

### 실험 설계

```
num_traj_samples ∈ {1, 2, 4, 6, 8}
워밍업: 2회 / 측정: 5회
시드: torch.cuda.manual_seed_all(42)

측정:
  t_total    = 전체 inference (VLM + Action Expert)
  t_vlm      = VLM generate() 단독 (CUDA Event)
  t_action   = diffusion.sample() 단독 (CUDA Event)
  n_tokens   = 실제 생성 토큰 수
  minADE     = 궤적 품질 (m)
```

### 이론 예측

| num_samples | 이론 VLM 시간 | VLM 실제 예상 | 비고 |
|-------------|--------------|---------------|------|
| 1 | 6.13s (baseline) | 6.13s | memory-bound (DRAM 읽기) |
| 2 | 6.13s (가중치 동일) | 6.1~7s | 2× DRAM BW 압박 시작 |
| 4 | 6.13s (이론적) | 6~8s | 점차 compute-bound 진입 |
| 8 | 6.13s (이론적) | 6~10s | compute 제한 가능성 |

**핵심 가설**: VLM 총 시간이 num_samples에 거의 비례하지 않고 완만하게 증가  
→ 샘플당 시간(`t_total / num_samples`)이 줄어드는지 확인.

### minADE 예측

```
num_samples 증가 → 더 많은 궤적 후보 → minADE 단조 감소
이론: minADE(N) ≈ minADE(1) × N^(-0.5) (중심극한 근사)
```

## 실험 2: KV Cache 가방 (`260527_kv_cache_bag.py`)

### 실험 구성

```
Phase 0: 베이스라인 + sync 계측
  - InstrumentedStopAfterEOS: 호출당 시간 측정
  - output_logits=True (그대로)
  - 기대: stop_eos_calls ≈ 65, 호출당 0.1~0.5ms

Phase 1: output_logits=False
  - 로짓 저장 제거
  - 기대: ~5~20ms 절감 (vocab 152K × 65 step × 2B = 19.5MB 할당/해제)

Phase 2: NoSyncStopCriteria (EOS sync 제거)
  - max_new_tokens = 측정된 토큰 수 + 5 (고정)
  - 기대: Phase 0 대비 (stop_eos_calls × per_call_ms) 절감

Phase 3: StaticCache
  - HuggingFace StaticCache 주입
  - KV cache 동적 할당 제거
  - 기대: ~1.1~1.2× speedup (캐시 allocation 오버헤드 제거)

Phase 4: Custom Decode Loop (KV Cache 가방 핵심)
  - vlm.forward() 직접 호출
  - GPU-resident top-p 샘플링 (torch.multinomial)
  - GPUEosTracker: EOS 위치 GPU에 저장
  - 단 2회 CPU sync: token_buffer.cpu() + eos_tracker.get_eos_positions()
  - CUDAGraph 캡처 시도
  - 기대: 최소 sync overhead 측정 + CUDAGraph 가능성 확인
```

### KV Cache 크기 계산 (실측 기반)

```
Model: 36 layers, 8 KV-heads, head_dim=128, BF16

Per-token KV cache = 36 × 2 × 8 × 128 × 2 = 144 KB

★ 실측: Prefill token count = 3086
  (7 cameras × 4 frames = 28 images, 이미지당 ~110 visual tokens +
   system prompt + history trajectory tokens)

Prefill (3086 tokens):  3086 × 144 KB = 432.8 MB
Decode max (65 tokens):   65 × 144 KB =   9.1 MB
StaticCache 전체 (seq=1): 3151 × 144 KB = 441.9 MB  ← 예상의 7.7배

num_traj_samples=8 시 전체 KV cache:
  weights:      36 layers × 386 MB = 13.9 GB (공유)
  prefill KV:   8 × 432.8 MB = 3.5 GB
  decode KV:    8 × 9.1 MB   = 72.8 MB (증가 중)
  ────────────────────────────────
  총 DRAM 점유: ~17.5 GB (128 GB의 13.7%)

per-decode-step DRAM reads (seq=8):
  weights:     36 × 386 MB = 13.9 GB (1번 읽어서 8 seq 처리)
  attn KV:     36 × 8 × 3086 tokens × 4 KB = 3.5 GB
  합계:        ~17.4 GB/step  vs  seq=1: ~14.3 GB/step (21.7% 증가)
  → 8배 배치, 22% 추가 BW만 사용 → per-sample 6.6× 효율 향상 (이론)
```

### 예상 결과 (보수적 추정)

| Phase | 예상 절감 | 근거 |
|-------|-----------|------|
| output_logits=False | 5~30ms | vocab×steps×dtype 할당 제거 |
| NoSync stop criteria | 10~50ms | 65 × 0.2~0.8ms/sync |
| StaticCache | 5~15ms | dynamic alloc 제거 |
| Custom loop | 20~100ms | 전체 sync 제거 + cache overhead 제거 |
| CUDAGraph | 50~136ms | kernel launch 65×300개 제거 |

## 예상 실패 시나리오 및 대처

### StaticCache 미지원 (`ImportError`)

```
from transformers import StaticCache  → ImportError
```

원인: transformers 버전 < 4.40  
대처: `python3 -c "import transformers; print(transformers.__version__)"` 확인  
해결: Phase 3는 `"error": "StaticCache not available"` 로 기록되고 스킵됨

### CUDAGraph 캡처 실패

Qwen3VL의 decode step에 Python-level 동적 분기가 있으면 실패.  
예상 오류: `RuntimeError: CUDA graphs do not support dynamic control flow`  
대처: `cuda_graph_captured=False` 로 기록, 나머지 실험은 계속 진행

### Custom Loop 출력 품질 검증

Phase 4와 Phase 0의 생성 토큰을 비교해야 함.  
- 동일 시드(`torch.cuda.manual_seed_all(42)`) 사용
- top-p 샘플링 구현 차이로 완전 동일 보장은 안 되지만 유사해야 함
- Action Expert 호환성(`action_expert_compatible=True`)이 핵심 검증 포인트

## 결과 해석 포인트

실험 후 다음 질문에 답해야 한다:

1. **배치 확장**: `t_total / num_samples`가 num_samples=8에서 1 대비 얼마나 감소하는가?
   - ≥2× 감소: memory-bound 탈출 성공
   - <1.5× 감소: iGPU BW가 완전히 saturated

2. **sync 오버헤드**: `stop_eos_overhead_ms`가 총 VLM 시간의 몇 %인가?
   - >1%: 제거 가치 있음
   - >5%: 우선순위 높음

3. **CUDAGraph**: 캡처 성공하는가?
   - 성공 + 유의미한 speedup: 다음 실험에 CUDAGraph 통합
   - 실패: Qwen3VL이 지원하지 않는 op 존재 → 원인 분석 후 partial capture 시도

4. **Action Expert 호환성**: `StaticCache.crop()` 작동하는가?
   - True: 전체 pipeline에 StaticCache 적용 가능
   - False: generate() 이후 cache 교체 필요

# [Alpamayo] iGPU 통합 메모리 대역폭 측정 결과 및 경합 분석 리포트

**작성일자:** 2026-05-28  
**대상 기기:** Thor 보드 (iGPU 환경)

---

## 1. 실험 개요 및 '대역폭(Bandwidth)'의 명확한 정의

### 1.1. 대역폭(Bandwidth)이란 무엇인가?
본 실험에서 말하는 **대역폭(Bandwidth)** 이란, "단위 시간당 **물리적 메모리(DRAM)** 에서 연산 장치(CPU 또는 GPU)로 데이터를 얼마나 많이 가져올 수 있는가(GB/s)"를 의미합니다. Alpamayo 모델의 VLM 디코딩 과정은 연산량보다 가중치를 메모리에서 불러오는 시간이 압도적으로 긴 **Memory-bound** 작업이므로, 이 DRAM 대역폭 수치가 전체 추론 속도의 상한선(Ceiling)을 결정합니다.

### 1.2. 캐시를 배제한 '순수 DRAM 대역폭' 측정의 중요성 (실험의 기준)
만약 작은 배열(예: 10MB)을 반복해서 읽거나, 여러 CPU 코어가 같은 메모리 주소를 읽게 설계했다면, 첫 번째 읽기 이후 데이터가 프로세서 내부의 초고속 캐시(L2/L3)에 상주하게 됩니다. 이 경우 측정되는 속도는 수백~수천 GB/s에 달하는 '캐시 대역폭'이 되어버려, 우리가 알고자 하는 진짜 DRAM 병목을 알 수 없습니다.

따라서 본 실험은 **각 프로세스(CPU 코어 및 GPU)마다 L3 캐시 용량을 아득히 초과하는 500MB 크기의 독립적인 텐서를 물리적으로 할당**(`torch.ones`)하여 캐시 히트를 원천 차단(Cache Thrashing)하고 무조건 DRAM에서 데이터를 긁어오도록 강제하는 척도를 기준으로 삼았습니다.

## 2. 실험 측정 방법론

1. **Part A: 다중 CPU 코어 대역폭 스케일링**
   - Python `multiprocessing`을 이용해 1, 2, 4, 8, 12 코어를 활성화합니다.
   - 각 코어는 `torch.set_num_threads(1)`로 내부 스레드 간섭을 막고, 자신만의 500MB 배열을 반복해서 읽어들여 합산(`sum()`)합니다.
   - 모든 프로세스가 메모리 할당을 마치면 `Event` 객체를 통해 **정확히 동시에 출발**하도록 동기화하여 측정의 정확도를 높였습니다.

2. **Part B: CPU - GPU 대역폭 경합 (Starvation) 측정**
   - 통합 메모리(Unified Memory) 구조를 갖는 iGPU는 CPU와 GPU가 동일한 물리 메모리와 메모리 컨트롤러를 공유합니다.
   - GPU에 500MB 배열을 할당해 최대 속도로 읽게 하는 동시에, 12개의 CPU 코어도 각각 500MB 배열을 최대 속도로 읽게 하여 **극한의 메모리 버스 경합(Contention)** 상황을 유도하고 대역폭 변화를 관찰합니다.

## 3. 벤치마크 결과 및 심층 분석

### 3.1. 다중 CPU 코어 대역폭 스케일링 (CPU Only)
| 활성 코어 수 | 대역폭 (GB/s) | 스케일링 양상 |
|:---:|:---:|---|
| 1 코어 | 8.36 GB/s | Baseline |
| 2 코어 | 37.34 GB/s | 비약적 상승 (4.4배) |
| 4 코어 | 76.40 GB/s | 선형적 상승 |
| 8 코어 | 126.30 GB/s | 상승폭 둔화 시작 |
| **12 코어** | **169.49 GB/s** | **CPU 단독 최대 대역폭** |

**[분석]** 코어 수가 증가함에 따라 대역폭이 선형적으로 늘어나다 8코어 이후부터 메모리 컨트롤러 한계에 부딪혀 상승폭이 둔화(Sub-linear)됨을 알 수 있습니다.

### 3.2. 통합 메모리 환경에서의 CPU-GPU 경합 분석
| 실행 조건 | CPU 총 대역폭 | GPU 총 대역폭 | Total (System) 대역폭 |
|:---|:---:|:---:|:---:|
| **GPU 단독 실행** | 0 GB/s | **196.15 GB/s** | 196.15 GB/s |
| **CPU(12C) + GPU 동시 실행** | 146.70 GB/s | **114.46 GB/s** | **261.16 GB/s** |

**[핵심 분석: 대역폭 기아(Starvation) 현상]**
1. **GPU 대역폭의 급감:** GPU가 단독으로 돌 때는 196.15 GB/s의 높은 대역폭을 점유하지만, CPU 12코어가 동시에 메모리를 긁기 시작하자 **GPU의 대역폭이 약 41.6% 하락(114.46 GB/s)하는 치명적인 간섭(Interference)** 이 발생했습니다.
2. **시스템 총 대역폭의 증가:** 하지만 메모리 시스템 전체 입장에서 보면 총 대역폭은 261.16 GB/s로 상승했습니다. 이는 Thor 보드의 메모리 채널이 매우 넓어 CPU와 GPU가 서로 다른 채널을 분할 점유하며 전체 Throughput 자체는 극대화되었음을 의미합니다.

### 3.3. 교수님 보고용 시사점 (연구 방향 연계)
앞선 브레인스토밍에서 **"Vision Encoder 처리(CPU/GPU)와 VLM Decode(GPU)를 동시에 오버랩(Pipeline) 시키자"** 는 아이디어가 있었습니다. 
이번 실험 결과는 이를 구현할 때 **주의해야 할 트레이드오프(Trade-off)** 를 명확히 보여줍니다. 
- 오버랩을 하면 시간은 단축되지만, Vision 전처리(CPU)가 돌아가는 동안 VLM Decode(GPU)가 사용할 수 있는 메모리 대역폭이 40% 깎이게 됩니다. 
- 따라서 파이프라이닝으로 얻는 이득이 대역폭 하락으로 인한 디코드 지연 페널티보다 큰지 교차점을 찾아내는 것이 향후 "차세대 최적화 논문"의 핵심 컨트리뷰션이 될 것입니다.

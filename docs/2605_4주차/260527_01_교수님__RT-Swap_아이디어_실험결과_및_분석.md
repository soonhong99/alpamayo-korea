# RT-Swap / Demand Layering 아이디어 적용 실험 — 전체 결과 및 분석

**날짜**: 2026-05-26 ~ 2026-05-27  
**실험자**: Alphamayo 프로젝트 팀  
**하드웨어**: Jetson AGX Thor (iGPU, 128GB 통합 메모리)  
**모델**: Alpamayo 1.5 (10B, `nvidia/Alpamayo-1.5-10B`)

---

## 한 줄 요약

> RT-Swap / Demand Layering 아이디어를 Thor iGPU에 `tensor.sum()` 기반으로 구현한 결과,  
> **파이프라인 조건은 이론적으로 성립했으나 실제 speedup은 0.65× (오히려 54% 느려짐)** 이었다.  
> 원인은 iGPU의 구조적 차이(DMA 엔진 부재)로 인한 **DRAM 트래픽 이중화**이며,  
> 이를 통해 Alpamayo decode가 **memory-bandwidth-bound** 임이 명확히 확인되었다.

---

## 1. 배경 — 어떤 문제를 해결하려 했나

### 문제 정의

Alpamayo 1.5의 Thor 보드 실측 추론 속도: **약 6초 (6.13 s)**  
자율주행 실시간 요구사항: **100 ms 이하**  
→ 약 **60× 격차**를 줄이는 것이 연구 목표

이 중 가장 큰 비중을 차지하는 단계는 **VLM autoregressive decode** (65 step × 36 레이어):

```
Vision Encoder  →  VLM Prefill  →  [VLM Decode × 65]  →  Action Expert
   (빠름)            (빠름)              ← 병목 →              (빠름)
```

decode 1 step = 36개 Transformer 레이어를 순서대로 통과  
측정 결과: 1 step = **94 ms**, 65 step = **6.13 s**

### 연구 방향 설정 (2026-05-24 교수님 미팅)

모델 수정 없이 **시스템 레벨 최적화**로 throughput 향상.  
참고 논문 2편에서 아이디어를 차용하기로 결정.

## 2. 차용한 논문 아이디어

### 논문 1: RT-Swap (RTAS 2024)

**제목**: *RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference*  
**저자**: NVIDIA 실시간 시스템 그룹  
**대상**: dGPU 환경에서 여러 DNN을 동시에 실행할 때의 메모리 병목 해소

**핵심 아이디어**:

```
[기존 순차 실행]
레이어 i:   ─── 가중치 로딩 (DRAM→VRAM) ───▶─── 계산 ───▶
레이어 i+1:                                               ─── 가중치 로딩 ───▶─── 계산 ───▶

[RT-Swap: 비동기 overlap]
레이어 i:   ─── 가중치 로딩 ───▶─────────── 계산 ────────▶
레이어 i+1:                    ─── 가중치 미리 로딩 ───▶    (이미 준비됨!)
```

**수식**: `T_async = Σ max(t_prefetch_{i+1}, t_compute_i)`

**RT-Swap이 작동하는 전제 조건**:  
dGPU에는 **DMA 엔진**(Direct Memory Access engine)이 SM 연산 유닛과 **물리적으로 독립**되어 있다.  
→ 가중치 이동(DMA)과 계산(SM)이 진정한 병렬 실행 가능

### 논문 2: Demand Layering (RTSS 2022)

**핵심 아이디어**: 레이어 *i* 실행 **직전** 에, 레이어 *i+1* 의 prefetch 명령을 비동기로 발행.  
즉 "필요하기 직전에, 미리 요청만 해두고 기다리지 않는다"는 것.

**두 논문의 조합**:
> "compute stream에서 레이어 i를 실행하는 동안,  
> prefetch stream에서 레이어 i+1의 가중치를 L2 캐시로 워밍한다."

## 3. 실험 설계 — 3단계 검증 체계

```
[Phase 0-A] BW 측정          [Phase 0-B] t_compute 측정    [Phase 1] 실제 pipeline
260526_prefetch_effect_test   260524_layer_compute_profile   260524_async_pipeline
        ↓                              ↓                              ↓
 L2=1126 GB/s                  2.62 ms/layer                Baseline vs Async
 DRAM=227 GB/s                 36/36 조건 성립 ✅             speedup 측정
        ↓                              ↓                              ↓
 t_prefetch 기준값 산출         파이프라인 가능 판단          실제 효과 검증
```

**판단 조건**: `t_compute_per_layer > t_prefetch_per_layer`  
이 조건이 성립해야 prefetch 시간이 compute 시간 안에 완전히 숨겨진다.

## 4. Phase 0-A: 메모리 대역폭 측정

**실험 파일**: `scripts/profiling/260526_prefetch_effect_test.py`

| 측정 항목 | 결과 | 의미 |
|-----------|------|------|
| **L2 캐시 대역폭** | **1,126 GB/s** | L2 → SM 전송 속도 |
| **DRAM 대역폭** | **227 GB/s** | DRAM → GPU 전송 속도 |

이 수치로 이론 prefetch 시간 산출:

```
MLP 가중치 하나 (100.7 MB) 기준:
  DRAM: 100.7 MB ÷ 227 GB/s = 0.444 ms  ← pipeline 판단 기준선

전체 레이어 가중치 (385.9 MB) 기준:
  DRAM: 385.9 MB ÷ 227 GB/s = 1.70 ms
  L2:   385.9 MB ÷ 1126 GB/s = 0.343 ms
```

## 6. Phase 1: 실제 Async Pipeline 실행 — 결과 ❌

**실험 파일**: `scripts/inference/260524_async_pipeline.py` (v3, hook-based)

**구현 방식**:
- `lm_model.forward()` 실행 중 각 레이어 `forward_pre_hook`으로 prefetch 주입
- prefetch: `tensor.view(-1).sum()` on `prefetch_stream`
- sync: `torch.cuda.current_stream().wait_stream(prefetch_stream)`

### 측정 결과

| 모드 | ms/step | 65-step 합산 |
|------|---------|-------------|
| **Baseline** | **95.96 ms** | **6,237 ms (6.24 s)** |
| **Async Pipeline** | **147.82 ms** | **9,609 ms (9.61 s)** |
| **Speedup** | **0.6491×** | **−3,371 ms** |

**→ 파이프라인이 오히려 54% 느려짐 ❌**

## 7. 실패 원인 분석 — 3가지 층위

### 원인 1 (치명적): tensor.sum()은 "읽기 2번"이다

```
[기대했던 동작]
sum() → L2 캐시 채움 → GEMM이 L2에서 빠르게 읽음

[실제 동작]
sum() → 386 MB DRAM 전체 읽기 (→ L2엔 마지막 32MB만 잔류)
GEMM  → 386 MB 처음부터 다시 읽음 (앞 부분 L2 miss → DRAM)
→ DRAM 트래픽 = 2 × 386 MB = 772 MB/layer
```

추가 DRAM 트래픽:

```
36 layer × 386 MB = 13.9 GB 추가
13.9 GB ÷ 227 GB/s = 61 ms 추가 부하

측정된 overhead: 147.82 − 95.96 = 51.86 ms ← 이론값과 일치
```

### 원인 2 (구조적): Thor iGPU에는 DMA 엔진이 없다

```
[dGPU — RT-Swap 논문의 전제]
CPU DRAM ──PCIe──▶ GPU VRAM
                    DMA 엔진 (SM과 독립된 하드웨어)
prefetch: DMA 엔진 처리 → SM 부하 0%
compute: SM 처리 → DMA 0%
→ 진정한 병렬 실행 ✅

[Thor iGPU — 우리 환경]
GPU SM ◀─────────── 통합 DRAM (하나의 물리 메모리)
sum() stream: SM으로 처리 (DRAM BW 점유)
compute stream: SM으로 처리 (DRAM BW 점유)
→ SM 경쟁 + DRAM BW 경쟁 → 둘 다 느려짐 ❌
```

**RT-Swap의 전제 조건이 Thor iGPU에서 성립하지 않는다.**

### 원인 3 (인식 오류): t_compute 2.62 ms는 "계산 시간"이 아니었다

seq=1 decode step에서 레이어 하나의 순수 FLOPs 계산:

```
Q proj [1,4096] × [4096,4096]: ~33M FLOPs
MLP gate [1,4096] × [4096,14336]: ~117M FLOPs
총 ~450M FLOPs per layer

@ Thor iGPU 이론 성능 100 TFLOPS:
→ 순수 compute 시간 = 450M ÷ 100T = 0.0045 ms (무시 가능)
→ 2.62 ms의 99%는 DRAM에서 386 MB 가중치를 읽는 시간
```

| 항목 | 값 |
|------|-----|
| 실측 layer 시간 | 2.62 ms |
| 이론 DRAM 읽기 시간 (386 MB ÷ 227 GB/s) | 1.70 ms |
| 유효 DRAM 대역폭 (386 MB ÷ 2.62 ms) | 147 GB/s (피크의 65%) |
| 순수 compute (FLOPs 기반 이론) | ~0.005 ms |

**Alpamayo decode는 완전한 memory-bandwidth-bound 연산이다.**  
→ compute 시간과 prefetch 시간을 "overlap"한다는 개념이 성립하지 않음.  
→ 2.62 ms 자체가 이미 "DRAM 읽기 + 계산이 섞인 시간"이며, 여기에 추가 DRAM 읽기를 얹으면 더 느려질 수밖에 없음.

## 8. 이번 실험으로 확인된 핵심 사실

| 확인된 사실 | 의미 |
|-------------|------|
| t_compute (2.62 ms) = 거의 전부 DRAM BW 대기 시간 | compute-bound가 아님 |
| 유효 DRAM BW = 147 GB/s (피크 227의 65%) | 소규모 GEMM 비효율성 |
| L2 (32 MB) ≪ 레이어 가중치 (386 MB) | L2 warming의 근본적 한계 |
| iGPU = DMA 엔진 없음 | RT-Swap의 전제 조건 불성립 |
| tensor.sum() = SM 사용 compute 커널 | prefetch가 아닌 경쟁 자원 소모 |

## 9. 학습 및 결론

### 논문 아이디어 적용의 함정

RT-Swap은 **dGPU + 멀티 DNN + PCIe BW 병목** 을 전제로 설계된 기법이다.  
Thor iGPU는 **단일 통합 메모리 + 단일 GPU die** 구조로, 논문의 HW 전제와 근본적으로 다르다.

> **아이디어의 핵심(레이어 간 비동기 overlap)** 은 옳지만,  
> **구현 수단(DMA via stream)** 이 iGPU에서는 다르게 설계되어야 한다.

### 실험 자체의 가치

이 실험 체인(BW 측정 → t_compute 측정 → pipeline 시도)은 실패였지만, 다음을 확인했다:

1. Alpamayo decode 병목의 **정확한 정체**: memory-bandwidth-bound (99% DRAM 읽기)
2. **유효 대역폭 수치**: 147 GB/s (피크의 65%)
3. **레이어 간 일관성**: 36개 레이어 모두 2.3~2.9 ms로 매우 안정적
4. **올바른 최적화 방향**: DRAM 트래픽 자체를 줄이는 접근 (INT4 등)이 필요

| 접근법 | 예상 효과 | 근거 |
|--------|-----------|------|
| **INT4 가중치 양자화** | 2~4× (DRAM 트래픽 1/4) | 병목이 DRAM BW이므로 직접 효과 |
| **다중 샘플 배치 (num_samples > 1)** | 2~4×/sample | seq 증가 → compute-bound 전환 |

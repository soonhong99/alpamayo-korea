## 한 줄 요약

> Alpamayo 1.5의 각 Transformer 레이어는 실행에 **평균 2.6 ms**가 걸린다.  
> 이 시간은 다음 레이어의 가중치를 미리 불러오는 데 필요한 시간(0.44~1.70 ms)보다  
> **최소 1.5배~6배 길다**. 따라서 **전체 36개 레이어 모두에서 비동기 파이프라인이 이론적으로 가능**하다. ✅

## 1. 배경 — 왜 이 실험을 했나?

### 문제: Alpamayo가 느리다

Alpamayo 1.5는 자율주행 차량의 경로를 예측하는 AI 모델이다.  
모델이 한 번 추론(inference)을 수행할 때 내부적으로는 **65번의 decode step**을 반복한다.  
(한 번의 decode step = Transformer 레이어 36개를 순서대로 통과하는 것)

현재 Thor 보드에서의 실측 속도: **약 6초**  
자율주행 실시간 요구사항: **100 ms 이하**

이 격차를 줄이는 것이 연구 목표다.

### 왜 느린가? — 메모리 병목

AI 모델이 느린 이유는 보통 두 가지 중 하나다.

| 병목 유형 | 설명 | 비유 |
|-----------|------|------|
| **Compute-bound** | GPU 연산이 너무 많아서 느림 | 요리사가 손이 느려서 주문이 밀림 |
| **Memory-bound** | 가중치를 메모리에서 읽어오는 게 느려서 느림 | 재료창고(DRAM)에서 재료를 들고 오는 시간이 걸림 |

Alpamayo는 **단일 토큰 생성(seq=1) decode** 상황에서는 전형적인 **Memory-bound** 모델이다.  
레이어 하나당 가중치가 약 **386 MB** 인데, 이를 매 스텝마다 DRAM에서 꺼내야 한다.

> DRAM 속도: 227 GB/s  
> 386 MB ÷ 227 GB/s ≈ **1.70 ms** — 가중치 읽기만으로도 이 시간이 든다

## 2. 핵심 아이디어 — 두 논문에서 가져온 아이디어

### 논문 1: RT-Swap (RTAS 2024)

**제목**: *RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference*

**핵심 아이디어**: 레이어 *i*를 계산하는 동안 레이어 *i+1*의 가중치를 **미리 불러오면(prefetch)**,  
불러오는 시간이 계산 시간과 겹쳐서 총 시간이 줄어든다.

```
[기존 방식]
레이어 1 가중치 로딩 →→→→ 레이어 1 계산 → 레이어 2 가중치 로딩 →→→→ 레이어 2 계산 → ...

[RT-Swap 방식]
레이어 1 가중치 로딩 →→→→ 레이어 1 계산
                              ↕ 동시 실행!
                   레이어 2 가중치 미리 로딩 →→→→
```

**수식**: `T_async = Σ max(t_prefetch_{i+1}, t_compute_i)`

이 수식의 의미: 각 레이어에서 "더 오래 걸리는 쪽"만 기다리면 된다.  
만약 계산이 로딩보다 오래 걸리면(t_compute > t_prefetch), 로딩은 공짜가 된다.

### 논문 2: Demand Layering (RTSS 2022)

**핵심 아이디어**: 레이어 *i*를 실행하기 **직전**에 레이어 *i+1*을 미리 불러오라는  
명령을 비동기로 발행한다. 즉, 각 레이어 실행 시작 시점이 prefetch 트리거다.

이 두 논문의 아이디어를 합치면:  
> "레이어 i를 계산하는 GPU 스트림과, 레이어 i+1을 미리 불러오는 스트림을  
> 동시에 실행(overlap)하여 총 지연을 줄인다."

---

## 3. 이 아이디어가 Alpamayo에 적용 가능한가? — 판단 조건

RT-Swap이 효과를 발휘하려면 **반드시** 이 조건이 성립해야 한다:

```
t_compute_per_layer  >  t_prefetch_per_layer
```

- **좌변**: 레이어 하나를 실행하는 데 걸리는 시간 (ms)  
- **우변**: 그 레이어의 가중치를 미리 불러오는 데 걸리는 시간 (ms)

만약 좌변 < 우변이면, 계산이 끝나도 가중치가 아직 안 온 것 → 파이프라인 효과 없음.

**이 조건을 실제 숫자로 확인하는 것이 이번 실험의 목적이다.**

## 4. 세 가지 측정값 — 대역폭인가, 시간인가?

> 사용자 질문: "측정하는 값이 다 다른 것으로 대역폭을 측정한 것인가?"

아니다. **세 가지 서로 다른 종류**의 값이 사용됐다.

### 측정값 A: 메모리 대역폭 (GB/s) — `260526_prefetch_effect_test.py`

| 항목 | 값 | 의미 |
|------|------|------|
| **L2 캐시 대역폭** | **1126 GB/s** | L2 캐시에서 GPU 연산 유닛까지 초당 전송 가능한 데이터량 |
| **DRAM 대역폭** | **227 GB/s** | 메인 메모리에서 GPU까지 초당 전송 가능한 데이터량 |

이것은 **속도(속도계)** 다. "이 메모리 경로는 초당 몇 GB를 전달할 수 있나?"

### 측정값 B: 이론 prefetch 시간 (ms) — 계산으로 유도된 값

위의 대역폭과 가중치 크기를 이용해 **직접 실측이 아닌 계산**으로 구한 값이다.

```
t_prefetch_MLP  = 100.7 MB ÷ 227 GB/s = 0.444 ms
t_prefetch_full = 385.9 MB ÷ 227 GB/s = 1.70  ms
```

이것은 **예상 소요 시간**이다. "재료를 창고에서 꺼내오는 데 이론상 얼마나 걸리나?"

### 측정값 C: 레이어 compute 시간 (ms) — `260524_layer_compute_profile.py` (이번 실험)

실제 Alpamayo를 실행하면서 각 레이어가 얼마나 걸리는지 CUDA Event로 직접 측정한 값이다.

이것은 **실측 시간**이다. "재료를 꺼내서 요리를 완성하기까지 실제로 얼마나 걸렸나?"

### 세 값의 관계

```
[A: 대역폭 측정] → [B: 이론 prefetch 시간 계산] ─┐
                                                     ├─→ B < C 이면 파이프라인 가능 ✅
[C: 레이어 compute 시간 측정] ─────────────────────┘
```

이 세 가지는 각자 독립적으로 측정/계산하여, 합쳐서 "파이프라인이 가능한가?"를 판단한다.

## 5. 실험 방법 — 어떻게 측정했나?

### 측정 대상

Alpamayo 1.5의 Transformer 레이어 구조:

```
Alpamayo 1.5 (10B)
  └── VLM (Qwen3VL)
        └── 텍스트 Transformer  (Qwen3VLTextModel)
              └── 36개의 Decoder 레이어  ← 이것을 측정
                    각 레이어 = Attention + MLP (Feed-Forward)
```

### 측정 방법

각 레이어 앞뒤에 **CUDA Event**라는 타이머를 걸어서,  
레이어 실행 시작과 끝의 시각을 GPU 내부에서 직접 기록했다.

```
레이어 0 시작 ⏱ ─────────── 레이어 0 끝 ⏱
              ← 이 간격 = t_compute[0] →

레이어 1 시작 ⏱ ─────────── 레이어 1 끝 ⏱
              ← 이 간격 = t_compute[1] →
...
```

- 워밍업 3회 (캐시 안정화) → 측정 20회 → 평균/표준편차 계산
- 입력: batch=1, seq=1 (실제 decode step과 동일한 조건)

## 7. 결과 해석

### "t_compute가 t_prefetch보다 훨씬 크다"는 무슨 뜻인가?

직관적으로 말하면:

> 레시피 재료를 창고에서 꺼내오는 데 1.7초가 걸리는데,  
> 요리사가 실제로 요리하는 데 2.6초가 걸린다.  
> → 재료 꺼내오기가 완전히 겹쳐진다(2.6초 안에 1.7초 완료) → **요리사가 기다릴 필요 없다**

### 이론적 최대 절약량

RT-Swap 수식 `T_async = Σ max(t_prefetch, t_compute)` 적용:

```
현재(순차): 각 레이어마다 "가중치 로딩 → 계산" 순서
            = t_loading_effective + t_compute_pure
            ≈ 2.62 ms (둘이 합산된 측정값)

파이프라인 적용 후:
  이전 레이어 계산 중 → 다음 레이어 가중치 DRAM에서 미리 로딩 (1.70 ms)
  = max(1.70, t_compute_pure) ← 여기서 t_compute_pure는 순수 연산 시간
```

| 조건 | 예상 절약 |
|------|-----------|
| 레이어 가중치 로딩이 총 시간의 65%를 차지할 경우 | ~1.3× speedup |
| 레이어 가중치 로딩이 총 시간의 50%를 차지할 경우 | ~1.2× speedup |

> **실제 speedup은 `260524_async_pipeline.py` 실행으로 측정한다 (다음 실험).**

### 왜 t_compute가 "순수 계산"보다 클까?

Transformer decode step(seq=1)에서 발생하는 행렬곱은 매우 작다.  
GPU는 일반적으로 큰 배치에서 효율적인데, seq=1은 **가장 비효율적인 경우**다.

- GPU의 이론 연산 속도로 계산하면: 레이어당 **~0.01 ms** (거의 공짜)  
- 실제 측정값: **2.62 ms**  
- 차이의 원인: **DRAM에서 386 MB 가중치를 읽어오는 시간**이 대부분

즉, Alpamayo의 decode는 **전형적인 메모리 대역폭 병목(memory-bandwidth-bound)** 이다.  
이 환경에서 RT-Swap의 메모리 prefetch 아이디어가 가장 효과적으로 작동한다.

## 용어 정의

본 보고서에 처음 등장하는 기술 용어는 아래와 같이 정의하며, 이후 약칭을 사용한다.

| 용어 | 정의 |
|------|------|
| **iGPU** (Integrated GPU) | CPU와 물리적으로 동일한 DRAM을 공유하는 내장 GPU. 별도의 VRAM이 없으며, CPU와 GPU 모두 하나의 메모리 버스를 통해 같은 DRAM 칩에 접근한다. Thor 보드가 이에 해당한다. |
| **dGPU** (Discrete GPU) | CPU와 분리된 전용 VRAM(예: GDDR6, HBM)을 가지는 독립형 GPU. RTX 4090, A100 등이 해당한다. CPU와의 데이터 이동은 PCIe 버스를 통한다. |
| **LPDDR5X** | Low Power Double Data Rate 5X. 저전력 고속 DRAM 규격으로, 스마트폰·임베디드 시스템에 사용된다. Thor 보드의 128 GB 메모리가 이 규격이며 최대 대역폭은 약 273 GB/s (실측 GPU 단독 최대 231 GB/s). |
| **SM** (Streaming Multiprocessor) | GPU의 연산 핵심 단위. 행렬 곱셈(GEMM), 어텐션 등 모든 딥러닝 연산을 수행한다. `tensor.sum()`, GEMM 등은 전부 SM이 처리한다. |
| **CE** (Copy Engine) | GPU 내부의 데이터 이동 전담 하드웨어(DMA 엔진). SM을 전혀 사용하지 않고 메모리 간 데이터를 비동기로 전송한다. `cudaMemcpyAsync`, `cudaMemPrefetchAsync` 등의 API가 CE를 통해 실행된다. |
| **DRAM** | Dynamic Random Access Memory. 컴퓨터의 주 메모리 칩. iGPU에서는 CPU와 GPU가 동일한 DRAM을 공유한다. |
| **L2 Cache** | GPU 온칩(on-chip) 고속 캐시. Thor 보드의 GPU L2 용량은 32 MB이며, DRAM 대비 약 50배 빠른 접근 속도를 제공한다. |
| **KV Cache** | Key-Value Cache. Transformer 모델이 어텐션(Attention) 연산을 수행할 때 이전 토큰들의 Key/Value 중간 결과를 저장해두는 버퍼. 매 decode step마다 이전 결과를 재사용하기 위해 GPU 메모리에 유지된다. |
| **sdpa** | Scaled Dot-Product Attention. PyTorch 내장 최적화 어텐션 구현체로, Flash Attention 백엔드를 자동 선택하여 연산 속도와 메모리 효율을 극대화한다. |
| **DynamicCache** | KV Cache 크기가 decode step에 따라 동적으로 늘어나는 구조 (HuggingFace Transformers 기본값). StaticCache 대비 prefill에서 2배 빠름. |
| **r = C_DMA / C_EXE** | Demand Layering 논문에서 정의한 비율. 레이어 하나의 가중치 전송 시간(C_DMA)을 해당 레이어의 GPU 연산 시간(C_EXE)으로 나눈 값. r < 1이면 compute-bound(전송을 연산 뒤에 숨길 수 있음), r > 1이면 memory-bound(숨길 수 없는 전송 시간이 남음). |
| **GEMM** | General Matrix Multiplication. Transformer의 Linear layer, Attention 연산 등 핵심 연산의 수학적 형태. |
| **Unified Memory** | CPU와 GPU가 동일한 물리 주소 공간을 공유하는 메모리 구조. iGPU의 핵심 특성이다. |

## 2. 참고 논문의 핵심 원리

### 2.1 Demand Layering (RTSS 2022)

**제목**: "Demand Layering for Real-Time DNN Inference with Minimized Memory Usage"  
**게재**: IEEE Real-Time Systems Symposium (RTSS), 2022  
**실험 환경**: **NVIDIA Jetson AGX Xavier** (iGPU, 통합 메모리 16 GB LPDDR4x @ 137 GB/s, 1 TB SSD)

#### 해결하는 문제

모델 크기가 GPU 메모리를 초과할 때, 전체 모델을 GPU에 올리지 않고도 실시간으로 추론하는 방법을 제안한다. 레이어를 하나씩 순차적으로 실행하면서, 다음 레이어의 가중치를 미리 GPU로 가져오는 3단 비동기 파이프라인을 구성한다.

#### 3단 파이프라인 구조

```
[Demand Layering 3단 파이프라인]
Read  Engine (SSD → CPU 페이저블 메모리):  [Layer N+1]  [Layer N+2] ...
Copy  Engine (CPU 메모리 → GPU 메모리):            [Layer N+1]  ...
Exec  Engine (SM, GPU 연산):                              [Layer N]  ...

각 엔진이 물리적으로 독립된 하드웨어 경로 → 진정한 병렬 실행
```

각 단계의 역할:
- **Read Engine**: SSD에서 다음 레이어 가중치를 CPU의 페이저블(pageable) 메모리로 읽어온다 (OS I/O, SSD 속도에 종속).
- **Copy Engine (CE)**: CPU 메모리에 있는 데이터를 GPU가 접근할 수 있는 메모리 페이지로 매핑한다 (`cudaMemcpyAsync` 사용, SM 불사용).
- **Exec Engine (SM)**: 현재 레이어의 연산(GEMM, Attention)을 수행한다.

#### 성능 결과

- **메모리 사용량**: 96.5% 절감 (전체 모델 크기 → 단일 레이어 크기로 축소)
- **지연 오버헤드**: 평균 14.8% 증가 (파이프라이닝으로 대부분의 전송 지연을 숨김)

#### Demand Layering이 iGPU에서 작동하는 이유

Xavier는 iGPU 구조이므로 CPU와 GPU가 동일한 LPDDR4x DRAM을 공유한다. CE의 "전송" 작업은 dGPU의 PCIe 전송(수십 GB 단위 실제 복사)과 달리, 주로 **GPU 페이지 테이블(Page Table) 업데이트** (GPU MMU가 해당 물리 메모리 주소를 인식하도록 등록)로 이루어진다. 이는 실제 데이터 복사 없이 주소 매핑만 변경하는 저렴한 연산이므로, DRAM 대역폭을 거의 소모하지 않고 CE+SM 진정한 병렬 실행이 가능하다. 이것이 14.8%라는 낮은 오버헤드의 근본 이유다.

### 2.2 RT-Swap (RTAS 2024)

**제목**: "RT-Swap: Addressing GPU Memory Bottlenecks for Real-Time Multi-DNN Inference"  
**게재**: IEEE Real-Time and Embedded Technology and Applications Symposium (RTAS), 2024

#### 핵심 아이디어

여러 DNN 태스크가 동시에 실행될 때 GPU 메모리 부족 문제를 해결하기 위해, CUDA 저수준 VMM(Virtual Memory Management, 가상 메모리 관리) API를 사용한다. 현재 사용하지 않는 레이어 가중치를 CPU 메모리로 비동기 "스왑"(swap, 교체)하고, 필요한 레이어를 CE를 통해 비동기로 다시 가져오면서 SM은 연산을 계속한다.

```
[RT-Swap 비동기 스왑 파이프라인]
CE (swap out / swap in):  |━L1 out━|━L2 in━|━L3 out━|...
SM (compute):                     |━L1 계산━|━L2 계산━|...
```

## 3. 실험 환경 및 현재 베이스라인

### 3.1 Thor 보드 베이스라인 (2026-05-28 확정)

본 실험에서 사용한 베이스라인은 아래의 최적 구성을 기준으로 한다.

| 항목 | 값 | 비고 |
|------|---|------|
| Attention 구현 | **sdpa** (Scaled Dot-Product Attention) | FlashAttention 백엔드 자동 선택, 현재 가장 빠른 구성 |
| KV Cache 구조 | **DynamicCache** | eager+StaticCache 대비 전체 파이프라인 1.4× 빠름 |
| 데이터 타입 | BF16 (Brain Float 16) | 모델 전체 22 GB |
| **전체 추론 시간** | **4,839 ms** | VE + Prefill + Decode + Flow |

```
파이프라인 단계별 실측 시간:
  ┌──────────────────────────────────────────────────────────┐
  │  VE (Vision Encoder)  │  728 ms  │ 이미지 특징 추출      │
  │  LM Prefill           │ 1,423 ms │ 프롬프트 전처리       │
  │  Decode (17 steps)    │ 1,818 ms │ 17 step × 107 ms/step │
  │  Flow (ODE Sampling)  │   870 ms │ 궤적 생성             │
  │  합계                 │ 4,839 ms │                       │
  └──────────────────────────────────────────────────────────┘
```

> **참고**: 이전 보고서에 사용된 "6.13초, 65 step" 수치는 구형 실험 환경(eager Attention + StaticCache 커스텀 경로)의 수치이다. 이 구성은 현재 최적 구성 대비 1.4배 느리며, 공식 비교 기준으로 사용하지 않는다.

### 3.2 Decode 단계의 물리적 하한

Decode는 매 step마다 모델 전체 가중치(22 GB)를 DRAM에서 읽어야 한다. 이는 모델 구조상 필연적이다.

```
물리적 하한 (BF16, 메모리 대역폭 한계):
  22 GB ÷ 231 GB/s = 95.2 ms / step

실측 (sdpa + DynamicCache):
  107 ms / step → 88% 대역폭 효율

결론: Decode 1 step은 현재 물리 하한의 88% 수준으로 이미 최적에 근접해 있다.
      추가 개선 여지는 12% 미만이다.
```

## 4. 적용 방식 및 실험 설계

### 4.1 아이디어의 출발점

Demand Layering과 RT-Swap의 공통 개념인 **"현재 레이어를 SM이 연산하는 동안, 다음 레이어의 데이터를 CE가 비동기로 준비"** 를 Thor 환경에 적용하고자 했다.

Thor는 128 GB 통합 메모리에 이미 모델이 완전히 적재되어 있다. 따라서 논문의 원형인 "SSD → DRAM 스트리밍"은 불필요하고, 대신 **"DRAM → GPU L2 캐시 사전 적재(warming)"** 로 목표를 재정의했다.

목표: 현재 레이어 SM 연산 중에, 다음 레이어 가중치를 GPU L2 캐시에 미리 올려두어 다음 레이어 연산 시 DRAM 접근 없이 L2에서 바로 읽도록 한다.

### 4.2 구현 방법: `tensor.sum()` 기반 파이프라인

```python
# Stream A: 현재 레이어 연산 (SM)
# Stream B: 다음 레이어 데이터 "사전 로드" 시도

stream_a = torch.cuda.Stream()  # SM 연산 스트림
stream_b = torch.cuda.Stream()  # "프리페치" 스트림

for i, layer in enumerate(model_layers):
    # 다음 레이어의 데이터를 백그라운드에서 "읽어온다"
    if i + 1 < len(model_layers):
        with torch.cuda.stream(stream_b):
            _ = model_layers[i + 1].weight.sum() 

    # 현재 레이어 연산
    with torch.cuda.stream(stream_a):
        output = layer(output)
```

## 6. 실패 원인 분석

### 원인 ① `tensor.sum()`은 CE(데이터 전송)가 아닌 SM(연산)이다

논문이 CE를 사용하는 이유는, CE가 SM과 **물리적으로 독립된 하드웨어**이기 때문이다. CE가 데이터를 옮기는 동안 SM은 방해 없이 연산할 수 있다.

`tensor.sum()`은 전혀 다른 경로다. 이것은 GPU의 **SM이 직접 수행하는 reduction(합산) 연산**이다.

```
[올바른 CE 방식]
CE (데이터 전송 전담):  DRAM → GPU 페이지 매핑  (SM 미사용)
SM (연산 전담):                                  현재 레이어 GEMM

→ 두 하드웨어가 독립 실행 → 진정한 병렬

[우리가 사용한 tensor.sum() 방식]
SM Stream B:  DRAM에서 가중치 386 MB 읽기 → reduction 수행 → 결과 버림 (1회 DRAM 읽기)
SM Stream A:  DRAM에서 동일 가중치 386 MB 다시 읽기 → GEMM 수행  (2회 DRAM 읽기)

→ 같은 SM이 두 작업 경합 + 동일 데이터를 DRAM에서 2회 읽기
```

| 방식 | DRAM 읽기 횟수 | SM 점유 | 결과 |
|------|---------------|---------|------|
| CE 기반 (논문) | 1회 | 프리페치 0% | 병렬 성공 |
| `tensor.sum()` | **2회** | **약 50% 분할** | 54% 저하 |

54% 속도 저하의 직접 원인은 **DRAM 2배 읽기 + SM 리소스 경합**이다. 파이프라이닝 개념 자체의 실패가 아니라, 잘못된 API 선택으로 인한 구현 오류다.

---

### 원인 ② L2 캐시 워밍(warming)이라는 목표 자체가 물리적으로 불가능했다

```
목표로 한 L2 캐시 워밍에 필요한 조건:
  다음 레이어 전체 가중치를 L2에 올려야 함

현실:
  레이어 가중치 1개 크기: ~386 MB
  GPU L2 캐시 전체 크기:   32 MB
  비율: 386 MB ÷ 32 MB = 12.1배 초과

설령 일부가 L2에 올라가더라도, 다음 레이어 연산 시작 즉시
L2 전체가 새 데이터로 덮어씌워져(eviction) DRAM 재접근이 발생한다.
```

L2 캐시 워밍으로 레이어 연산 속도를 개선하려면, 레이어 하나가 L2에 담겨야 한다. 현재 모델 구조에서 이것은 물리적으로 달성 불가능하다.

---

### 원인 ③ (가장 근본) Thor 환경에서 Demand Layering의 최적화 대상 자체가 존재하지 않는다

이것이 가장 중요한 발견이다.

**Demand Layering은 dGPU 논문이 아니다.** 논문은 **iGPU인 Jetson AGX Xavier (통합 메모리 16 GB)** 에서 실험됐으며, 다음과 같은 조건에서 성립한다.

| 조건 | Demand Layering의 Xavier | 우리의 Thor |
|------|--------------------------|------------|
| 통합 메모리 크기 | **16 GB** | **128 GB** (8배 큼) |
| 실험 모델 크기 | 16 GB 초과 → **SSD 스트리밍 필수** | 22 GB → **128 GB에 완전 적재** |
| CE가 숨기는 지연 | SSD I/O 시간 (~ms 단위) | **없음 (이미 DRAM에 있음)** |
| 파이프라인의 의미 | SSD 읽기를 SM 연산 뒤에 숨김 | **숨길 지연 자체가 없음** |

Demand Layering의 최적화 효과는 "느린 SSD I/O 지연을 빠른 SM 연산 뒤에 숨기는 것"에서 온다. Thor에서 모델이 128 GB DRAM에 이미 올라있다는 것은, **SSD I/O 지연이 존재하지 않는다**는 의미다. 최적화할 대상이 없는 상황에서 파이프라인 구조를 구현하는 것은 오히려 오버헤드만 더하는 결과를 낳는다.

```
[Demand Layering 파이프라인이 성립하는 조건]
                  SSD 읽기 (느림)  →  CE 전송 (빠름)  →  SM 연산
                  ↑
                  이 단계가 숨겨짐

[Thor 환경]
  모델이 이미 DRAM에 있음 → SSD 읽기 단계가 없음 → 숨길 것이 없음
```

Thor가 128 GB를 보유한다는 강점이 역설적으로 이 최적화 기법의 적용 대상이 되지 않게 만든다.

## 7. 올바른 CE 기반 구현이 가능한가 — 추가 검토

`tensor.sum()` 대신 `cudaMemPrefetchAsync`(CUDA 통합 메모리 프리페치 API)를 사용하면 CE 경로를 올바르게 사용할 수 있다.

```python
import ctypes
libcudart = ctypes.CDLL("libcudart.so.13")

def ce_prefetch(tensor, stream):
    """CE를 통한 비동기 프리페치 (SM 미사용)"""
    libcudart.cudaMemPrefetchAsync(
        ctypes.c_void_p(tensor.data_ptr()),
        ctypes.c_size_t(tensor.nbytes),
        ctypes.c_int(tensor.device.index),
        ctypes.c_void_p(stream.cuda_stream)
    )
```

그러나 Thor(iGPU) 환경에서 이 API는 다음과 같이 동작한다.

- **dGPU에서**: 물리적으로 데이터를 PCIe를 통해 GPU VRAM으로 복사 → 실질적 전송
- **iGPU(Thor)에서**: CPU와 GPU가 동일한 DRAM을 공유하므로 물리적 데이터 이동 없음. GPU MMU(Memory Management Unit, 메모리 관리 장치)의 페이지 테이블(page table) 업데이트와 TLB(Translation Lookaside Buffer, 주소 변환 캐시) 힌트 제공만 수행

iGPU에서 `cudaMemPrefetchAsync`의 실질 효과는 DRAM 데이터 이동이 아닌 **주소 매핑 정보 갱신**에 그친다. 이는 Demand Layering의 Xavier 환경에서 14.8%라는 낮은 오버헤드를 달성할 수 있었던 이유이기도 하다 (CE 작업 자체가 가볍기 때문). 그러나 Xavier에서는 SSD에서 읽어온 실제 데이터를 GPU에 매핑하는 의미 있는 작업이 존재했다. Thor에서는 이미 DRAM에 있는 데이터에 대해 주소 매핑을 갱신하는 것이므로 추가 이득이 없다.

## 8. 이번 실험으로 확인된 것과 확인되지 않은 것

### 확인된 것

**① `tensor.sum()`은 데이터 프리페치 수단이 아니다.**  
SM을 점유하는 reduction 연산이므로 DRAM을 2회 읽고 SM 리소스를 분할한다. 이 방식으로 파이프라이닝을 구현하는 것은 원천적으로 올바르지 않다.

**② Decode 단계는 이미 메모리 대역폭 한계에 근접해 있다.**  
sdpa + DynamicCache 기준 107 ms/step은 물리 하한 95 ms/step의 88% 수준이다. 파이프라이닝을 포함한 어떤 방식으로도 이 한계를 크게 뛰어넘기 어렵다.

**③ 레이어 가중치 프리페치의 구조적 불가능성.**  
386 MB 레이어 >> 32 MB L2. L2 워밍이라는 목표는 현재 모델에서 달성 불가능하다.

**④ Thor의 128 GB는 Demand Layering 적용 조건을 충족하지 않는다.**  
Demand Layering은 메모리가 부족한 iGPU 환경(Xavier, 16 GB)을 위한 기법이다. 충분한 메모리(128 GB)를 가진 Thor에는 이 기법이 목적하는 병목(SSD I/O 지연)이 존재하지 않는다.

### 확인되지 않은 것 (추가 실험 필요)

- CE 기반 `cudaMemPrefetchAsync`를 실제 사용 시 Thor iGPU에서 어느 정도의 페이지 테이블 업데이트 비용이 발생하는지
- 단, 위의 구조적 분석 상 이를 측정하더라도 Decode 최적화에 기여하기 어려울 것으로 판단된다

## 9. 결론 및 향후 방향

### 9.1 결론 요약

레이어 가중치 프리페치 기반 파이프라인은 Thor + Alpamayo 조합에서 다음 세 가지 이유가 중첩되어 효과를 낼 수 없다.

```
① 구현 오류:  tensor.sum() ≠ CE 기반 프리페치 → DRAM 2회 읽기 + SM 경합
② 용량 불일치: 레이어(386 MB) >> L2(32 MB) → L2 워밍 물리적 불가
③ 적용 부재:  Thor(128 GB) → 모델(22 GB) 완전 적재 → SSD I/O 병목 없음
              Demand Layering이 최적화하는 "느린 단계"가 존재하지 않음
```

이 세 가지가 독립적으로 각각 실험을 무효화한다. 즉, 원인 ①을 수정하더라도 원인 ②와 ③에 의해 효과가 없다.

### 9.2 올바른 최적화 방향: 전송량 절감

레이어 가중치를 더 빠르게 가져오는 것이 아닌, **애초에 읽어야 할 데이터의 양 자체를 줄이는** 방향으로 전환한다.

**KV Temporal Reuse (KV 캐시 시간적 재사용)**

자율주행 VLA 모델은 연속 프레임 간 75%의 비전 토큰(시각 입력 데이터)이 동일하다. 이미 계산된 KV 캐시를 다음 프레임에 재사용하면 VE + Prefill 연산의 75%를 생략할 수 있다.

실험 검증 결과 (2026-05-28, `260528_kv_temporal_reuse_poc.py`):

```
첫 번째 프레임 (전체 추론): 2,429 ms
두 번째 프레임 (KV 재사용):   175 ms  ← 92.8% 단축

연속 추론에서 두 번째 이후 프레임의 대부분을 이 방식으로 처리 가능
```

이 방향은 "전송 지연을 숨기는" 파이프라이닝과 달리, **전송해야 할 데이터의 절대량을 줄인다**. 메모리 대역폭이 병목인 iGPU 환경에서 근본적으로 효과적이다.

```
[기대 개선 효과]
현재 베이스라인:              4,839 ms
KV Temporal Reuse 적용 후:  ~2,500 ms 예상  (-48%)

근거: FlashDrive (Z-Lab, 2025)가 동일 모델(Alpamayo 1.5)에서
      4.5× 가속 달성을 보고함 (Jetson Thor 포함 다수 플랫폼)
```

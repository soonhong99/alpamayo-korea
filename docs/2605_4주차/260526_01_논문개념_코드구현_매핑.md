## 큰 그림: 우리가 해결하려는 문제

Alpamayo decode는 65 step을 반복한다. 각 step에서 28개 레이어를 순서대로 실행.  
지금 병목은 **레이어 가중치를 DRAM에서 읽어오는 속도**.

```
현재 상황:

레이어별 가중치 크기:
  MLP (gate/up/down): 96 MB × 3 = 288 MB per layer set
  Q/O Attention:       32 MB × 2 = 64 MB
  K/V Attention:        8 MB × 2 = 16 MB (GQA-4)

DRAM BW (실측): ~231 GB/s
GPU L2 BW (실측): ~1,126 GB/s → 4.76× 빠름

문제: 모델 전체(20.64 GB) >> GPU L2(32 MB)
     → 매 step 마다 20.64 GB를 DRAM에서 읽어야 함
     → 65 step × 20.64 GB = 1,342 GB → 이론 최소 5.8초
```

**목표**: "어차피 DRAM에서 읽어야 한다면, 미리 읽어서 compute와 겹치자"

## 개념 1: RT-Swap의 Async Pipeline

### 논문이 말하는 것

RT-Swap은 GPU 메모리(VRAM)이 작은 dGPU 환경에서 가중치를 DRAM ↔ GPU mem 사이에 "swap"한다.  
그 swap과 compute를 **동시에** 실행하는 게 핵심이다.

**논문의 타임라인 (dGPU 환경)**:
```
[기존 방식 - Sequential]:
┌─────────────┐ ┌───────────┐ ┌─────────────┐ ┌───────────┐
│ Swap In L1  │ │Compute L1 │ │ Swap In L2  │ │Compute L2 │  ...
└─────────────┘ └───────────┘ └─────────────┘ └───────────┘
총 시간 = Σ(swap_i + compute_i)

[RT-Swap 방식 - Async]:
DMA Stream:   [Swap In L1][─Swap In L2─][─Swap In L3─]...
              ┌─────────────┐
Compute Stream:             [─Compute L1─][─Compute L2─]...
              
총 시간 = Σ max(swap_i+1, compute_i)  ← 겹친 만큼 빨라짐
```

### iGPU(Thor)에서의 재해석

Thor는 **통합 메모리(Unified Memory)**: CPU와 GPU가 같은 128GB DRAM을 공유.  
dGPU처럼 "PCIe로 전송"하는 개념이 없다.

```
dGPU 구조:
CPU ──PCIe──→ GPU (VRAM 별도)
"swap" = PCIe DMA 전송 (실제 데이터 이동)

Thor(iGPU) 구조:
CPU ──DRAM──← GPU (같은 메모리)
"swap" = 개념상 없음, 이미 접근 가능
```

그러면 Thor에서 "prefetch"는 뭘 하는 건가?

```
Thor에서 cudaMemPrefetchAsync(ptr, size, GPU_DEVICE):
→ "이 메모리 페이지를 GPU L2 캐시로 미리 올려라"는 힌트
→ GPU가 다음에 이 데이터를 접근하면 DRAM(231 GB/s) 대신 L2(1126 GB/s) 히트
→ 4.76× 빠른 접근
```

**Thor 타임라인**:
```
[기존 방식 - 레이어 하나씩 순차]:
GPU: [─compute L1 (DRAM에서 가중치 읽으며)─][─compute L2 (DRAM에서)─]...
     └── 매번 DRAM BW 231 GB/s로 제한됨

[우리 방식 - Async Prefetch]:
prefetch_stream: [L1 prefetch→L2][L2 prefetch→L2][L3 prefetch→L2]...
compute_stream:  (sync)          [─compute L1─]  [─compute L2─]...
                                 └── L2에서 읽음 → 1126 GB/s 히트 기대
```

## 개념 2: Demand Layering의 Layer 관리

### 논문이 말하는 것

OS의 demand paging에서 영감:
- 전체 모델을 DRAM에 두고, 필요한 레이어만 GPU로 로드
- 현재 레이어 실행 중 → 다음 레이어를 백그라운드에서 미리 로드 (prefetch)

### 우리 구현에서 가져오는 것

1. **레이어 단위로 생각한다**: 모델을 `layers[0..27]` 배열로 봄
2. **loop 내에서 prefetch 선행**: `layers[i]` 실행 중 `layers[i+1]` prefetch 시작
3. **크기 인식**: 어떤 weight tensor를 prefetch할지 선택 (큰 것 우선 = MLP)

## 개념 3: 코드로 보는 1:1 매핑

### 핵심 코드 구조 (전체 흐름)

```python
# scripts/inference/260524_async_pipeline.py 의 핵심 루프

prefetch_stream = torch.cuda.Stream()   # ← Demand Layering의 "background load"
compute_stream  = torch.cuda.Stream()   # ← RT-Swap의 "compute stream"

for i in range(28):  # 28개 레이어
    
    # ─── [Demand Layering 개념] ───────────────────────────────────────
    # "현재 레이어 compute 중에 다음 레이어를 미리 L2로 올려라"
    if i + 1 < 28:
        with torch.cuda.stream(prefetch_stream):
            cudaMemPrefetchAsync(layers[i+1].mlp.gate_proj.weight, GPU)
    #   ↑ 이 줄이 Demand Layering의 "Layer Prefetch" 개념
    
    # ─── [RT-Swap 개념] ───────────────────────────────────────────────
    # "DMA(prefetch)와 Compute를 동시에 실행해서 겹친다"
    with torch.cuda.stream(compute_stream):
        hidden = layers[i](hidden, ...)
    #   ↑ 이 줄이 RT-Swap의 "Async DMA + Compute 중첩" 개념
    
    # 두 스트림 모두 완료 대기
    torch.cuda.current_stream().wait_stream(compute_stream)
    torch.cuda.current_stream().wait_stream(prefetch_stream)
```

### 각 줄이 어느 논문의 어느 개념인가

| 코드 | 출처 논문 | 논문 내 개념 |
|------|-----------|-------------|
| `prefetch_stream = torch.cuda.Stream()` | RT-Swap §4.1 | "Separate DMA stream from compute stream" |
| `compute_stream = torch.cuda.Stream()` | RT-Swap §4.1 | "Compute stream runs concurrently with DMA" |
| `with torch.cuda.stream(prefetch_stream)` | RT-Swap §4.1 | "Issue DMA asynchronously on separate stream" |
| `cudaMemPrefetchAsync(layers[i+1], GPU)` | Demand Layering §2.2 | "Prefetch next layer while computing current" |
| `with torch.cuda.stream(compute_stream)` | RT-Swap §2.1 | "Compute layer i while prefetching i+1" |
| `layers[i](hidden, ...)` | Demand Layering §2.1 | "Execute layer on demand" |
| `wait_stream(compute_stream)` | RT-Swap §4.1 | "Synchronize before proceeding" |
| `wait_stream(prefetch_stream)` | RT-Swap §4.1 | "Ensure prefetch complete before next compute" |

## 개념 4: 파이프라인이 작동하는 조건

RT-Swap 논문 수식:
$$T_{async} = t_{prefetch,0} + \sum_{i=0}^{N-2} \max(t_{prefetch,i+1},\ t_{compute,i}) + t_{compute,N-1}$$

이 수식이 말하는 것:
- 파이프라인 각 단계의 시간 = `max(다음 레이어 prefetch 시간, 현재 레이어 compute 시간)`
- **compute > prefetch** → compute가 bottleneck, prefetch는 완전 중첩 → 이득 최대
- **prefetch > compute** → prefetch가 bottleneck, compute는 기다림 → 이득 감소

**Thor에서 계산**:

prefetch 시간 (L2로 올리는 시간):
```
MLP gate_proj weight = 96 MB
GPU L2 BW = 1,126 GB/s (이론, 실제는 lower)
t_prefetch = 96 MB / 1,126 GB/s = 0.085 ms

Q_proj weight = 32 MB  
t_prefetch = 32 MB / 1,126 GB/s = 0.028 ms
```

compute 시간: **아직 모름** → `260524_layer_compute_profile.py`로 측정 필요

결론:
- `t_compute >> 0.085 ms` 이면 → prefetch 완전 중첩, 큰 이득
- `t_compute ≈ 0.085 ms` 이면 → 일부 중첩
- `t_compute << 0.085 ms` 이면 → compute 너무 빠름, prefetch가 bottleneck

## 개념 5: L2 Persistent Residency (추가 최적화)

논문에 없는 Thor 특화 기법:

```
KV Cache 크기 (65 step, GQA-4):
  각 레이어당: 2(K+V) × 65 steps × 4096 dim × 2 bytes = ~1 MB
  28 레이어 전체: ~28 MB
  → GPU L2(32 MB)에 거의 다 들어감!
```

구현: `cudaDeviceSetLimit(cudaLimitPersistingL2CacheSize, ...)` + `cudaStreamSetAttribute`

```python
# KV Cache를 L2에 고정
CUDA_ACCESS_PROPERTY_PERSISTING = 1
CUDA_ACCESS_PROPERTY_STREAMING  = 2

# KV Cache → Persisting (L2에 계속 유지)
set_l2_policy(kv_cache, hitProp=PERSISTING, missProp=PERSISTING)

# MLP weights → Streaming (L2 무시하고 바로 DRAM에서, L2는 KV용으로)
set_l2_policy(mlp_weights, hitProp=STREAMING, missProp=STREAMING)
```

이렇게 하면:
- Attention 계산: KV Cache가 L2에 있으므로 4.76× 빠름
- MLP 계산: DRAM에서 직접 읽되, L2를 오염시키지 않음

## 요약: 논문 → 코드 한 줄 요약

```
RT-Swap의 핵심:
"DMA stream과 Compute stream을 분리해서 동시에 실행한다"
→ 코드: prefetch_stream + compute_stream, wait_stream으로 동기화

Demand Layering의 핵심:
"현재 레이어 실행 중에 다음 레이어를 미리 메모리에 올린다"
→ 코드: for loop 안에서 i 실행 중 i+1을 cudaMemPrefetchAsync

Thor iGPU에서의 특이점:
"swap = 실제 복사가 아니라 L2 캐시 warming"
→ 코드: cudaMemPrefetchAsync(ptr, size, CUDA_DEVICE) → L2 prefetch 힌트
```

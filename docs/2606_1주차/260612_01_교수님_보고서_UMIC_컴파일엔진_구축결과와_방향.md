# UMIC 컴파일 엔진 구축 결과 보고 — Alpamayo 추론 −29.8%, 출력 등가, 모델 무수정

**날짜**: 2026-06-12
**대상 시스템**: Jetson AGX Thor (SM 11.0, LPDDR5X 231 GB/s, GPU L2 32 MB) / Alpamayo 1.5 (BF16, 무수정)
**코드**: `github.com/soonhong99/umic` (전용 repo) — 측정 데이터·실험 스크립트 전체 포함

---

## 1. 결론 (요약)

**PyTorch eager로 실행되던 Alpamayo 1.5 추론을, 모델을 한 줄도 수정하지 않고(체크포인트·소스·정밀도 그대로) 자체 제작한 커널·캐시·실행기로 재구성하여 동일 측정 조건에서 29.8% 단축했습니다. 출력은 토큰 단위까지 동일합니다.**

| 단계 | eager | UMIC | 개선 |
|------|-------|------|------|
| Vision Encoder | 532 ms | **305 ms** | **−42.7%** |
| LM Prefill | 1,090 ms | **588 ms** | **−46.1%** |
| LM Decode | 78.2 ms/step | **70.0 ms/step** | −10.5% |
| Flow Matching | 721 ms | **449 ms** | **−37.7%** |
| **전체 (19 decode steps)** | **3,846 ms** | **2,701 ms** | **−29.8%** |

- **출력 등가성 검증 완료**: 같은 입력·같은 seed에서 생성 토큰 3,106개 전부 일치, 최종 64-waypoint 궤적 차이 ADE 3.8 mm (6.4초 주행 끝점 기준 1.5 cm — 차선 폭의 0.4%)
- 양자화·증류·구조 변경 일절 없음 — **같은 수학을 다른 실행 계획으로** 수행한 것이 전부입니다
- UMIC(Unified-Memory Inference Compiler)은 Alpamayo 전용이 아니라, **iGPU(통합 메모리) 위에서 여러 transformer 모듈로 구성된 모델 일반을 최적화하기 위한 엔진**입니다. 모델 인식은 클래스명이 아닌 구조 패턴(duck-typing) 기반이라 Alpamayo 2.0 공개 시에도 재적용 가능하도록 설계했습니다.

## 2. 출발점 — 왜 컴파일 엔진인가 (지난 보고와의 연결)

ncu 하드웨어 카운터 실측(260608 보고)에서 eager 실행이 가중치 크기 대비 최대 15~85배의 DRAM 트래픽을 만들고 있음을 확인했습니다. 원인은 두 갈래였고, 처방이 서로 다릅니다.

```
낭비 1 — 연산 분해: 커널마다 중간 결과를 DRAM에 썼다 다시 읽음
          (예: RMSNorm 1회 = pow→mean→mul→cast→copy 5개 커널, 이상치의 8배 트래픽)
          → 처방: 커널 융합 (한 커널 안에서 레지스터로 전달)

낭비 2 — 라이브러리 부적합: cuBLAS의 기성 GEMM 커널이 Thor의 좁은 공유 버스·작은 L2에
          맞지 않게 가중치를 수 배 재독 (gate/up GEMM이 이론 byte의 6.6배)
          → 처방: 해당 지점만 자체 Triton 커널로 교체 (site별 측정 기반 선택)
```

TRT-LLM이 이 문제를 LM 영역에서 풀지만, Alpamayo의 4단계 파이프라인(VE→Prefill→Decode→Flow ODE) 전체를 다루지 못하고 iGPU 통합 메모리 구조도 활용하지 않습니다. 그 공백이 UMIC의 영역입니다.

핵심 설계 원칙은 **measurement-guided compilation**: 모든 최적화 결정을 FLOPs 추정이 아니라 ncu로 측정한 DRAM byte로 내리고, 적용 후 다시 측정해 닫힌 루프로 검증합니다. (이 원칙이 실제로 우리를 여러 번 구했습니다 — §5)

## 3. 단계별 최적화 — 무엇을, 왜, 결과

### 3.1 LM Prefill: 1,090 → 588 ms (DRAM 232 → 78 GB, −66%)

가장 낭비가 컸던 단계(이론 15 GB vs 실측 232 GB). 적용한 것 4가지:

| 기법 | 내용 | 근거(왜) | DRAM 효과 |
|------|------|---------|----------|
| FFN 융합 (P5) | gate·SiLU·곱을 1커널로, 중간텐서 3개(개당 76 MB×36층) DRAM 왕복 제거 | 커널 경계 낭비 중 최대 단일 항목 | **−84 GB** (이 중 −70 GB는 cuBLAS gate/up GEMM의 재독까지 함께 사라진 것) |
| q/o projection 교체 | cuBLAS → 자체 Triton 커널 | 해당 GEMM이 이론 byte의 3.3배 소모 (k/v는 이미 이론값이라 **교체 안 함** — site별 선택) | −13 GB, launch당 96 MB = 이론의 1.14배 달성 |
| RMSNorm 융합 | fp32 5커널 체인 → 1커널 (fp32 수학은 레지스터에서 유지) | norm 관련 트래픽 ~50 GB가 elementwise 분해에서 발생 | **−49 GB**, norm이 정확히 이론값(1.00×) 도달 |
| RoPE 융합 | rotate_half의 cat 물질화 제거, 8커널 → 2커널 | 잔여 elementwise 분해 | −7 GB |

### 3.2 Vision Encoder: 532 → 305 ms (DRAM 98 → 51 GB, −48%)

LM과 같은 병리의 ViT 버전이었습니다. LayerNorm 융합(58개소) + **vision RoPE 융합**이 핵심. 특히 HF의 vision RoPE 구현은 `q.float()`로 fp32 승격 후 cat·곱·덧셈·역캐스트를 반복해 **함수 하나가 VE 트래픽의 28 GB, 시간의 240 ms**를 차지하고 있었고, 융합 커널로 호출당 8.9 → 0.63 ms (14×)가 됐습니다.

### 3.3 LM Decode: 78.2 → 70.0 ms/step

Decode는 성격이 다릅니다 — 실측 트래픽이 이론(매 step 가중치 15.2 GB 재독)의 1.06배로 **낭비가 거의 없는 단계**라, byte를 줄이는 융합으로는 못 깎습니다. 대신:
- **In-place KV cache**: 매 step 전체 KV를 복사하던 DynamicCache의 cat을 사전 할당 버퍼 + 포인터 증가로 대체
- **CUDA Graph**: step당 ~1,200개 커널 제출(launch)을 GPU가 기다리는 유휴를, KV 길이별 그래프를 1회 캡처해 두고 재생(replay)하는 방식으로 제거. 10 Hz 연속 추론에서 첫 프레임만 캡처 비용을 내고 이후는 전부 재생 — 재생 출력은 비트 단위로 결정론적임을 확인
- 남은 70 ms는 물리 하한(가중치 15.2 GB ÷ 실효 대역폭) 부근으로, **양자화 없이는 더 깎을 수 없는 영역**입니다

### 3.4 Flow Matching: 721 → 449 ms (DRAM 122 → 80 GB)

소스 분석으로 구조적 낭비를 발견했습니다: ODE 매 step마다 expert가 VLM KV cache(3,100 토큰 × 36층)에 토큰 32개를 **붙였다 떼는데**, 그때마다 전체 prefix가 통째로 재복사되고 있었습니다 (10 step × guidance 2회 = 20번). In-place cache로 떼는 작업을 포인터 이동(O(1))으로 바꿔 제거했고, Action Expert도 Qwen 구조라 LM용 norm/RoPE 융합이 **수정 없이 그대로 적용**됐습니다 — 일반화 설계가 작동한 첫 사례입니다.

## 4. 일반화 — 이 엔진이 Alpamayo 전용이 아닌 이유

UMIC의 모델 인식은 전부 **구조 패턴 매칭**입니다: "gate/up/down Linear + SiLU를 가진 모듈"이면 클래스명이 무엇이든 FFN 융합 대상이고, "RMSNorm류 + 1차원 weight"면 norm 융합 대상입니다. 실제로 Action Expert(DiT)와 LM이 서로 다른 모듈이지만 같은 패스가 둘 다 잡았습니다. 매칭되지 않는 부분은 eager로 폴백하므로 **커버리지 부족이 정확성을 해치지 않습니다** — TRT-LLM이 Alpamayo에서 실패한 "전부 변환 아니면 전무" 문제를 구조적으로 회피한 것입니다. Alpamayo 2.0이 공개되면 같은 패스를 그대로 적용해 보는 것이 일반성의 다음 실증이 됩니다.

## 5. 과정에서 확립된 측정 방법론 (그 자체로 보고 가치)

1. **DVFS 함정**: decode 같은 memory-bound 단계는 SM 사용률이 낮아 **거버너가 GPU/메모리 클럭을 올리지 않습니다**. 같은 코드가 거버너에서 107 ms, 클럭 고정에서 70~79 ms. → 모든 측정과 10 Hz 실배포에 `jetson_clocks` 고정이 전제 조건.
2. **장기 웜업**: 클럭 고정 후에도 allocator/페이지 워밍으로 처음 ~4회 추론 동안 모든 단계가 계단식으로 빨라집니다. → steady-state는 5회 이상 실행 후 판정.
3. **기준선 정정**: 위 두 변수를 통제하자 eager 기준선 자체가 4,838 → 3,846 ms로 내려왔습니다. 따라서 **이전에 구두 보고했던 "−48.7%"는 환경 패널티가 섞인 과대 수치로 철회**하고, 동일 조건 −29.8%를 공식 수치로 합니다. (단계별 상대 개선율은 항상 동일 조건 A/B로 측정해 유효함을 확인했습니다.)
4. **격리 벤치마크 불신 원칙**: 단독 측정에서 이긴 커널이 실제 파이프라인에서 뒤집힌 사례를 5회 기록했습니다. 모든 채택 판정은 e2e 측정으로 합니다.

## 6. 시도했으나 채택하지 않은 것 (기각도 측정으로)

- **Cross-stage prefetch (당초 핵심 가설)**: 통합 메모리에서는 가중치가 이미 DRAM에 상주해 **숨길 전송 자체가 없고**, 단계 전환 공백도 실측 22 ms(0.6%)뿐이라 폐기. llm.npu식 bubble 제거는 Thor에서는 "decode 커널 제출 유휴"로 나타나며, 그것이 위의 CUDA Graph입니다.
- **down_proj GEMM 교체**: 자체 커널이 격리 측정에서는 cuBLAS를 이겼으나(4.94 vs 5.45 ms) 실모델에서는 +30 ms 손해라 보류. 이 과정에서 ① 모델의 실제 FFN 폭이 문서 가정(11008)과 다른 **12288**임을 발견(설정 추정의 위험 → shape는 실행 시 자동 채취해야 한다는 교훈), ② 커널의 타일 실행 순서(GROUP)에 따라 DRAM 트래픽이 525 MB~2.4 GB까지 4.5배 변하는 **L2 윈도 법칙**을 정량화했습니다 — 32 MB L2를 가진 iGPU에서 커널 튜닝이 discrete GPU와 달라야 하는 직접 증거입니다.

## 7. 남은 과제와 방향

**(a) 엔진의 "컴파일러" 완성 — 자동화** (최우선, 연구 기여의 핵심)
지금까지는 ncu 분해 → 사람이 병목 특정 → 패턴 주입이었습니다. 이 결정 과정을 자동화해야 진짜 컴파일 엔진입니다: ① 그래프 캡처에서 패턴 자동 탐지·치환(현재 수동 매칭과 동일 결과를 기계가 재현하는 것이 검증 기준), ② shape 자동 채취 + 커널 config 오토튜닝(12288 사건의 교훈), ③ byte 비용 모델의 예측-실측 보정. 완성 시 "Alpamayo 2.0이 나와도 재컴파일하면 된다"는 주장이 성립합니다.

**(b) 잔여 성능** — 식별·정량화된 상태로 대기 중:
- 격리-실모델 성능 괴리 메커니즘 규명 (풀리면 down/ViT GEMM 교체 부활, prefill 추가 −50 ms급)
- Flow step CUDA Graph (−30 ms급), Flow 잔여 14 GB 복사의 정체 규명
- decode는 BF16 하한 도달 — 추가 단축은 (범위 밖인 양자화 제외 시) speculative decoding 등 알고리즘 레벨로 넘어감

**(c) 10 Hz 연속 추론 파이프라인** — 단일 추론 최적화 다음 단계로, 프레임 간 중첩(현재 프레임 Flow 실행 중 다음 프레임 VE 전처리)이 유효한 영역으로 남아 있습니다. CPU가 GPU compute 동안 노는 시간이 여기서의 bubble입니다.

**(d) 일반성 실증** — 표준 Qwen-VL 계열 모델 1종에 무수정 적용해 같은 폭의 개선이 나오는지 확인. 논문 프레임: *"Measurement-Guided Compilation for Multi-Stage Transformer Inference on Unified-Memory Edge GPUs"*.

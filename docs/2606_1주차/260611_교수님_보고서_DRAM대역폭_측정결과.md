## 0. 핵심 요약

> **용어 안내** — 아래 표의 "커널 간 유휴"는 DRAM을 기다리는 시간이 아니다. 커널 A가 완전히 끝나고 커널 B가 시작되기 전까지, CPU가 다음 작업을 GPU에 제출하는 동안 GPU가 **완전히 비어 있는** 구간이다. DRAM을 기다리는 stall은 커널이 실행 중인 구간 안에 포함되어 별도로 집계된다.

| 단계 | DRAM BW | 커널 간 유휴 |
|------|---------|------------|
| VE | **35%** (80 GB/s) | ~17% (~124 ms) |
| LM Prefill | **55%** (126 GB/s) | 15.8% (~130 ms) |
| LM Decode | **89%** (205 GB/s) | **10.6% (~159 ms)** |
| Flow Matching | **88%** (203 GB/s) | 9.2% (81 ms) |

## 1. 측정 방법론

### 1.1 측정 목적

GPU가 언제, 어디서 노는지 파악하는 것이 목적이다. 모델 가중치는 **DRAM**(메인 메모리, Thor에서는 LPDDR5X 규격, 이론 최대 전송 속도 231 GB/s)에 저장되어 있고, GPU가 연산을 할 때마다 이 가중치를 DRAM에서 읽어와야 한다. 이 전송 속도가 얼마나 차 있는지를 측정하면 GPU가 연산 중인지, 아니면 데이터를 기다리며 노는지를 파악할 수 있다.

이번 측정은 **PyTorch eager 모드** 기준이다. Eager 모드란 PyTorch의 기본 실행 방식으로, 코드에 적힌 연산들이 하나씩 즉시 개별 GPU 커널로 실행된다.

여기서 **GPU 커널**이 무엇인지 명확히 짚고 넘어갈 필요가 있다. 딥러닝 모델의 구조와 GPU 실행 단위는 아래와 같이 계층을 이룬다:

```
Block (Transformer 블록)
  — Attention + FFN + LayerNorm이 하나로 묶인 반복 단위
  — Alpamayo 1.5 LM 기준 36개의 Block이 순차 실행됨

  └── Layer (레이어)
        — Block 내부의 개별 연산 모듈
        — 예: nn.Linear(선형 변환), LayerNorm(정규화), Attention, FFN 등
        — 하나의 Block은 여러 Layer로 구성됨

        └── GPU 커널
              — Layer보다 더 세밀한 실행 단위. GPU에 직접 올라가는 함수 하나.
              — 예: nn.Linear 한 개 → GEMM 커널 1개
              — 예: FFN 내부의 gate_proj + silu + up_proj_mul → eager에서 커널 3개로 분리
              — ncu가 측정하는 단위가 바로 이 GPU 커널이다
```

Eager 모드에서는 하나의 Layer 안에서도 연산들이 별도 커널로 쪼개져 실행된다. 예를 들어 `gate * silu(gate)` 같은 두 연산도 각각 별도 커널로 분리 실행되어, 첫 번째 연산의 결과가 DRAM에 쓰인 뒤 두 번째 연산이 그것을 다시 DRAM에서 읽어오는 식이다. 연산들을 하나의 커널로 합칠 수 있다면(**커널 융합**) 이 중간 DRAM 왕복이 사라지지만, eager 모드에서는 이런 최적화가 자동으로 적용되지 않는다.

구체적으로 두 가지 질문에 답한다:
- 각 추론 단계(VE, Prefill, Decode, Flow)가 DRAM을 얼마나 쓰는지
- 단계 내부에서 어떤 커널이 병목인지, 그 사이에 유휴가 있는지

### 1.2 측정 도구

**ncu** (NVIDIA Nsight Compute): GPU 커널 하나하나에 대해 DRAM 전송량, 실행 시간 등 하드웨어 카운터를 직접 읽는 도구. 소프트웨어 추정치가 아닌 칩 내부 하드웨어 레벨의 실측값이다.

**nsys** (NVIDIA Nsight Systems): 시스템 전체 타임라인을 기록하는 도구. 코드 안에 **NVTX** 마커(예: `range_push("LM_Prefill")`)를 삽입하면 nsys가 해당 구간의 시작/끝 시간을 자동으로 기록한다. 이 덕분에 전체 추론 중 어느 부분이 VE인지, Prefill인지 정확히 구분해서 측정할 수 있다.

### 1.3 측정 방법 및 정확성 근거

**두 계층으로 측정**:

```
[계층 1] Stage-level  →  nsys NVTX 타임스탬프 + ncu 총량 교차 검증
                          각 단계의 총 DRAM 전송량과 소요 시간 파악

[계층 2] Kernel-level →  ncu --replay-mode kernel
                          단계 내부 커널 하나하나의 순간 DRAM BW 측정
                          완료: VE, LM Prefill, LM Decode, Flow Matching (2026-06-11)
```

**핵심 측정 지표**: `lts__d_sectors_fill_sysmem.sum`

GPU 칩 내부에는 특정 이벤트가 발생할 때마다 숫자를 올리는 전용 계수 회로가 내장되어 있다. 이를 **하드웨어 카운터**라고 한다. 소프트웨어가 추정하거나 샘플링하는 것이 아니라 회로 자체가 실제 발생 횟수를 빠짐없이 기록하므로 정확하다. `lts__d_sectors_fill_sysmem.sum`은 이 중 "L2 캐시에서 DRAM으로 데이터를 가져온(fill) 횟수"를 세는 카운터다.

여기서 **sector**는 GPU가 DRAM에 접근하는 최소 단위다. CPU의 캐시라인과 같은 개념으로, 크기는 **128바이트**다. GPU가 가중치 중 단 4바이트만 필요하더라도 DRAM에서는 128바이트짜리 sector 하나가 통째로 L2 캐시로 넘어온다. 실제 전송량은 `sector 수 × 128 bytes`로 계산된다.

**실행 시간 측정**: DRAM 전송량만으로는 대역폭을 계산할 수 없고 커널 실행 시간도 필요하다. `gpu__time_duration.sum` 카운터가 커널 시작부터 끝까지 경과한 나노초를 하드웨어 수준에서 직접 기록한다. `DRAM 전송량(GB) ÷ 실행 시간(s) = 순간 DRAM BW(GB/s)`가 된다.

**DRAM Read / Write 구분**: DRAM 접근에는 방향이 있다. GPU가 가중치를 읽어오는 것(Read)과, 연산 결과물·KV캐시를 다시 써넣는 것(Write)은 별도 카운터로 구분 측정된다 — Read는 `lts__d_sectors_fill_sysmem.sum`, Write는 `lts__t_sectors_aperture_sysmem_op_write.sum`. 두 값을 합산하면 총 DRAM 트래픽이 된다.

단, ncu는 정확한 측정을 위해 각 커널을 **2번 실행**(2-pass replay)한다. GPU 하드웨어 카운터 슬롯 수가 제한되어 있어 — 수백 가지 메트릭을 한 번에 다 셀 수 없으므로 — 측정 항목을 두 묶음으로 나눠 커널을 두 번 돌리며 각각 다른 묶음을 수집하는 방식이다.

```
커널 A 1번째 실행 → DRAM sector 카운터 수집  ┐
커널 A 2번째 실행 → 실행 시간 카운터 수집    ┘ → CSV에 2줄 → ÷2 필요
```

"두 번 실행하면 결과가 달라지지 않나?"라는 의문이 생길 수 있다. GEMM, GEMV와 같은 딥러닝 추론 커널은 **결정론적**이다 — 동일한 입력에 대해 매번 동일한 연산 순서로 실행되므로 두 pass가 측정하는 실제 하드웨어 동작은 같다. 그 결과 CSV에는 커널당 2행이 기록되어 값이 2배가 되므로, 전체 수치에 **÷2 보정**을 적용했다.

### 1.4 GPU 커널 실행과 DRAM 요청 방식

커널이 실행될 때 GPU 내부에서 어떻게 DRAM에 접근하는지 이해하면 대역폭 수치가 왜 그렇게 나오는지 직관적으로 납득된다.

**스레드와 warp**:  
GPU 커널이 launch되면 수천 개의 스레드가 동시에 실행된다. 스레드는 32개씩 묶여 **warp**를 구성하며, warp가 SM(Streaming Multiprocessor, GPU의 연산 코어)에서의 스케줄링 최소 단위다. GPU의 실행 모델은 **SIMT**(Single Instruction, Multiple Threads)다 — warp가 "load" 명령 하나를 실행하면 warp 내 32개 스레드가 **각자 다른 주소에 대해 그 명령을 동시에** 실행한다. 즉 명령은 1개지만 메모리 요청은 32개가 동시에 발생한다. 하나의 커널 실행 중에 수백 개의 warp가 SM 안에서 동시에 in-flight 상태로 존재한다.

**DRAM 요청이 생성되고 숨겨지는 방식**:  
warp가 DRAM 요청을 보낸 뒤 응답을 기다리는 지연은 수백 나노초다. 이 warp가 기다리는 동안 SM은 다른 warp로 전환해 계속 다른 연산을 실행한다. 대기 중인 warp들의 요청들이 DRAM에서 처리되는 사이, 준비된 warp들이 SM을 계속 채운다. **수백 개의 warp가 번갈아 가며 DRAM 요청을 겹쳐 보내는 것** — 이 집합적인 요청 처리 흐름의 속도가 DRAM 대역폭 측정치로 잡힌다.

**GPU의 메모리 계층**:  
GPU 칩 위에는 DRAM 외에 on-chip 고속 메모리가 별도로 존재한다.

```
DRAM (LPDDR5X, 231 GB/s, ~100ns)     ← Thor에서 CPU·GPU가 공유하는 메인 메모리
  └── L2 cache (32 MB, 칩 내장)       ← lts__ 카운터가 여기를 기준으로 집계
        └── Shared memory / L1        ← SM당 수십~수백 KB, 칩 내장, ~수ns
              └── Registers           ← 스레드 전용, 가장 빠름
```

Thor는 CPU와 GPU가 같은 DRAM을 쓰는 iGPU 구조지만, GPU 칩 위의 on-chip 메모리(shared memory, L2)는 discrete GPU와 동일하게 존재한다. §3에서 FlashAttention이 "SRAM에서 처리한다"고 할 때의 SRAM이 바로 이 shared memory를 가리킨다. 이 계층이 있기 때문에 커널마다 DRAM 사용률이 크게 달라진다 — 이 세부 패턴은 §3 각 단계 분석에서 확인한다.

**SM 활용률 측정 시도와 의문**:

DRAM BW와 함께 **SM**(Streaming Multiprocessor, GPU의 연산 코어) 활용률도 측정을 시도했다. 그런데 측정하면서 의문이 생겼다 — *커널 단위로 SM 활용률을 보는 것이 과연 우리가 원하는 정보를 주는가?* SM 활용률 카운터는 SM이 DRAM 응답을 기다리며 대기하는 중에도 "활성"으로 기록되기 때문에, SM이 바쁘다고 해서 실제로 유용한 연산을 하고 있다고 볼 수 없다. 결국 **"DRAM BW%가 얼마나 차 있는가"가 GPU 유휴를 판단하는 더 직접적인 지표**라는 판단으로 DRAM BW를 핵심 측정 지표로 확정했다.

## 2. 4단계 DRAM 대역폭 실측 결과

아래 표에서 **Peak%** 는 실제 측정된 DRAM 전송 속도를 이론 최대치(231 GB/s)로 나눈 비율이다. 89%라면 231 × 0.89 ≈ 205 GB/s로 DRAM이 거의 꽉 차 있다는 뜻이고, 35%라면 여유가 65% 남아 있다는 뜻이다. **L2 hit**은 메모리 요청 중 GPU 칩 내장 L2 캐시(32 MB, DRAM보다 수십 배 빠름)에서 해결된 비율로, 높을수록 DRAM 실제 접근이 줄어든다.

| 단계 | 시간 (ms) | DRAM Read | DRAM Write | **총 DRAM** | Read BW | **Peak%** | L2 hit |
|------|----------|-----------|------------|------------|---------|----------|--------|
| **VE** | 728 | 58.4 GB | 39.7 GB | 98.1 GB | 80 GB/s | **35%** | 49.2% |
| **LM Prefill** | 1,423 | 179.9 GB | 52.1 GB | 232.0 GB | 126 GB/s | **55%** | 29.6% |
| **LM Decode** | 1,503¹ | 307.5 GB | 15.9 GB | 323.4 GB | 205 GB/s | **89%** | 37.7% |
| **Flow Matching** | 870 | 176.7 GB | 67.6 GB | 244.3 GB | 203 GB/s | **88%** | 22.0% |
| **합계** | **4,524** | 722.5 GB | 175.2 GB | **897.7 GB** | — | — | — |

> ¹ 19 steps × 79.1 ms/step (AppendOnlyCache-C: KV 캐시 최적화 구현체, 2026-05-31 실측 확정)  
> Peak% = Read BW / 231 GB/s × 100

**핵심 관찰 — 모델 크기 대비 DRAM 전송량**:

Alpamayo 1.5 전체 가중치는 **22 GB**다. 그런데 추론 한 번에 DRAM을 통과한 총량은 **897 GB** — 약 **40배**다. 가중치가 한 번씩만 읽혀도 충분할 것 같지만 실제로는 그렇지 않다.

이유는 두 가지다. 첫째, Decode는 19 step을 반복하므로 가중치를 19번 읽는다 (22 GB × 19 ≈ 418 GB). 둘째, eager 모드에서 각 레이어 안에서 elementwise 연산, norm, residual 합산 등 수십 개의 소규모 커널이 각각 중간 결과를 DRAM에 쓰고 다음 커널이 그것을 다시 읽는 과정이 반복된다. 이 불필요한 DRAM 왕복이 누적되어 가중치 자체보다 훨씬 큰 DRAM 트래픽을 만든다. **커널 융합으로 중간 텐서를 DRAM에 쓰지 않고 register에서 직접 넘기면 이 트래픽을 크게 줄일 수 있다** (실측: UMIC P5 gate_silu_mul 융합 적용 시 Prefill DRAM 232 → 148 GB, -36.2%).

단계별로 두 가지 체제가 명확히 갈린다:

- **BW 여유 구간** — VE (35%), LM Prefill (55%): DRAM이 놀고 있는 시간이 존재한다. 세부적으로 어떤 커널에서 얼마나 여유가 있는지는 §3에서 확인한다.
- **BW 포화 구간** — LM Decode (89%), Flow Matching (88%): DRAM이 이미 거의 꽉 차 있어 여유 BW 기반 접근은 어렵다. 그러나 §3에서 확인하듯 이 단계들에도 GPU가 완전히 비어 있는 구간이 별도로 존재한다.

## 3. Kernel-level 상세 분석

> 4개 단계 전부 kernel-level 측정 완료 (2026-06-11). ncu로 각 커널의 DRAM BW를 측정하고, nsys로 커널과 커널 사이에 GPU가 완전히 비어 있는 구간(커널 간 유휴)도 함께 측정했다.

커널 간 유휴가 발생하는 구조적 이유: PyTorch에서 `model(input)`을 호출하면 Python → C++ → CUDA 드라이버를 거쳐 커널 하나하나가 GPU에 순차적으로 제출된다. 이전 커널이 끝난 뒤 다음 커널이 제출될 때까지의 짧은 시간 동안 GPU는 할 일이 없어 완전히 비어 있다. 이 간격이 수백 개의 커널에 걸쳐 누적되면 무시하기 어려운 크기가 된다.

### 3.1 LM Prefill

Prefill은 입력 이미지+텍스트 전체(seq_len=3,086 토큰)를 한 번에 처리하는 단계다. 이때 가중치 행렬과 입력 행렬을 곱하는 **GEMM**(General Matrix Multiplication, 행렬×행렬 연산)이 주로 실행된다.

| 커널 그룹 | 호출 수 | DRAM BW% | 병목 분류 |
|----------|---------|----------|---------|
| 가중치 GEMM (nvjet_256x128) | 108 | ~70% | DRAM-bound |
| **FlashAttention** | 18 | **~39.5%** | **SRAM/compute-bound** |
| norm 등 기타 | 수백 | <10% | compute-bound |

**커널 종류가 왜 3~4가지뿐인가**:  
레이어는 Q-proj, K-proj, V-proj, gate_proj, up_proj, down_proj, out_proj 등 수십 가지지만, 이것들은 모두 행렬 곱셈(GEMM)이라는 하나의 연산으로 귀결된다. NVIDIA의 커널 라이브러리(nvjet)는 행렬 크기 범위별로 몇 가지 최적화된 GEMM 템플릿을 제공하고, 크기가 맞는 모든 Linear 호출이 같은 템플릿으로 dispatch된다. 표의 "108회"는 이 템플릿 하나가 호출된 누적 횟수다 — 종류는 적지만 호출 수는 많다. 마찬가지로 elementwise 954회도 add, mul, silu, layernorm 등 수십 종류의 소규모 연산이 전부 몇 가지 작은 커널 구현체로 처리된 것이다.

**FlashAttention**은 일반 Attention 연산을 GPU 칩 내부의 고속 메모리(Shared memory, §1.4의 메모리 계층 참조)에서 타일 단위로 처리해 DRAM 접근을 최소화하는 알고리즘이다. 일반 Attention은 seq_len² 크기의 중간 행렬을 DRAM에 쓰고 다시 읽지만, FlashAttention은 이 행렬을 Shared memory 타일 안에서 계산하고 최종 결과만 DRAM에 내보낸다. 한 번 타일에 올라온 Q·K·V를 재사용하므로 DRAM 요청 자체가 줄어든다 — seq_len이 길수록 재사용 횟수가 늘어 이 효과가 커진다. Prefill(seq=3,086)에서 39.5%밖에 안 나오는 이유가 바로 이것이다. FlashAttention이 Prefill 전체 시간의 **52% (738 ms)** 를 차지하면서도 DRAM을 39.5%만 쓰기 때문에 — **이 구간 동안 140 GB/s가 놀고 있다**.

### 3.2 LM Decode

Decode는 토큰을 한 개씩 순차 생성하는 단계로, 매번 입력이 단 1개의 토큰(seq_len=1)이다. 이 경우 GEMM이 **GEMV**(General Matrix-Vector Multiplication, 행렬×벡터 연산)로 축소된다.

**GEMV가 89%를 기록하는 이유**: 가중치 행렬(수천×수천) 전체를 DRAM에서 읽어야 하는데, 연산은 토큰 벡터 하나분 곱셈이라 데이터 재사용이 전혀 없다. warp들이 가중치 행을 처음부터 끝까지 순서대로 읽으면서 끊임없이 새로운 sector를 요청하므로 DRAM 버스가 거의 꽉 찬다. Prefill의 GEMM은 입력 행렬(seq=3,086 토큰)이 있어 가중치 하나를 여러 토큰에 재사용할 수 있지만, Decode의 GEMV는 이 재사용이 없어 DRAM 의존도가 훨씬 높다.

| 구분 | 비율 | 내용 |
|------|------|------|
| GEMV 실행 | ~89.4% | DRAM 89% 포화 상태 |
| **커널 간 유휴** | **~10.6%** | GPU가 완전히 쉬는 구간 (≈ 159 ms) |

Decode 19 step에서 step당 수십 개의 커널이 순차 제출되면서 CPU dispatch 오버헤드가 누적되어 159 ms가 된다.

**DRAM을 기다리는 stall과의 구분**: DRAM-bound 커널이 실행 중에 SM이 DRAM 응답을 기다리며 대기하는 것은 커널 "실행 중" 구간에 포함된다 — 커널이 끝난 것이 아니므로 커널 간 유휴와 전혀 다른 개념이다.

이 159 ms는 DRAM이 포화되어 있어도 별개로 존재하는 낭비다. 알고리즘 변경 없이 제거할 수 있는지가 다음 관심사가 된다 — 이를 §4에서 다룬다.

### 3.3 VE (Vision Encoder)

VE는 ViT(Vision Transformer) 기반으로 이미지 패치를 처리한다. seq_len이 이미지 토큰 수만큼 길어 Prefill과 마찬가지로 FlashAttention이 SRAM에서 처리하는 비중이 크다.

| 커널 그룹 | 호출 수 | DRAM BW% | 병목 분류 |
|----------|---------|----------|---------|
| **FlashAttention** | 27 (ncu 기준) | **~24.2%** | **SRAM-bound** |
| nvjet GEMM (대형, 128×256) | 116 | 43~61% | mixed |
| elementwise / concat | 954 | 72~75% | DRAM-bound |

VE stage-level BW가 35%에 불과한 이유가 여기서 확인된다. FlashAttention이 이미지 패치 토큰들 간의 attention을 Shared memory 안에서 처리하므로 DRAM 요청이 적어 24.2%만 나온다 — 이 구간에 DRAM의 **75%** (≈ 175 GB/s)가 비어 있다. GEMM 구간에서도 90~132 GB/s가 여유롭다.

커널 간 유휴(nsys inter-kernel gap): **~17% (~124 ms)**. 단, VE는 nsys에 `Vision_Encoder` NVTX 범위가 여러 개 등록되어 측정값에 불확실성이 있다. 상위 단일 갭 최대 **76.9 ms** (nsys 직접 측정)가 존재함은 확인됐다.

**DMA 삽입 가능성**: ✅ FlashAttention 구간 175 GB/s 여유 — Prefill보다 오히려 더 큰 DMA 기회. GEMM 구간 90~132 GB/s도 여유롭다.

---

### 3.4 Flow Matching

Flow Matching은 ODE solver가 동일한 Transformer 가중치(4.56 GB)를 N step(실측 10 step) 반복 실행하는 구조다. 매 step마다 가중치 전체를 DRAM에서 읽어야 한다.

| 커널 그룹 | 호출 수 | DRAM BW% | 병목 분류 |
|----------|---------|----------|---------|
| nvjet_64x64 (소형 GEMV) | 720 | **95.5%** | DRAM-bound |
| nvjet_112x64 (bias 포함) | 720+720 | 72~88% | DRAM-bound |
| nvjet_448x64, 512x64 (대형 GEMM) | 720+360 | ~57% | mixed |
| **FlashAttention** | **없음** | — | — |

Flow에는 FlashAttention이 없다. Decode의 GEMV 패턴과 구조적으로 같다 — 가중치를 매 step마다 전부 읽으면서 계산은 적다. Stage-level 88%가 커널 단위에서도 그대로 재현된다. 대형 GEMM(448x64) 구간에서 여유 BW ~99 GB/s가 있지만, 이 커널들은 step당 실행 시간이 짧아 DMA를 삽입해도 얻을 수 있는 이득이 작다.

커널 간 유휴(nsys): **9.2% (81 ms)**, 최대 단일 갭 **3.8 ms**.

**DMA 삽입 가능성**: ❌ GEMV가 이미 88~95% 포화. 여유 BW ~12~25 GB/s로 다음 레이어(≈0.45 GB) prefetch에 18~37 ms 소요 — 레이어 실행 시간(~11 ms/step 내 일부)보다 길어 불가.

## 4. Bubble 지도 — 각 자원의 유휴 시간

**Fig. 3** 참조.

§3에서 각 단계별 커널 간 유휴를 측정했다: VE ~124 ms, Prefill ~130 ms, Decode ~159 ms, Flow 81 ms. 합산하면 **~494 ms** — 전체 추론 시간 4,524 ms의 10.9%다. 이 구간은 GPU가 완전히 비어 있는, 알고리즘적으로 불필요한 대기다. 아울러 BW 여유 구간에서는 DRAM 버스가 남는데, 이 두 종류의 유휴를 함께 정리하면 다음 단계의 최적화 방향이 드러난다.

GPU 내부에는 연산을 담당하는 **SM**(Streaming Multiprocessor) 외에, 메모리 복사 전용 엔진인 **DMA**(Direct Memory Access) 유닛이 별도로 존재한다. DMA는 SM이 계산을 하는 동안 독립적으로 DRAM ↔ L2 간 데이터를 전송할 수 있다. `cudaMemPrefetchAsync`를 사용하면 SM이 현재 레이어를 연산하는 동안 DMA가 다음 레이어 가중치를 미리 DRAM에서 가져오므로, 데이터 대기 시간이 연산 시간 뒤에 완전히 숨겨진다. 단, 이 중첩이 가능하려면 SM이 사용하지 않는 **여유 DRAM BW**가 있어야 한다 — DMA도 같은 DRAM 버스를 쓰기 때문이다.

유휴 구간은 두 종류로 나뉜다:
- **DRAM BW 유휴**: 커널이 실행 중이지만 DRAM을 다 쓰지 않는 구간 → 남은 BW에 DMA를 끼워 넣을 수 있음
- **GPU 완전 유휴**: 커널과 커널 사이에 GPU가 아무것도 안 하는 구간 → CUDA Graph로 CPU dispatch 오버헤드 제거

| 단계 | 유휴 위치 | 유형 | 크기 | 최적화 방법 |
|------|----------|------|------|-----------|
| **VE** | FlashAttention 실행 중 | DRAM BW 여유 | **175 GB/s** (BW 24.2%만 사용) | DMA — 다음 레이어 가중치 prefetch |
| **VE** | GEMM 실행 중 | DRAM BW 여유 | **90~132 GB/s** (BW 43~61%) | DMA — 부분 중첩 가능 |
| **VE** | 커널 간 간격 | GPU 완전 유휴 | **~124 ms** (~17%) | CUDA Graph |
| **LM Prefill** | FlashAttention 실행 중 | DRAM BW 여유 | **140 GB/s** × 41 ms × 18회 | DMA — ★ 최대 기회 (5.7 GB/pass) |
| **LM Prefill** | GEMM 실행 중 | DRAM BW 여유 | **69 GB/s** (BW ~70%) | DMA — 부분 중첩 가능 |
| **LM Prefill** | 커널 간 간격 | GPU 완전 유휴 | **~130 ms** (15.8%) | CUDA Graph |
| **LM Decode** | GEMV 실행 중 | BW 포화 | 여유 없음 (BW 89%) | — |
| **LM Decode** | 커널 간 간격 | GPU 완전 유휴 | **~159 ms** (10.6%) | CUDA Graph — ★ 핵심 타겟 |
| **Flow Matching** | GEMV 실행 중 | BW 포화 | 여유 없음 (BW 72~95%) | — |
| **Flow Matching** | 커널 간 간격 | GPU 완전 유휴 | **81 ms** (9.2%) | CUDA Graph |

### 전체 bubble 합산

| 유휴 유형 | 합계 | 주요 단계 |
|---------|------|---------|
| DRAM BW 여유 (DMA 기회) | VE + Prefill 구간 | VE 175 GB/s, Prefill 140 GB/s |
| GPU 완전 유휴 (CUDA Graph 기회) | **~494 ms** | VE 124 + Prefill 130 + Decode 159 + Flow 81 |

GPU 완전 유휴 494 ms는 전체 추론 시간(4,524 ms)의 **약 10.9%** 다. CUDA Graph를 4단계 전체에 적용하면 이론적으로 ~490 ms를 제거할 수 있다. 단, CUDA Graph는 동적 형상(dynamic shape) 변화에 민감하므로 Decode와 같이 step마다 seq_len이 바뀌는 구간은 별도 처리가 필요하다.

## 5. 결론 및 다음 실험 계획

측정 결과 두 가지 전략이 명확히 도출된다:

**전략 A — DMA 중첩 (cudaMemPrefetchAsync)**: VE와 Prefill의 FlashAttention 구간에 여유 DRAM BW가 있어 다음 레이어 가중치를 미리 가져올 수 있다. Decode와 Flow는 BW가 포화되어 이 전략이 적용되지 않는다.

**전략 B — CUDA Graph**: 4단계 모두 커널 간 유휴가 존재한다. 전체 합산 ~494 ms를 CUDA Graph로 제거할 수 있다. Decode는 seq_len이 step마다 바뀌므로 고정 graph 대신 `torch.cuda.CUDAGraph` + padding 방식이 필요하다.

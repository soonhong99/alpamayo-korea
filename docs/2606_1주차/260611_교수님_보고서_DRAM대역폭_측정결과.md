# Alpamayo 1.5 on Jetson AGX Thor — DRAM 대역폭 실측 및 Async Pipeline 설계 가능성 분석

**보고 날짜**: 2026-06-11  
**보고 대상**: 지도교수님  
**실험 환경**: Jetson AGX Thor (GB10B, SM 11.0, LPDDR5X 231 GB/s, 128 GB 공유 메모리)  
**모델**: Alpamayo 1.5 (Cosmos Reason2 8.2B + Action Expert 2.3B, BF16)  
**측정 도구**: NVIDIA ncu (`lts__d_sectors_fill_sysmem.sum`) + nsys NVTX 타이밍

---

## 0. 핵심 요약 (결론부터)

**연구 목표**: llm.npu (ASPLOS 2025) 핵심 원리 — *"의존성을 지키면서 각 자원의 유휴 시간(bubble)을 제거하는 스케줄링"* — 을 Alpamayo on Thor에 적용하기 위한 실측 기반 분석.

**이번 분석에서 밝혀진 것**:

| 발견 | 의미 |
|------|------|
| Decode BW **89%** (204 GB/s), Flow BW **88%** (203 GB/s) | 이 두 단계는 DRAM이 이미 포화 → prefetch 여지 없음 |
| Prefill BW **55%** (126 GB/s), VE BW **35%** (80 GB/s) | 여유 BW 존재 → DMA 삽입 가능한 구간 |
| Prefill FlashAttention: BW **39.5%** → 여유 **140 GB/s** | 41ms 구간에 5.7 GB DMA 가능 → **layer prefetch 실현 가능** |
| Decode inter-kernel 유휴 **10.6%** (약 160 ms) | CUDA Graph 적용 시 제거 가능한 bubble |
| Decode layer-level prefetch: **4.4 ms** layer << **33 ms** prefetch 소요 | layer-level DMA는 Decode에 불가 → inter-inference 전략 필요 |

**다음 단계**: Prefill layer-level DMA 중첩 구현 + Decode CUDA Graph으로 inter-kernel bubble 제거.

## 1. 측정 방법론

### 1.1 Block vs Layer 개념 정의

> **교수님이 "36 layers는 block을 의미하는 건가?"라고 질문하셨습니다.**

이 연구에서 **Block = Layer = Transformer Block** (동일 개념, 혼용 가능):

```
Transformer Block (1개) 구성:
  ┌─ RMSNorm
  ├─ Multi-Head Attention
  │    ├─ Q_proj / K_proj / V_proj  (GEMM)
  │    ├─ FlashAttention             (SRAM-bound, O(n²) FLOPs)
  │    └─ O_proj                     (GEMM)
  ├─ RMSNorm
  └─ FFN (SwiGLU)
       ├─ gate_proj                  (GEMM)
       ├─ up_proj                    (GEMM)
       └─ down_proj                  (GEMM)
```

- VLM LM (Qwen3VL Text Decoder): ncu 실측 기준 **18 transformer blocks** 확인  
  (FlashAttention 호출 수 18회, GEMM 108회 = 6 per block × 18)
- VE (Vision Encoder): 별도 ViT 아키텍처 (block 수 상이)
- Flow (ODE solver): 별도 diffusion 아키텍처 (block 개념 다름)

> ⚠ 모델 아키텍처 문서에는 "36 layers"로 기재되어 있으나, ncu 실측에서 18 FlashAttention 호출이 확인됨. NVTX 필터가 LM Prefill 전체를 커버하는지 재검증 필요 (NVTX 계측 점검 예정).

### 1.2 측정 도구 및 방법

**두 가지 계층으로 측정**:

```
[계층 1] Stage-level (nsys NVTX)
  → 각 단계(VE, LM Prefill, Decode, Flow)의 총 시간과 DRAM 총량 파악
  → 도구: ncu + nsys NVTX timestamp 교차 검증

[계층 2] Kernel-level (ncu per-kernel)
  → 각 커널(FlashAttention, GEMM, GEMV 등)의 순간 BW + 점유율 측정
  → 도구: ncu --replay-mode kernel (2-pass per kernel)
  → 핵심 메트릭: lts__d_sectors_fill_sysmem.sum (DRAM read), lts__t_sectors...write.sum (DRAM write)
```

**GB10B SM 메트릭 이슈**: Ampere/Hopper에서 사용하는 `sm__active_cycles.sum`은 GB10B(SM 11.0)에 미존재 → 항상 0 반환. 대신 **`smsp__cycles_active.sum / smsp__cycles_elapsed.sum`** (warp 점유율, "smsp_occupancy") 사용.  
단, smsp_occupancy는 "SM이 연산 중"이 아니라 "SM 내 warp가 스케줄링 대기 중"을 나타냄 — DRAM 대기 중인 warp도 포함됨. 따라서 **DRAM BW%가 병목 판단의 실질 지표**.

## 2. 4단계 DRAM 대역폭 실측 결과

### 2.1 Stage-level 요약

| 단계 | 시간 (ms) | DRAM Read | DRAM Write | **총 DRAM** | Read BW | **Peak%** | L2 hit |
|------|----------|-----------|------------|------------|---------|----------|--------|
| **VE** | 728 | 58.4 GB | 39.7 GB | 98.1 GB | 80 GB/s | **35%** | 49.2% |
| **LM Prefill** | 1,423 | 179.9 GB | 52.1 GB | 232.0 GB | 126 GB/s | **55%** | 29.6% |
| **LM Decode** | 1,503¹ | 307.5 GB | 15.9 GB | 323.4 GB | 205 GB/s | **89%** | 37.7% |
| **Flow ODE** | 870 | 176.7 GB | 67.6 GB | 244.3 GB | 203 GB/s | **88%** | 22.0% |
| **합계** | **4,524** | 722.5 GB | 175.2 GB | **897.7 GB** | — | — | — |

> ¹ 19 steps × 79.1 ms/step (AppendOnlyCache-C 실측, 2026-05-31 확정)  
> Peak% = Read BW / 231 GB/s × 100 (LPDDR5X 이론 단방향 최대치)

```
DRAM BW 포화도 (Peak%)
─────────────────────────────────────────────────────────────────
VE       ████░░░░░░░░░░░░░░░░░░░░░  35%   (여유 BW 150 GB/s)
Prefill  ██████████████░░░░░░░░░░░  55%   (여유 BW 105 GB/s)
Decode   █████████████████████████  89%   (여유 BW  25 GB/s)
Flow     █████████████████████████  88%   (여유 BW  28 GB/s)
─────────────────────────────────────────────────────────────────
         0%       50%      100% (231 GB/s)
```

### 2.2 두 가지 체제 식별

**체제 A — Compute/SRAM-limited** (BW 여유 많음, async DMA 삽입 가능):
- VE (35%), LM Prefill (55%): GPU SM이 연산하는 동안 DRAM이 놀고 있음
- → DMA 엔진을 동시에 돌려 다음 데이터를 미리 올릴 수 있음

**체제 B — Memory-saturated** (BW 거의 꽉 참, DMA 삽입 여지 없음):
- Decode (89%), Flow (88%): DRAM이 이미 포화 상태
- → DMA 더 넣으면 오히려 간섭 발생, 다른 전략 필요

## 3. Kernel-level 상세 분석

### 3.1 LM Prefill — 커널별 측정 결과

**Prefill 내 주요 커널 그룹** (ncu 2-pass replay, 2026-06-11 실측):

| 커널 그룹 | 호출 수 | DRAM BW% | warp occ | **병목 분류** | 지속 시간 |
|----------|---------|----------|---------|------------|---------|
| **nvjet_256x128 (가중치 GEMM)** | 108 | ~70% | ~95% | **DRAM-bound** | ~14ms/kernel |
| **FlashAttention** | 18 | ~39.5% | ~92% | **SRAM/compute-bound** | ~41ms/kernel |
| norm/miscellaneous | 수백 | <10% | — | compute-bound | <1ms/kernel |

**FlashAttention 분석** — Prefill에서 가장 중요한 커널:
- 총 실행 시간: 18 calls × 41 ms ≈ **738 ms** (Prefill 전체 1,423 ms의 52%)
- DRAM BW: 39.5% → 여유 BW = **140 GB/s**
- 이유: seq_len=3,086 → attention matrix O(n²) 연산이 SRAM(Flash 알고리즘)에서 처리
  - SRAM 내 재사용이 많아 DRAM 접근이 줄어듦
  - SM은 바쁘지만 DRAM은 놀고 있음

**핵심 기회**: FlashAttention 41ms 실행 중 140 GB/s가 놀고 있음  
→ 이 시간에 **다음 layer 가중치(~0.84 GB)를 10ms 만에 prefetch** 가능

### 3.2 LM Decode — 커널별 측정 결과

Decode는 seq_len=1의 autoregressive 생성 → 모든 가중치 GEMM이 **GEMV**(행렬-벡터 곱)로 전환:

| 구분 | 비율 | 특성 |
|------|------|------|
| GEMV 실행 시간 | ~89.4% | 89% BW 포화, 여유 없음 |
| **커널 간 유휴 (bubble)** | **~10.6%** | GPU가 완전히 쉬는 구간 |

- 커널 간 유휴 시간 총합: 10.6% × 1,503 ms ≈ **159 ms**
- 19 step 중 매 step마다 반복되는 CPU dispatch overhead, CUDA stream sync 대기
- → **CUDA Graph 적용 시 대부분 제거 가능**

## 4. llm.npu 스케줄링 원칙 적용

### 4.1 핵심 원리

llm.npu (ASPLOS 2025)가 NPU에서 한 것:

> bubble 37% → 0.7% 달성  
> 방법: out-of-order subgraph execution으로 의존성을 지키면서 각 자원의 유휴 시간 제거

Thor에 적용하면:

```
[NPU 관점]          [Thor 관점]
NPU core       →   GPU SM pool
DMA engine     →   cudaMemPrefetchAsync (DMA 엔진, SM과 독립)
SRAM           →   GPU L2 캐시 (32 MB)
DRAM           →   LPDDR5X (128 GB, 231 GB/s)
bubble         →   (1) 커널 간 유휴, (2) SM이 기다리는 DRAM stall
```

**Thor 단일 SM 풀 제약**: CUDA multi-stream 실험(260604_04)에서 SM‖SM = 0.97~1.00× (거의 직렬). 따라서:
- Level 0 (intra-layer SM 병렬화): ❌ Thor 단일 SM 풀로 불가
- **Level 1 (inter-layer DMA-SM 중첩)**: ✅ DMA 엔진은 SM과 독립
- Level 2 (CPU-GPU 중첩): 추후 검토
- **Level 3 (inter-inference streaming)**: ✅ Decode의 근본적 병목 해결책

### 4.2 Level 1: Prefill Layer Prefetch (★ 실현 가능)

```
현재 (직렬):
  [Layer 0 compute, 79ms] [Layer 1 compute, 79ms] ...
  ↕ DRAM에서 매번 새로 로딩

목표 (async pipeline):
  [Layer 0 compute, 79ms]
          [DMA: Layer 1 prefetch, ~10ms] ← SM 연산 중 DMA가 동시에 돌아감
                          [Layer 1 compute, 79ms]
                                  [DMA: Layer 2 prefetch, ~10ms]
```

**수치 근거**:

| 항목 | 값 | 근거 |
|------|---|------|
| Layer 실행 시간 | ~79 ms | 1,423 ms / 18 blocks |
| Layer 가중치 크기 | ~0.84 GB | 15.2 GB (LM) / 18 blocks |
| 사용 가능 DMA BW | ~85 GB/s | FlashAttention 구간 140 GB/s 여유 중 보수 추정 |
| Layer prefetch 소요 시간 | **~10 ms** | 0.84 GB / 85 GB/s |
| **DMA 완료 여유** | **69 ms** | 79 ms - 10 ms ✅ |

결론: 다음 layer 가중치를 현재 layer 실행 중에 완전히 prefetch 가능. 이론적 이득: FlashAttention 구간(738 ms)에서 DMA latency가 compute 뒤에 숨음 → **최대 ~180 ms Prefill 단축** 가능 (DMA 10 ms × 18 layers).

### 4.3 Level 1: Decode Layer Prefetch (❌ 불가, 이유 명확)

| 항목 | 값 |
|------|---|
| Decode step당 실행 시간 | 79.1 ms |
| 1 step 내 layer당 시간 | **4.4 ms** (79.1 / 18) |
| Layer 가중치 크기 | ~0.84 GB |
| Decode 여유 BW (11%) | ~25 GB/s |
| Layer prefetch 소요 시간 | **33 ms** |
| **DMA 완료 가능?** | 33 ms >> 4.4 ms ❌ |

Decode에서 layer-level prefetch가 불가한 근본 이유:
- Decode는 seq=1 GEMV → arithmetic intensity ≈ 1 op/byte → DRAM BW가 이미 89% 포화
- Layer 하나가 4.4 ms 만에 끝나지만 가중치를 DRAM에서 읽는 데만 33 ms 필요
- → **Decode에서는 "layer 하나가 끝나기 전에 다음 layer를 가져올 수 없음"**

### 4.4 Level 3: Inter-Inference Streaming (Decode의 근본 해결책)

Decode bubble의 실질적 해결책:

```
현재 (단일 추론, 직렬):
  [VE 728ms] → [Prefill 1,423ms] → [Decode 1,503ms] → [Flow 870ms]
                                          ↑ 89% BW 포화, layer prefetch 불가

목표 (rolling pipeline, inter-inference):
  추론 T:   [VE] → [Prefill] → [Decode step 1..N] → [Flow]
  추론 T+1:               [VE] → [Prefill] → [Decode step 1..N] → [Flow]
  (T의 Flow 실행 중 T+1의 VE/Prefill 동시 진행)
```

이는 llm.npu Level 3에 해당하며, Alpamayo 10Hz 연속 추론 구조와 자연스럽게 맞아떨어짐.

## 6. 결론 및 다음 실험 계획

### 6.1 확정된 발견

1. **Prefill (55% BW)과 Decode (89% BW)는 근본적으로 다른 체제**:
   - Prefill: GEMM + FlashAttention 혼재, DMA 삽입 여지 충분
   - Decode: GEMV 지배, DRAM 거의 포화, layer-level DMA 불가

2. **FlashAttention이 Prefill 최적화의 핵심**:
   - Prefill 시간의 52% 차지 (738/1,423 ms)
   - SRAM-bound이라 DRAM 여유 140 GB/s 존재 → DMA 기회 최대

3. **Decode는 inter-kernel bubble (10.6% = 159 ms)이 유일한 삭제 가능 구간**:
   - CUDA Graph 적용 목표
   - Layer-level DMA는 물리적으로 불가 (33 ms >> 4.4 ms per layer)

4. **llm.npu 스케줄링 원칙이 Thor에 직접 적용 가능**:
   - Level 1 (DMA-SM 중첩): Prefill에서 구현 가능
   - Level 3 (inter-inference streaming): Decode 병목의 근본 해결책

### 6.2 다음 실험 계획

| 우선순위 | 실험 | 예상 효과 |
|---------|------|---------|
| ★ 1순위 | Prefill layer-level cudaMemPrefetchAsync 구현 | ~-180 ms (-13% Prefill) |
| ★ 2순위 | Decode CUDA Graph 적용 (inter-kernel bubble 제거) | ~-159 ms (-11% Decode) |
| 3순위 | NVTX 계측 점검 (18 vs 36 layers 불일치 해소) | 정확도 향상 |
| 4순위 | VE/Flow 커널별 측정 (VE BW 35% 원인 분석) | 추가 최적화 기회 탐색 |
| 5순위 | Inter-inference streaming pipeline 설계 | 근본적 Decode 병목 해결 |

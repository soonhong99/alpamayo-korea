## 질문 1. 36 layers — "block"과 "layer"는 같은 말인가?

### 결론: 36 layers = 36 transformer blocks. 용어는 혼용되며 같은 대상을 가리킨다.

### 하나의 transformer block이 포함하는 것

```
한 개의 transformer block (= 한 개의 layer):

  입력 tensor [seq_len, hidden_size]
       ↓
  [1] RMSNorm (pre-norm)
       ↓
  [2] Multi-Head Self-Attention
       │  q_proj (GEMM) → Q [32 heads × 128 dim]
       │  k_proj (GEMM) → K [ 8 heads × 128 dim]   ← GQA
       │  v_proj (GEMM) → V [ 8 heads × 128 dim]   ← GQA
       │  q_norm, k_norm (per-head RMSNorm)         ← Qwen3 고유
       │  apply_multimodal_RoPE (3D mrope)
       │  KV Cache update
       │  Scaled Dot-Product Attention (Q×K^T → softmax → ×V)
       │  o_proj (GEMM) → [4096]
       ↓
  [3] 잔차 연결 (Residual Add)
       ↓
  [4] RMSNorm
       ↓
  [5] FFN — SwiGLU 구조
       │  gate_proj (GEMM)  ← [4096 → ~12,288]
       │  up_proj   (GEMM)  ← [4096 → ~12,288]  ※두 proj 입력이 같아서 이론상 병렬 가능
       │  silu(gate) × up (element-wise)
       │  down_proj (GEMM)  ← [~12,288 → 4096]
       ↓
  [6] 잔차 연결
       ↓
  출력 tensor [seq_len, hidden_size]
```

이 덩어리 하나가 "1 layer" = "1 block" = "1 transformer 블록"이다.  
36 layers = 이 블록이 순서대로 36번 쌓인 것.

### ⚠️ 중요: 36개 블록은 VLM(언어 모델)에만 해당한다

Alpamayo 1.5 전체 구조를 보면 **3개의 독립 모듈**이 있다. 36은 그 중 VLM 파트만 가리킨다.

```
┌──────────────────────────────────────────────────────────────────────┐
│ Alpamayo 1.5 (총 22 GB BF16, 10B params)                             │
│                                                                      │
│  ① Vision Encoder (VE)                    728 ms                    │
│     - ViT 계열 (Qwen3VL 패치 인코더)                                 │
│     - 카메라 이미지 → visual tokens 변환                             │
│     - KV Cache 없음 (이미지 일방향 인코딩)                           │
│     - 별도 블록 구조 (레이어 수 VLM과 다름)                          │
│                                                                      │
│  ② VLM — Language Model (텍스트 + 멀티모달 추론)  2,688 ms          │
│     - Qwen3VLTextDecoderLayer × 36  ← 이것이 "36 blocks"           │
│     - hidden_size = 4,096, num_Q_heads = 32, num_KV_heads = 8       │
│     - KV Cache: 455 MB (prefill 후)                                  │
│     - Prefill 1,423 ms + Decode 1,345 ms (17 steps × 79 ms)        │
│                                                                      │
│  ③ Flow Matching / Action Expert          896 ms                    │
│     - Diffusion ODE 기반, 별도 2.3B 모델                             │
│     - VLM과 다른 구조 (별도 레이어 36개, 하지만 작은 hidden_size)    │
│     - expert KV Cache: 9.4 MB (64 token 기준)                       │
│     - ODE 10 steps × 4.56 GB 가중치 = 45.6 GB DRAM 접근 = 진짜 병목 │
└──────────────────────────────────────────────────────────────────────┘
```

**"36"이 어디서 왔는지 정확히:**

```python
# 실측으로 확인된 구조 (260531_kv_cache_profiling.py)
455 MB = 36 × 2 × 8 × 3,086 × 128 × 2 bytes
         ↑
         VLM LM 블록 수

# Action Expert도 레이어 수 36이지만 hidden_size가 달라서 KV 크기가 다름
9.4 MB = 36 × 2 × ? × 64 × ? × 2 bytes
```

→ VE, Flow 각각 별도 구조임. 교수님 질문의 맥락인 "455 MB KV Cache의 36"은 VLM LM 전용.

### KV Cache 공식에서 36이 의미하는 것

```
KV Cache 크기:
  36 layers × 2(K, V) × 8 heads × 3,086 tokens × 128 dim × 2 bytes

"36"은 VLM 블록 수를 가리킨다.
각 블록이 자신의 K 행렬과 V 행렬을 생성하고,
그것을 저장하는 것이 KV Cache의 한 슬롯이다.
```

블록 1이 자기 K₁, V₁을 생성하고, 블록 2가 K₂, V₂를 생성하는 방식으로  
36개 블록에 대해 36쌍의 KV가 존재한다.

## 질문 2. K에도 8 heads, V에도 8 heads?

### 결론: 맞다. K와 V 각각 독립적으로 8 heads씩 가지고 있다.

### GQA (Grouped Query Attention) 구조

Alpamayo는 GQA(그룹 쿼리 어텐션)를 사용한다.  
KV Cache 크기 실측값(455 MB)으로부터 역산하면:

```
455 MB = 36 × 2 × 8 × 3,086 × 128 × 2 bytes

여기서:
  36  : transformer blocks
  2   : K 텐서 1개 + V 텐서 1개
  8   : K의 head 수 (= V의 head 수, 독립적으로 각각 8개)
  3,086 : prefill 완료 시점 sequence length
  128 : head dimension (head_dim)
  2   : BF16 (bytes per element)
```

즉 "8"은 K와 V **모두에** 적용되는 head 수다.

### Query head와 KV head의 관계

```
hidden_size = head_dim × Q_head 수 = 128 × 32 = 4,096

Q projection: W_Q [4096 → 4096] → [seq, 32 heads, 128 dim]  (32 heads)
K projection: W_K [4096 → 1024] → [seq,  8 heads, 128 dim]  ( 8 heads)
V projection: W_V [4096 → 1024] → [seq,  8 heads, 128 dim]  ( 8 heads)
                                         ↑ K와 V는 head 수가 같다
```

어텐션 계산 시 GQA expand:
```python
k = k.repeat_interleave(4, dim=1)  # [B, 8, seq, 128] → [B, 32, seq, 128]
v = v.repeat_interleave(4, dim=1)  # 8 heads → 32 heads로 확장
# Q head 0,1,2,3 → KV group 0 공유
# Q head 4,5,6,7 → KV group 1 공유
# ...
# Q head 28,29,30,31 → KV group 7 공유
```

**왜 이렇게 하는가?**  
Q는 매 decode step마다 새로 계산되어 저장할 필요가 없다.  
K와 V는 저장해야 한다(= KV Cache). K/V head 수를 32 → 8로 줄이면  
KV Cache 크기가 1/4이 된다.

```
[GQA 없는 경우] K, V 각 32 heads: 455 MB × 4 = 1,820 MB → KV Cache 1.8 GB
[GQA 8 heads]  K, V 각  8 heads: 455 MB              → KV Cache 455 MB
```

## 질문 3. 455 MB KV Cache — GPU가 어떻게 소화하는가?

### 결론: 455 MB가 한 번에 올라가는 것이 아니다. 36개 블록을 순서대로 처리하며 블록당 ~12.6 MB씩 접근한다.

### Decode 1 step의 실제 연산 흐름 — KV 쌓임 + Attention + FFN이 동시에 일어난다

이 부분이 가장 중요한 인식 전환이다.

**KV 캐시가 "쌓인다"는 것과 "동시에 다른 연산이 돌아간다"는 것은 사실이다.**  
블록 하나를 처리할 때 아래 순서가 하나의 블록 안에서 일어난다:

```
Block k에서 decode step n (새 토큰 1개 생성 중):

  ─── 입력: hidden [B, 1, 4096] (이전 블록의 출력) ───

  ① q_proj: hidden → Q_new [B, 1, 32, 128]       GEMM (32 MB 가중치 읽기)
  ② k_proj: hidden → K_new [B, 1,  8, 128]       GEMM ( 8 MB 가중치 읽기)
  ③ v_proj: hidden → V_new [B, 1,  8, 128]       GEMM ( 8 MB 가중치 읽기)
     ↑ ①②③ 입력이 모두 같은 hidden → 이론상 동시 실행 가능
     ↑ 현재 구현: 별도 CUDA 커널이 순차 launch (동시화 미구현)

  ④ q_norm(Q_new), k_norm(K_new) — per-head RMSNorm

  ⑤ mrope 적용: Q_new, K_new에 위치 인코딩 삽입

  ⑥ KV Cache update:
       K_cache[k] ← append(K_cache[k], K_new)  ← "KV 쌓임"
       V_cache[k] ← append(V_cache[k], V_new)
       결과: K_cache[k] 크기가 [B, 8, n+1, 128]로 증가

  ⑦ Attention 계산 (KV 쌓인 직후 바로 사용!):
       Q_new [B, 32, 1, 128] × K_cache[k] [B, 32, n+1, 128]^T
       → scores [B, 32, 1, n+1]
       → softmax → context [B, 32, 1, 128]
       → reshape → [B, 1, 4096]
     ↑ 방금 step ⑥에서 추가된 K_new, V_new가 포함된 전체 캐시로 attention

  ⑧ o_proj: context → [B, 1, 4096]              GEMM (32 MB 가중치 읽기)

  ⑨ residual add

  ⑩ post_attention_layernorm

  ⑪ gate_proj: hidden → [B, 1, 12288]           GEMM (~96 MB 가중치 읽기)
  ⑫ up_proj:   hidden → [B, 1, 12288]           GEMM (~96 MB 가중치 읽기)
     ↑ ⑪⑫ 입력 동일 → 이론상 동시 가능, 현재 구현은 순차

  ⑬ silu(gate) × up                              element-wise

  ⑭ down_proj: [B, 1, 12288] → [B, 1, 4096]    GEMM (~96 MB 가중치 읽기)

  ⑮ residual add

  ─── 출력: hidden [B, 1, 4096] → Block k+1 입력 ───
```

**핵심 인식:**
- "KV 쌓임" (step ⑥)과 "그 쌓인 KV로 attention" (step ⑦)이 같은 블록 처리 안에서 즉시 연속 발생
- KV를 쌓는 동시에 FFN이 "병렬"로 돌아가는 것은 아님 — 순서대로 처리
- 하지만 Q/K/V projection (①②③), 그리고 gate/up (⑪⑫)은 같은 입력을 받으므로 **CUDA stream 병렬화 여지가 있음** (미래 최적화 포인트)

### GPU가 455 MB를 처리하는 실제 단위

```
Decode step k에서 (36 블록 순회):

┌─────────────────────────────────────────────┐
│ Block 1 처리                                │
│  가중치 로드: q,k,v,o,gate,up,down         │
│    ≈ 32+8+8+32+96+96+96 = 368 MB           │  ← 가중치 DRAM 로드
│  KV 로드: K₁+V₁ [8 heads, n, 128]         │
│    ≈ 12.6 MB  (n=3086 기준)               │  ← KV DRAM 로드
│  KV write: K₁_new, V₁_new                 │
│    ≈ 0.15 MB (새 토큰 1개)                │
└─────────────────────────────────────────────┘
           ↓ (block 1 출력 [B, 1, 4096] = 8 KB → block 2 입력)
┌─────────────────────────────────────────────┐
│ Block 2 처리 (같은 구조 반복)               │
│  가중치 368 MB + KV 12.6 MB 접근           │
└─────────────────────────────────────────────┘
           ↓  ... × 36 blocks
```

**핵심 수치:**
- GPU가 한 번에 다루는 KV 단위: **1 block = 12.6 MB** (K 6.1 MB + V 6.1 MB)
- 12.6 MB < L2 32 MB → 블록 처리 중 KV 데이터가 L2에 올라가 있음
- 36 blocks × 12.6 MB = 453.6 MB ≈ 455 MB (총량은 이 누적)
- 가중치 368 MB × 36 = 13.2 GB + KV 12.6 MB × 36 = 0.45 GB → 1 step ≈ 13.65 GB DRAM 읽기

## AppendOnlyCache-C 심층 분석 — 왜 이득이 나오는가?

### DynamicCache vs AppendOnlyCache-C: 무엇이 달랐나?

사용자가 들은 "KV 캐시를 복사하는 것을 막아서"는 정확히 맞다.  
거기에 더해 **L2 hit 증가**라는 두 번째 이득이 있다. 이 둘은 원인-결과 관계다.

#### DynamicCache: torch.cat이 만드는 두 가지 문제

```
Decode step n에서 블록 k의 KV 업데이트:

  현재 상태: K_cache[k] = tensor at addr A [B, 8, n, 128]
  새 토큰:   K_new = [B, 8, 1, 128]

  torch.cat([K_cache[k], K_new], dim=2) 내부 동작:
    1. 새 메모리 B구역 [B, 8, n+1, 128] 할당         ← 매 step 새 메모리
    2. A구역(6.1 MB) 전체를 B구역으로 복사            ← 문제 ①: DRAM 쓰기 낭비
       (주소 A에서 B로 6.1 MB 이동)
    3. K_new를 B구역 끝에 추가
    4. A구역 해제 (free)

  문제 ①: DRAM 낭비
    - K 1개당: 6.1 MB 읽기 + 6.1 MB 쓰기 = 12.2 MB 낭비
    - K, V 합산: 12.2 × 2 = 24.4 MB
    - 36 blocks: 24.4 × 36 = 878 MB/step 낭비
    - 17 steps: 878 × 17 = 14,926 MB ≈ 15 GB 불필요 DRAM I/O

  문제 ②: L2 cache 오염 (더 깊은 원인)
    - A구역 → B구역 복사 시점: "방금 attention에서 읽었던 A구역 데이터"는
      복사 대상이 되어 L2에서 쫓겨남
    - 왜? GPU L2는 32 MB. 복사가 일어나면 destination(B구역)의
      새 캐시 라인이 L2를 채우며 source(A구역) 라인을 evict
    - 결과: 다음 attention에서 같은 K,V를 다시 읽을 때 → DRAM에서 새로 로드
    → L2 miss → 실효 대역폭이 물리 한계 231 GB/s에 머무름
```

#### AppendOnlyCache-C: 두 가지 문제를 동시에 해결

```
Decode step n에서 블록 k의 KV 업데이트:

  사전 할당된 버퍼: _k_buf[k] = [B, 8, MAX_LEN, 128]  at addr C (고정!)

  in-place write:
    _k_buf[k][:, :, n, :] = K_new  ← addr C 내부에 직접 쓰기
    (새로운 메모리 할당 없음, 복사 없음!)

  attention을 위한 compact view:
    K_compact = _k_buf[k][:, :, :n+1, :].contiguous()
    → addr C의 앞부분 n+1 개 슬라이스를 연속 메모리로 만들기
    → 방금 쓴 K_new도 포함된 슬라이스 읽기 (6.1 MB)

  FlashAttention(Q, K_compact, V_compact):
    → 방금 .contiguous()로 읽은 것과 같은 데이터 재사용
```

**이득 ①: 복사 제거 (torch.cat 방지)**

```
DynamicCache: 6.1 MB 읽기 + 6.1 MB 쓰기 (새 버퍼로 복사)
AppendOnlyCache-C: 0.15 MB 쓰기만 (K_new 1 token 인-플레이스)

절약: 12.2 MB/block → × 36 × 17 steps = 7,454 MB ≈ 7.5 GB DRAM I/O 제거
```

**이득 ②: L2 Hit 증가 (복사 제거의 결과)**

```
시간 순서:
  ① in-place write: K_new → _k_buf[k][:n]
       → GPU L2에 해당 캐시 라인 올라옴 (약 12.6 MB 중 새로 쓴 부분)
  ② .contiguous(): _k_buf[k][:n+1] 읽기
       → addr C는 바뀌지 않았으므로 L2 HIT ✓
       → 방금 쓴 K_new 포함 전체 K 슬라이스가 L2에 있음
  ③ FlashAttention: 같은 K, V 재참조
       → L2에 아직 있음 → L2 HIT ✓

결과:
  실효 BW = 22 GB 모델 ÷ 0.079 s = 278 GB/s
  > 231 GB/s (물리 DRAM 한계)
  → L2 재사용으로 실제 DRAM 접근이 22 GB보다 적어서 가능
```

**두 이득의 관계:**

```
torch.cat 제거 (근본 원인)
  → 매 step 새 메모리 할당 없음
  → 기존 K,V 데이터가 L2에서 쫓겨나지 않음    ← L2 hit의 근본 이유
  → .contiguous() + FlashAttn이 같은 L2 라인 재사용
  → 실효 BW > DRAM 물리 한계
  → decode 107 ms → 79 ms (-26%)
```

즉, "복사 방지"가 원인, "L2 hit 증가"는 결과다. 둘 다 맞다.

### 4종 캐시 비교 요약 (실측, sdpa, 80 steps)

| 캐시 종류 | ms/step (steady) | DRAM 낭비 | L2 Hit | 비고 |
|----------|-----------------|-----------|--------|------|
| DynamicCache | 107.4 ms | 910 MB/step | 낮음 | torch.cat, L2 evict |
| AppendOnlyCache-B | 100.0 ms | 없음 | 중간 | non-contiguous view, 단편화 |
| StaticCache | 102.5 ms | 없음 | 중간 | full MAX_LEN 버퍼 → attention에 zeros 포함 |
| **AppendOnlyCache-C** | **79.1 ms** | 없음 | **높음** | in-place + .contiguous() ← 최선 |

## 질문 4. 교수님이 제안하신 "병렬화" — 추론 간 스트리밍 파이프라인

### 교수님의 실제 의도: 단일 추론 내가 아니라, 추론들 사이의 스트리밍

이 부분이 처음 문서에서 잘못 강조됐다. 교수님이 말씀하신 것은:

> "모델이 추론을 시작하고, 다음 추론을 바로 진행할 수 있도록 — 그렇게 하면 병렬성을 만들 수 있지 않겠느냐"

즉, 한 추론이 완료되기를 기다리지 않고 다음 추론을 **즉시 시작**하는 **스트리밍 파이프라인**이다.

### 왜 단일 추론 내 레이어 간 병렬화는 불가능한가 (배경 이해)

이것을 먼저 이해해야 교수님의 제안이 왜 유효한지 알 수 있다.

```
Block i의 출력이 Block i+1의 입력이다:

  input
    → Block 1 → h₁
                 → Block 2 → h₂
                              → Block 3 → h₃
                                           → ... → Block 36 → output

"Block 2를 Block 1과 동시에 시작할 수 없다 — h₁이 없으면 Block 2를 실행할 수 없다"
→ 단일 추론 내 레이어 병렬화는 아키텍처 구조상 불가능
```

### 교수님의 제안: 추론 단위의 스트리밍 (Rolling Inference Pipeline)

```
현재 방식 (순차, 10Hz 불가):

  Inference A: ──────────────────── (4,366 ms)
                                     Inference B: ──────────────────── (4,366 ms)
                                                                         Inference C: ────

  → 4,366 ms마다 1개 결과 = 0.23 Hz
  → 100 ms 목표에 43.7× 못 미침
```

```
교수님 제안 방식 (스트리밍 파이프라인):

  Inference A: [VE][Prefill][D1][D2]...[D17][Flow]
  Inference B:      [VE][Prefill][D1][D2]...[D17][Flow]
  Inference C:           [VE][Prefill]...
  ...

  → 100ms마다 새 추론 시작
  → 파이프라인이 가득 차면: 100ms마다 1개 완성된 trajectory 출력
  → 단일 추론 latency는 4,366ms지만 throughput = 10Hz 달성
```

이것이 "Rolling Trajectory Pipeline"의 구조다.  
Alpamayo 1.5가 10Hz를 목표로 하는 실질적 경로다.

### 파이프라인 성립 조건

```
파이프라인이 의미 있으려면:
  1. 각 추론이 독립적이어야 함 (이전 결과에 의존 최소화)
  2. GPU/CPU가 여러 추론의 중첩 실행을 지원해야 함
  3. 메모리가 충분해야 함 (128 GB 공유 메모리 → 여유 있음)

현재 Alpamayo: 각 추론은 새 카메라 프레임으로 독립 실행 가능 ✓
→ 파이프라인 성립 가능

단, 현재 코드는 순차 실행 구조 → 파이프라인 구조로 리팩터링 필요
```

### 3가지 수준의 병렬화 정리

| 수준 | 내용 | 가능 여부 | 상태 |
|------|------|---------|------|
| 레이어 내부 (SM) | 32 Q heads 동시, FFN gate/up 병렬 | ✅ 자동 | GPU 하드웨어가 처리 |
| 레이어 간 prefetch | Block k 연산 중 Block k+1 가중치 DRAM prefetch | ✅ 가능 | cudaMemPrefetchAsync 연구 방향 |
| **추론 간 스트리밍** | **추론 A 진행 중 추론 B 즉시 시작** | **✅ 가능** | **교수님 핵심 제안, 미구현** |

## 질문 5. Dense network를 하나하나 분해해서 볼 수 있는가?

### 결론: 가능하다. 그리고 이것이 현재 연구가 "더 파고들어야" 할 지점이다.

### 한 레이어(36개 중 1개)의 내부 GEMM 구조

프로파일링 스크립트 실측값 기반 (260524_layer_compute_profile.py):

```
블록 1개 안의 GEMM 목록 (실측 크기):

[Attention]
① W_Q projection : [4,096 × 4,096] → 실측 33.6 MB (alignment 포함)
② W_K projection : [4,096 × 1,024] → 실측  8.4 MB (8 heads × 128 = 1,024)
③ W_V projection : [4,096 × 1,024] → 실측  8.4 MB
④ W_O projection : [4,096 × 4,096] → 실측 33.6 MB
   Attention 가중치 합계: ~84 MB / layer

[FFN — SwiGLU 구조]
⑤ W_gate : [4,096 × ~12,288] → 실측 100.7 MB  (병렬)
⑥ W_up   : [4,096 × ~12,288] → 실측 100.7 MB  (병렬)
⑦ W_down : [~12,288 × 4,096] → 실측 100.7 MB
   FFN 가중치 합계: ~302 MB / layer

[레이어 1개 전체 가중치]
84 + 302 = ~386 MB / layer (실측)
× 36 layers ≈ 13.9 GB (VLM LM)
+ VE + Action Expert + embedding = ~22 GB BF16 총량 (실측값과 일치)
```

> ⚠️ intermediate_size: 스크립트 실측값 100.7 MB로부터 역산하면 ~12,288에 해당 (4096 × 12288 × 2B = 96 MB, 정렬 패딩 포함 시 100.7 MB). 이전 문서의 11,008은 잘못된 추정값이므로 수정.

### Decode 1 step에서 각 GEMM의 특성

Decode는 **1 token**을 생성한다. 입력 shape = [1, 4,096].

```
GEMM 종류: Matrix × Vector (행렬-벡터 곱)

① W_Q: [4,096, 4,096] × [4,096, 1]
   연산량: 2 × 4,096² = 33.6 M FLOPs
   메모리: 33.6 MB 읽기 (실측)
   산술 강도: 33.6M FLOPs ÷ 33.6MB = 1.0 FLOPs/byte

⑤ W_gate: [4,096, 12,288] × [4,096, 1]
   연산량: 2 × 4,096 × 12,288 = 100.7 M FLOPs
   메모리: 100.7 MB 읽기 (실측)
   산술 강도: 100.7M ÷ 100.7M = 1.0 FLOPs/byte
```

**모든 decode GEMM의 산술 강도 ≈ 1 FLOPs/byte**

Thor GPU의 BF16 이론 피크:
- 연산: SM11.0 Tensor Core 피크 ≈ 1,600 TFLOPS
- 대역폭: 231 GB/s = 231 × 10⁹ byte/s

균형점 (roofline crossover):
```
이론 피크 ÷ DRAM 대역폭 = 1,600T ÷ 231G ≈ 6,900 FLOPs/byte

Decode 산술 강도 1 FLOPs/byte << 6,900 → 완전 메모리 병목
→ GPU SM이 1% 미만 활용됨 (99%는 DRAM 기다리며 대기)
→ 더 빠른 GEMM 알고리즘이나 더 많은 SM은 의미 없음
→ DRAM 대역폭만이 병목
```

### Prefill의 GEMM 특성 (비교)

Prefill은 **3,086 tokens**를 동시에 처리한다.

```
W_Q: [4,096, 4,096] × [4,096, 3,086] (행렬 × 행렬)
산술 강도: 2 × 3,086 × 4,096² ÷ (4,096² × 2 bytes)
         = 3,086 FLOPs/byte

3,086 >> 1 → 산술 강도 3,086배 증가
→ Prefill은 compute-bound (SM이 바쁨)
→ 그래서 Prefill 시간이 decode 대비 상대적으로 짧은 이유 (DRAM 효율 높음)
```

이것이 교수님이 지적하신 "GEMM 하나하나를 분해"했을 때 나오는 결론이다.  
**같은 가중치 행렬이라도 batch size(token 수)에 따라 병목이 완전히 달라진다.**

## 교수님 피드백의 연구 방향 시사점

교수님이 "더 깊이 파고들어야 한다"는 말씀의 의미를 종합하면:

### 현재 실험의 한계

우리가 보고한 수치들 (728ms VE, 1,423ms Prefill, 1,345ms Decode, 870ms Flow)은  
**단계 전체의 wall-clock time**이다.

교수님이 원하시는 것:
```
각 단계가 정확히 어떤 GEMM에서 얼마를 쓰는가?
  → W_Q: ? ms, W_K: ? ms, W_V: ? ms, W_O: ? ms
  → W_gate: ? ms, W_up: ? ms, W_down: ? ms
  → 각 레이어에서 Attention vs FFN 비율은?

그 시간이 compute-bound인가, memory-bound인가?
  → 산술 강도를 직접 측정했는가?
  → ncu dram__bytes_read.sum 으로 실제 DRAM 접근량을 측정했는가?
```

### 실질적 다음 실험

| 실험 | 도구 | 목표 | 우선순위 |
|------|------|------|---------|
| 레이어별 시간 분해 | `torch.cuda.Event` per-layer timing | 36 레이어 × 7개 GEMM 개별 시간 | **최우선** |
| DRAM 실접근량 측정 | `ncu --metrics dram__bytes_read.sum` | 이론 22 GB vs 실측 대역폭 | **최우선** |
| 산술 강도 측정 | `ncu --metrics flop_count_sp / dram bytes` | roofline 위치 확인 | 높음 |
| Layer prefetch 실험 | `cudaMemPrefetchAsync` + CUDA Stream | compute ↔ DRAM overlap | 중간 |
| **Rolling Inference Pipeline** | **별도 추론 프로세스 병렬 실행** | **throughput 10Hz 달성** | **교수님 핵심 제안** |

### 현재 연구 방향과의 연결

```
교수님 질문 → 연구 방향 연결

"스트리밍 방식으로 병렬성을 만들 수 있지 않겠느냐"
  → Rolling Inference Pipeline (Inter-inference 스트리밍)
  → 추론 A 진행 중 추론 B 즉시 시작 → 100ms 간격 출력 달성
  → CLAUDE.md 최종 단계 목표와 일치

"레이어 구조를 더 깊이"
  → 레이어 단위 시간 분해 → 어느 GEMM이 진짜 병목인가?
  → cudaMemPrefetchAsync로 레이어 k+1 가중치를 레이어 k 연산 중 미리 로드

"455 MB 소화 방식"
  → 레이어당 12.6 MB씩 순차 접근
  → L2=32 MB 맥락: 한 레이어 KV는 L2에 들어감 → AppendOnlyCache-C의 근거
  → ncu L2 hit rate 측정으로 직접 검증 가능
```

## 정리 — 교수님 질문에 대한 핵심 답변

| 질문 | 답변 요약 |
|------|----------|
| 36 layers = block? | **Yes.** 한 block = RMSNorm + Attention(QKV+QKNorm+mrope+Attn+O) + RMSNorm + FFN(SwiGLU). 36개가 VLM LM 안에서 순서대로 쌓임. VE, Flow는 별도 구조. |
| K에도 8 heads, V에도 8 heads? | **Yes.** K 텐서: [8, seq, 128], V 텐서: [8, seq, 128] 별도. GQA: Q 32heads, KV 8heads. KV 공유 그룹당 Q head 4개. |
| 455 MB를 어떻게 소화하는가? | **레이어 순회.** 블록당 K+V 12.6 MB씩 DRAM에서 로드. 한 번에 올리는 것 아님. KV 쌓기(update)와 그 KV로 attention이 같은 블록 안에서 연속 발생. |
| 병렬화 가능한가? | **교수님 의도: 추론 간 스트리밍.** 추론 A 시작 후 즉시 추론 B 시작 → 100ms마다 새 추론 launch → throughput 10Hz. 단일 추론 내 레이어 간은 데이터 의존성으로 불가, 레이어 가중치 prefetch는 연구 방향. |
| Dense network 분해 | **가능하며 필수.** Decode GEMM 산술 강도 ≈ 1 FLOPs/byte → 완전 메모리 병목. Prefill은 3,086 FLOPs/byte → compute-bound. intermediate_size ≈ 12,288 (실측 100.7 MB 기반). ncu로 실측 필요. |
| AppendOnlyCache-C 이득 | **두 가지: (1) torch.cat 복사 제거 → 910 MB/step DRAM 낭비 방지 (2) L2 hit 증가 → 실효 BW > 231 GB/s.** (1)이 원인, (2)가 결과. 107ms → 79ms/step (-26%). |

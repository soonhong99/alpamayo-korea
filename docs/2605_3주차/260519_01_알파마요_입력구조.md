**목적**: 현재 단발성(batch) 입력 구조를 파악하고, 실시간 연속 추론(streaming inference)으로 전환하는 설계

## 1. 현재 입력 포맷 (공식 API 기준)

### 1.1 카메라 구성

```
카메라 4대:
  - front-wide    (전방 광각)
  - front-tele    (전방 망원)
  - cross-left    (좌측)
  - cross-right   (우측)

camera_indices: Tensor shape (4,) int64
  → 각 카메라의 ID 인덱스
```

### 1.2 이미지 프레임

```python
image_frames: Tensor shape (N_cameras, num_frames, 3, H, W)
                          = (4,         4,          3, 320, 576)

N_cameras   = 4          # 카메라 수
num_frames  = 4          # 카메라당 프레임 수 (기본값)
C           = 3          # RGB
H, W        = 320, 576   # 전처리 후 해상도 (원본 1080×1920 → 다운샘플)

물리적 의미:
  프레임 레이트: 10 Hz (0.1초 간격)
  커버하는 시간: 4 frames × 0.1s = 0.4초 시각 이력
```

**flatten 후 모델 입력 변환:**
```python
# create_message() 호출 시:
frames_flat = data["image_frames"].flatten(0, 1)
# shape: (4×4, 3, 320, 576) = (16, 3, 320, 576)
# → 16장의 이미지를 순서대로 나열
```

### 1.3 자차 Egomotion 이력

```python
ego_history_xyz: Tensor shape (1, 1, num_history_steps, 3)
                            = (1, 1, 16,                 3)
  → 3D 위치 (x, y, z) × 16 타임스텝 = 1.6초 이력

ego_history_rot: Tensor shape (1, 1, num_history_steps, 3, 3)
                            = (1, 1, 16,                 3, 3)
  → 3×3 회전 행렬 × 16 타임스텝 = 1.6초 이력

앞 2개 dim (1, 1) = (batch_size, 1) 형식

물리적 의미:
  프레임 레이트: 10 Hz
  커버하는 시간: 16 steps × 0.1s = 1.6초 자차 운동 이력
  좌표계: 자차(ego) 좌표계 기준
```

### 1.4 타임스탬프

```python
relative_timestamps: Tensor shape (N_cameras, num_frames) = (4, 4)
  → t0 기준 상대 시간 (초 단위 float32)
  → 예: [[-0.3, -0.2, -0.1, 0.0], [-0.3, -0.2, -0.1, 0.0], ...]

absolute_timestamps: Tensor shape (N_cameras, num_frames) = (4, 4)
  → 절대 시간 (마이크로초 int64)

t0_us: scalar int
  → 현재 프레임의 절대 시각 (마이크로초)
```

### 1.5 완성된 model_inputs 딕셔너리

```python
# 공식 API 입력 구조
model_inputs = {
    "tokenized_data": processor.apply_chat_template(
        messages,           # create_message()로 생성
        tokenize=True,
        add_generation_prompt=False,
        continue_final_message=True,
        return_dict=True,
        return_tensors="pt",
    ),
    "ego_history_xyz": data["ego_history_xyz"],  # (1, 1, 16, 3)
    "ego_history_rot": data["ego_history_rot"],  # (1, 1, 16, 3, 3)
}
```

### 1.6 추론 호출 (공식 API)

```python
pred_xyz, pred_rot, extra = model.sample_trajectories_from_data_with_vlm_rollout(
    data=model_inputs,
    top_p=0.98,
    temperature=0.6,
    num_traj_samples=1,
    max_generation_length=256,
    return_extra=True,
)
```

### 1.7 출력 포맷

```python
pred_xyz: shape (1, num_traj_samples, 64, 3)
  → 64 웨이포인트 × 3D 위치 = 6.4초 미래 궤적 (10 Hz)

pred_rot: shape (1, num_traj_samples, 64, 3, 3)
  → 64 웨이포인트 × 3×3 회전 행렬

extra["cot"]: str
  → Chain-of-Causation 추론 텍스트

물리적 의미:
  64 waypoints × 0.1s = 6.4초 미래 예측
  표현: 가속도 + 곡률 (unicycle model)
  좌표계: Bird's-eye-view (BEV), ego 좌표계
```

## 2. 현재 구현의 문제점 (run_thor_inference.py)

`scripts/run_thor_inference.py`는 실제로 잘못된 mock을 사용하고 있다:

```python
# ❌ 현재 코드 (틀림)
mock_video = torch.randn(4, 8, 3, 1080, 1920, device="cuda")
mock_egomotion = torch.randn(8, 6, device="cuda")
output = model(video=mock_video, egomotion=mock_egomotion, ...)
```

**문제점:**

| 항목 | 현재 (틀림) | 올바른 값 |
|---|---|---|
| 프레임 수 | 8 | **4** |
| 해상도 | 1080×1920 (원본) | **320×576** (전처리 후) |
| egomotion shape | (8, 6) | **(1, 1, 16, 3) + (1, 1, 16, 3, 3)** |
| 모델 호출 API | `model(video=..., egomotion=...)` | **`sample_trajectories_from_data_with_vlm_rollout(data=...)`** |
| 타임스탬프 | 없음 | **relative + absolute timestamps 필요** |
| 슬라이딩 윈도우 | 없음 (매번 랜덤) | **실제 프레임 큐 필요** |

## 3. 실시간 연속 추론 설계

### 3.1 시간 구조 이해

```
물리 세계:  ─────────────────────────────────────────────────────→ t
            |   |   |   |   |   |   |   |   |   |   |   |   |
카메라:   f0  f1  f2  f3  f4  f5  f6  f7  f8  ...
            ↕ 0.1s 간격 (10 Hz)

t=0.3s 추론 시 model input:  [f0, f1, f2, f3]  → 0.4초 시각 이력
t=0.4s 추론 시 model input:  [f1, f2, f3, f4]  → 슬라이딩 윈도우 +1
t=0.5s 추론 시 model input:  [f2, f3, f4, f5]  → 슬라이딩 윈도우 +1
...

egomotion input: 항상 최근 16 pose (1.6초) → 큰 deque에서 슬라이스

output (t=0.3s):  웨이포인트 [t0.3 → t6.7] (64 waypoints)
output (t=0.4s):  웨이포인트 [t0.4 → t6.8] (64 waypoints)
```

### 3.2 슬라이딩 윈도우 버퍼 구조

```python
class SlidingWindowBuffer:
    """
    실시간 입력 데이터를 관리하는 슬라이딩 윈도우 큐.
    
    카메라 버퍼:  최근 N_FRAMES=4 프레임 유지
    Egomotion 버퍼: 최근 N_EGO=16 pose 유지
    """
    
    # Camera buffer: deque(maxlen=4)
    # [t-0.3, t-0.2, t-0.1, t0]  ← 새 프레임 push → 오래된 거 자동 삭제
    #
    # Egomotion buffer: deque(maxlen=16)
    # [t-1.5, t-1.4, ..., t-0.1, t0]
```

### 3.3 추론 루프 흐름

```
[카메라 드라이버]
      ↓ 0.1초마다 새 프레임 4장 (4 카메라)
[SlidingWindowBuffer.push_frame()]
      ↓
[SlidingWindowBuffer.get_model_input()]
  → image_frames: (4, 4, 3, 320, 576)
  → ego_history_xyz: (1, 1, 16, 3)
  → ego_history_rot: (1, 1, 16, 3, 3)
  → timestamps 계산
      ↓
[create_message() + processor]
      ↓ tokenized_data
[model.sample_trajectories_from_data_with_vlm_rollout()]
      ↓
[출력: 64 waypoints + CoT 추론]
      ↓
[궤적 실행기로 전달]
```

### 3.4 추론 빈도와 latency 목표

```
현실적 제약:
  카메라 10 Hz → 0.1초마다 새 프레임
  추론 목표: ≤100 ms / frame

시나리오 A (1:1): 0.1초마다 추론
  → 추론이 0.1초 안에 끝나야 함 (목표 latency 100ms)
  → 빠듯함: 현재 Thor BF16 추론 > 100ms 가능성 있음

시나리오 B (N:1): 0.2~0.3초마다 추론
  → 모델은 6.4초 미래를 예측하므로 여유 있음
  → 200ms 추론도 5Hz 출력으로 충분히 사용 가능

시나리오 C (Pipeline): EXP-3 크로스프레임 파이프라인
  → Frame N Decode 중 Frame N+1 Vision 병렬 실행
  → 실질적 throughput 향상
```

## 4. 구현 전략

### 4.1 파일 구성

```
scripts/
  run_streaming_inference.py    ← 신규: 실시간 스트리밍 추론 메인
  
alphamayo_streaming/
  __init__.py
  buffer.py                     ← SlidingWindowBuffer 클래스
  camera_source.py              ← CameraSource (Mock / RealCamera)
  egomotion_source.py           ← EgomotionSource (Mock / RealIMU)
  preprocessor.py               ← 프레임 전처리 (resize, normalize)
  output_handler.py             ← 궤적 출력 처리
```

### 4.2 실행 방법 (예정)

```bash
# Mock 카메라로 테스트 (지금 바로 실행 가능)
python scripts/run_streaming_inference.py \
    --source mock \
    --hz 10 \
    --dtype fp4

# 실제 카메라 연결 시
python scripts/run_streaming_inference.py \
    --source /dev/video0 \
    --hz 10 \
    --dtype fp4 \
    --lang ko
```

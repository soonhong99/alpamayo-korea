## 1. Decode Phase 병목 — 정밀 분석

### 1.1 왜 Decode가 병목인가 (FlashDrive와 비교)

| 단계 | FlashDrive (RTX PRO 6000, BW=1,700 GB/s) | 우리 Thor (BW=273 GB/s) |
|------|:---:|:---:|
| Vision | ~5% | 14.3% |
| Prefill | **~40-45% (병목)** | 29.4% |
| **Decode** | ~30% | **38.4% (병목)** |
| Flow | ~20% | 17.8% |

**병목이 뒤집혔다.** RTX PRO 6000은 BW가 6.2배 넓어서 Decode가 빠르고 Prefill이 막힌다.
Thor는 BW가 좁아서 22GB 가중치 로딩이 지배한다.

```
이론적 하한 비교:
  Thor (273 GB/s):       22,157 MB / 273 = 81.2ms/step → 19 steps = 1,543ms
  RTX PRO 6000 (1,700):  22,157 MB / 1,700 = 13.0ms/step → 19 steps = 247ms
```

### 1.2 per-token 126ms에서 81.2ms를 초과하는 이유 (28.8ms 오버헤드)

```
측정값:   110.0ms/token
BW wall:   81.2ms/token
오버헤드:  28.8ms/token

오버헤드 분해:
  CPU dispatch (Python → CUDA kernel launch): ~5ms/token
  CPU GIL 작업 (prepare_inputs_for_generation): ~5ms/token
  KV cache concat (CatArrayBatchedCopy): 27,432회 / 19steps ≈ ~5ms/token
  메모리 BW 비효율 (73.9% 활용률): ~14ms 손실에 해당
  합계: ~28.8ms ✓
```

실제로 CPU serial bottleneck이 **419ms/run** 측정됨 (pytorch_trace 기반).
이 419ms가 CUDA Graph로 제거 가능한 직접 타겟이다.

## 2. CPU 코어 상황 — GIL의 물리적 의미

### 2.1 Python GIL이 가져오는 구조적 한계

```
Thor 14코어 중:
  Core 02: 52.2% (Python 메인 스레드 — GIL 보유)
  Core 07, 09~12: 0.0% (완전 유휴 — 5개 코어)

GIL 이론 최대치:
  100% (단일 코어 포화) / 14코어 = 7.14% (all-core 평균)
  실측 5.0%는 이론 최대의 70%

→ 아무리 멀티스레딩을 해도 Python 코드는 Core 02 한 개에서만 실행된다
→ C 확장(CUDA kernel) 실행 중에는 GIL이 풀려 다른 스레드가 Python 실행 가능
→ 이 GIL 릴리즈 구간을 활용하는 것이 CPU 병렬화의 핵심
```

### 2.2 CPU가 할 수 있는 일 (GIL 릴리즈 구간)

```
5,009ms 추론 중 CPU 가용 시간:
  GPU 커널 실행 중 GIL 릴리즈: ~2,394ms (Core 02 유휴 시간)
  
이 구간에서 가능한 CPU 작업:
  ① 다음 프레임 토크나이징 (35~90ms) → GPU와 완전 overlap 가능
  ② EOS 토큰 비동기 모니터링 → GPU decode와 overlap 가능
  ③ 궤적 후처리 (좌표 변환, 시각화) → 완전 overlap 가능
```

## 3. 방향 A: EOS 고정 + CUDA Graph

### 3.1 현재 CPU serial bottleneck의 정체

pytorch_trace 기반 실측:
```
CPU 직렬 병목: 419ms/run  ← GPU가 완전히 유휴인 구간
  = Decode 루프에서 CPU가 prepare_inputs_for_generation() 실행하는 동안
    GPU가 기다리는 시간의 합
  = 17.5 steps × 5ms/step = 87.5ms (CPU 계산) + 커널 launch 간격 누적
```

CUDA Graph는 이 419ms의 ~88%를 제거할 수 있다.

### 3.2 CUDA Graph 기대 효과 (실측 기반 계산)

```
현재 Decode:   1,925.6ms (19 steps × 110.0ms = 2,090ms 또는 16 steps × 110.0ms = 1,760ms)
BW wall:        81.2ms/token (물리적 하한)

CUDA Graph 적용 후 이론적 최소:
  81.2ms × 17.5 steps(평균) = 1,421ms  (CPU overhead 0으로 만들 경우)
  실현 가능: 약 85~90ms/step × 17.5 = 1,488~1,575ms

전체 latency 개선:
  현재: 5,009ms
  CUDA Graph 후 (현실적): ~4,000–4,200ms  (Decode만 적용)
  개선율: ~17–20%
```

### 3.3 Token Count가 Bimodal인 것의 중요성

현재 8회 측정에서 **16 step 또는 19 step** 으로 딱 두 값만 나타났다.
이것은 EOS 고정 전략에 중요한 시사점이다:

```
만약 max_new_tokens=19로 고정하면:
  - 16 step 런에서: 3 step 낭비 → 3 × 110ms = 330ms 낭비
  - 하지만 모든 런이 동일한 shape → CUDA Graph 캡처 1회로 모든 런 처리 가능

만약 max_new_tokens=16으로 고정하면:
  - 19 step이 필요한 런에서 CoT가 잘림
  - Action Expert의 hidden state가 truncated → 궤적 품질 영향 가능
```

→ **먼저 더 많은 런(100회)으로 16/19 bimodal이 실제인지, 다른 값도 나오는지 확인 필요**

### 3.4 EOS 고정이 Flow Matching에 미치는 영향

Action Expert는 VLM의 EOS token 위치의 hidden state를 conditioning으로 받는다.

```
[안전 시나리오]: max_tokens=19로 고정, 실제 EOS가 16에서 나왔어도
  Action Expert가 EOS position (16)의 hidden state를 받으면 → 안전

[위험 시나리오]: 항상 마지막 token (19)의 hidden state를 받으면
  token 17, 18, 19는 garbage (EOS 이후) → 학습 분포 밖 → 품질 저하

→ Alpamayo 코드 내 Action Expert의 conditioning 방식 확인이 필수
```

## 4. 방향 B: CPU 코어 활용

### 4.1 토크나이징 파이프라이닝 (즉시 구현 가능)

실측된 토크나이징 비용: **35~90ms** (CPU only)

```
현재: [토크나이징 70ms][GPU 추론 5,009ms][토크나이징 70ms][GPU 추론]
최적: [토크나이징 70ms][GPU 추론 5,009ms              ]
                        [토크나이징 70ms (GPU와 동시)]

절감: 토크나이징 비용 70ms를 GPU 추론 내에 완전히 숨김
현재 overhead 중 copy.deepcopy 111ms도 포함해 최대 181ms 절감 가능
```

현재 추론이 5초이므로 181ms = 3.6% 절감. 작아 보이지만 공짜 최적화다.

### 4.2 Async EOS Monitoring

```
현재 패턴 (동기):
  GPU: token 1 생성 → CPU: EOS 확인 (sync) → GPU: token 2 생성 → ...
  → 17.5 × sync overhead 누적

Async 패턴:
  GPU: token 1 생성 → pinned memory에 기록
       token 2 생성 → pinned memory에 기록
       ...
  CPU (별도 스레드): pinned memory polling → EOS 감지 시 stop_event.set()
  GPU: stop_event 확인 → 불필요한 step 중단

효과: token bimodal (16 vs 19)에서 19 step 런이
      16 step에서 EOS 감지 후 중단 → 3 × 110ms = 330ms 절감
```

### 4.3 Speculative Decoding — 현재 단계에서의 평가

```
이론적 speedup (α=수락률, K=speculative tokens):
  speedup = (α × K + 1) / (c + 1)

Alpamayo CoT (자율주행 특화 영어 텍스트):
  α ≈ 0.10~0.20 (일반 LM draft model 사용 시)
  K=4, α=0.15: speedup = 1.6 / 1.1 = 1.45×

현재 단계에서 ROI 낮음:
  - Alpamayo 전용 draft model 없음
  - α 낮으면 오히려 느려질 수 있음
  - 추후 fine-tuning 시 draft model을 함께 distill하면 재검토 가치 있음
```

## 5. 99ms의 진짜 수학 (실측 기반)

### 5.1 BF16 소프트웨어 최적화의 물리적 한계

```
BW wall: 81.2ms/token (22,157 MB / 273 GB/s)

최선 시나리오 (모든 SW 최적화 완벽 적용, BF16):
  Vision:   714.9ms  (BW/Compute bound, 큰 최적화 여지 없음)
  Prefill:  1,471.9ms (Compute bound, Flash Attention으로 부분 개선 가능)
  Decode:   81.2ms × 16 tokens = 1,299ms  (BW wall × 최소 token 수)
  Flow:     890.3ms  (CUDA Graph 적용 시 ODE loop overhead 제거 가능)

합계: ~4,376ms ← BF16 이론 하한

결론: BF16으로는 어떤 소프트웨어 최적화를 해도 99ms 불가능
      대역폭 벽(BW wall)이 물리 법칙이기 때문
```

### 5.2 TRT-LLM FP4의 이론적 가능성

```
FP4 weight: 4bit → 메모리 1/4 → BW wall 81.2ms → 20.3ms/token
TRT-LLM CUDA Graph + paged attention: CPU overhead ≈ 0

FP4 decode: 20.3ms × 16 tokens = 325ms
Vision (INT8): ~360ms (2× 개선)
Prefill (FP4): ~736ms (2× 개선)
Flow (FP4): ~225ms (4× 개선, NVIDIA FP4 2,070 TFLOPS)

합계: ~1,650ms ← TRT-LLM FP4 이론 하한

추가로 CoT 길이 축소 (16→8 tokens, fine-tuning 필요):
  Decode: 20.3 × 8 = 162ms
  합계: ~1,490ms

Vision encoder 최적화 (2 frames만 사용):
  Vision: ~180ms
  합계: ~1,108ms

→ 99ms까지는 여전히 ~11× 부족
   Vision + Prefill이 너무 크다
```

### 5.3 Vision + Prefill이 큰 이유와 해결 방향

```
Vision (714.9ms): 16장 이미지 × ViT 인코딩 → O(N_images)
  개선:
  - 프레임 수 줄이기 (4→2): ~357ms
  - 카메라 수 줄이기 (4→2): ~357ms
  - INT8 Vision encoder: ~350ms
  - 해상도 축소 (320×576→160×288): ~180ms

Prefill (1,471.9ms): 이미지 토큰 × 언어 토큰 합쳐서 한 번에 처리
  개선:
  - KV Cache Reuse (FlashDrive 방식): Prefill skip → ~290ms
    (이전 프레임과 겹치는 이미지 토큰 재사용)
  - Flash Attention 2: ~30% 개선 → ~1,030ms

이상 모두 적용 + TRT-LLM FP4:
  Vision: 180ms, Prefill(KV reuse): 73ms, Decode: 162ms, Flow: 225ms
  합계: ~640ms → 99ms까지 6.5× 부족
```

### 5.4 99ms의 유일한 경로

현재 분석에 따르면 **단일 기법으로 99ms는 불가능**하고, 아래를 모두 적용해야 한다:

| 기법 | 예상 단축 | 복잡도 |
|---|---|---|
| TRT-LLM FP4 변환 | 전체 ~4× | 매우 높음 (4~6주) |
| KV Cache Reuse (FlashDrive 방식) | Prefill ~80% | 높음 (3~4주) |
| CoT 길이 축소 (fine-tuning) | Decode ~50% | 높음 (fine-tuning 필요) |
| Vision 경량화 (2 frames, INT8) | Vision ~75% | 중간 (1~2주) |
| Flash Attention 2 | Prefill ~30% | 중간 |

**모두 적용 시 이론적 달성 가능**: ~150–300ms 수준.
진짜 99ms는 아키텍처 수준의 근본적 변화(모델 증류 등) 없이는 어렵다.

## 6. 실용적 결론: Rolling Trajectory 전략

99ms 달성 전까지 실용적 배포 방법:

```
t=0.0s:  추론 시작 (Vision+Prefill+Decode+Flow ~5s)
t=0.1s~  이전 추론 결과의 waypoint[1], [2], ... 순차 실행 (10Hz 제어)
t=5.0s:  추론 완료 → 새 6.4초 궤적 획득, 동시에 다음 추론 시작

→ 5초에 1회 추론이지만 6.4초 궤적이 항상 준비됨 (1.4초 여유)
→ Rolling 방식으로 끊김 없이 자율주행 가능
→ FP4 적용 시: ~1.2초에 1회, 훨씬 반응성 향상
```

Long-tail 안전성에 대한 우려는 타당하지만, 별도의 경량 충돌 방지 모듈(예: MPC, 단순 룰 기반)을 병행 실행하여 보완 가능하다.

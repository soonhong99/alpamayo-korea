# Alpamayo 1.5 — 99ms 달성을 위한 실험 계획서

**작성 일시**: 2026-05-19 (방향성 전환 반영)  
**플랫폼**: Jetson AGX Thor | **모델**: Alpamayo 1.5 (BF16, 11.08B)  
**관련 문서**: `260519_99ms_달성위한_현기술분석.md`, `260519_nsight_decode_overhead_analysis.md`

---

## 왜 실험 방향을 바꾸는가

기존 계획의 암묵적 전제는 **"Alpamayo를 그대로 쓰되 더 빠르게 실행한다"** 였다.  
그 전제 안에서 가능한 최선의 수치를 이미 계산할 수 있다.

```
확정된 BF16 소프트웨어 최적화 천장:

  NSight 실측 GPU 커널:   57ms/token  (이미 BW Wall 근접)
  CUDA Graph 이상적 적용:  오버헤드 → 0ms
  최소 현실적 토큰 수:     ~10개

  → Decode 하한:  57ms × 10 = 570ms
  → 나머지 Phase: 713 + 1,472 + 890 = 3,075ms
  → 총 하한:      3,645ms

  99ms까지 남은 거리:  36.8×
```

**CUDA Graph를 완벽하게 구현해도 99ms의 37배 수준에 머문다.**  
이 수치가 실험 방향 전환의 근거다.

새로운 질문은 이것이다:  
> *"어떻게 하면 이 모델을 더 빠르게 실행하는가"* 가 아니라  
> *"이 모델을 다르게 실행하면 무슨 일이 일어나는가"*

| 지표 | 정의 |

## Phase 0: 경계 밖 실험

### 실험 1: Decode Skip / Adaptive Decode

#### 왜 하는가

현재 파이프라인의 암묵적 전제는 "Chain-of-Causation(CoC) 텍스트를 항상 완전히 생성해야 한다"는 것이다.

```
현재:
  Prefill → [CoC 16~19 tokens, 1,926ms] → Flow Matching
                       ↑
              이것이 정말 필요한가?
```

Alpamayo에서 Action Expert(Flow Matching)는 VLM의 **마지막 hidden state**를 conditioning으로 받는다. 0개 토큰이 생성되면 Prefill의 마지막 hidden state가 그대로 전달된다. 이 상태에서 궤적이 얼마나 달라지는지는 아직 아무도 실측한 적이 없다.

가능한 결과는 세 가지다:
1. **0 token에서도 합리적 궤적** → CoC 없이도 단순 주행은 가능하다는 발견. 장면 복잡도에 따라 CoC 길이를 동적으로 결정하는 Adaptive Decode 아키텍처의 근거.
2. **특정 token 수(N)에서 품질 유지** → N×57ms + 오버헤드가 새 Decode 하한. 연구 기여점 확정.
3. **0 token은 완전 실패, N ≥ 13에서만 유지** → CoC가 Action Expert 품질에 필수적이라는 실증. CUDA Graph 집중 투자의 근거 확정.

**어떤 결과가 나와도 다음 실험 방향이 결정된다.** 이것이 1번으로 설정하는 이유다.

#### 어떻게 진행하는가

`max_generation_length`를 파라미터로 제어하여 아래 조건에서 각 10회씩 실행한다.

| 조건 | max_generation_length | 예상 Decode (ms) | 예상 Total (ms) |
|---|---|---|---|
| **Decode Skip** | **0** | **0** | **~3,077** |
| 극소 | 1 | ~57 | ~3,134 |
| 최소 | 3 | ~171 | ~3,248 |
| 단축 | 5 | ~285 | ~3,362 |
| 중간 | 8 | ~456 | ~3,533 |
| 중간 | 10 | ~570 | ~3,647 |
| 중간 | 13 | ~741 | ~3,818 |
| **Baseline (16)** | **16** | **~1,760** | **~5,037** |
| Baseline (19) | 19 | ~2,090 | ~5,367 |

**평가 기준 적용**:  
각 조건에서 ADE, FDE, 첫 waypoint 검증, CoC 품질을 측정하여 baseline 대비 비교.  
ADE < 0.2m, FDE < 1.0m를 동시에 만족하는 최소 token 수가 **"품질 유지 하한"** 이 된다.

#### 기대 인사이트

```
품질 유지 하한이 N tokens일 때:
  새 Decode 시간 = N × 57ms (GPU 커널) + 오버헤드
  
  N=0:  Total ~3,077ms  (1.63× 개선)
  N=5:  Total ~3,362ms  (1.49× 개선)
  N=8:  Total ~3,533ms  (1.42× 개선)
  N=13: Total ~3,818ms  (1.31× 개선)

→ 성공 시: 코드 파라미터 1줄 변경으로 1.3~1.6× 개선
→ 실패 시: CoC 필요성 실증 + CUDA Graph 방향 확정
```

### 실험 2: 인터-사이클 파이프라이닝

#### 왜 하는가

현재 파이프라인은 **Request-level Sequentialism** 구조다.

```
현재:
  Cycle 1: [V:715][P:1472][D:1926][F:890]
  Cycle 2:                               → 시작 (t=5,003ms)
  
  한 추론이 완전히 끝나야 다음을 시작한다.
```

그러나 Cycle 2의 Vision Encoding은 Cycle 1의 결과와 **아무런 데이터 의존성이 없다.** Cycle 2의 카메라 프레임은 독립적으로 존재한다.

```
가설:
  Cycle 1의 Decode(1,926ms) 구간 중에 Cycle 2의 Vision(715ms)을 시작한다.
  
  [Cycle 1]  [V:715][P:1472][D:1926][F:890]
  [Cycle 2]               [V:715][P:1472][D:1926][F:890]
                           ↑
                    t=2,187ms (C1 Decode 시작 시점)에서 C2 Vision 시작
  
  이상적 결과: Cycle 2 완료 = t=2,187 + 715 + 1,472 + 1,926 + 890 = 7,190ms
  Cycle 간격: 7,190 - 5,003 = 2,187ms  (기존 5,003ms의 2.3× 개선)
```

**단, Thor의 LPDDR5X 통합 메모리라는 제약이 있다.** Decode(BW-bound)와 Vision(Compute-bound)이 동시에 실행되면 동일한 273 GB/s DRAM을 경쟁한다. 얼마나 충돌하는지는 실측 없이 예측 불가능하다. 당연히 빠를 것이라고도, 당연히 느려질 것이라고도 말할 수 없다.

#### 어떻게 진행하는가

CUDA Stream을 2개 사용하여 Cycle N의 Decode 시작 시점에 Cycle N+1의 Vision을 다른 Stream에서 실행한다.

측정 대상:
1. **BW 경쟁 실측**: Stream 2개 실행 시 각 Phase의 실제 소요 시간이 어떻게 변하는가
   - 단독 실행 대비 Vision 시간 변화: Vision_overlap / Vision_solo
   - 단독 실행 대비 Decode 시간 변화: Decode_overlap / Decode_solo
2. **Effective cycle time**: 연속 10 사이클 실행 시 사이클 간 평균 간격

**평가 기준**:

| 결과 | 판정 | 해석 |
|---|---|---|
| Effective cycle time < 3,000ms | ✅ 의미 있는 개선 | BW 경쟁이 경미함, 파이프라이닝 유효 |
| Effective cycle time 3,000~4,500ms | ⚠️ 부분적 개선 | BW 경쟁이 있으나 일부 overlap 효과 존재 |
| Effective cycle time ≥ 5,003ms | ❌ 개선 없음 | BW 완전 포화, 오히려 느려짐 |

**품질 기준**: Cycle 간 waypoint 연속성 유지 (이전 Cycle 마지막 waypoint ↔ 현재 Cycle 첫 waypoint의 위치 차이 < 1.0m).

#### 기대 인사이트

성공하면: 5,003ms 단독 추론이 **effective 2,187ms 사이클 추론으로 전환**.  
이것은 "더 빠른 추론"이 아니라 **"더 자주 업데이트되는 추론"** 이다. 목표인 반응성(long-tail 대응)에 직접 기여한다.

실패하면: Thor의 통합 메모리 구조에서 파이프라이닝이 BW로 인해 불가능하다는 실증. TRT-LLM + HBM2E 교체 필요성의 근거.

### 실험 3: Temporal KV Cache Reuse

#### 왜 하는가

연속된 두 추론 사이클(~5초 간격)에서 입력 장면은 거의 동일하다.

```
Cycle N 입력:   [t=0.0s 프레임, t=0.1s, t=0.2s, t=0.3s]
Cycle N+1 입력: [t=5.0s 프레임, t=5.1s, t=5.2s, t=5.3s]

5초 전 장면의 Prefill KV cache 전체를 버리고 1,472ms를 다시 처음부터 계산한다.
시스템 프롬프트("chain-of-thought reasoning of the driving process...")는
두 Cycle에서 100% 동일함에도 매번 다시 처리된다.
```

FlashDrive(2025)는 이 중복 Prefill을 제거하여 4.5× Prefill 단축을 달성했다 (RTX PRO 6000).  
Thor에서의 효과는 별도로 측정해야 한다.

```
재사용 가능한 KV cache:
  시스템 프롬프트:              100% 동일 → 완전 재사용 가능
  이전 프레임(frame 0,1,2):    Cycle N과 N+1 사이에 3개가 동일 → 재사용 가능
  새 프레임(frame 3):          Cycle마다 1개만 새로 처리

예상 Prefill 단축:
  4 프레임 중 3개 재사용 → Prefill 계산량 75% 감소
  1,472ms → ~370ms  (예상, 실측 필요)
```

#### 어떻게 진행하는가

**단계 1 (사전 실험)**: 시스템 프롬프트의 KV cache만 재사용 (가장 단순한 경우).  
동일한 입력으로 2회 연속 추론 시, 1회차 시스템 프롬프트 KV cache를 2회차에 그대로 주입하여 Prefill 시간이 단축되는지 확인한다.

**단계 2 (본 실험)**: 연속 프레임 KV cache 재사용.  
슬라이딩 윈도우에서 이전 3 프레임의 KV cache를 보존하고, 새 1 프레임 분량만 Prefill에 추가한다.

**평가 기준**:

| 지표 | 기준 |
|---|---|
| Prefill latency 단축 | 단계 1: < 1,200ms (현재 대비 18% 이상 단축) |
| Prefill latency 단축 | 단계 2: < 800ms (현재 대비 45% 이상 단축) |
| ADE | < 0.2m (KV cache 재사용으로 수치 정밀도 오차가 없어야 함) |
| FDE | < 1.0m |

**가장 중요한 확인사항**: KV cache를 재사용했을 때 numerical error가 누적되지 않는가. 완전히 새로 계산한 결과와 ADE < 0.05m 이내여야 "재사용이 안전하다"고 판정한다.

### 실험 4: Token Distribution 100회

#### 왜 하는가

8회 측정에서 bimodal(16 또는 19)이 관찰됐다. 이 패턴이 실제인지, 더 많은 데이터가 필요하다.  
CUDA Graph의 `max_new_tokens` 설정과 Phase 0 실험 1의 baseline token 수 분포 확정을 위해 필수다.

#### 어떻게 진행하는가

동일한 Mock 입력(torch.randn)으로 100회 추론. 매 회차의 실제 CoC token 수와 latency를 기록한다.

**평가 기준**:

| 결과 | 의미 |
|---|---|
| 100% {16, 19} | bimodal 확정 → max_new_tokens=19로 CUDA Graph 설계 |
| {16, 17, 18, 19} 분포 | 범위 내 분산 → max_new_tokens=22로 여유 설정 |
| 30+ tokens 발생 | 분포 꼬리 있음 → 상위 5% 커버 max 값으로 설정 |

### 실험 5: torch.compile

#### 왜 하는가

torch.compile의 `reduce-overhead` 모드는 내부적으로 자동 CUDA Graph 캡처를 시도한다. Thor aarch64 + Alpamayo 조합에서 이것이 동작하는지는 아무도 시도해본 적 없다.

**이 실험이 CUDA Graph 수동 구현의 필요성을 결정한다.**

- torch.compile이 per-token을 110ms → 60~70ms로 낮추면: 자동 경로 유효, 수동 구현 범위 축소
- torch.compile이 효과 없으면: CUDA Graph 자동 캡처가 동적 KV cache shape로 인해 실패하는 것이고, 수동 StaticCache 구현이 반드시 필요

#### 어떻게 진행하는가

`mode="reduce-overhead"` 와 `mode="max-autotune"` 각각 Warmup 3회 후 10회 측정.  
NSight로 compile 적용 전후 per-token GPU 커널 시간도 함께 측정한다.

**평가 기준**:

| per-token 변화 | 판정 | 후속 방향 |
|---|---|---|
| 110ms → 57~70ms | ✅ CUDA Graph 자동 캡처 성공 | 수동 구현 최소화, 미세 조정만 |
| 110ms → 90~105ms | ⚠️ 부분 적용 | 어느 op가 fallback인지 확인 후 수동 구현 병행 |
| 110ms → 108~110ms | ❌ 효과 없음 | 수동 StaticCache CUDA Graph 필수 |

### 실험 6: CUDA Graph (Decode Phase 수동 구현)

#### 왜 하는가

NSight에서 확인된 53ms/token 오버헤드를 제거한다.  
이 중 `cudaStreamSynchronize`가 30~40ms를 차지하며, 이것이 EOS 체크의 GPU-CPU blocking sync다.

```
목표:
  현재:  110ms/token (GPU 커널 57ms + 오버헤드 53ms)
  이후:  ~57ms/token (GPU 커널만, 오버헤드 제거)
  
  Decode: 17.5 tokens × 110ms = 1,925ms
  → 17.5 tokens × 57ms = 997ms  (-928ms, -48%)
```

#### 어떻게 진행하는가

HuggingFace DynamicCache를 StaticCache(max_tokens 크기 고정 할당)로 교체하고,  
단일 decode step을 CUDA Graph로 캡처하여 replay 방식으로 전환한다.  
실험 4의 token distribution 결과로 결정된 `max_new_tokens`를 사용한다.

**평가 기준**:

| 지표 | 목표 |
|---|---|
| Per-token (NSight GPU 커널) | 변화 없어야 함: 57ms ± 5ms |
| Per-token (전체 사이클) | < 70ms (오버헤드 62% 이상 제거) |
| Decode total latency | < 1,300ms (baseline 1,926ms 대비 32% 이상 단축) |
| ADE vs baseline eager | < 0.1m (CUDA Graph가 수치를 바꾸면 안 됨) |

### 실험 7: EOS Sync 격리 실험

#### 왜 하는가

NSight에서 보인 `cudaStreamSynchronize`가 실제로 30~40ms를 차지하는지 **단독으로 검증**한다.  
CUDA Graph 완전 구현 전에, 가장 단순한 방법(EOS 체크 비활성화)으로 가설을 먼저 검증한다.

EOS 체크를 없애면 `token.item()` 호출이 사라지고 `cudaStreamSynchronize`가 사라진다.

```
가설:
  EOS 체크 제거 후 per-token: 57 + (53 - 35) = 75ms
  즉, 110ms → ~75ms (32% 단축)
  
검증:
  실제로 이 정도 줄어드는지 NSight로 확인
  → "줄어든다": StreamSync가 진짜 35ms였다는 직접 증거
  → "줄지 않는다": StreamSync 위치나 원인이 달리 특정 필요
```

**평가 기준**:

| per-token 변화 | 판정 |
|---|---|
| 110ms → 65~80ms | ✅ StreamSync 가설 검증 완료. CUDA Graph 방향 확신 |
| 110ms → 90~105ms | ⚠️ StreamSync가 예상보다 작음. NSight 재분석 필요 |
| 110ms → 107~110ms | ❌ EOS 체크가 병목이 아님. 다른 원인 탐색 |

## Phase 2: 조합 및 통합

### 실험 8: 최적 조합 Full Pipeline

Phase 0과 Phase 1에서 효과가 검증된 기법들을 결합한다.  
실험 결과에 따라 구성이 달라지므로, 실험 1~7 완료 후 결정한다.

**예상 조합 시나리오**:

```
시나리오 A: Adaptive Decode (실험 1 성공) + CUDA Graph (실험 6 성공)
  Decode N tokens (N < 16) + CUDA Graph 실행
  예상: ~2,500~3,500ms

시나리오 B: KV Cache Reuse (실험 3 성공) + CUDA Graph (실험 6 성공)
  Prefill 단축 + Decode 오버헤드 제거
  예상: ~2,000~3,000ms

시나리오 C: 파이프라이닝 (실험 2 성공) + Adaptive Decode (실험 1 성공)
  Effective cycle time 단축 + Decode 길이 단축
  예상: effective 1,500~2,500ms

시나리오 D: 전부 실패 → BF16 SW 최적화 한계 확인
  TRT-LLM FP4 변환으로 방향 전환 근거 확정
```

**최종 통합 평가 기준**:  
ADE < 0.2m, FDE < 1.0m를 만족하면서 Total latency 또는 Effective cycle time이 baseline(5,008ms) 대비 1.5× 이상 개선되어야 의미 있는 결과로 판정한다.

## 1. 실험 배경

### 왜 캐시를 보는가?

딥러닝 추론에서 GPU는 매 연산마다 모델 가중치를 메모리에서 읽어야 한다. 이때 가중치가 **DRAM(주 메모리)** 에서 오느냐, **L2 캐시(GPU 내부 SRAM)** 에서 오느냐에 따라 속도가 크게 달라진다. 캐시가 DRAM보다 빠르다면, 자주 접근하는 가중치를 캐시에 유지할 수 있다면 추론 속도를 향상시킬 수 있다.

Alpamayo 1.5는 자동회귀 디코딩(autoregressive decoding) 방식으로 65개의 토큰을 생성한다. 이 과정에서 매 디코딩 스텝마다 동일한 가중치 행렬 전체를 반복 접근한다. 이 반복 접근 패턴이 캐시 활용의 핵심 관심 대상이다.

### 핵심 질문

1. Thor GPU에 SRAM(L2 캐시)이 있는가? 크기는 얼마인가?
2. CPU의 캐시 계층(L1/L2/L3) 구성과 크기는?
3. 모델 가중치(또는 일부)를 캐시에 올려 빠르게 접근할 수 있는가?

## 3. 실험 방법론

### 실험 원리: "Bandwidth vs Working Set Size" 기법

메모리 계층의 경계를 실험적으로 찾는 방법이다. 작은 데이터(예: 4 MB)를 읽으면 캐시에 담겨 빠른 대역폭이 나오고, 큰 데이터(예: 256 MB)를 읽으면 캐시를 넘어 DRAM에서 읽어 낮은 대역폭이 나온다. 이 "절벽(cliff)"이 발생하는 크기가 곧 캐시 용량이다.

### 측정 도구의 선택과 실패 과정

실험 중 측정 도구 선택이 결과의 정확도에 결정적 영향을 미친다는 것을 확인했다.

**[시도 1 실패] CPU: `numpy.sum()` 사용**

초기 시도에서 `arr.sum()`으로 대역폭을 측정했더니 모든 크기에서 약 15 GB/s로 flat하게 나왔다. 원인: `sum()`은 메모리를 읽는 동시에 부동소수점 덧셈을 수행한다. ARM64에서 스칼라 float32 덧셈의 이론 한계가 약 10–15 GB/s이므로, 데이터가 L1 캐시 안에 있어도 연산 속도가 병목이 되어 캐시 효과가 보이지 않는다. → **해결**: 순수 메모리 복사인 `numpy.copyto(dst, src)` 사용 (덧셈 없음).

**[시도 2 실패] GPU: 단일 커널 반복 측정**

`x.sum()`을 50번 반복 호출해 평균값을 구했더니 512 KB에서 16 GB/s, 256 MB에서 251 GB/s로 bandwidth가 단조증가하는 이상한 결과가 나왔다. 원인: CUDA 커널 호출마다 고정 오버헤드(~5 µs)가 붙는다. 4 MB 텐서의 실제 읽기 시간이 16 µs인데 오버헤드 5 µs가 더해지면 측정 대역폭이 실제보다 크게 낮아진다. → **해결**: CUDA Graph 사용 — 100번 반복을 단일 그래프에 묶어 오버헤드를 100분의 1로 줄임.

### 최종 측정 방식

```
CPU: numpy.copyto(dst, src)
     Working set = src + dst = 2 × 배열크기
     BW = 2 × 배열크기 / 측정시간
     단일 코어 고정 (taskset)

GPU: CUDA Graph + y.copy_(x) (100회 묶음)
     Working set = x + y = 2 × 텐서크기
     그래프 1회 재생 시간 ÷ 100 = 1회 평균 시간
     BW = 2 × 텐서크기 / 평균시간
     → 그래프 내 2번째 반복부터 L2 캐시 히트 발생
```

### 왜 이 측정 방식을 선택했는가

**목표: "순수한 메모리 대역폭"만 격리해서 측정하는 것**

딥러닝 추론에서 병목이 메모리인지 연산인지를 판단하려면, 먼저 메모리 대역폭 자체를 정확하게 알아야 한다. 이를 위해 연산(compute) 요소를 완전히 제거한 순수 메모리 복사(memory copy) 벤치마크가 필요하다.

Bandwidth vs Working Set Size 기법을 선택한 이유는 두 가지다:

1. **캐시 계층 경계(cliff)를 직접 실험으로 확인할 수 있다.** 이론 스펙(공식 문서)이 없거나 틀릴 수 있는 상황에서, 실제 동작하는 하드웨어에서 직접 경계를 측정한다. 실제로 Thor GPU L2(32 MB)는 공식 문서에서 32 MB라고 명시되어 있지만, CPU L3(16 MB)는 NVIDIA 공식 스펙에서 제공되었고 `/sys` 파일시스템에서는 보고되지 않았다 — 이 방법으로 간접 확인했다.

2. **배열 크기를 연속적으로 바꾸면, 각 크기에서 어느 계층이 병목인지를 하나의 실험에서 한번에 볼 수 있다.** 단일 크기만 측정하면 "L2에 들어가는지 아닌지"를 알 수 없다. 연속 sweep을 통해 cliff 발생 위치를 정밀하게 찾는다.

### 왜 이 방법이 정확한가

측정 정확도는 두 가지에 달려있다: **연산 배제**와 **오버헤드 배제**.

**연산 배제 — `numpy.copyto(dst, src)` 선택 이유**

`sum()`, `mean()`, `dot()` 같은 reduction 연산은 메모리를 읽으면서 동시에 부동소수점 연산을 한다. ARM64에서 스칼라 float32 덧셈의 이론 한계는 약 10–15 GB/s인데, 이것이 실제 메모리 대역폭(107 GB/s)보다 훨씬 낮기 때문에 연산이 병목이 되어 캐시 효과가 숨겨진다. `copyto()`는 읽기(load) + 쓰기(store)만 수행하므로 연산 병목이 없다.

```
sum()   : 읽기 + 덧셈 →  연산이 병목 → 측정값 ≈ 15 GB/s  (잘못된 값)
copyto(): 읽기 + 쓰기 → 메모리가 병목 → 측정값 ≈ 107 GB/s (올바른 값)
```

Working set = `src + dst = 2 × 배열크기`로 계산하므로, 읽기와 쓰기 모두 트래픽에 정확히 반영된다.

**오버헤드 배제 — GPU CUDA Graph 선택 이유**

GPU에서 Python 함수나 CUDA 커널을 호출할 때마다 약 5–10 µs의 고정 오버헤드가 발생한다. 4 MB 텐서를 1 회 복사하는 실제 시간은 약 4 µs이므로, 오버헤드(5 µs)가 실제 시간(4 µs)보다 크다 — 측정값이 실제 대역폭의 절반 이하로 나온다.

CUDA Graph는 여러 번의 커널 실행을 하나의 그래프로 컴파일한 뒤, 그래프를 한 번 실행한다. 오버헤드는 그래프 실행 당 한 번만 발생하므로 100회 반복을 묶으면 오버헤드가 1/100로 줄어든다.

```
단순 반복 100회: (실제시간 + 오버헤드) × 100  →  오버헤드가 큰 비중
CUDA Graph 100회: 실제시간 × 100 + 오버헤드 × 1  →  오버헤드 ≈ 무시 가능
```

### 이것이 업계 표준 방법인가

**그렇다. 이 방식은 메모리 계층 벤치마킹의 표준 방법론이다.**

**CPU 쪽: STREAM benchmark (1995–현재)**

CPU 메모리 대역폭 측정의 사실상 표준은 John McCalpin의 **STREAM benchmark** (University of Virginia, 1995)다. STREAM은 네 가지 커널(Copy, Scale, Add, Triad)을 사용하는데, 그중 Copy 커널이 바로 `a[i] = b[i]` — 즉 덧셈 없는 순수 메모리 복사다. CPU 제조사(Intel, ARM, AMD)도 공식 성능 문서에서 STREAM 결과를 제시한다. 우리 실험의 `copyto()`가 STREAM Copy와 동일한 원리다.

**GPU 쪽: Microbenchmark 논문들**

GPU 메모리 계층 측정은 다음 논문들에서 확립된 방법론을 사용한다:

- **Volkov & Demmel (2008)** "Benchmarking GPUs to Tune Dense Linear Algebra": Working set sweep으로 L1/L2/DRAM 경계를 최초로 체계적으로 측정
- **Wong et al. (2010)** "Demystifying GPU Microarchitecture through Microbenchmarking": CUDA 커널로 캐시 용량·레이턴시·대역폭을 working set sweep으로 측정하는 방법 정립
- **NVIDIA Roofline Model 가이드라인**: DRAM 대역폭 측정에 `cudaMemcpy` 또는 단순 copy 커널 사용을 권장

CUDA Graph를 활용한 반복 실행 + 평균화는 NVIDIA의 공식 프로파일링 가이드(Nsight Systems)에서도 권장하는 방식이다.

**요약:**

| 구분 | 우리 방법 | 업계 표준 |
|------|---------|---------|
| CPU 대역폭 | `numpy.copyto()` + working set sweep | STREAM Copy benchmark |
| GPU 대역폭 | CUDA Graph + `y.copy_(x)` + sweep | Wong et al. (2010) 방식 |
| 오버헤드 처리 | CUDA Graph 100회 묶음 | CUDA event / graph 활용 |
| 계층 경계 탐지 | cliff 발생 위치 = 캐시 크기 | 동일 원리 |

이 방법은 H/W 스펙을 "읽어서 아는" 것이 아니라 "실제 동작으로 검증"하기 때문에, 공식 문서가 없거나 틀린 경우에도 정확한 값을 얻을 수 있다. Thor처럼 공식 문서가 충분하지 않은 신규 하드웨어에서 특히 유효하다.

## 4. 실험 결과

### 실험 1: CPU 캐시 계층 대역폭 측정

**결과 표:**

| 배열 크기 | BW (GB/s) | 해석 |
|----------|-----------|------|
| 8 KB | 33.3 | 측정 오버헤드 영향 (배열 너무 작음) |
| 32 KB | 83.0 | L1D 접근 중 |
| **64 KB** | **95.7** | **L1D 경계** |
| **128 KB** | **106.6** | **측정 피크 (최고 대역폭) — L1D/L2 경계, ARM 프리페처 효과** |
| **512 KB** | **89.3** | **L2 경계 진입** (working set 1 MB = L2 크기) |
| 1 MB | 67.7 | L2 초과, L3 진입 |
| 4 MB | 48.4 | L3 영역 |
| **16 MB** | **39.7** | **L3 공식 경계** |
| 128 MB | 33.3 | DRAM 바닥 |
| 256 MB | **33.2** | **단일 코어 DRAM 실측 BW** |

**핵심 발견:**

- **측정 피크 대역폭: ~107 GB/s** (128 KB 배열, working set 256 KB)
  - ⚠️ **주의**: 128 KB 배열은 working set이 256 KB이므로 L1D(64 KB/코어)를 4배 초과한다. 물리적으로는 **L2 영역**이다.
  - 그럼에도 L1D급 대역폭이 나오는 이유: **ARM Neoverse V3AE 하드웨어 프리페처**가 순차 접근 패턴을 감지하여 L1D 이상의 크기에서도 한동안 고대역폭을 유지한다. 즉, 107 GB/s는 "L1D 내부" 수치가 아니라 "L1D/L2 경계에서 프리페처 최대 효율" 수치다.
  - **결론**: 107 GB/s는 이 CPU에서 순차 복사로 달성 가능한 단일 코어 최대 대역폭이며, 그 위치는 L2 캐시 영역이다.
- **L2 경계**: 512 KB 배열(= working set 1 MB)에서 대역폭 급락. L2 용량 1 MB/코어와 정확히 일치. ✅
- **L3 경계**: 16 MB 부근에서 cliff가 없고 완만하게 감소함. `/sys` 파일시스템에도 L3가 보고되지 않음.
  - **이유**: ARM Neoverse V3AE의 하드웨어 프리패처(prefetcher)가 순차 접근 패턴을 감지해 DRAM을 미리 가져온다. Sequential streaming 패턴에서는 L3와 DRAM의 bandwidth 차이가 blur된다. 실제 L3 경계를 뚜렷하게 보려면 랜덤 접근(pointer chasing) 기반 레이턴시 측정이 필요하다.
- **단일 코어 DRAM 대역폭: ~33 GB/s**. Thor 전체 DRAM 273 GB/s의 약 12%.

**CPU 캐시 계층 요약:**
```
L1D  (64 KB/코어)    : ~107 GB/s  ─ 단일 코어 순차 피크
                                    (실제 피크 위치는 128 KB = L2 영역,
                                     ARM 프리페처 효과로 L1D 초과 후에도 유지됨)
L2   (1 MB/코어)     : ~67 GB/s   ─ L2 히트 안정 영역 (512 KB~1 MB)
L3   (16 MB 공유)    : ~40 GB/s   ─ 완만한 전환 (프리패처 효과)
DRAM (공유)          : ~33 GB/s   ─ 단일 코어 실측 한계
```

### 실험 2: GPU L2 vs DRAM 대역폭 측정

**결과 표:**

| 텐서 크기 | BW (GB/s) | 해석 |
|----------|-----------|------|
| 512 KB | 114.2 | GPU 메모리 파이프라인 미포화 |
| 1 MB | 206.5 | 파이프라인 채워지는 중 |
| 4 MB | 344.9 | L2 영역이나 아직 미포화 |
| **8 MB** | **1,106.0** | **GPU L2 피크 대역폭** |
| **10 MB** | **1,125.5** | **GPU L2 피크 (최고점)** |
| 12 MB | 1,036.5 | L2 내 |
| **16 MB** | **455.3** | **L2 경계** (x+y = 32 MB = L2) |
| **18 MB** | **201.9** | **DRAM으로 전환** |
| 32 MB | 244.9 | DRAM (포화 상태) |
| 96 MB | 233.7 | DRAM |
| 256 MB | **231.0** | **DRAM 실측 대역폭** |

> **L2 boundary 주석**: `y.copy_(x)` 연산의 working set은 `x + y = 2 × 텐서크기`. L2(32 MB) = 2 × 16 MB이므로 cliff는 16 MB에서 발생.

**핵심 발견:**

| 항목 | 실측값 |
|------|--------|
| **GPU L2 피크 대역폭** | **~1,106–1,125 GB/s** |
| **GPU DRAM 실측 대역폭** | **~227–232 GB/s** |
| **L2 / DRAM 속도 비** | **4.76×** |
| L2 경계 (실증) | **16 MB** (working set 32 MB) |

**"소형 텐서에서 L2 대역폭이 낮은 이유":** 512 KB(114 GB/s)가 8 MB(1,106 GB/s)보다 훨씬 낮다. 두 텐서 모두 L2에 들어가는데 왜 차이가 나는가? → GPU는 수천 개의 스레드가 병렬로 메모리를 요청해야 고대역폭을 달성한다. 512 KB(128K elements)는 20개 SM에 분산하면 SM당 6.4K elements, 처리 시간이 ~0.5 µs에 불과해 메모리 파이프라인을 채울 시간이 없다. 8 MB는 처리 시간 ~7 µs로 충분히 파이프라인을 포화시킨다.

**L2 내부에서도 크기에 따라 달성 가능한 BW가 다름을 실험적으로 확인.**

### 실험 3: GPU L2 Eviction 실험 — 디코딩 중 가중치가 L2에 남는가?

#### 실험 설계

디코딩 1 스텝을 다음과 같이 모사했다:
1. **flush 텐서 읽기** (다른 레이어 연산 모사, L2를 밀어낼 수 있음)
2. **대상 weight 텐서를 CUDA Graph(100회)로 읽고 대역폭 측정**

4가지 weight 크기 × 4가지 flush 크기 = 16가지 조합 측정. DRAM 기준선: 227.2 GB/s.

#### 결과

| Weight 크기 | L2 적합? | Flush 크기 무관 평균 BW | DRAM 대비 |
|------------|---------|----------------------|----------|
| **4 MB** | ✅ 적합 | **~970 GB/s** | **4.3×** → L2 resident |
| **16 MB** | ✅ 적합 | **~466 GB/s** | **2.1×** → partial L2 |
| **33 MB** | ❌ 초과 | ~228 GB/s | ~1.0× → DRAM |
| **200 MB** | ❌ 초과 | ~230 GB/s | ~1.0× → DRAM |

**상세 결과 (4 MB weight):**

| Flush 크기 | BW (GB/s) | 해석 |
|-----------|-----------|------|
| 4 MB | 983.0 | L2 resident ✓ |
| 32 MB | 978.7 | L2 resident ✓ |
| 64 MB | 961.6 | L2 resident ✓ |
| **200 MB** | **957.4** | **L2 resident ✓** |

**매우 중요한 발견**: 4 MB weight는 **200 MB flush 이후에도** L2에 남아 있다. Flush 크기(DRAM의 여러 배)가 변해도 결과가 거의 바뀌지 않는다.

**해석**: CUDA Graph 내에서 100번 반복이 연속 실행된다. Flush로 인해 첫 번째 접근이 DRAM에서 올지라도, 두 번째부터는 L2에 올라가 있으므로 평균 BW는 L2 수준을 유지한다. 즉 "L2 eviction test"는 **"반복 접근 시 steady-state L2 대역폭"**을 측정한다.

**16 MB weight의 "partial L2" (2.1× DRAM) 해석**: 16 MB는 L2(32 MB)에 들어가지만, 앞서 실험 2에서 16 MB 텐서의 L2 대역폭도 ~455 GB/s였다. 이는 16 MB 접근 시 GPU 파이프라인이 완전히 포화되지 않음을 의미한다. "L2에 있지만 small kernel이라 달성 BW가 낮음"이 2.1× 결과의 원인이다.

#### 실제 Alpamayo 디코딩에 적용하면?

```
실제 decode 1 step:
  레이어 0 가중치 읽기 (MLP: 96 MB) → L2 flush 발생
  레이어 1 가중치 읽기 (MLP: 96 MB) → L2 flush 발생
  ...
  레이어 27 가중치 읽기             → 누적 총 20.64 GB
  
  → 레이어 0으로 돌아오기까지 20.64 GB가 L2를 통과
  → L2(32 MB)는 완전히 치환됨
  → 다음 decode step에서 레이어 0은 반드시 DRAM에서 다시 읽어야 함
```

**결론: 실제 Alpamayo 추론에서 model weight는 decode step 사이에 L2에 남아있지 않는다. 매 step마다 20.64 GB를 DRAM에서 읽는다.**

### 실험 4: Alpamayo 1.5 모델 레이어별 캐시 적합성 분석

#### 실측 아키텍처

모델을 직접 로드하여 레이어별 가중치 크기를 측정했다.

| 항목 | 값 |
|------|-----|
| 총 파라미터 수 | 11.08B |
| 총 가중치 크기 (BF16) | **20.64 GB** |
| Language Model 레이어 수 | 28개 |
| d_model (임베딩 차원) | 4,096 |
| FFN 히든 차원 | 12,288 (= 3 × d_model) |
| Q/O attention projection | 4,096 × 4,096 → **32 MB 각** |
| MLP gate/up/down projection | 12,288 × 4,096 → **96 MB 각** |
| K/V projection (GQA) | < 32 MB (측정) |
| embed_tokens / lm_head | **1.19 GB 각** |

**모델 전체 vs 캐시:**

```
Model (20.64 GB) / GPU L2 (32 MB)  = 660배 → 모델 전체를 L2에 올리는 것은 불가능
Model (20.64 GB) / CPU L3 (16 MB)  = 1,322배 → 동일하게 불가능
```

#### 레이어 분류 (총 979개 linear layer)

| 분류 | 수 | 비고 |
|------|-----|------|
| L2 초과 (>32 MB) | 226개 | MLP layer 전부, embed/lm_head |
| **L2 이하 (≤32 MB)** | **753개** | Q/O attention, K/V (GQA), 소형 projection 등 |
| SMEM/SM 이하 (≤228 KB) | 353개 | LayerNorm weight, bias 등 |

**중요 발견**: Q_proj와 O_proj (4096×4096)의 크기가 **정확히 32 MB = L2 크기**와 일치한다. 경계선에 있다.  
MLP layer (96 MB)는 L2의 3배이므로 항상 DRAM bound.

#### KV 캐시 분석 — 가장 흥미로운 발견

KV 캐시는 디코딩 중 축적되는 이전 토큰들의 Key, Value 텐서다. 65 decode step 기준:

| 시나리오 | KV 캐시 크기 | L2(32 MB) 대비 |
|---------|------------|---------------|
| Full attention (32 KV heads) | 28.44 MB | **0.89× → L2에 들어감** |
| GQA-4 (8 KV heads, 실제 추정) | 7.11 MB | **0.22× → 여유롭게 들어감** |
| Action Expert | 4–8 MB | **들어감** |

**실측 k_proj가 32 MB 이하(top-30 목록에 없음)** → 모델이 GQA(Grouped Query Attention)를 사용하고 있음을 간접 확인. 즉, **실제 KV 캐시는 7–28 MB 수준으로 GPU L2에 충분히 들어간다.**

## 5. 종합 결론

### 결론 1: GPU L2 캐시(SRAM)는 존재하며 DRAM보다 4.76배 빠르다

| 구분 | 크기 | 대역폭 |
|------|------|-------|
| GPU L2 캐시 (SRAM) | 32 MB | **~1,100 GB/s** |
| GPU DRAM | 122.8 GB | **~230 GB/s** |
| **속도비** | — | **4.76×** |

GPU 안에는 32 MB의 L2 SRAM이 있으며, 이것이 DRAM보다 약 5배 빠르다. 이 캐시가 추론 성능에 미치는 영향이 있는지가 다음 질문이다.

### 결론 2: 모델 전체를 캐시에 올리는 것은 불가능하다

Alpamayo 1.5의 가중치는 20.64 GB (BF16)로, GPU L2(32 MB)보다 **660배** 크다. 어떤 양자화를 적용해도 캐시에 전체를 올리는 것은 불가능하다. INT4 양자화를 하면 ~2.6 GB이지만, 여전히 L2보다 81배 크다.

### 결론 3: 매 디코딩 스텝마다 20.64 GB를 DRAM에서 읽는다 (현재 상태)

Alpamayo 디코딩은 65 step으로 구성된다. 각 step에서 28개 레이어 전체를 순차 접근하므로, step당 20.64 GB 읽기가 발생한다. 총 디코딩 전체:
```
20.64 GB × 65 steps = 1,342 GB
시간: 1,342 GB ÷ 230 GB/s ≈ 5.8초 (DRAM bandwidth 한계)
```
실측 decode latency와 일치한다. **Alpamayo 디코딩은 DRAM bandwidth bottleneck이다.**

### 결론 4: KV 캐시는 L2에 들어가지만, 자동으로 유지되지는 않는다

65 decode step의 KV 캐시는 7–28 MB로 GPU L2 크기 이하다. 그러나 매 decode step에서 레이어 전체 가중치(20.64 GB)가 L2를 통과하면서 KV 캐시를 evict시킨다. 따라서 자동으로는 KV 캐시가 L2에 남아있지 않다.

## 6. 향후 연구 방향 — CUDA L2 Persistent Residency

위의 결론을 뒤집을 수 있는 CUDA 기능이 존재한다: **L2 Persistent Residency** (CUDA 11.2+, SM 8.0+ 지원 → Thor SM 11.0 지원).

### 개념

GPU의 `cudaStreamSetAttribute` API를 사용하면:
- 특정 메모리 영역을 "persisting" 모드로 설정 → L2에서 evict되지 않도록 힌트
- 다른 메모리 접근을 "streaming" 모드로 설정 → L2를 거치지 않고 DRAM 직접 접근 (L2 bypass)

```
[현재 상태]
  decode step:  Layer0 weight → L2 → evicts KV cache
                Layer1 weight → L2 → evicts everything
                ...
                KV cache access → DRAM (evicted)

[L2 Persistent 적용 후]
  decode step:  KV cache → persistent (L2에 잠금)
                Layer weights → streaming mode (L2 bypass, DRAM 직접)
                KV cache access → L2 resident ✓ (4.76× faster)
```

### 예상 효과 계산

KV cache(7 MB, GQA 기준)를 L2에 고정하면 attention 연산 시 DRAM 접근 횟수가 줄어든다.

각 decode step의 attention 연산에서 KV cache 읽기:
```
기존:  7 MB × 65 steps ÷ 230 GB/s = 1.98 ms
개선:  7 MB × 65 steps ÷ 1,100 GB/s = 0.41 ms
절감:  1.57 ms (전체 decode에서 attention bandwidth cost 79% 절감)
```

KV cache bandwidth cost가 전체 decode latency에서 차지하는 비중은 작지만 (총 5.8초 중 ~2%), MLP 이전에 실행되는 attention이 latency-critical path에 있다면 효과가 있을 수 있다.

### 실험 계획

실제 추론 코드에 L2 persistent residency를 적용하여 decode latency 변화를 측정하는 실험이 필요하다. 이는 ctypes를 통해 CUDA Runtime API를 직접 호출하는 방식으로 구현 가능하다.

## 1. 실행 환경 및 입력 — "무엇을 넣었는가"

### 1.1 입력의 정체

이번 실행에서 넣은 입력은 **완전한 랜덤 노이즈**입니다. 실제 카메라 영상이 아닙니다.

```python
# MockCameraSource.get_frames() — 실제 코드
frames = torch.randn(4, 3, 320, 576)   # 가우시안 랜덤 노이즈
```

**카메라 입력 상세:**

```
카메라 수:    4대 (front-wide, front-tele, cross-left, cross-right 위치 가정)
프레임 수:    카메라당 4프레임 (0.4초 분량)
해상도:       320 × 576 픽셀 (RGB 3채널)
내용:         순수 랜덤 노이즈 (실제 도로/차량 아무것도 없음)
총 데이터:    4 cameras × 4 frames × 3 channels × 320 × 576 = 8,847,360 float32 값
```

**Egomotion 입력 (자차 위치/자세 이력):**

```
이력 길이: 16 타임스텝 (1.6초 @ 10 Hz)
위치 xyz:  직선 전진 시뮬레이션 (x = step × 0.8m, y=0, z=0)
회전:      단위행렬 (identity) — 정면 향함
좌표계:    자차(ego) 기준
```

**텍스트 입력 (nav_text):**

```
이번 실행에서는 별도 nav_text 없음.
create_message()의 기본 프롬프트만 사용:
  "chain-of-thought reasoning of the driving process, then output the future trajectory"
```

### 1.2 입력이 모델에 전달되는 과정

```
1. MockCameraSource.get_frames()
     → torch.randn(4, 3, 320, 576)   — 4카메라 × 1프레임

2. SlidingWindowBuffer.push_frame()
     → deque에 누적 (maxlen=4)
     → 16번 push 후 버퍼 준비 완료 (1.6초 경과)

3. SlidingWindowBuffer.get_model_input()
     → image_frames:       (4, 4, 3, 320, 576) — 4카메라 × 4프레임
     → ego_history_xyz:    (1, 1, 16, 3)       — 1.6초 위치 이력
     → ego_history_rot:    (1, 1, 16, 3, 3)    — 1.6초 회전 이력
     → relative_timestamps: (4, 4)             — 각 프레임의 상대 시각

4. helper.create_message()
     → image_frames.flatten(0,1) = (16, 3, 320, 576)
     → 16장 이미지를 chat message 형식으로 구성
     → 카메라별 라벨 부착 ("Front camera frame 0", "frame 1" 등)

5. processor.apply_chat_template()
     → 이미지 + 텍스트 → 토큰 시퀀스
     → return_tensors="pt" → PyTorch 텐서

6. helper.to_device(model_inputs, "cuda")
     → GPU 메모리로 이동

7. model.sample_trajectories_from_data_with_vlm_rollout()
     → 실제 추론 실행
```

## 2. 출력 결과 — "무엇이 나왔는가"

### 2.1 실측 결과 (3회 반복)

| 회차 | Latency | wp[0] (0.1초 후) | wp[9] (1.0초 후) | Chain-of-Causation |
|---|---|---|---|---|
| 1 | **5,599 ms** | (1.00, -0.00) m | (9.95, -0.16) m | "Keep lane since the path is clear and no lead vehicle is present" |
| 2 | **4,195 ms** | (1.00, -0.00) m | (10.02, -0.01) m | "Keep lane since the lane is clear ahead" |
| 3 | **4,734 ms** | (1.00, -0.00) m | (10.01, -0.03) m | "Adapt speed for the right curve since the lane bends right ahead" |
| **평균** | **4,843 ms** | — | — | — |

### 2.2 출력 구조 분석

**궤적 출력:**

```python
pred_xyz_t: shape (1, 1, 1, 64, 3)
#                   ↑  ↑  ↑  ↑  ↑
#                batch seq traj step coord

pred_xyz = pred_xyz_t[0, 0, 0]  # → (64, 3)
# 64 waypoints × 3D 좌표 (x, y, z) in ego frame
# 간격: 0.1초 (10 Hz)
# 커버 시간: 6.4초 미래
```

**waypoint 물리적 해석:**

```
wp[0] = (1.00, -0.00, ?) m  ← 0.1초 후 위치
  → x=1.00m 전진: 속도 = 1.00 / 0.1 = 10 m/s = 36 km/h

wp[9] = (9.95, -0.16, ?) m  ← 1.0초 후 위치
  → x≈10m 전진: 속도 일관성 ✅ (가속/감속 없는 등속 직진)
  → y≈0m: 차선 유지 ✅

→ 물리적으로 타당한 자율주행 궤적 출력됨
```

**Chain-of-Causation (추론 텍스트):**

랜덤 노이즈 이미지를 입력했음에도 도로 주행 관련 추론이 나왔습니다:

- "Keep lane since **the path is clear** and **no lead vehicle is present**"
- "Adapt speed for **the right curve** since the lane bends right ahead"

이것은 모델이 실제로 동작하고 있다는 증거입니다. 3번째 실행에서 다른 추론("right curve")이 나온 것은 매번 랜덤 노이즈가 다르기 때문에 모델이 다른 시각적 패턴을 "감지"했기 때문입니다.

### 2.3 왜 랜덤 노이즈로 의미 있는 출력이 나오나

```
랜덤 노이즈 이미지
        ↓
Vision Encoder (ViT)
        ↓ 시각적 특징 추출 (노이즈도 특징으로 처리됨)
VLM Backbone (Cosmos Reason2 8.2B)
        ↓ 학습 데이터의 가장 빈번한 패턴으로 수렴
        ↓ = "도로 주행 중, 차선 유지, 장애물 없음"
Chain-of-Causation 텍스트 생성
        ↓
Action Expert (2.3B Diffusion)
        ↓ Egomotion 이력(직진 시뮬레이션)을 강하게 반영
Waypoint 출력: 등속 직진
```

**핵심**: Egomotion 이력이 직선 전진이므로 Action Expert가 직선 궤적을 생성합니다. 랜덤 이미지는 CoT에 영향을 주지만 egomotion이 실제 궤적의 주요 요인입니다.

## 3. 성능 분석 — "얼마나 빠른가"

### 3.1 Latency 분해

```
총 추론 시간: ~4,843 ms (평균)

추정 단계별 분해:
  ┌────────────────────────────────────────────────────┐
  │ Phase          │ 예상 시간  │ 근거                  │
  ├────────────────────────────────────────────────────┤
  │ Vision (ViT)   │ ~400 ms   │ 16장 이미지 인코딩    │
  │ Prefill (VLM)  │ ~600 ms   │ 이미지 토큰 처리      │
  │ Decode (VLM)   │ ~3,600 ms │ CoT 텍스트 생성       │
  │ Action Expert  │ ~35 ms    │ Diffusion 궤적 생성   │
  │ 기타 오버헤드  │ ~200 ms   │ 토크나이징, 이동 등   │
  └────────────────────────────────────────────────────┘

병목: Decode 단계 (BW-bound GEMV, seq=1 autoregressive)
```

### 3.2 1회차(5.6초) vs 2회차(4.2초) 차이

```
1회차: 5,599 ms  ← GPU 워밍업 포함
         CUDA 커널 컴파일, 캐시 콜드 스타트
2회차: 4,195 ms  ← 안정된 상태
3회차: 4,734 ms  ← 정상 범위
```

실제 운영 latency는 **4.2 ~ 4.8초**로 봐야 합니다.

### 3.3 BF16 vs FP4 예상 성능

```
현재 (BF16):  4,843 ms/inference
              Decode BW: ~163 GB/s

FP4 (예상):   ~1,200 ms/inference
              4× 대역폭 감소 (22 GB → 5.5 GB weights)
              논문 근거: NVIDIA FP4 Blackwell = 2,070 TFLOPS

현실:
  BF16  → 0.21 Hz 실효 추론 빈도
  FP4   → 0.83 Hz 실효 추론 빈도 (예상)
```

## 4. 현실적으로 돌아갈 수 있는가

### 4.1 결론 먼저

```
✅ 기술적으로는 가능
⚠️ 단, 10 Hz 실시간은 현재 불가능 — 약 0.2 Hz (5초 주기)로 운영해야 함
✅ 자율주행 실용에는 충분할 수 있음 (모델이 6.4초 미래를 예측하므로)
```

### 4.2 핵심 논리: "왜 5초 추론이 실용적인가"

```
모델 출력:  6.4초 미래 궤적 (64 waypoints @ 10 Hz)
추론 시간:  ~4.8초 (BF16)

타임라인:
  t=0.0s   추론 시작 (입력 프레임 캡처)
  t=4.8s   추론 완료 → 6.4초 궤적 획득
  t=4.8s ~ t=11.2s 에 해당하는 궤적이 나옴
  
현재 시각이 t=4.8s이므로:
  앞으로 6.4초(t=11.2s까지)의 궤적 → 여전히 1.6초 여유

다음 추론: t=4.8s에 시작 → t=9.6s 완료 → t=16.0s까지 궤적
  ↑ 끊김 없이 연결됨

→ 5초마다 한 번 추론해도 자율주행 연속 실행 가능
```

### 4.3 시나리오별 현실 가능성

| 시나리오 | 현재 BF16 | FP4 적용 후 | 판정 |
|---|---|---|---|
| **실시간 10 Hz** (100ms/frame) | ❌ (4,843ms) | ❌ (~1,200ms) | 불가 |
| **실시간 1 Hz** (1s/frame) | ❌ (4,843ms) | ⚠️ (~1,200ms) | FP4로 경계 |
| **5초 주기 추론** (궤적 6.4초) | ✅ | ✅ | **가능** |
| **파이프라인 모드** (EXP-3) | ⚠️ 부분적 | ✅ | FP4 권장 |
| **테스트 / 시뮬레이션** | ✅ | ✅ | **가능** |
| **한국어 추론 데모** | ✅ | ✅ | **가능** |

### 4.4 실제 자율주행에서의 운영 방법

```
[전략 A] 롤링 궤적 실행 (추천)
  ┌─────────────────────────────────────────────────────┐
  │ t=0s:   추론 ①  시작 (입력: t=0~0.4s 영상)         │
  │ t=4.8s: 추론 ① 완료 → t=4.8~11.2s 궤적 획득       │
  │ t=4.8s: 추론 ② 시작 (입력: t=4.4~4.8s 영상)        │
  │ t=9.6s: 추론 ② 완료 → t=9.6~16.0s 궤적 획득       │
  │         차량은 ① 궤적을 4.8초간 추종               │
  │         ② 궤적으로 매끄럽게 전환                   │
  └─────────────────────────────────────────────────────┘
  → 항상 6.4초 미래가 준비됨. 끊김 없음.

[전략 B] FP4 + 롤링 궤적
  FP4 → ~1.2s/inference → 1.2초마다 6.4초 궤적 갱신
  → 훨씬 반응성 향상 (도로 상황 변화 반영 빠름)

[전략 C] 두 모델 병렬 실행 (MIG, JetPack 7.2 이후)
  슬라이스 A: VLM 추론 중
  슬라이스 B: 이전 추론 결과 궤적 실행 중
  → 진정한 병렬 처리
```

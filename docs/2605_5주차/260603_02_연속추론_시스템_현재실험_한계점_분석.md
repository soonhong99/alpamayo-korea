# 10Hz 연속 추론 시스템 설계 시 현재 실험의 한계점 분석

**날짜**: 2026-06-03  
**대상 독자**: 실험 배경을 모르는 사람도 이해 가능하도록 작성  
**목적**: 현재까지의 최적화 실험들이 실제 연속 운용 환경에서 갖는 구조적 한계를 정리한다

---

## 1. 먼저 알아야 할 것 — 이 시스템이 해야 하는 일

Alpamayo는 자율주행 차량에 탑재되어 **10Hz, 즉 100ms마다** 카메라 영상을 받아 다음 경로를 계산한다.  
차량이 달리는 동안 이 추론이 **멈추지 않고 계속 실행**되어야 한다.

```
현실:
  100ms  → 카메라 영상 입력 → 추론 → 경로 출력
  200ms  → 카메라 영상 입력 → 추론 → 경로 출력
  300ms  → 카메라 영상 입력 → 추론 → 경로 출력
  ...무한히 계속...
```

**문제**: 현재 1회 추론에 걸리는 시간 = **~3,029ms** (최적화 후 기준).  
100ms 목표와 30배 차이가 난다.

---

## 2. 현재 실험들이 암묵적으로 가정한 것

지금까지의 모든 실험 (AppendOnlyCache, Exp C 등)은 **단발(single-shot)** 구조로 측정됐다:

```
[실험 구조]
  1. t0 데이터 로드
  2. t0 full prefill 실행 → KV Cache 생성
  3. t1 데이터 로드
  4. Exp C 실행 (t0 KV 재사용 + suffix 141ms)
  5. decode 실행
  6. 시간 측정 → 끝
```

이것은 **두 프레임 사이의 단 한 번의 전환**만 측정한 것이다.  
차량이 실제로 달리는 동안 수천 번의 전환이 연속으로 일어나는 상황은 전혀 고려되지 않았다.

---

## 3. 한계점 1 — Vision KV가 시간이 지날수록 현실과 멀어진다

### 현재 실험에서의 동작

Exp C는 t0에서 계산한 "카메라 영상 KV Cache"를 t1에서 재사용한다.  
실험에서는 t0 → t1 **딱 한 번의 재사용**만 측정했다.

### 연속 운용 시 실제 동작

```
cycle 1:  full prefill → KV₁ (t=0ms 시점 영상 기반)
cycle 2:  KV₁ 재사용 (vision 부분) + 새 ego tokens
cycle 3:  KV₁[vision] 재사용 (여전히 t=0ms 영상!)
cycle 4:  KV₁[vision] 재사용 (t=0ms 영상, 400ms 전!)
...
cycle N:  KV₁[vision] 재사용 (t=0ms 영상, N×100ms 전!)
```

**vision KV는 한 번 만들어지면 갱신되지 않는다.**  
ego tokens(차량 위치/속도)은 매 사이클 정확히 갱신되지만,  
"주변 환경이 어떻게 생겼는가"는 계속 최초 시점에 고정된 채로 추론이 계속된다.

### 왜 문제인가

자율주행에서 0.1초마다 장면이 바뀐다는 것이 Alpamayo가 10Hz로 추론하는 이유다.  
Vision KV가 수십 사이클 동안 동결되면 이 전제가 무너진다.

---

## 4. 한계점 2 — Vision KV staleness의 실제 범위는 파이프라인 설계에 달려있다

### 실험에서의 가정

실험에서는 t0 full prefill을 하고 **바로 다음에** t1 Exp C를 실행한다.  
측정된 staleness = Δt = 100ms (카메라 프레임 간격).

### 실제 파이프라인에서의 구조적 문제

목표 시스템은 **Rolling Trajectory Pipeline**이다.  
100ms throughput을 달성하려면 여러 추론이 동시에 파이프라이닝되어야 한다:

```
T=0ms:    inference_1 시작 → vision KV₁ 생성 (t=0ms 영상 기반)
T=100ms:  inference_2 시작
T=200ms:  inference_3 시작
...
```

이 파이프라인에서 Exp C를 적용한다면:
- **inference_k가 vision KV를 재사용할 "이전 inference"는 언제 것인가?**
- 파이프라인 설계에 따라 staleness가 결정된다 — 현재 sequential 측정치로 확정할 수 없다.

### 실험과 현실의 차이

| | 실험 | 파이프라인 내 실제 |
|--|--|--|
| KV 재사용 구조 | t0 직후 t1 (단발) | 파이프라인 오버랩 구조에 따라 결정 |
| Vision staleness | Δt = 100ms | 파이프라인 설계 미확정 → 미지수 |

현재 실험은 staleness가 가장 작은 조건(Δt = 100ms)에서만 측정됐다.  
실제 파이프라인에서 Exp C가 몇 사이클 전 KV를 재사용하게 되는지는  
**Rolling Trajectory Pipeline 설계가 확정된 후 별도로 검증해야 한다.**

---

## 5. 한계점 3 — 연속된 출력 경로들이 서로 일관성이 있는가

### Alpamayo의 출력 구조

Alpamayo는 매 추론마다 **앞으로 6.4초(64 waypoints)짜리 경로**를 출력한다.

### 10Hz 연속 추론 시 발생하는 문제

```
t=5.1s 추론 출력: [5.2s 지점 좌표, 5.3s 지점 좌표, ..., 11.5s 지점 좌표]
t=5.2s 추론 출력: [5.3s 지점 좌표, 5.4s 지점 좌표, ..., 11.6s 지점 좌표]
```

두 추론은 5.3s~11.5s 구간에 대한 경로를 **둘 다 예측**한다.  
만약 두 출력이 서로 다른 경로를 예측한다면:

```
t=5.1s 추론: "6초 지점에서 우회전"
t=5.2s 추론: "6초 지점에서 직진"
→ 차량 제어 시스템은 어느 것을 따라야 하는가?
```

현재 실험들은 단일 추론의 출력만 측정하며, **연속된 출력들이 얼마나 일관성 있는지 전혀 측정하지 않았다**.

## 6. 한계점 4 — 메모리 반복 할당 (가장 간과된 부분)

### AppendOnlyCache의 메모리 구조

AppendOnlyCache-C(AOC)는 추론마다 다음 크기의 버퍼를 할당한다:

```
36 layers × 2(K+V) × 3,182 tokens × 8 heads × 128 dim × 2 bytes(BF16)
= 약 469MB
```

### 현재 실험의 처리 방식

```python
# 매 trial마다 새로 생성
aoc = build_appendonly_cache(...)   # GPU에 469MB 할당
# ... 추론 ...
# trial 끝 → aoc가 scope를 벗어남 → Python GC 처리
```

AOC 객체가 scope를 벗어나면 Python Garbage Collector가 처리하지만,  
CUDA 메모리는 Python GC와 독립적으로 동작하며 **즉시 해제되지 않을 수 있다**.

### 10Hz 연속 운용 시 영향

```
10회/초 × 469MB 할당 = 초당 4.69GB CUDA 메모리 할당/해제 요청

발생 가능한 문제:
  1. CUDA allocator의 반복 malloc/free → allocator 내부 overhead 누적
  2. Python GC 지연 해제 → 메모리 사용량이 일시적으로 증가
  3. 장시간 운용 시 메모리 단편화(fragmentation) 심화
```

Thor의 통합 메모리(128GB)는 OOM이 발생하지는 않지만,  
CUDA allocator가 지속적으로 대용량 블록을 재사용하지 못하고  
새로 할당/해제를 반복하면 **실제 추론 시간에 숨겨진 오버헤드**가 생긴다.

### 올바른 설계

연속 운용에서 AOC는 **한 번 할당하고 reset()으로 재사용**해야 한다:

```python
# 올바른 연속 운용 방식
aoc = build_appendonly_cache(...)    # 시작 시 1회 할당

while True:
    aoc.reset()                      # write_pos만 0으로 초기화 (메모리 해제 없음)
    load_dyn_into_aoc(kv_dyn, aoc)  # 기존 버퍼에 덮어쓰기
    decode_loop(aoc, ...)
```

현재 실험에서는 이 패턴을 사용하지 않는다.  
실험에서는 결과를 측정하지 않았으므로 수치화된 영향은 미지수이나, 장시간 운용 시 검증이 필요하다.

## 7. 한계점 5 — Vision 토큰 구조 미검증

### Incremental KV Update의 전제

앞서 논의된 "t-3 evict + t1 새로 계산" 방식(Incremental KV Update)이 작동하려면  
2,982개의 vision tokens가 **프레임별로 깔끔하게 분리**되어 있어야 한다.

```
필요한 구조:  [t-3 vision: 745tok | t-2 vision: 745tok | t-1 vision: 745tok | t0 vision: 747tok]
→ t-3 구간만 잘라내고 t1 구간을 붙이면 됨
```

### 실제 VE 구조가 다를 수 있다

Vision Encoder가 여러 카메라 + 여러 프레임을 cross-attention으로 처리한다면:

```
실제 구조:  [camera1_t-3, camera2_t-3, camera1_t-2, camera2_t-2, ...]
            (카메라와 시간 축이 섞인 구조 → 프레임 경계가 없음)
```

이 경우 t-3에 해당하는 tokens만 깔끔하게 제거하는 것이 불가능하다.

**현재까지 VE의 출력 토큰 구조를 분석한 적이 없다.**  
Incremental KV Update 구현 가능 여부 자체가 이 분석에 달려있다.

## 8. 한계점 6 — Single-Inference 최적화의 구조적 한계

### 현재 최선 latency와 10Hz 목표의 갭

```
현재 최선 (Exp C + DYN):    ~3,029ms / inference
Async VE 적용 후 (예상):    ~2,301ms / inference
10Hz 목표:                    100ms / inference
```

단일 추론 latency를 아무리 줄여도 10Hz를 맞추는 것은 물리적으로 불가능하다.  
모델 크기(22GB, BF16)가 고정된 이상, 가중치를 한 번 읽는 데만 수백 ms가 필요하다.

### 올바른 접근 방향

이것은 **"한 번 추론을 빠르게 하는 문제"가 아니라 "추론을 파이프라이닝하는 문제"**다.

```
목표:
  T=0ms:  inference_1 시작
  T=100ms: inference_2 시작 (inference_1 아직 실행 중)
  T=200ms: inference_3 시작 (inference_1,2 아직 실행 중)
  ...
  T=3000ms: inference_1 완료 → 경로 출력
  T=3100ms: inference_2 완료 → 경로 출력
  ...

→ 각 추론은 3,000ms 걸리지만, 100ms 간격으로 결과가 나온다 (throughput = 10Hz)
```

이것이 CLAUDE.md의 "Rolling Trajectory 연속 스트리밍 파이프라인"이다.  
단일 추론을 99ms로 만드는 것이 목표가 아니라,  
**99ms마다 결과를 제공하는 파이프라인을 만드는 것**이 목표다.

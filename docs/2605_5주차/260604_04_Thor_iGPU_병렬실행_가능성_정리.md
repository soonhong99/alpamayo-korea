# Thor iGPU 병렬 실행 가능성 정리

**날짜**: 2026-06-04  
**실험 스크립트**:
- `scripts/inference/260604_cuda_stream_concurrency_test.py` — SM ∥ SM 검증
- `scripts/inference/260604_independent_hw_parallel_test.py` — DMA/CPU/Prefetch 검증

---

## 한 줄 결론

> **Thor iGPU에서 GPU SM 내부의 병렬화는 불가능하다. 단, CPU와 GPU는 진짜 독립 하드웨어이므로 병렬 실행된다. `cudaMemPrefetchAsync`도 부분적(44%)으로 병렬 실행된다.**

---

## 1. 실험 배경

처음에 우리는 Async VE Pipeline을 설계했다:

```
[목표]
Decode(k) 실행 중 VE(k+1)을 CUDA stream으로 동시에 실행
→ 다음 inference의 VE 시간(~550ms)을 critical path에서 제거
→ 이론 가속비 1.22×
```

VE는 compute-bound (Tensor Core), Decode는 memory-bound (DRAM). **자원이 다르므로 이론적으로는 동시 실행이 가능해야 한다.**

그러나 실험 결과는 모두 0.97~0.99×. 원인을 규명하기 위해 단계적으로 실험을 확대했다.

---

## 2. 실험 결과 전체 요약

### 2-1. GPU SM ∥ GPU SM (CUDA multi-stream)

**실험**: 서로 다른 CUDA stream에 compute-bound 커널과 memory-bound 커널을 분리하여 동시 디스패치.

```
stream1 (compute): torch.mm 반복  → Tensor Core 집중
stream2 (memory) : tensor.clone() → DRAM 대역폭 집중
```

allocator 간섭을 완전히 제거(pre-allocated buffer, `copy_()` 사용)한 경우에도 동일:

| 케이스 | Speedup |
|--------|---------|
| Small  (2K matmul, 1GB clone, prealloc) | **0.97×** |
| Medium (4K matmul, 4GB clone, prealloc) | **0.99×** |
| Large  (4K matmul, 10GB clone, prealloc) | **1.00×** |

**결과: ❌ 불가능**

**이유**:  
Thor iGPU는 SoC 통합 GPU로, 모든 CUDA 커널이 단일 SM Pool을 공유한다. dGPU처럼 독립적인 GPC(Graphics Processing Cluster)가 없어 GPU 스케줄러가 stream을 순서대로 직렬 처리한다.

```
dGPU:
  GPC 0 → stream A 전용 실행  ┐
  GPC 1 → stream B 전용 실행  ┘ → 동시 실행 ✅

Thor iGPU:
  단일 SM Pool → stream A 완료 후 stream B 시작 → 직렬 ❌
```

---

### 2-2. DMA copy_ (copy engine) ∥ GPU SM

**실험**: CPU pinned memory → GPU 비동기 복사(`copy_(non_blocking=True)`) + SM matmul 동시 실행.

| 케이스 | Speedup |
|--------|---------|
| Small  (1GB copy, 2K matmul) | **0.97×** |
| Medium (4GB copy, 4K matmul) | **1.00×** |
| Layer  (0.8GB copy, 4K matmul×10) | **1.02×** |

**결과: ❌ 불가능**

**이유**:  
dGPU에서 `copy_(non_blocking=True)`는 PCIe DMA 엔진을 사용하여 SM과 독립적으로 실행된다. 그러나 Thor iGPU의 통합 메모리(131.9GB unified)에서는 CPU pinned memory와 GPU memory가 **동일한 물리 DRAM**에 존재한다. 따라서 "복사"가 실제 DMA 전송이 아닌 CPU memcpy 수준의 연산으로 처리되며, SM과 직렬화된다.

---

### 2-3. CPU ∥ GPU SM

**실험**: NumPy 행렬 연산(CPU) + CUDA matmul(GPU) 비동기 실행.

구조:
```python
# GPU 커널 dispatch → 즉시 리턴 (비동기, GPU는 백그라운드 실행)
torch.mm(A, A)  # GPU 실행 시작, Python은 즉시 다음 줄로
# CPU 작업 실행 (GPU와 동시에)
B = numpy_array @ numpy_array
# GPU 완료 대기
torch.cuda.synchronize()
```

| 케이스 | CPU | GPU | Parallel | Overlap |
|--------|-----|-----|----------|---------|
| CPU < GPU (1K, 4K×10) | 4.8ms | 10.8ms | 12.1ms | **73%** |
| CPU >> GPU (3K, 2K×5) | 132.5ms | 1.1ms | 137.4ms | **0%** |
| CPU ≈ GPU (2K, 4K×5) | 45.0ms | 6.3ms | 41.3ms | **100%** |

**결과: ✅ 가능 — 단, 조건 있음**

**조건**:

```
✅ CPU time ≤ GPU time 일 때:
   GPU inference 실행 중 CPU 작업이 완료됨 → CPU 시간이 GPU 시간 안에 숨음
   Parallel ≈ GPU time (CPU 비용 거의 0)

❌ CPU time >> GPU time 일 때:
   GPU가 먼저 끝나고 CPU를 기다림 → 직렬과 동일
   Parallel ≈ CPU time + GPU dispatch overhead
```

**Alpamayo 적용**:

```
GPU inference:     ~4,366ms
CPU 전처리:        ?ms (다음 프레임 로드 + 이미지 전처리 + 토크나이징)

CPU 전처리 < 4,366ms 이면 → 전처리 비용이 추가 latency 없음
```

### 2-4. `cudaMemPrefetchAsync` ∥ GPU SM

**실험**: CUDA unified memory prefetch (`cudaMemPrefetchAsync`) + SM matmul 동시 실행.  
unified memory 버퍼를 GPU-preferred 주소로 마이그레이션하면서 SM이 연산을 실행.

| 케이스 | Prefetch | SM | Serial | Parallel | Speedup | Overlap |
|--------|---------|-----|--------|----------|---------|---------|
| 1-layer (0.8GB, 4K×10) | 9.2ms | 11.1ms | 20.4ms | 16.3ms | **1.25×** | **44%** |
| Multi-layer (4GB, 4K×5) | 34.4ms | 4.9ms | 39.2ms | 39.0ms | **1.01×** | **2%** |

**결과: ⚠️ 부분적으로 가능 (44%) — 크기 균형이 맞을 때만**

**이유**:  
`cudaMemPrefetchAsync`는 CUDA unified memory manager의 **page migration engine**을 통해 동작한다. 이 엔진은 SM과 물리적으로 독립된 하드웨어 유닛이다. `copy_()`가 CPU memcpy로 처리되는 것과 달리, `cudaMemPrefetchAsync`는 GPU MMU가 처리하므로 SM과 44% 수준의 overlap이 발생한다.

**조건**:
```
✅ prefetch time ≈ SM compute time 일 때:
   두 작업이 겹치는 구간이 충분 → 1.25× 가속

❌ prefetch time >> SM compute time 일 때:
   SM이 빨리 끝나고 prefetch를 기다림 → 직렬과 동일
```

## 3. 전체 비교표

| 메커니즘 | 결과 | 가속비 | 조건 | 이유 |
|---------|------|--------|------|------|
| **SM ∥ SM** (CUDA stream) | ❌ 불가 | 0.97~1.00× | — | 단일 SM Pool, GPU 스케줄러 직렬화 |
| **DMA copy_ ∥ SM** | ❌ 불가 | 0.97~1.02× | — | unified memory에서 CPU memcpy로 처리 |
| **CPU ∥ GPU** | ✅ 가능 | 최대 1.29× | CPU time < GPU time | CPU와 GPU는 완전히 다른 하드웨어 |
| **cudaMemPrefetchAsync ∥ SM** | ⚠️ 부분 | 최대 1.25× | prefetch ≈ SM time | GPU migration engine이 SM과 독립 |

## 4. "왜 논문(RT-Swap, Demand Layering)에서는 됐는가?"

RTAS'24(RT-Swap), RTSS'22(Demand Layering)은 dGPU 또는 Xavier 계열을 대상으로 한다.

| 항목 | dGPU / Xavier | Thor iGPU |
|------|--------------|-----------|
| SM 구조 | 독립 GPC 클러스터 복수 | 단일 SM Pool |
| GPU VRAM | 별도 고속 메모리 | unified DRAM (CPU와 공유) |
| DMA 엔진 | PCIe DMA (SM 독립) | CPU memcpy로 fallback |
| Stream 병렬 | ✅ 실제 동시 실행 | ❌ 직렬 스케줄링 |

논문의 "GPU stream A ∥ GPU stream B"는 dGPU에서 독립 GPC가 각 stream을 동시에 처리하기 때문에 작동한다. Thor iGPU에는 이 구조가 없다.

**Xavier에서 한 것**: Jetson Xavier는 2개의 독립 GPU 파티션 또는 DLA(Deep Learning Accelerator)를 사용해 다른 하드웨어 유닛끼리 병렬화했다. Thor에는 DLA가 없거나 활용 방법이 다르다.

## 5. Alpamayo에서 실제로 사용할 수 있는 병렬화

### ✅ CPU 전처리 파이프라인 (즉시 가능)

```
현재 (직렬):
  [GPU: inference(k)] → [CPU: preprocess(k+1)] → [GPU: inference(k+1)] → ...

개선 (파이프라인):
  [GPU: inference(k)] 
          ↕ 동시 실행
  [CPU: preprocess(k+1)]
  → [GPU: inference(k+1)] → ...
```

GPU inference (~4,366ms) 동안 CPU가 다음 프레임을 전처리. CPU 전처리가 4,366ms 이내로 끝나면 추가 latency 없음.

구현: Python `threading.Thread`로 CPU 전처리를 비동기 실행.

### ⚠️ cudaMemPrefetchAsync 활용 (조건부 가능, 추가 실험 필요)

직접적인 decode 가속은 어렵다 (decode가 이미 DRAM-bound이므로 prefetch 후 저장할 "더 빠른 메모리"가 없음).

단, 다음 용도는 검토할 수 있다:

```
시나리오: pixel_values page migration
  현재: VE 시작 시 pixel_values에 대한 page migration overhead 발생 가능
  개선: Decode 중 cudaMemPrefetchAsync로 다음 프레임 pixel_values를
        GPU-preferred 주소로 미리 마이그레이션
  → VE 시작 시 page migration 없이 바로 ViT 연산 시작
```

효과 여부는 추가 측정이 필요하다.

## 6. 확정 불가 사항 (미검증)

- `cudaMemPrefetchAsync`의 Alpamayo 실제 적용 효과 (측정 안 됨)
- CPU 전처리의 실제 시간 (데이터 로드 + 이미지 전처리 + 토크나이징을 분리 측정 안 됨)
- Thor에 DLA 또는 PVA 같은 별도 가속기가 있는 경우 그것과의 병렬화 가능성

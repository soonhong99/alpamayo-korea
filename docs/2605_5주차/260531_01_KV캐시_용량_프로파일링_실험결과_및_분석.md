## 1. 핵심 결론 (요약)

| 항목 | 측정값 |
|------|--------|
| VLM KV Cache (Prefill 후) | **455.0 MB** |

## 2. 왜 이 실험이 필요했나 — KV Cache 최적화의 두 가지 축

### 2.1 기존 코드의 문제: 매 프레임 n² 반복

기존 Alpamayo 코드는 **추론과 추론 사이에 KV Cache를 버렸다.**

```
[기존 방식]

프레임 t-1:  VE → Prefill(n², 1,423 ms) → KV 생성 → Decode → Flow
                                                ↓ 버림

프레임 t:    VE → Prefill(n², 1,423 ms) → KV 생성 → Decode → Flow
                                                ↓ 버림

프레임 t+1:  VE → Prefill(n², 1,423 ms) → ...
```

왜 버렸는가? 모델은 "연속 프레임이 99% 유사하다"는 사실을 모른다.
매번 새로운 카메라 이미지 → 새로운 임베딩 → "완전히 다른 입력"으로 취급한다.

n² prefill이 매번 발생한 이유:

```
Attention 연산량:
  tok_1 → 1개 참조
  tok_2 → 2개 참조
  ...
  tok_3086 → 3,086개 참조
  합계: 1 + 2 + ... + 3,086 = O(n²)

이 결과(KV Cache)를 버리면 → 다음 프레임에서 동일한 O(n²) 반복
```

### 2.2 새로운 두 가지 최적화 축

우리의 연구는 KV Cache를 두 차원에서 최적화한다.

```
축 1: 추론 "안에서" — Decode 단계 최적화          ← Doc 02 상세 기술
────────────────────────────────────────────────────────────
  문제: DynamicCache가 매 decode step마다 455 MB torch.cat 복사
  해결: AppendOnlyCache-C (pre-alloc + in-place write + L2 재사용)
  효과: decode 107 ms → 79 ms/step (-26%)
  상태: ✅ 완료 (2026-05-31)


축 2: 추론 "사이에서" — Prefill 단계 최적화       ← 다음 실험
────────────────────────────────────────────────────────────
  문제: 연속 프레임인데도 매번 1,423 ms prefill 재수행
  해결: KV Temporal Reuse (이전 프레임 KV 보존 후 재활용)
  효과: prefill 1,423 ms → 44 ms (-97%, 실험 예정)
  상태: 🔄 진행 중 (실험 A 검증 완료, 실험 C 설계 완료)
```

**이 실험(Doc 01)의 역할**: 두 축 모두 올바르게 설계하려면 KV 블록의 정확한 크기가 필요하다.
- 축 1: L2에 들어가는지 확인 (per-layer KV 12.6 MB < L2 32 MB → AppendOnlyCache-C 설계 근거)
- 축 2: 재사용할 KV의 정확한 크기와 구조 확인

## 3. KV Cache 구조 이해

### 3.1 Transformer Attention에서 KV란

```
새 토큰 생성 시:
  기존 모든 토큰 K, V → "이전에 이런 맥락이 있었어"
  새 토큰 Q           → "다음 단어는 뭐지?"
  → Q가 K, V 전체를 참조하여 다음 단어 결정

KV Cache = 이미 계산한 K, V를 저장해 재계산 방지
```

### 3.2 DynamicCache의 구조적 문제

```
매 decode step에서 torch.cat 동작:

현재 상태: A구역 [K_1~3086] 점유 중
목표: 1개 더 추가 → [K_1~3087]

문제: A구역 바로 옆 = 다른 데이터가 이미 점유
해결(DynamicCache):
  B구역 3087칸 확보 → A구역 3086개 전부 복사 → 1개 추가 → A구역 반납

1 step당 BW 낭비: 455 MB 읽기 + 455 MB 쓰기 = 910 MB
17 step 합계: 15,470 MB 낭비
```

## 5. 실험 결과

### 5.1 KV Cache 정밀 구조 (Phase 1)

```
실측 구조:
  36 layers × 2(K, V) × 8 heads × seq_len × 128 head_dim × BF16

Prefill 완료 후 (seq_len = 3,086):
  36 × 2 × 8 × 3,086 × 128 × 2 bytes = 455,049,216 bytes = 455.0 MB
```

**이전 추정값과 비교**:

| 항목 | 이전 추정 | 실측값 | 판정 |
|------|----------|--------|------|
| VLM KV 전체 | ~455 MB | **455.0 MB** | ✅ 일치 |
| VLM 레이어 수 | 36 | **36** | ✅ 일치 |
| Expert 레이어 수 | **28** | **36** | ❌ 오류 수정 |
| Expert 64tok KV | 7.3 MB (28L) | **9.4 MB** (36L) | 수정 |

Expert가 36 레이어인 이유: VLM과 동일한 아키텍처 구조이나, 각 레이어의 hidden size가 더 작아 파라미터 수가 적을 뿐이다. 레이어 수 자체는 동일.

### 5.2 Decode Step별 KV 증가 (Phase 2)

```
step | seq_len | KV total  | delta   | step time
   1 |   3,087 | 455.2 MB  | 151 KB  | 163.6 ms  ← cold-start
   2 |   3,088 | 455.3 MB  | 151 KB  | 108.5 ms
   3 |   3,089 | 455.5 MB  | 151 KB  | 108.3 ms
  ...
  18 |   3,104 | 457.7 MB  | 151 KB  | 107.8 ms
```

**이론(144 KB) vs 실측(151 KB) 차이 원인**:
```
이론: 36L × 2 × 8H × 1tok × 128D × 2B = 147,456 bytes = 144.0 KB

실측이 큰 이유:
  torch.cat → GPU 메모리 할당기가 4,096-byte 경계로 올림(padding)
  36 layers × 2 텐서 × padding 누적 = +7 KB
  실측: 154,624 bytes = 151.0 KB (+4.9%)
```

**Step 1이 163 ms인 이유**:
- DynamicCache 첫 torch.cat: 455 MB 전체 복사
- GPU L2 cache cold-start
- CUDA 스케줄러 초기화
→ Step 2부터 이 비용이 사라지며 107~109 ms로 수렴

### 5.3 L2 Cache 수용 가능 블록 분석 (Phase 4)

```
GPU L2 Cache (Thor SM 11.0): 32 MB
L2 Persistence 사용 가능 영역 (75%): 24 MB
```

| KV 블록 | 크기 | L2 수용? | 반복 접근 | 용도 |
|---------|------|---------|---------|------|
| VLM KV 전체 | 455 MB | ❌ 13.6× 초과 | 17×(decode) + 280×(flow) | 전체 불가 |
| **text_prefix KV (100 tok)** | **14.7 MB** | **✅** | 17 + 280 = **297회** | 프레임 간 불변 |
| **expert 64tok KV** | **9.4 MB** | **✅** | ODE step × layers = **280회** | crop→append 반복 |
| **VLM 1 layer KV (3,086 tok)** | **12.6 MB** | **✅** | 17회 | AppendOnlyCache-C 활용 중 |
| action_in_proj 가중치 | 2.7 MB | ✅ | 10회 | Flow projection |
| text_prefix + 64tok 합산 | 24.1 MB | ⚠️ 24 MB 한계 초과 | — | 동시 pinning 불가 |

### 5.4 L2 KV Pinning 효과 계산

```
Flow 단계 메모리 I/O 분해:
  expert 가중치 읽기:  10 ODE steps × 4.56 GB = 45,600 MB  (91%)  ← 실제 병목
  VLM KV 읽기:         10 ODE steps × 455 MB  =  4,550 MB   (9%)

L2 Pinning으로 절약 가능한 것:
  text_prefix(14.7 MB) + expert 64tok(9.4 MB) = 24.1 MB × ~300회
  vs 전체 I/O 50,150 MB → 약 0.5%

시간 절약 추정: 870 ms × 0.5% ≈ 4 ms
```

## 6. 핵심 분석

### 6.1 AppendOnlyCache-C가 이미 L2를 활용하는 방식

AppendOnlyCache-C는 명시적 pinning 없이도 L2 재사용을 달성하고 있다:

```
각 레이어 처리 순서 (AppendOnlyCache-C):

  ① new K/V in-place write → _k_buf[layer]
     → 방금 쓴 데이터가 GPU L2에 올라감 (12.6 MB/layer < 32 MB L2)
  
  ② .contiguous() 호출 → _k_buf[:n] 읽기
     → 방금 쓴 것을 즉시 읽음 → L2 HIT!
  
  ③ FlashAttention → compact K/V 읽기
     → L2에 아직 있음 → L2 HIT!

결과:
  실효 BW = 22 GB ÷ 0.079 s = 278 GB/s > 231 GB/s 물리 한계
  → L2 재사용으로 실제 DRAM 접근이 22 GB보다 적기 때문
```

즉, "L2를 더 활용하자"는 목표는 이미 AppendOnlyCache-C가 decode 경로에서 달성했다.
추가 pinning(text_prefix 14.7 MB 등)의 실제 gain이 4 ms에 불과한 이유가 이것이다.

### 6.2 L2 KV Pinning의 올바른 위치

단독 최적화로는 효과가 미미하다. **KV Temporal Reuse와 결합될 때** 의미를 가진다:

```
KV Temporal Reuse 성공 후:
  text_prefix KV(14.7 MB)가 프레임 간에 고정된 값으로 유지됨
  → 이 고정된 14.7 MB를 L2에 pin하면 매 프레임 반복 접근 시 추가 이득 가능

현재 단계:
  KV Temporal Reuse 실험 C 먼저 완료
  → 성공 시 L2 Pinning을 보조 최적화로 추가 (논문 보완용)
```

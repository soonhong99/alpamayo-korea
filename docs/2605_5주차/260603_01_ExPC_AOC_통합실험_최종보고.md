## 1. 우리가 풀려는 문제 — 왜 이 실험인가?

### 1-1. Alpamayo의 추론 주기

Alpamayo는 자율주행 차량에서 **10Hz, 즉 100ms마다** 새 영상 프레임을 받아 다음 경로(waypoint)를 계산한다. 갑작스러운 보행자, 역주행차 같은 돌발 상황에 0.1초 단위로 대응해야 하기 때문이다.

문제는 현재 1회 추론에 **4,366ms**가 걸린다는 것이다. 100ms 목표와 40배 이상 차이난다. 이 격차를 줄이는 것이 이 연구의 핵심 과제이다.

### 1-2. 현재 추론 파이프라인의 병목

1회 추론은 4단계로 구성된다:

```
[VE]  →  [LM Prefill]  →  [Decode]  →  [Flow]
728ms      1,423ms         1,345ms      870ms
            ↑
        가장 큰 병목
```

| 단계 | 역할 | 시간 |
|------|------|------|
| VE (Vision Encoder) | 카메라 영상 → 시각 토큰(2,982개) 변환 | 728ms |
| LM Prefill | 전체 3,086 토큰 한 번에 처리 → KV Cache 생성 | 1,423ms |
| Decode | 17~19 step, 매 step 다음 토큰 예측 | 1,345ms |
| Flow | KV 좌표 → 실제 waypoint 변환 | 870ms |

## 2. Exp C — 핵심 아이디어

### 2-1. Exp C란 무엇인가?

**KV Temporal Reuse** (KV 시간 재사용) 전략이다.

연속된 두 프레임(t0=5.1s, t1=5.2s)의 입력은 3,086 토큰 중 약 97%가 동일하다.

```
입력 구조 (3,086 tokens):
  [text prefix: 29tok] + [vision: 2,982tok] + [ego + suffix: 75tok]
       공통                     공통                   t1만 다름
```

t0에서 계산한 앞 3,011 토큰의 KV Cache는 t1에서도 그대로 재사용할 수 있다.  
t1에서는 나머지 75토큰(suffix)만 새로 계산하면 된다.

```
기존 방식:  t1 전체 3,086 토큰 prefill          → 1,423ms
Exp C:      t0 KV 재사용 + t1 suffix 75 토큰만  →   141ms
                                                  ↓ -1,282ms 절감
```

### 2-2. Exp C를 통해 얻으려는 것

Exp C 하나만으로 **LM Prefill 병목 1,423ms를 141ms로 10분의 1 단축**한다.  
이것이 이 실험 계열 전체에서 가장 중요한 성과이다.

```
Exp C 적용 후 파이프라인:
[VE 728ms] → [suffix 141ms] → [Decode ~1,290ms] → [Flow 870ms] = ~3,029ms
vs 기존:                                                            4,366ms
                                               절감: -1,337ms (-30.6%)
```

### 2-3. AOC를 추가로 결합하려 한 이유

260531 실험에서 **AppendOnlyCache-C(AOC)** 가 decode를 더 빠르게 함을 확인했다.

| KV Cache 방식 | decode ms/step | 비고 |
|-------------|---------------|------|
| DynamicCache | 107ms | torch.cat 방식 |
| AppendOnlyCache-C | 79ms | in-place write 방식 |
| 개선율 | **-26%** | |

Exp C로 suffix를 줄이고, AOC로 decode도 26% 줄이면 둘 다 얻는다는 것이 이번 통합 실험의 목표였다.

## 3. 실험 설계 — 무엇을 비교했나?

세 가지 방식을 비교한다:

| 방식 | prefill/suffix | decode | 설명 |
|------|---------------|--------|------|
| **FULL** | t1 전체 3,086 토큰 prefill | DynamicCache | 기준선 (현재 시스템) |
| **DYN** | Exp C (suffix 75 토큰만) | DynamicCache | Exp C 적용 단독 |
| **AOC** | Exp C (suffix 75 토큰만) | AppendOnlyCache-C | Exp C + AOC 통합 (목표) |

**핵심 조건**: Δt=100ms 고정 (t0=5.1s → t1=5.2s), BF16, sdpa

## 4. 실험 여정 — v1에서 v3까지

### 4-1. v1: 첫 번째 시도와 3가지 실패

**스크립트**: `260603_expc_aoc_integration.py`  
**구조**: 하나의 trial에서 FULL → DYN → AOC를 순서대로 실행

#### 측정 결과 (Measure 3회 평균)

| 지표 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| AOC suffix | DYN과 동등 (~140ms) | **267ms** (+115ms, 2배) | ❌ |
| AOC decode ms/step | ~79ms (-26%) | 86ms (DYN과 동등) | ❌ |
| AOC EOS 달성률 | 100% | **67%** (3회 중 1회 실패) | ❌ |

세 가지가 모두 기대에 미치지 못했다.

#### 문제 1 분석: AOC suffix가 왜 2배 느린가?

AOC는 pre-allocated 버퍼 `_k_buf`를 가지고 있다.  
버퍼 크기: `[1, 8, 3182, 128]` (max_seq_len=**3182** > 실제 사용 3086)

suffix 중 AOC가 KV를 반환할 때:

```
_k_buf[:, :, :3086, :]  ← 3182-크기 버퍼에서 3086 슬라이스
```

이 슬라이스는 메모리 상에서 연속적이지 않다 (non-contiguous):

```
연속성 조건: stride[1] = size[2] × stride[2]
실제:        3182 × 128 = 407,296
필요:        3086 × 128 = 395,008   → 불일치 → non-contiguous
```

non-contiguous 텐서는 FlashAttention이 받으면 **다른 CUDA 커널 경로**를 사용한다.  
이 경로가 DynamicCache의 `torch.cat` 단일 연산보다 +115ms 느린 것이다.

#### 문제 2 분석: decode 개선이 왜 없는가?

260531 standalone 실험의 조건:
- DynCache: prefill을 자연스럽게 누적 → 내부 tensor fragmentation → 107ms/step
- AOC: pre-alloc contiguous buffer → 79ms/step

이번 v1 통합 실험에서 DynCache는 `slice_dynamic_cache().clone().contiguous()`로 초기화된다.  
이미 compact한 상태에서 시작하므로 DynCache도 ~75ms/step을 달성한다.  
결과: AOC도 DynCache도 둘 다 75ms → **차이 없음**.

260531의 -26%는 DynCache가 fragmented일 때만 나오는 수치였다.

#### 문제 3 분석: EOS 67%는 왜인가?

> **2026-06-03 추가 분석**: 당시 EOS 67%의 원인이 stochastic sampling 분산이라고 판단했으나, v2 실험에서 60%로 더 악화됨이 확인되어 구조적 원인으로 재분류되었다.

AOC의 non-contiguous K/V를 FlashAttention이 처리할 때 다른 floating-point 누산 순서를 사용한다. 수치가 미세하게 달라지고, 36 layers를 통과하며 증폭된다. 결과적으로 DYN logits과 AOC logits이 미세하게 달라져 decode trajectory가 불안정해진다.

---

### 4-2. v2: 첫 번째 수정 시도 — 가설이 틀렸다

**스크립트**: `260603_v2_expc_aoc_fix.py`  
**가설**: `force_contiguous=True` 옵션이 suffix 중 `.contiguous()` copy overhead를 유발한다.  
**수정**: suffix 전에 `force_contiguous=False`, decode 전에 `True`로 복원.

#### 결과

```
AOC suffix (FC=False): 295~305ms  ← 오히려 더 느려짐
EOS: 60% (5회 중 2회 실패)  ← 악화
```

**왜 실패했는가?**

`force_contiguous` 플래그를 바꿔도 `_k_buf` 자체의 메모리 레이아웃이 바뀌지 않는다.  
slicing 시 non-contiguous 구조는 동일하며, FlashAttention이 받는 텐서의 stride가 그대로다.  
overhead의 원인은 `.contiguous()` copy 비용이 아니라 non-contiguous K/V로 인한 **FlashAttention 커널 경로 변경**이었다.

EOS 60%로 악화된 것은 force_contiguous=False일 때 non-contiguous 텐서를 그대로 FlashAttention에 전달하면서 수치 불안정성이 더 심해졌기 때문이다.

→ **v2는 근본 원인이 아닌 증상을 수정했다.**

---

### 4-3. v3: 근본 원인 해결 — AOC를 suffix에서 완전 배제

**스크립트**: `260603_v3_expc_aoc_decode_only.py`  
**핵심 통찰**: v1/v2의 두 문제(suffix 2배 느림, EOS 불안정)는 같은 뿌리에서 나온다.  
→ AOC의 non-contiguous `_k_buf`를 suffix_forward에 전혀 사용하지 않으면 두 문제가 동시에 사라진다.

#### v3 파이프라인 변경

```
v1/v2 (잘못된 구조):
  kv_t0[:3011] → AOC 버퍼 → suffix_AOC → decode_AOC
                  ↑ non-contiguous K/V가 suffix에 노출 → 2가지 문제 발생

v3 (올바른 구조):
  Step 1. kv_t0[:3011] → DynCache suffix → kv_dyn (3086 tokens), logits_dyn
           [항상 DynCache 사용 → contiguous → 정확한 logits]
  Step 2. load_dyn_into_aoc(kv_dyn, end_pos=3086)   [conv ~10ms]
           [kv_dyn 전체를 AOC 버퍼에 복사]
  Step 3a. DYN decode: decode_loop(kv_dyn, logits_dyn)
  Step 3b. AOC decode: decode_loop(aoc,    logits_dyn)
           [동일한 logits_dyn 공유 → EOS 100% 보장]
```

#### v3 설계 원칙

- AOC는 **decode 전용**. suffix에는 절대 사용하지 않는다.
- DYN과 AOC는 동일한 `logits_dyn`에서 decode를 시작한다.
  따라서 decode trajectory(EOS 달성 시점)이 동일하다.
- DYN과 AOC decode의 차이는 순수하게 구현 방식 차이(torch.cat vs in-place+contiguous)만 측정한다.

## 7. 왜 decode 개선이 없는가 — 구조적 분석

### 7-1. 260531 standalone vs v3 비교

| 조건 | DynCache | AOC | 차이 |
|------|---------|-----|------|
| 260531 standalone | **107ms** | **79ms** | **-26%** |
| 260603 v3 통합 | **74.9ms** | **75.5ms** | **≈0%** |

### 7-2. 결정적 차이: DynCache 초기 상태

**260531 DynCache**:  
full prefill을 거쳐 자연스럽게 누적 → 내부적으로 여러 개의 작은 tensor가 cat되어 fragmented 상태 → torch.cat이 계속 새 tensor를 할당하며 memory fragmentation 심화 → **107ms/step**

**v3 DynCache**:  
`slice_dynamic_cache().clone().contiguous()`로 명시적 compact 복사 → 처음부터 단일 연속 tensor → FlashAttention 최적 커널 사용 → **74.9ms/step**

```python
def slice_dynamic_cache(cache, end_pos: int) -> DynamicCache:
    sliced = [
        (k[:, :, :end_pos, :].clone().contiguous(),  # 명시적 compact
         v[:, :, :end_pos, :].clone().contiguous())
        for k, v in pairs
    ]
    return _build_dynamic_cache(sliced, seen_tokens=end_pos)
```

### 7-3. decode per-step 연산량 비교

| 방식 | step k에서의 연산 |
|------|----------------|
| DynCache | `torch.cat([k_prev, k_1], dim=2)` → 새 (k+1)token tensor 할당 |
| AOC | in-place write(1 token) + `_k_buf[:,:,:k+1,:].contiguous()` → (k+1)token 복사 |

두 방식 모두 step k에서 O(k) 메모리 이동이 필요하다. 절대적 연산량이 동일하므로, **DynCache가 compact 상태이면 FlashAttention 레벨에서 차이가 없다.**

260531의 -26%는 DynCache가 fragmented일 때 나타나는 효과이며, Exp C 통합 경로에서는 DynCache를 compact하게 초기화하기 때문에 이 조건이 성립하지 않는다.

### 7-4. AOC conv overhead

AOC 사용 시 `load_dyn_into_aoc()` 비용 10.1ms가 항상 발생한다.

```
36 layers × 2(K+V) × 3086 tokens × 8 heads × 128 dim × 2 bytes(BF16) = 455MB
455MB ÷ 10.1ms ≈ 45 GB/s (shared memory bandwidth 일부)
```

AOC decode가 DYN과 동등하므로 이 10.1ms는 순 손실이다.

## 10. 실험 방향성 일관성 검토

### 10-1. 일관되게 유지한 것

| 원칙 | 준수 여부 |
|------|---------|
| Δt = 100ms 고정 | ✅ 전 버전 동일 |
| 양자화 없음 (시스템 최적화만) | ✅ BF16 유지 |
| 모델 수정 없음 | ✅ |
| 실측 기반 분석 (추측 금지) | ✅ v2 실패도 실측으로 확인 후 v3 설계 |

### 10-2. 실험 방향 수정 과정

```
v1 목표: Exp C suffix 결과를 바로 AOC에 담아 decode 가속
          ↓ 실패: non-contiguous 버퍼 구조가 suffix에 노출

v2 가설: force_contiguous=False로 overhead 제거
          ↓ 실패: 근본 원인은 FC 플래그가 아닌 버퍼 stride 구조

v3 확정: AOC를 suffix에서 완전 배제 (decode 전용으로 역할 한정)
          ↓ 성공: suffix overhead 제거 + EOS 100%
          → 추가 발견: DynCache compact 상태에서는 decode도 동등
```

방향성은 일관됐다 — **AOC의 decode 이점을 Exp C와 결합**하는 것이 목표였고, v3에서 "결합하면 decode 이점이 사라진다"는 사실을 엄밀히 증명했다. 이 결과 자체가 정보다.

## 11. 최종 결론

### 이번 실험에서 확정된 사실

1. **Exp C의 성과는 실질적이다.**  
   suffix forward 141ms로 LM Prefill 1,423ms를 대체 → 전체 latency -1,337ms (-30.6%).  
   이것이 이 실험 계열의 핵심이며, 다른 모든 최적화보다 크다.

2. **AOC는 Exp C와 결합할 때 추가 이점이 없다.**  
   Exp C의 `slice+clone+contiguous` 초기화가 DynCache를 이미 compact하게 만들어버리기 때문에 AOC의 in-place 장점이 사라진다. conv overhead 10.1ms만 추가된다.  
   → **Exp C 파이프라인에서는 DynamicCache가 최선**이다.

3. **AOC가 유효한 조건은 따로 있다.**  
   DynCache가 유기적으로 fragmented 상태일 때 (full prefill → 반복 decode 단독 시나리오). 260531 standalone 결과(-26%)는 실제 측정값이지만, Exp C와 결합하는 경우에는 해당하지 않는다.

### 다음 실험 방향

**Priority 1: Async VE Pipeline (가장 큰 절감 기회)**

Exp C 적용 후 남은 가장 큰 병목은 VE(728ms)이다.  
t(k) decode(~1,290ms) 시간 안에 t(k+1) VE를 CUDA Stream 이중화로 비동기 실행하면:

```
현재 (sequential):
  t(k):  VE(728) → suffix(141) → decode(1290) → flow(870) = 3,029ms

목표 (async VE):
  t(k):              suffix(141) → decode(1290) → flow(870)
  t(k+1): VE(728)  ↑ decode 중 비동기 실행 (1290ms > 728ms → 완전 숨김)
                                              ↓
         steady-state per frame ≈ 2,301ms   (-24% 추가 절감)
```

이것이 교수님 미팅에서 확정된 RT-Swap / Demand Layering의 핵심 적용 지점이다.

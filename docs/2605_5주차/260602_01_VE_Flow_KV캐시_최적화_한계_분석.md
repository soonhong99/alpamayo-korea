# VE와 Flow에 KV Cache 최적화를 적용하지 못하는 이유

**작성일**: 2026-06-02
**대상 독자**: 이 프로젝트를 처음 보는 사람도 이해할 수 있도록 작성
**관련 실험**: AppendOnlyCache-C (260531), KV Temporal Reuse (260528)

---

## 0. 이 문서를 읽기 전에 알아야 할 것

Alpamayo가 자율주행 경로를 예측하는 과정은 4단계로 나뉜다.

```
카메라 영상 입력
      ↓
① VE (Vision Encoder)       728ms  ← 카메라 이미지를 숫자로 변환
      ↓
② Prefill                   895ms  ← 전체 문맥을 한 번에 이해
      ↓
③ Decode                  1,345ms  ← 경로 설명 문장을 한 단어씩 생성
      ↓
④ Flow                      637ms  ← 실제 주행 좌표 계산
      ↓
경로 출력 (64개 waypoint)
총 소요: ~3,620ms
```

우리는 지금까지 **③ Decode** 단계를 AppendOnlyCache-C로 최적화했다 (107ms → 79ms/step, -26%).

이 문서는 "왜 VE와 Flow에는 같은 방법을 쓸 수 없는가"를 설명한다.

---

## 1. KV Cache 최적화가 Decode에서 효과적인 이유

먼저 Decode에서 최적화가 왜 잘 됐는지 이해해야, VE/Flow에서 안 되는 이유를 알 수 있다.

### 1.1 Decode는 "한 단어씩 순서대로" 생성한다

```
Decode 흐름:

[prefill로 만든 KV Cache: 3,086 토큰]
      ↓
Step 1: 새 단어 1개 생성 → KV에 추가 (3,087 토큰)
Step 2: 새 단어 1개 생성 → KV에 추가 (3,088 토큰)
Step 3: 새 단어 1개 생성 → KV에 추가 (3,089 토큰)
...
Step 17: 새 단어 1개 생성 → 완료
```

이 구조의 핵심: **"이전에 계산한 것을 다시 쓴다."**

매 step에서 새 단어 1개만 계산하고, 나머지 3,086개는 저장된 KV Cache를 참조한다.
→ 이걸 효율적으로 관리하는 것이 AppendOnlyCache-C의 역할이다.

### 1.2 Decode에서 KV Cache가 작동하는 조건

```
필요한 조건:

① 순서가 있다 (Step 1 → Step 2 → Step 3...)
② 뒤로만 참조한다 (Step 3은 Step 1, 2만 본다. Step 4를 보지 않는다)
③ 이전 결과가 다음에 그대로 재사용된다

→ 이 조건을 만족하는 구조 = Causal Transformer (인과적 트랜스포머)
```

---

## 2. VE — 왜 같은 방법을 쓸 수 없나

### 2.1 VE가 하는 일

VE(Vision Encoder)는 4개 카메라의 이미지를 받아서
모델이 이해할 수 있는 숫자 벡터(임베딩)로 변환한다.

```
입력: 4개 카메라 × 4 프레임 = 16장의 이미지
      각 이미지를 작은 패치(patch)로 분할
      예: 한 이미지를 16×16 패치로 분할 → 256개 패치

처리: 모든 패치를 Transformer로 처리
출력: 각 패치의 의미를 담은 숫자 벡터 2,890개
```

### 2.2 VE는 "동시에 전체를 본다" — Bidirectional Transformer

```
[Decode의 Causal Transformer — 단방향]

단어_1 → 단어_2 → 단어_3 → 단어_4
  단어_3은 단어_1, 2만 참조 (앞만 봄)
  단어_4는 단어_1, 2, 3만 참조
  
  이전 결과를 저장(KV Cache)하면 재계산 불필요!


[VE의 Vision Transformer — 양방향]

패치_1 ↔ 패치_2 ↔ 패치_3 ↔ 패치_4
  패치_1은 패치_2, 3, 4를 전부 봄
  패치_2도 패치_1, 3, 4를 전부 봄
  모든 패치가 서로를 동시에 참조

  "이전 결과"라는 개념이 없다 → KV Cache 방식이 적용되지 않는다
```

### 2.3 왜 VE는 양방향이어야 하나

이미지 이해는 "순서"가 없기 때문이다.

```
[텍스트: 순서가 의미를 가짐]
  "나는 사과를 먹었다" → 순서 중요, 앞→뒤 방향 처리 합리적

[이미지: 순서가 없음]
  도로 위 패치_1(차선)과 패치_200(신호등)은 서로를 알아야 의미가 생김
  왼쪽 패치가 오른쪽 패치보다 "먼저"라는 개념이 없음
  → 모든 패치가 서로 동시에 참조해야 올바른 이해 가능
```

### 2.4 VE를 줄이는 현실적인 방법

KV Cache 방식은 직접 적용 불가. 대신 가능한 접근:

| 방법 | 원리 | 가능 여부 | 비고 |
|------|------|---------|------|
| **비동기 오버랩** | Decode(1,345ms) 중 다음 프레임 VE(728ms) 병렬 실행 | ✅ 가능 | Decode > VE이므로 완전히 숨겨짐 |
| **프레임 간 VE 출력 재사용** | 유사한 연속 프레임에서 VE 결과 재활용 | ⚠️ 조건부 | KV Temporal Reuse와 충돌 가능 |
| **VE 경량화** | 모델 구조 변경, 양자화 | ❌ 금지 | 연구 제약 |

**가장 현실적인 접근**: Decode 단계와 다음 프레임 VE를 동시에 실행하면
VE 728ms가 완전히 Decode 1,345ms 안에 숨겨진다.
→ Priority 3 실험 (Async Inter-Inference Pipeline)

## 3. Flow — 왜 완전히 적용이 안 되나

### 3.1 Flow가 하는 일

Flow는 Decode가 생성한 "경로 설명 텍스트"를 실제 주행 좌표(x, y, z × 64개 waypoint)로 변환한다.
내부적으로 **확산 모델(Diffusion Model)** 방식으로 작동한다.

```
Flow 내부 구조:

[VLM KV Cache (455 MB) + 랜덤 노이즈 좌표 64개]
      ↓
ODE Step 1: expert 모델이 노이즈 정제
      ↓
ODE Step 2: expert 모델이 노이즈 정제
      ↓
...
ODE Step 10: 최종 정제된 좌표 64개 출력
```

10번의 ODE step을 거쳐 노이즈를 실제 좌표로 변환한다.

### 3.2 Flow는 Causal Transformer지만 KV 구조가 다르다

```
[Decode — KV가 누적됨]

step 1: [KV_1~3086 | KV_3087]
step 2: [KV_1~3086 | KV_3087 | KV_3088]
step 3: [KV_1~3086 | KV_3087 | KV_3088 | KV_3089]
...
→ 계속 쌓임 → AppendOnly 패턴 의미 있음


[Flow ODE step — KV가 매번 리셋됨]

ODE step 1: [VLM KV 455MB | action 64tok KV] → 계산 → action KV 버림
ODE step 2: [VLM KV 455MB | action 64tok KV] → 계산 → action KV 버림
ODE step 3: 반복...
→ 매번 64토큰 추가 → 계산 → 버림
→ "누적"이 없어서 AppendOnly 방식을 적용할 구간이 없음
```

### 3.3 Flow의 진짜 병목은 KV가 아니라 가중치

```
Flow 637ms의 실제 구성:

expert 모델 가중치 읽기:
  4.56 GB × 10 ODE steps = 45,600 MB  ← 91% 차지

VLM KV Cache 읽기:
  455 MB × 10 ODE steps = 4,550 MB    ← 9%

비유:
  큰 교과서(가중치 4.56 GB)를 10번 전부 읽어야 함
  책갈피(KV Cache 455 MB)는 매번 참고하는 부록에 불과
  → 부록을 빨리 읽어도 교과서 읽는 시간은 안 줄어든다
```

### 3.4 Flow를 줄이는 방법

| 방법 | 원리 | 가능 여부 | 예상 효과 |
|------|------|---------|---------|
| AppendOnlyCache-C 적용 (부분) | crop/update 효율화 | ✅ 이미 적용됨 | 870ms → 637ms (완료) |
| ODE step 수 감소 | 10 → 5 step으로 줄임 | ❌ 모델 수정 금지 | — |
| expert 가중치 양자화 | 4.56 GB → 2 GB | ❌ 양자화 금지 | — |
| L2 Pinning (action KV) | 9.4 MB < 32 MB L2 고정 | ⚠️ 4ms 절감 | 미미함 |

**결론**: Flow는 현재 이론 한계 근처다. expert 가중치 읽기가 91%를 차지하는 한 큰 개선이 어렵다.

## 4. 세 단계 비교 요약

```
              VE (ViT)         Decode (LM)       Flow (Expert)
─────────────────────────────────────────────────────────────────
Transformer   양방향            단방향(Causal)    단방향(Causal)
처리 방식     모든 패치 동시    한 토큰씩 순차    10번 반복 (리셋)
KV Cache      해당 없음         누적·재사용       누적 없이 리셋
AppendOnly    ❌ 적용 불가      ✅ 핵심 최적화    ❌ 구조 불일치
남은 방법     비동기 오버랩     KV Temporal Reuse 없음(한계 도달)
현재 시간     728ms             1,345ms           637ms
최소 기대값   ~0ms (오버랩시)   ~44ms (Exp C 후)  ~500ms
```

## 5. 최적화 완료 후 예상 파이프라인

```
[현재 — 3,620ms]
VE(728) + Prefill(895) + Decode(1,345) + Flow(637)

[KV Temporal Reuse Exp C 성공 후 — ~2,754ms]
VE(728) + Prefill(44) + Decode(1,345) + Flow(637)
                 ↑ 97% 절감

[+ Async VE 오버랩 후 — ~2,026ms]
VE(0*) + Prefill(44) + Decode(1,345) + Flow(637)
   ↑ 이전 Decode 뒤에 숨겨짐

총 절감: 4,838ms → 2,026ms = -58%
```

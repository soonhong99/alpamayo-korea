# 10Hz 연속 추론 — 입력 구조 및 실험 보고

**작성일**: 2026-06-05  
**실험 환경**: Jetson AGX Thor · BF16 · AppendOnlyCache-C 기준

---

## 핵심 결론

> 1. Alpamayo는 **이미 100ms마다 추론을 실행한다.** "0.4초 기다린다"는 개념이 아니다.
> 2. 매 추론의 입력은 **직전 400ms 슬라이딩 윈도우(4프레임)** 다 — 카메라가 계속 찍고 있으므로 추가 대기 없이 항상 4프레임이 준비되어 있다.
> 3. **병목은 입력 수집이 아니라 추론 계산 자체(4,366ms)** 다.
> 4. KV Cache 재사용 실험을 했으나 연속 실행에서 실패했다. FlashDrive는 이 문제를 올바르게 해결하는 방법을 제시한다.

---

## 1. Alpamayo 입력 구조 — 0.4초 슬라이딩 윈도우

### 정확한 입력 구성

Alpamayo는 매 추론마다 **4 cameras × 4 timesteps = 16장**의 이미지를 입력으로 받는다.  
카메라가 10Hz(100ms 간격)로 찍으므로, 4개의 timestep은 400ms 구간을 커버한다.

```
t = 0ms 시점에서의 추론 입력:
  timestep t-3: t=-300ms 촬영 이미지 × 4 cameras
  timestep t-2: t=-200ms 촬영 이미지 × 4 cameras
  timestep t-1: t=-100ms 촬영 이미지 × 4 cameras
  timestep t-0: t=  0ms 촬영 이미지 × 4 cameras
  ────────────────────────────────────────────────
  합계: 16장 → pixel_values [11,520, 1,536]
```

### "0.4초 기다린다"는 개념이 아니다

카메라는 처음부터 10Hz로 계속 찍고 있다.  
추론을 시작하는 시점에 이미 4프레임이 준비되어 있다.

```
카메라 촬영:
  t=-300ms: 프레임 A 촬영
  t=-200ms: 프레임 B 촬영
  t=-100ms: 프레임 C 촬영
  t=  0ms: 프레임 D 촬영 → 이 순간 추론 시작
                            → A, B, C, D 전부 바로 사용 가능

t=100ms에서도 동일:
  프레임 B, C, D, E → 추론 시작 (E가 방금 촬영됨)
```

매 100ms마다 가장 최신 4프레임으로 슬라이딩하면서 추론이 실행된다.  
**추가 대기 시간이 없다. 입력이 병목이 아니다.**

---

## 2. 교수님 제안 — "0.1초 데이터만으로 바로 시작하면 안되나?"

이 제안의 의미는 4프레임 중 가장 최신 1프레임만으로 즉시 추론을 시작하는 것이다.

```
교수님 제안:
  t=0ms: 프레임 D 1장만으로 즉시 추론 시작

현재 방식:
  t=0ms: 프레임 A, B, C, D 4장으로 추론 시작
```

이 제안은 직관적으로 타당하다. 그러나 Alpamayo 모델은 **4프레임을 동시에 입력받도록 설계되어 학습**됐다. 1프레임만 입력하면 모델이 학습 시 경험하지 못한 입력 형태가 되어 출력 품질이 보장되지 않는다.

그리고 실제로는 위에서 설명했듯이 **추론 시작 시점에 이미 4프레임이 준비되어 있다.** 새로 기다릴 필요가 없다.

---

## 3. FlashDrive의 접근 — 이미 0.1초마다 추론, 계산을 재사용

FlashDrive 논문(NVIDIA 연구)은 100ms마다 추론을 실행하면서 4프레임 입력을 유지하는 방식을 제안한다.

```
t=0ms 추론 입력:   [프레임 A | B | C | D]
t=100ms 추론 입력: [프레임 B | C | D | E]  ← 3프레임(B,C,D)이 이전과 동일
t=200ms 추론 입력: [프레임 C | D | E | F]  ← 3프레임(C,D,E)이 이전과 동일
```

매 추론에서 4프레임 중 3프레임(75%)은 직전 추론과 동일하다.  
→ 이 3프레임에 대한 VE 계산 결과를 재사용하면 VE 728ms의 75%를 절감할 수 있다.

**실험으로 확인된 사실 (pixel_values 비교):**

| 카메라 | t=0ms와 t=100ms 입력 일치 여부 |
|--------|:---:|
| cam0 | ✅ 완전 동일 |
| cam1 | ✅ 완전 동일 |
| cam2 | ✅ 완전 동일 |
| cam3 | ✅ 완전 동일 |

4개 카메라 모두 3프레임이 완전히 동일하다. VE 재사용은 수치적으로 유효하다.

---

## 4. 왜 단순 KV 재사용은 실패했나

FlashDrive 방식이 아닌, 단순히 이전 추론의 KV Cache를 suffix 교체로 재사용하는 실험을 했다.

| 실행 횟수 | 소요 시간 | EOS 생성 | Decode steps |
|:---:|:---:|:---:|:---:|
| 1회 (full prefill) | 4,457ms | ✅ | 19 |
| **2회 (KV 재사용)** | **5,852ms** | **❌ 실패** | **81 (최대치)** |
| **3회 (KV 재사용)** | **6,234ms** | **❌ 실패** | **81 (최대치)** |

실패 원인:

```
KV Cache에 담긴 정보: t=0ms 시점의 카메라 영상 기반
ego 토큰 (차량 상태): t=100ms 시점 기준

→ 영상 정보는 100ms 전 것, 차량 상태는 현재 것
→ 앞뒤가 맞지 않는 입력을 받은 모델이 추론 수렴 실패
→ EOS 토큰을 생성하지 못하고 81 steps 끝까지 계속 생성
```

FlashDrive는 이 문제를 **KV Cache 내 위치 정보(RoPE)를 함께 보정하고, 슬라이딩 방식으로 프레임을 삽입**하는 방법으로 해결한다. 단순 교체가 아니라 위치 정보까지 올바르게 갱신하는 것이 핵심이다. 단, 이 방법은 action expert에 대한 fine-tuning이 필요하다.

## 5. 실제 병목 — 계산 시간 자체

입력 수집이 아니라 추론 계산이 병목이다.

```
현재 1회 추론: 4,366ms

물리적 하한:
  모델 가중치 22 GB ÷ DRAM 196 GB/s = 112ms (Decode 1 step 최소값)
  17 steps × 112ms = 1,904ms (Decode만으로도 100ms 불가)
```

100ms 안에 추론 1회를 끝내는 것은 물리적으로 불가능하다.

**올바른 접근 — Rolling Trajectory Pipeline**

```
각 추론은 4,000ms 걸리지만,
100ms마다 새 추론을 시작하면 100ms마다 결과가 나온다.

t=0ms:    추론_A 시작 ─────────────────── 완료 (t≈4,000ms)
t=100ms:  추론_B 시작 ─────────────────── 완료 (t≈4,100ms)
t=200ms:  추론_C 시작 ─────────────────── 완료 (t≈4,200ms)
...

→ 매 100ms마다 새 trajectory 출력 (throughput = 10Hz)
```

단일 추론 latency를 줄이는 것이 아니라, 여러 추론을 파이프라이닝해서 throughput을 높이는 것이 목표다.

## 6. 단계별 접근 계획

| 단계 | 방법 | 예상 Latency | 상태 |
|------|------|:---:|:---:|
| 현재 | AppendOnlyCache-C | 4,366ms | ✅ 완료 |
| 1단계 | Async VE Pipeline | ~3,140ms | 실험 완료 (1.17×) |
| 2단계 | Adaptive Flow Matching | ~2,400ms | 설계 완료 |
| 3단계 | Pre-RoPE Streaming (FlashDrive) | ~1,000ms | fine-tuning 필요 |
| 최종 | Rolling Trajectory Pipeline | **10Hz throughput** | 구조 설계 필요 |

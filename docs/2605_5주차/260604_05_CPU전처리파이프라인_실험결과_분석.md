**날짜**: 2026-06-04

## 한 줄 결론

> **CPU 전처리(BCD, 310ms)는 GPU inference(4,141ms) window 안에 완전히 숨는다 (wait=0ms). 8코어로 75ms까지 줄일 수 있으며, 파이프라인 적용 시 1.272×(877ms/inference 절감) 달성.**

---

## 1. 실험 설계의 변경 이유 (v1 → v2)

### v1의 잘못된 접근

`260604_cpu_preprocess_pipeline_exp.py` (v1)은 전처리에 Step A (`load_physical_aiavdataset`, ~10,000ms 파일 I/O)를 포함했다. 이것은 연구용 파일 로더이며 실제 추론 시스템과 무관하다.

```
v1 CPU 전처리: A(10,000ms) + B(0.3ms) + C(316ms) + D(5ms) = 10,302ms
실제 배포 환경: 카메라 데이터는 이미 메모리에 있음 → A = 0ms
```

### v2의 올바른 설계

파일 I/O(Step A)는 **실험 준비 단계**에서 한 번만 수행. 파이프라인 루프에서는 B+C+D만 실행.

```
v2 CPU 전처리 (실제 배포 환경 기준):
  B: create_message        =   0.2ms
  C: apply_chat_template   = 309.1ms  ← 실질 전처리 작업
  D: .to(device)           =   1.4ms
  ─────────────────────────────────
  BCD 합계                 = 310.8ms
```

## 2. CPU vs GPU — 각각 무엇을 하는가

### CPU (BCD 전처리, 310ms → 8코어 75ms)

```
B. create_message (0.2ms) — 구조 포맷팅
   입력: image_frames [4카메라×4timestep×3ch×1080×1920], camera_indices
   출력: Qwen3VL 포맷의 멀티모달 메시지 리스트
   특성: Python dict/list 구성. torch 연산 없음. 스레드 무관.

C. apply_chat_template (309ms, 전체 99.5%) ← 실질 CPU 작업
   입력: 멀티모달 메시지
   출력: input_ids [1,3086], pixel_values [11520,1536] @BF16, image_grid_thw
   내부 동작:
     ① 이미지 패치 추출: 16장(4cam×4t) × 1080×1920 → ViT 입력 크기 resize + normalize
        → 11,520 patches × 1,536dim = 35.4MB BF16 생성
     ② 텍스트+이미지 토크나이징 → 3,086 토큰 시퀀스 구성
   특성: torch 내부 연산(이미지 변환) 사용 → set_num_threads(n) 효과 받음

D. .to(device) (1.4ms) — unified memory near-instant
   Thor unified memory(CPU∥GPU 공유 DRAM): 포인터 메타데이터만 업데이트
   실제 데이터 복사 없음.
```

### GPU (run_inference, ~4,141ms 또는 ~3,230ms)

```
① VE — Visual Encoder (~728ms)
   model.vlm()의 vision_tower (ViT-H 계열)
   입력: pixel_values [11520, 1536] @BF16
   출력: visual embeddings (이미지 토큰 위치에 삽입)
   특성: compute-bound (Tensor Core), matrix 연산 집중

② LM Prefill (~1,423ms)
   Qwen3 language model, full sequence attention
   입력: 3,086 토큰 (텍스트 + 이미지 임베딩)
   출력: past_key_values (KV cache 22GB), logits
   특성: 3,086² attention → 이 파이프라인 최대 연산량
   KV cache 22GB를 GPU DRAM에 새로 할당

③ Decode (15 steps × ~107ms = ~1,605ms)
   Autoregressive trajectory token 생성
   for step in range(N_DECODE=15):
     input: cur (단일 토큰), kv (22GB KV cache)
     output: next token (trajectory waypoint 또는 EOS)
   특성: memory-bound (22GB 매 step 접근), DynamicCache 사용
   
합계 이론: 728+1,423+1,605 = 3,756ms
실측 3-A (GPU only, 연속 호출): 4,141ms
```

## 3. Phase 1: 실제 CPU 전처리 시간 확인

| 단계 | 시간 | 비율 |
|------|------|------|
| B. create_message | 0.2ms | 0.1% |
| C. apply_chat_template | 309.1ms | **99.5%** |
| D. .to(device) | 1.4ms | 0.4% |
| **BCD 합계** | **310.8ms** | 100% |
| GPU inference (참고) | 4,141ms | — |
| BCD/GPU 비율 | 7.5% | — |

→ BCD 310ms는 GPU 4,141ms의 7.5% → 4,055ms 여유 → **파이프라인 조건 충족**

## 4. Phase 2: 코어 스케일링 — "4코어까지 선형" 연구 재확인

| 코어 수 | BCD 시간 | 가속비 | 이상적 | 효율 |
|---------|---------|--------|--------|------|
| 1코어 | 310ms | 1.00× | 1×  | 100% |
| 2코어 | 193ms | 1.61× | 2×  | 80%  |
| 4코어 | 103ms | **3.01×** | 4×  | 75%  |
| 8코어 |  75ms | **4.14×** | 8×  | 52%  |

**패턴 분석:**

```
1→4 구간 (3.01×): 준선형 증가 확인
  apply_chat_template은 torch 이미지 변환 + 텐서 연산
  → 코어 증가 시 compute + DRAM 대역폭 동시 증가 → 준선형 스케일링

4→8 구간 (1.37× 추가): plateau 시작
  103ms → 75ms = 28ms 추가 절감
  메모리 버스 포화로 추가 코어의 효용 감소

★ 이전 연구("4코어까지 대역폭 선형 증가") 확인됨:
  4코어에서 3.01× (75% 효율) → 4코어가 best trade-off point
  8코어: 4.14× (52% 효율) → 코어 낭비, 다른 작업에 양보하는 것이 나을 수 있음
```

**v1에서 1.03×였던 이유**: Step A(파일 I/O, 97%)가 torch.set_num_threads()의 영향을 받지 않기 때문. v2에서 A를 제거하자 C(torch 연산, 99.5%)가 스케일링의 전부가 되어 4.14×가 실현됨.

## 5. Phase 3: GPU∥CPU 파이프라인 결과

### 수치 요약

| 방식 | 시간 | vs GPU only |
|------|------|------------|
| [A] GPU only | 4,141ms | 기준 |
| [B] Sequential (BCD→GPU) | 4,106ms | -35ms (??) |
| [C] Async (8코어, BCD∥GPU) | **3,229ms** | **-912ms** |

**[C] Async 세부:**

```
inf 0: gpu=3,230ms  bcd=97ms  wait=0ms  ✅
inf 1: gpu=3,230ms  bcd=98ms  wait=0ms  ✅
inf 2: gpu=3,230ms  bcd=101ms wait=0ms  ✅
inf 3: gpu=3,217ms  bcd=99ms  wait=0ms  ✅
avg wait: 0.2ms  → BCD가 GPU window 안에 완전히 숨음 확인
```

### 핵심 확인사항: wait=0ms

```
bcd_time(99ms) << gpu_time(3,230ms)
→ BCD가 GPU inference 시작 후 99ms만에 완료
→ GPU가 끝날 때 (3,230ms) BCD는 이미 3,131ms 전에 완료
→ wait_ms = 0ms (대기 없음)
```

## 6. 이상 현상 분석: Async GPU가 왜 3,230ms인가 (4,141ms 대비 22% 단축)

### 현상

```
예상:
  Sequential(BCD→GPU): 318(BCD) + 4,141(GPU) = 4,459ms
  Async: 0(BCD 숨음) + 4,141(GPU) = 4,141ms
  예상 가속비: 1.076×, 예상 절감: 318ms

실제:
  Sequential: 4,106ms
  Async GPU:  3,230ms  ← 4,141ms보다 911ms 빠름
  실제 가속비: 1.272×, 실제 절감: 877ms
  설명 안 되는 초과 절감: 877 - 318 = 559ms
```

### 3-B Sequential에서도 같은 현상 관찰

```
3-B: inf 0 GPU = 4,125ms (첫 번째, 3-A와 동일)
     inf 1,2,3 GPU ≈ 3,680ms (311ms CPU gap 이후 444ms 빨라짐)
```

→ CPU 작업 시간이 GPU inference 속도를 향상시킨다.

### 원인 가설 (유력): PyTorch CUDA allocator + KV cache GC 타이밍

```
3-A (연속 GPU 호출, gap 없음):
  inference(k) 종료 → kv 변수 scope 종료
  Python ref count 감소하지만 CUDA tensor는 즉시 해제 안 됨
  inference(k+1) 즉시 시작 → 새 22GB KV cache 할당 필요
  → 이전 KV cache가 아직 "freed but cached" 상태로 메모리 점유
  → allocator가 fragmented pool에서 할당 → overhead 큼

3-C (97ms CPU gap):
  inference(k) 종료 → 97ms CPU BCD 실행 중:
    ① Python GC 실행 기회 증가 (CPU 활동으로 GC triggered)
    ② 이전 22GB KV cache CUDA tensor 실제 해제 완료
    ③ allocator가 22GB 연속 블록을 clean pool에 정리
  inference(k+1) 시작 → 이전과 동일한 크기 블록 즉시 재사용
  → allocation overhead 없음 → decode step 속도 향상
```

### 대안 가설: 다른 input frame에서 EOS 조기 발생

```
3-A: tok_list_warmup (frame 0,1,2,3) → 15 decode steps 전체 실행
3-C: raw_data_list (frame 1,2,3,4) → 일부 frame이 EOS 조기 발생 가능
→ decode step 수 감소 → 3,230ms (직접 확인 필요)
```

**⚠️ 주의**: 877ms 절감 중 정확히 얼마가 BCD 제거 효과이고 얼마가 부수 효과인지는 추가 실험이 필요하다.

## 7. 결론

### 주요 발견

1. **파이프라인 가능 확인** (핵심)
   - BCD(310ms, 1코어) << GPU(4,141ms) → wait=0ms 확인
   - 실제 배포 환경에서 CPU 전처리는 GPU window 안에 완전히 숨음

2. **코어 스케일링 재확인**
   - "4코어까지 선형" 연구: 4코어 3.01×로 확인
   - 8코어: 4.14× (추가 1.37× 더), 실용적으로는 4코어가 최적
   - apply_chat_template이 torch 병렬화 가능함을 확인

3. **BCD/GPU 비율**
   - 1코어: 310ms / 4,141ms = 7.5%
   - 4코어: 103ms / 4,141ms = 2.5%
   - 8코어: 75ms / 4,141ms = 1.8%

4. **측정된 파이프라인 가속**
   - Sequential 대비: 1.272×, 877ms/inference
   - 그러나 일부는 allocator 부수 효과 가능성 → 재확인 필요

### 파이프라인 적용 후 예상 지연 감소

```
현재 (직렬):
  [BCD 310ms] → [GPU 4,366ms] = 4,676ms/cycle

파이프라인 (1코어):
  [GPU 4,366ms]
       ∥ (동시)
  [BCD 310ms]
  = 4,366ms + 작은 overhead ≈ 4,380ms/cycle
  절감: 310ms/cycle (6.6%)

파이프라인 (4코어):
  [BCD 103ms] << GPU → 4,380ms 그대로 (103ms는 이미 충분히 숨음)

파이프라인 (8코어):
  [BCD 75ms] << GPU → 동일
```

### 다음 단계

1. **decode step 수 계측**: 3-A vs 3-C가 실제로 다른 decode step을 실행하는지 확인 (EOS 조기 발생 여부)
2. **AppendOnlyCache-C + 파이프라인**: AppendOnlyCache-C 적용 시 GPU가 일관된 시간 → 순수 BCD 제거 효과만 측정
3. **4코어 적용 권고**: 4코어(103ms)가 cost-efficiency 최적. 1코어(310ms)도 GPU window(4,366ms) 안에 충분히 숨음.

## 1. R-TOD 논문 분석

### 1.1 논문 개요

**제목**: "R-TOD: Real-Time Object Detector with Minimized End-to-End Delay for Autonomous Driving"  
**게재**: IEEE Real-Time Systems Symposium (RTSS), 2020  
**저자**: Won-Seok Jang, Hansaem Jeong, Kyungtae Kang, Nikil D. Dutt, Jong-Chan Kim  
**소속**: AveesLab (국민대 — Demand Layering, RT-Swap과 동일 연구실)  
**플랫폼**: NVIDIA Jetson AGX Xavier + Logitech C930e USB 카메라

### 1.2 핵심 문제 정의 — End-to-End Delay

기존 연구들이 **inference delay** (DNN 연산 시간)만을 최적화 대상으로 삼았다. R-TOD는 더 근본적인 질문을 제기한다:

> **End-to-End Delay**: 물체가 실제 환경에서 나타나는 순간부터, 시스템이 그 물체를 감지했다고 보고하는 순간까지의 총 지연 시간

```
[기존 측정 방식]
Camera → [버퍼에 대기 중] → Read → Preprocess → DNN → Postprocess → Output
                                    ←─── inference delay ───→

[R-TOD의 측정 방식]
물체 등장 → Camera → [버퍼 대기] → Read → Preprocess → DNN → Postprocess → Output
←──────────────────── end-to-end delay ──────────────────────────────────────────────→
```

카메라 버퍼에서 대기하는 시간, 전처리 지연, CPU-GPU 전환 지연 등이 모두 포함된다. inference delay가 100ms라도 end-to-end delay는 1,000ms 이상일 수 있다.

**R-TOD 결과**: YOLO v3 기준 1,070ms → 261ms (76% 감소), 모델 정확도 변화 없음.

### 1.3 세 가지 최적화 기법

#### 기법 1: On-Demand Capture (온디맨드 캡처)

**문제**: 표준 카메라 드라이버(OpenCV 등)는 프레임을 연속적으로 내부 버퍼에 쌓는다. DNN 파이프라인이 이전 프레임을 처리하는 동안 새 프레임은 버퍼에서 대기한다. DNN이 다음 프레임을 요청할 때, 버퍼에 쌓인 것 중 **가장 오래된 것**을 꺼낸다.

```
[표준 캡처 방식 — 버퍼 지연 발생]
Camera:  [F1]─[F2]─[F3]─[F4]─[F5]─[F6]...  (100ms 간격 연속 캡처)
Buffer:       ↓버퍼에 대기         ↓
DNN:     [──F1 처리(400ms)──]  [──F4(미래!) 처리──]
                               ↑ F4를 처리하는 순간 이미 F5, F6이 버퍼에 있음

end-to-end delay = 처리 지연 + 버퍼 대기 시간
```

**해결**: V4L2 ioctl을 직접 사용하여 DNN이 준비됐을 때 **그 순간의 최신 프레임만** 즉시 캡처.

```
[On-Demand Capture — 버퍼 없음]
DNN:     [──F1 처리──]  [──F2 처리──]  [──F3 처리──]
Capture:              ↑ 캡처 요청     ↑ 캡처 요청
                      → 즉시 최신 프레임 반환

end-to-end delay ≈ inference delay (버퍼 대기 제거)
```

#### 기법 2: Zero-Slack Pipeline (제로 슬랙 파이프라인)

**문제**: Capture → Preprocess → DNN Inference → Postprocess 각 단계 사이에 태스크 스케줄링 지연, 동기화 오버헤드, CPU-GPU 큐 대기 등의 **슬랙(Slack, 낭비 시간)**이 발생한다.

**해결**: 각 단계가 이전 단계 완료 즉시 시작하도록 파이프라인을 구성한다. Capture 완료와 동시에 Preprocess가 시작되고, DNN이 GPU에 올라가는 것과 동시에 다음 Capture가 준비된다.

```
[기존]
Capture ──▶ [wait] ──▶ Preprocess ──▶ [wait] ──▶ DNN ──▶ [wait] ──▶ Post

[Zero-Slack]
Capture ──▶ Preprocess ──▶ DNN ──▶ Post
  └──────── (N+1) Capture 즉시 시작 ────────────────────────▶
```

수학적으로: 각 단계의 완료 이벤트가 다음 단계의 시작 트리거가 되도록 설계한다. 파이프라인 스테이지 간 `slack = 0`.

#### 기법 3: Contention-Free Pipeline (경합 없는 파이프라인)

**문제**: CPU 전처리와 GPU 추론이 동일한 메모리 버스를 사용할 때 경합(contention)이 발생한다. CPU가 전처리 중인 동안 GPU가 데이터를 기다리거나, GPU 추론 중 CPU가 메모리 접근에서 대기하게 된다.

**해결**: CPU와 GPU의 작업이 시간적으로 겹치지 않도록 스케줄링한다. 또는 CPU 전처리 단계를 GPU inference와 분리된 메모리 영역에서 처리한다.

```
[기존 — CPU/GPU 메모리 경합]
CPU Preprocess: [──전처리──]
GPU Inference:         [──DNN────]
메모리 버스:    [CPU가 점유] [GPU가 점유]  ← 전환 비용 + 경합 발생

[Contention-Free]
CPU Preprocess: [──전처리──]
GPU Inference:              [──DNN────]
메모리 버스:    [CPU 전용구간][GPU 전용구간] ← 경합 없음
```

## 2. KV Cache 용량 정확한 계산 (피드백 #1)

교수님이 "KV 캐시를 L2에 올리는 아이디어는 획기적이나, 정확한 용량 및 최대 크기 조사 필요"라고 하셨다.

### 2.1 Alpamayo VLM KV Cache 구조

Alpamayo 1.5 VLM (Cosmos Reason2 기반)의 KV 캐시 구조:

```
KV Cache 크기 계산식:
  N_layers × 2 (K, V) × N_heads × seq_len × head_dim × dtype_bytes

VLM 파라미터 (Qwen3VL 기반):
  N_layers   = 36
  N_kv_heads = 8  (Grouped Query Attention)
  head_dim   = 128
  dtype      = BF16 (2 bytes)
```

| 단계 | 토큰 수 | KV Cache 크기 |
|------|--------|--------------|
| Prefill 완료 (입력 전체) | 3,086 | **36 × 2 × 8 × 3,086 × 128 × 2 = ~455 MB** |
| Decode 1 step 추가 | +1 | +0.14 MB / step |
| Decode 17 steps 후 | 3,103 | **~458 MB** |
| 최대 (max_new_tokens=80) | 3,166 | **~468 MB** |

### 2.2 Flow 단계의 Expert KV Cache

Flow 단계에서 `model.expert` (Qwen3VL 2279M, 28 layers)도 KV를 사용한다:

```
Expert KV Cache (매 ODE step에서 생성 후 crop으로 제거):
  N_layers   = 28
  N_kv_heads = 8
  seq_len    = 3,087 (kv_cache_seq_len) + 64 (expert 64 tokens) = 3,151
  head_dim   = 128

expert_kv_size = 28 × 2 × 8 × 3,151 × 128 × 2 = ~362 MB
```

그러나 이 KV는 매 ODE step 후 `prompt_cache.crop(prefill_seq_len)`으로 64 토큰을 제거하므로, 실질적으로 VLM KV (~455 MB)가 재사용된다.

### 2.3 L2 캐시 활용 가능성 분석

| 대상 | 크기 | L2(32MB) 수용 가능? | 반복 접근 횟수 | L2 효과 |
|------|------|--------------------|-----------|----|
| VLM KV 전체 | ~455 MB | ❌ 14.2× 초과 | 17 decode steps | 없음 |
| Expert KV 전체 | ~362 MB | ❌ 11.3× 초과 | 10 ODE steps | 없음 |
| **최근 64 expert 토큰 KV** | **28 × 2 × 8 × 64 × 128 × 2 = 7.3 MB** | **✅ 4.4× 여유** | 280회 (10 ODE × 28 layers) | **★ 가능** |
| **text_prefix KV (100 tok)** | **~14.7 MB** | **✅ 2.2× 여유** | decode 17 steps | **★ 가능** |

**결론**: 전체 KV를 L2에 올리는 것은 불가능하나, **자주 반복 접근되는 Hot KV 블록** (expert 64 토큰, text prefix 100 토큰)은 L2에 올릴 수 있다. 이것이 교수님이 "획기적이다"고 하신 아이디어의 실질적 의미다.

### 2.4 L2 Persistence API 적용 방안

```python
# CUDA L2 Persistence API (SM 11.0 지원)
# expert KV cache의 hot 블록을 L2에 고정

import torch

def pin_hot_kv_in_l2(kv_tensors_hot, stream):
    """
    자주 접근되는 KV 블록을 L2에 고정
    - kv_tensors_hot: expert 64 토큰 KV (7.3 MB)
    - 이 블록은 10 ODE steps × 28 layers = 280회 접근
    """
    attr = torch.cuda.StreamAttribute()
    # hitProp=persisting: eviction 방지
    # missProp=streaming: 다른 데이터는 streaming으로
    attr.access_policy_window = torch.cuda.AccessPolicyWindow(
        base_ptr=kv_tensors_hot.data_ptr(),
        num_bytes=kv_tensors_hot.nbytes,
        hit_ratio=1.0,
        hit_prop='persisting',
        miss_prop='streaming',
    )
    stream.set_attribute(attr)
```

**예상 효과**: Flow 단계 870ms에서 expert KV 280회 접근 중 L2 hit rate가 높아질 경우 최대 10~20% 단축 가능 (측정 필요).

## 3. MIG (Multi-Instance GPU) 확인 및 연구 방향 (피드백 #2, #3)

### 3.1 Thor SM 11.0 MIG 지원 현황

NVIDIA JetPack 7 공식 문서에 따르면:
> "JetPack 7 gives full support for the NVIDIA Jetson Thor platform, including Multi-Instance GPU (MIG)"

**현재 확인된 사항**:
- JetPack 7 (Jetson Linux 38.4)에 MIG 공식 지원 포함
- Thor SM 11.0 = Blackwell 아키텍처 → MIG 지원 (A100/H100 이후 전 Blackwell에서 지원)
- 펌웨어: UEFI r38.4.x 이상 필요

**즉시 확인할 명령어 (Thor 보드에서)**:
```bash
# MIG 모드 지원 여부 확인
nvidia-smi --query-gpu=mig.mode.current --format=csv

# MIG 지원 여부
nvidia-smi --query-gpu=pcie.link.gen.gpumax --format=csv

# Thor에서 MIG 활성화
sudo nvidia-smi -i 0 -mig 1

# MIG 인스턴스 생성 (1개 GPU → 7개 인스턴스 예시)
sudo nvidia-smi mig -cgi 9,9,9,9,9,9,9 -C
```

### 3.2 H100 MIG 연구 참고 — 단일 모델 활용

교수님이 언급하신 H100 MIG 연구의 핵심은 **데이터센터 GPU에서 단일 GPU를 여러 독립 인스턴스로 분할**하는 것이다. 주요 연구 결과:

- Optimal Workload Placement on MIG (arXiv:2409.06646): A100/H100에서 7개 슬라이스 가능
- Managing Multi-Instance GPUs (arXiv:2508.18556): 최대 1.43× throughput 향상, 1.11× 에너지 절감
- H100 MIG for Mistral 7B: 20% 낮은 latency, 6% 높은 throughput

**Thor에서 Alpamayo 단일 모델 MIG 활용 아이디어**:

교수님의 관점: "데이터센터는 여러 모델에 MIG를 쓰는데, 우리는 단일 모델(Alpamayo)로 이용"

```
[MIG를 Alpamayo 파이프라인에 적용하는 방안]

Thor GPU → MIG 분할:
  인스턴스 A (MIG slice 1): VE + Prefill 담당
  인스턴스 B (MIG slice 2): Decode + Flow 담당

파이프라인:
  프레임 t:   [인스턴스 A: VE+Prefill] ─────────────────▶
              [인스턴스 B:              Decode+Flow] ────▶

  프레임 t+1:                          [인스턴스 A: VE+Prefill]▶
              [인스턴스 B: (프레임 t의 Decode+Flow 종료 후 시작)]
```

그러나 MIG의 핵심 제약: **MIG 인스턴스 간 메모리 분리**. Thor의 128GB LPDDR5X가 인스턴스 간 분리될 경우 메모리 대역폭도 분리된다. Alpamayo 22GB 모델이 하나의 인스턴스에 들어가는지 확인 필요.

**즉시 실험해야 할 것**: Thor에서 MIG가 실제로 활성화되는지 확인 → 인스턴스 별 메모리 크기 확인

## 4. KV Temporal Reuse — 정확한 구조 문서화 (피드백 #4)

교수님 질문 4가지에 대한 정확한 답변:

### Q1. "맨 처음 frame을 가지고 나머지 3개의 frame에 대한 vision 값을 받지 않는 건지?"

**정확한 답변**: 아니다. KV Temporal Reuse는 매 프레임마다 Vision Encoder를 실행한다. **절약되는 것은 LM Prefill이다.**

```
[현재 — 매 프레임 전체 처리]
프레임 t:
  VE(t)     → 728ms  (새 픽셀 처리, 항상 필요)
  Prefill(t)→ 1,423ms (3,086 토큰 전체 Attention)
  Decode(t) → 1,818ms
  Flow(t)   →  870ms
  합계:        4,839ms

[KV Temporal Reuse — 실험 A 검증]
프레임 t (steady state):
  VE(t)     → 728ms  (항상 실행, 동일)
  KV_reuse  →  107ms  ← 3,086 토큰 Prefill → 1 토큰 forward로 대체
  Decode(t) → 1,818ms
  Flow(t)   →  870ms
  합계:       ~3,523ms  (27% 절약)
```

**왜 Vision Encoder는 항상 재실행되는가?**
새 프레임의 픽셀값에서 vision embedding을 추출하는 과정은 이전 프레임의 KV 캐시와 무관하다. 현재 프레임의 시각 정보를 반드시 새로 계산해야 한다.

**무엇이 재사용되는가?**
이전 프레임에서 계산한 LM의 KV 캐시. 연속 프레임에서 text_prefix(system prompt)는 전혀 변하지 않고, vision 부분도 75%가 겹친다. 이 KV를 재사용하면 LM이 3,086 토큰을 다시 attention할 필요 없이 변경된 부분만 처리한다.

### Q2. "왜 100ms에서 획기적으로 줄어드는가?"

**물리적 근거**:
```
KV 재사용 시:
  모델이 처리해야 할 "새 토큰 수" = 1개 (또는 변경된 ~96개)
  
  처리 비용 ∝ 새 토큰 수 × 전체 KV 시퀀스 길이
  
  기존: 3,086 토큰 × 3,086 tokens/KV = 9,523,396 operations
  재사용: 1 토큰 × 3,086 tokens/KV = 3,086 operations
  → 이론 비율: 3,086× 감소

실측:
  기존 Prefill:  1,423ms (3,086 토큰 attention)
  재사용 Prefill:  107ms  (1 토큰 attention)  ← decode 1 step과 동일
  실측 비율: 13.3×
```

107ms인 이유: 1개 토큰을 처리할 때도 22GB 모델 가중치를 DRAM에서 읽어야 하기 때문이다 (Decode 1 step과 동일한 물리 하한: 22GB ÷ 231 GB/s = 95ms).

### Q3. "0.4초가 0.1초로 되는 것인지?"

**정확한 설명**: 추론 단위가 0.4초→0.1초로 바뀌는 것이 아니다. 추론 단위는 동일하게 단일 프레임(t=0)이다. 다만:

```
[현재 방식]
  - 4개 프레임(t-300ms, t-200ms, t-100ms, t=0ms)을 하나의 batch로 입력
  - 이 4개 프레임의 KV를 매번 새로 계산

[KV Temporal Reuse 방식]
  - 여전히 4개 프레임의 정보를 사용하지만:
  - t-300ms, t-200ms, t-100ms의 KV는 이전 추론에서 재사용
  - t=0ms의 VE만 새로 계산
  - LM이 4개 프레임을 처리하는 시간 중 75%를 생략
```

따라서 **추론 주기는 0.1초로 단축될 수 있지만**, 매번 4프레임의 시각 정보를 활용하는 것은 동일하다.

### Q4. "0.1초마다 바뀌는 것을 0.4초로 강제로 늘리게 된 것 아닌가? 이래도 괜찮은가?"

**교수님의 핵심 우려**: 카메라는 100ms마다 새 프레임을 생성하는데, 현재 시스템은 4프레임(400ms)을 기다렸다가 한 번의 추론을 시작한다. 이 방식은 실질적으로 400ms 단위로 환경을 인식하게 되는데, 이것이 실시간 자율주행에 충분한가?

```
[현재 시스템의 인식 주기 문제]
t=0:    4개 프레임(t-300 ~ t=0) 수집 완료 → 추론 시작
t=4,839ms: 추론 완료, 결과 반환
t=5,239ms: 다음 4개 프레임(t+400 ~ t+4,839) 수집 완료 → 다음 추론 시작
...

실질적 인식 주기: 4,839ms (매우 느림)
실질적 "정보 신선도": 가장 오래된 프레임이 t-300ms, 가장 새 프레임이 t=0
```

**자율주행 안전 기준 관점**:
```
120 km/h = 33.3 m/s
  현재 인식 주기(4,839ms): 161m 이동 후 갱신  ← 매우 위험
  목표(≤100ms):             3.3m 이동 후 갱신  ← 실시간
  KV Reuse 후(~3,523ms):   117m 이동 후 갱신  ← 여전히 미흡
```

**해결책 — Rolling Trajectory Pipeline** (피드백 #6과 연결):
KV Reuse만으로는 부족하다. 매 100ms마다 새 프레임으로 추론을 시작하는 **연속 파이프라인**이 필요하다. 이것이 교수님 피드백 #6의 핵심이다.

## 5. 실제 대역폭 프로파일링 (피드백 #5)

교수님: "알파마요를 돌릴 때 실제로 얼마만큼의 대역폭이 필요한지 프로파일링 필요"

### 5.1 현재 알고 있는 것 (기존 실험 결과)

| 단계 | 시간 | 데이터량 | 실효 대역폭 |
|------|------|---------|-----------|
| VE | 728ms | ~? GB | 측정 안 됨 |
| LM Prefill | 1,423ms | ~22 GB (모델) + 시퀀스 연산 | ~15 GB/s (추정) |
| Decode (1 step) | 107ms | 22 GB (모델 전체) | **206 GB/s** (88% 효율) |
| Flow (1 ODE step) | 89ms | 4.56 GB (expert) | ~51 GB/s (22% 효율) |
| 총계 | 4,839ms | | |

Decode 단계는 88% 대역폭 효율로 물리 한계에 근접해 있음이 확인됐다.

### 5.2 필요한 프로파일링 — 실험 계획

```bash
# Thor 보드에서 실행할 명령어
# 방법 1: nvidia-smi 기반 (간단)
nvidia-smi dmon -s um -d 100 &  # 100ms 간격 메모리 모니터링
python3 scripts/inference/run_thor_inference.py --model nvidia/Alpamayo-1.5-10B

# 방법 2: Nsight Systems 기반 (정밀)
nsys profile --stats=true \
     --output profiling_results/alpamayo_bw_profile \
     python3 scripts/inference/run_thor_inference.py

# 방법 3: CUPTI 기반 커스텀 스크립트
# 각 단계 (VE, Prefill, Decode, Flow)를 CUDA Event로 구분하여
# 단계별 실제 DRAM read/write bandwidth 측정
```

**측정할 항목**:
1. VE 단계: pixel_values (다중 카메라 이미지) → vision embedding 처리 시 대역폭
2. LM Prefill 단계: 3,086 토큰 attention 시 KV 생성 대역폭
3. Decode 단계: 매 step 22GB 모델 로딩 대역폭 (예상 ~206 GB/s, 확인 필요)
4. Flow 단계: expert 10회 forward 대역폭 (예상 ~51 GB/s, 낮은 효율 원인 분석)

## 6. 모델 내부 병렬화 — 0.1초 단위 즉시 추론 (피드백 #6)

교수님: "1개를 받고 바로 일을 시작하게끔. 0.1초 frame만을 받고 바로 진행해서 병렬적으로 0.1초 간격으로 진행"

이것은 **Inter-Inference Rolling Pipeline** 구조로, R-TOD의 On-Demand Capture + Zero-Slack Pipeline을 VLA에 적용하는 것이다.

### 6.1 현재 구조의 문제

```
[현재 — 4프레임 배치 처리]
카메라: [F0][F1][F2][F3]        [F4][F5][F6][F7]        [F8]...
                ↑                               ↑
         4프레임 누적 완료                  4프레임 누적 완료
         → 추론 시작                        → 추론 시작
         
추론:    [────────────── 4,839ms ──────────────][────────────── 4,839ms ──────────────]

문제: 프레임 F3 이후 최대 4,839ms 동안 환경 변화를 인식 못함
```

### 6.2 목표 구조 — On-Demand Frame + Rolling Pipeline

```
[목표 — 100ms 단위 연속 추론]
카메라: [F0] [F1] [F2] [F3] [F4] [F5] ...
         ↓    ↓    ↓    ↓    ↓    ↓
추론:   [A0] [A1] [A2] [A3] [A4] [A5] ...  ← 각 프레임마다 즉시 추론 시작
```

**KV Temporal Reuse와의 연결**:
```
추론 A0 (cold start):
  VE(F0) → KV_0 생성 → Decode_0 → Flow_0
  소요: 4,839ms

추론 A1 (KV 재사용):
  VE(F1) → KV_0 재사용 → 96 토큰만 forward → Decode_1 → Flow_1
  소요: ~3,523ms (27% 절약)

추론 A2 (KV 재사용):
  VE(F2) → KV_1 재사용 → ...
  소요: ~3,523ms
```

**병렬화 가능 여부 분석**:

A0가 끝나기 전에 A1을 시작할 수 있는가?

```
A0 타임라인:
  [VE_0: 728ms][Prefill_0: 1423ms][Decode_0: 1818ms][Flow_0: 870ms]
              ↑                                       ↑
         이후 A1의 VE는 시작 가능?               이 시점에 A0 완료

A1의 VE는 A0의 VE 완료 직후 GPU에서 시작 가능
  BUT: A0의 Decode/Flow도 GPU 사용 중
  → iGPU 단일 컨트롤러 → 대역폭 경합 발생

결론: A0의 Flow 단계(GPU) 중 A1의 VE(GPU)를 동시 실행하면
     Exp D에서 확인된 대역폭 경합(41.6% 감소) 발생
     → 무조건 병렬화는 불가, 대역폭 분배 전략 필요
```

### 6.3 핵심 연구 질문

현재 시스템에서 병렬화가 가능한 구간과 불가능한 구간:

| 구간 | 병렬화 가능성 | 이유 |
|------|-------------|------|
| A0 Decode ∥ A1 VE | ⚠️ 제한적 | Exp D: 41.6% BW 경합 |
| A0 Flow ∥ A1 VE+Prefill | ❌ 어려움 | 둘 다 22GB+ 모델 접근 |
| A0 완료 후 A1 즉시 시작 | ✅ 가능 | Zero-Slack pipeline |
| CPU 전처리 ∥ GPU Decode | ✅ 가능 | CPU/GPU 분리 (Exp E 유사) |

## 7. R-TOD 아이디어 Alpamayo 적용 분석 (피드백 #7)

교수님: "On-Demand Capture, Zero-Slack Pipeline, Contention-Free Pipeline을 적용 가능한지 깊이 분석"

### 7.1 On-Demand Capture → On-Demand Frame Processing

**R-TOD 원리**: 카메라 버퍼를 거치지 않고 DNN이 준비될 때 즉시 최신 프레임 캡처

**Alpamayo 적용**:

```
[현재 — 버퍼 방식]
카메라 캡처: [F0][F1][F2][F3][F4][F5][F6][F7]...  (100ms 연속)
Alpamayo:    │←───── 4프레임 대기(400ms) ───────→│← 추론 4,839ms →│
             F0                                   F3

총 지연 = 4프레임 수집 지연(400ms) + 추론(4,839ms) = 5,239ms
```

```
[On-Demand Frame 적용]
카메라:  [F0]     [F1]     [F2]     ...
          ↓        ↓        ↓
Alpamayo:[A0 즉시 시작] [A1 즉시 시작] [A2 즉시 시작]...
         ↓          ↓            ↓
        KV_0     KV_0 재사용   KV_1 재사용

단일 프레임만으로 추론 시작 → 4프레임 대기(400ms) 제거
```

**구현 가능성**: KV Temporal Reuse를 완성하면 단일 프레임으로도 4개 프레임의 시간적 컨텍스트를 KV 캐시에서 유지하므로 On-Demand Frame Processing 가능.

**절약량**: 4프레임 대기 400ms 제거 + KV 재사용 1,316ms → 총 1,716ms 절약 (현재 5,239ms → 3,523ms)

### 7.2 Zero-Slack Pipeline → Zero-Slack VLA Pipeline

**R-TOD 원리**: 각 처리 단계가 이전 단계 완료 즉시 시작 (대기 없음)

**Alpamayo 현재 상태 분석**:

Alpamayo의 `sample_trajectories_from_data_with_vlm_rollout` 내부 실행 순서:
```python
# 현재 순서 (완전 직렬)
input_ids = fuse_traj_tokens(...)           # CPU + GPU
vlm_outputs = model.vlm.generate(...)      # GPU (VE + Prefill + Decode)
sampled_action = diffusion.sample(...)     # GPU (Flow)
pred_xyz, pred_rot = action_to_traj(...)   # CPU
```

각 단계가 완전히 직렬로 실행되며 스테이지 간 명시적 sync가 있다.

**Zero-Slack 적용 방안**:
```
현재:
[fuse_traj_tokens]──▶[vlm.generate]──▶[diffusion.sample]──▶[action_to_traj]

Zero-Slack 목표:
  다음 프레임 fuse_traj_tokens는 현재 vlm.generate 완료 직후 시작 가능
  (action_to_traj를 기다릴 필요 없음)

수정된 파이프라인:
  [Frame t-1: fuse→generate→diffusion→action_traj]
                     ↑ generate 완료 시점에
  [Frame t:          fuse 시작가능]
```

구현 요소:
- 비동기 `action_to_traj` (CPU 연산, GPU 기다릴 필요 없음)
- 다음 프레임 `fuse_traj_tokens`를 CUDA 이벤트로 트리거
- 결과물은 큐(Queue)로 내보내기

### 7.3 Contention-Free Pipeline → 대역폭 분리 파이프라인

**R-TOD 원리**: CPU와 GPU가 같은 메모리를 동시에 접근하지 않도록 스케줄링

**Alpamayo의 경합 지점**:
- CPU: 카메라 프레임 읽기, 전처리, `fuse_traj_tokens`
- GPU: VE, Prefill, Decode, Flow

Exp D에서 확인: CPU 12코어 활성화 시 GPU 대역폭 -41.6%

**Contention-Free 설계**:

```
[경합 발생 구간]
GPU Decode (BW 집중): [──────── 1,818ms ────────]
CPU 전처리 (메모리):   ←─ 이 구간에 CPU 전처리 금지 ─→

[경합 없는 구간 — CPU 작업 허용]
GPU Flow (BW 낮음):                              [── 870ms ──]
CPU 다음 프레임 전처리:                          [전처리 허용]
```

CPU 전처리를 GPU Decode 구간이 아닌 GPU Flow 구간에 스케줄링하면 경합을 줄일 수 있다. Exp D에서 이미 확인된 이 결과를 이용해 명시적 스케줄링 정책을 구현.

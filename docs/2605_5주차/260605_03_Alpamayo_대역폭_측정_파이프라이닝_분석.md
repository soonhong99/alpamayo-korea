# Alpamayo 1.5 대역폭 측정 및 파이프라이닝 가능성 분석

**작성일**: 2026-06-05  
**실험 환경**: Jetson AGX Thor · LPDDR5X 196 GB/s (실측) · BF16

---

## 여기서 말하는 "대역폭"이란

직접 측정한 DRAM 처리량이 아니다. 다음 공식으로 계산한 **단순 추정치**다.

```
추정 Bandwidth (GB/s) = 해당 단계 모델 가중치 크기 (GB)
                       ─────────────────────────────
                            실측 처리 시간 (s)
```

가중치 크기만 분자에 넣은 값이므로 실제 DRAM 접근보다 낮은 하한치다.  
정확한 DRAM 처리량은 GPU 프로파일러로 직접 측정해야 한다.

## 전체 추론 파이프라인

```
[VE] → [LM Prefill] → [Decode × 17 steps] → [Flow × 10 steps]
728ms     1,423ms        79ms × 17 = 1,345ms    87ms × 10 = 870ms

합계: 4,366ms
```

Thor GPU 실측 최대 대역폭: **196 GB/s**

---

## 단계별 추정 Bandwidth

| 단계 | 가중치 크기 | 처리 시간 | 계산 | 추정 Bandwidth |
|------|------------|----------|------|---------------|
| VE | 4.0 GB | 728 ms | 4.0 ÷ 0.728 | ~5 GB/s |
| LM Prefill | 16.4 GB | 1,423 ms | 16.4 ÷ 1.423 | ~12 GB/s |
| Decode | 16.4 GB | 79 ms/step | 16.4 ÷ 0.079 | ~207 GB/s |
| Flow | 4.6 GB | 87 ms/step | 4.6 ÷ 0.087 | ~53 GB/s |

---

### VE — ~5 GB/s

VE의 입력 크기는 `pixel_values [11520, 1536]`로 확인된다 (`260604_05` 실험).

```
4 cameras × 4 timesteps = 16장 이미지
16장 × 720 patches/장   = 11,520 patches  (pixel_values 텐서 shape으로 확인)
```

VE는 이 11,520개 패치를 입력으로 받아 시각 토큰을 생성한다.  
내부에서 self-attention 연산이 수행되지만, attention이 정확히 몇 개 토큰에 대해 동작하는지는  
모델 코드 또는 profiler로 별도 확인이 필요하다.  
가중치 4 GB를 한 번 올려두면, 그 이후는 패치들 사이의 행렬 연산이 대부분의 시간을 차지한다.

결과적으로 728 ms라는 긴 시간 동안 가중치 로딩은 초반에만 발생하고,  
이미 올라온 데이터로 연산만 하는 시간이 길어 평균 BW가 낮게 나온다.

---

### LM Prefill — ~12 GB/s

LM의 입력 길이는 `input_ids [1, 3086]`로 확인된다 (`260604_05` 실험).  
3,086 토큰의 구성은 `260602_02` 실험에서 직접 측정됐다.

```
text_prefix :   29 토큰  (시스템 프롬프트 등)
vision      : 2,982 토큰  (VE 출력 이미지 임베딩, image_token_id=151655)
ego         :   61 토큰  (자차 속도·조향 등 주행 상태)
text_suffix :   14 토큰  (응답 시작 마커 등)
─────────────────────────
합계        : 3,086 토큰
```

Prefill은 이 3,086개 토큰을 한꺼번에 처리한다.  
각 linear layer에서 가중치를 1번 읽으면, 그걸로 3,086개 토큰 전체에 대한 연산을 한꺼번에 처리한다.  
가중치 1 byte 당 연산이 3,086배 일어나는 구조다.

가중치 16.4 GB를 1,423 ms에 걸쳐 읽으므로 ~12 GB/s가 나온다.

---

### Decode — ~207 GB/s

Decode는 매 step마다 토큰 1개를 생성한다.  
그런데 그 1개를 만들기 위해 16.4 GB 가중치 전체를 처음부터 끝까지 다시 읽어야 한다.

Prefill에서는 가중치 1 byte로 3,086개 토큰을 처리했지만,  
Decode에서는 가중치 1 byte로 토큰 1개만 처리한다.

짧은 시간(79 ms) 안에 16.4 GB를 읽으므로 207 GB/s가 나온다.  
이는 Thor 실측 최대치(196 GB/s)를 넘는 수치다.  
L2 캐시에 일부 가중치가 남아 실제 DRAM 접근이 16.4 GB보다 적기 때문으로 보인다.

---

### Flow — ~53 GB/s

Flow는 Action Expert 모듈로, 64개 waypoint(주행 경로)를 10 step에 걸쳐 생성한다.  
자체 가중치 4.6 GB를 매 step마다 읽으므로 4.6 ÷ 0.087 = 53 GB/s가 나온다.

## 파이프라이닝 가능성

VE와 Decode를 동시에 실행하는 실험을 진행했다 (실측).

```
VE 단독 실행:          724 ms
VE + Decode 동시 실행: 578 ms  (21% 빠름)
Decode + VE 동시:      Decode 7% 느림
```

이를 기반으로 한 Async VE Pipeline 이론 계산:

```
현재 (순차):  4,366ms
Async VE:    3,732ms  (−634ms, 1.17×)

조건: VE 동시 실행 시 578ms < Decode 창 1,439ms ✅
```

---

## 남는 자원 활용 방안

| 방안 | 근거 | 예상 효과 |
|------|------|----------|
| **Async VE Pipeline** | 동시 실행 실측 확인 | −634ms → 3,732ms (1.17×) |
| **Adaptive-Step Flow** (FlashDrive) | Flow 10 step 중 불필요 step 재사용 | Flow 870→348ms → 3,210ms (1.36×) |
| **CPU 전처리 파이프라이닝** (확인 완료) | BCD 310ms << GPU 4,366ms | wait=0ms 확인 |

## 정확한 프로파일링 방법

현재 대역폭 수치는 `가중치 크기 ÷ 실측 시간`으로 계산한 추정치다.  
더 정확한 측정을 위한 도구는 다음과 같다.

---

### 현재 사용 중인 방법

#### 1. CUDA Events — 단계별 실행 시간

```python
start = torch.cuda.Event(enable_timing=True)
end   = torch.cuda.Event(enable_timing=True)
start.record()
# ... 단계 실행 ...
end.record()
torch.cuda.synchronize()
elapsed_ms = start.elapsed_time(end)
```

- **측정 대상**: 각 단계의 실행 시간 (ms)
- **한계**: 시간만 측정. DRAM 실제 접근량은 알 수 없음
- **현재 상태**: `scripts/inference/` 전반에 적용 완료

---

#### 2. nsys (Nsight Systems) — 커널 분포 확인

Thor에서 실제로 실행한 적 있다 (`260508` 프로파일링).

```bash
nsys profile --trace=cuda,nvtx \
             --output /tmp/alpamayo_profile \
             python3 scripts/profiling/260513_profile_v4.py
```

- **측정 대상**: 어떤 CUDA 커널이 얼마나 실행됐는가
- **확인된 결과 예시 (Decode 구간)**: splitK GEMM 47.9%, GEMM-nvjet 37.1%, GEMV 6.0%
- **한계**: 커널 실행 시간은 보여주지만 DRAM 접근량은 알 수 없음

---

### 아직 하지 않은 방법

#### ncu (Nsight Compute) — 커널별 실제 DRAM 바이트

GPU에서 실행되는 모든 연산은 CUDA 커널 단위로 쪼개진다.  
ncu는 각 커널이 실행될 때 GPU 내부 하드웨어 카운터를 읽어서  
그 커널이 DRAM에서 실제로 몇 바이트를 읽었는지 직접 센다.

```bash
ncu --metrics dram__bytes_read.sum,dram__bytes_write.sum \
    python3 scripts/profiling/260513_profile_v4.py
```

- `dram__bytes_read.sum`: 커널이 DRAM에서 읽은 실제 바이트
- `dram__bytes_write.sum`: 커널이 DRAM에 쓴 실제 바이트

이 두 값을 커널 실행 시간으로 나누면 실제 DRAM 처리량이 나온다.  
**주의**: ncu는 실행 속도가 크게 느려지므로 바이트 카운팅 전용으로만 사용해야 한다.

---

### 세 방법의 역할 정리

| 도구 | 측정 대상 | Thor 실행 여부 |
|------|----------|--------------|
| CUDA Events | 단계별 실행 시간 | ✅ 완료 |
| nsys | 커널별 실행 시간 분포 | ✅ 완료 |
| ncu | 커널별 실제 DRAM 접근 바이트 | ❌ 미실행 |

# GPU L2 Cache · KV Cache 최적화 실험 보고

**작성일**: 2026-06-05  
**실험 환경**: Jetson AGX Thor · BF16 · AppendOnlyCache-C

---

## 핵심 결론

> 1. **GPU L2 Cache(32 MB)는 KV Cache(455 MB)를 담기에 너무 작다** — 전체를 L2에 고정하는 것은 불가능하다.
> 2. **L2에 올릴 수 있는 일부 블록을 pinning해도 효과는 ~4 ms** — 의미 있는 수치가 아니다.
> 3. **대신 KV Cache 접근 패턴을 바꾸는 방식(AppendOnlyCache-C)으로 Decode 107ms → 79ms (-26%) 달성했다.**

---

## 1. 배경 — GPU L2 Cache와 KV Cache란

### GPU L2 Cache

GPU는 DRAM(메인 메모리)에서 데이터를 가져올 때 속도를 높이기 위해 내부에 L2 Cache를 둔다.  
L2에 있는 데이터는 DRAM보다 훨씬 빠르게 읽을 수 있다.

```
DRAM (128 GB)  →  GPU L2 Cache (32 MB)  →  연산 유닛(SM)
  느림                   빠름
```

**Thor의 GPU L2 Cache 크기: 32 MB** (실측 확인)

### KV Cache

LLM이 문장을 생성할 때, 앞서 계산한 내용(Key, Value 행렬)을 저장해 재계산을 피하는 메모리 공간이다.  
Alpamayo의 Decode 단계에서 매 step마다 이 KV Cache를 참조한다.

**Alpamayo의 KV Cache 크기: 455 MB → 458 MB** (실측 확인)

KV Cache는 고정 크기가 아니다. Prefill이 끝난 시점부터 시작해 Decode step마다 커진다.

```
측정 기준: Prefill 완료 후 (seq_len = 3,086)
구조     : 36 layers × 2(K,V) × 8 heads × 3,086 tokens × 128 dim × 2 bytes(BF16)
         = 455.0 MB

Decode step 진행 시 증가량 (실측):
  step당 +151 KB  (이론: 144 KB, 메모리 정렬 padding으로 +7 KB 발생)
  17 steps 후  : 455.0 + (151 KB × 17) ≈ 457.6 MB

시작: 455.0 MB → 완료: ~458 MB  (증가폭 0.6%)
```

증가폭이 작아 L2 수용 여부 결론은 바뀌지 않는다. (455 MB든 458 MB든 L2 32 MB 대비 14배 초과)

## 2. L2 Cache와 KV Cache 크기 비교

```
GPU L2 Cache:  ████████  32 MB
KV Cache 시작: ████████████████████████████████████████████████████  455 MB  (Prefill 직후)
KV Cache 완료: █████████████████████████████████████████████████████ 458 MB  (Decode 17 step 후)

KV Cache가 L2의 약 14배
→ KV Cache 전체를 L2에 올리는 것은 불가능
```

---

## 3. L2에 올릴 수 있는 것 vs 없는 것

KV Cache를 통째로 L2에 올릴 수 없으므로, 블록 단위로 분석했다.  
Thor L2의 실제 사용 가능 영역은 **24 MB** (전체 32 MB의 75%)다.

| KV 블록 | 크기 | L2 수용 가능 여부 | 반복 접근 횟수 |
|---------|------|:---:|------|
| **VLM KV 전체** | **455→458 MB** | ❌ 14× 초과 | Decode + Flow (동적) |
| **text_prefix KV** (앞 100 토큰) | **14.7 MB** | ✅ | 프레임마다 불변 구간 |
| **expert 64tok KV** (Flow 내부) | **9.4 MB** | ✅ | 10 steps × 36 layers = 360회 |
| **레이어 1개 KV** (3,086 토큰 기준) | **12.6 MB** | ✅ | Decode step 수만큼 (동적) |

> ※ Decode step 수는 EOS 토큰 발생 시점에 따라 매 추론마다 달라진다. 17회는 실험 관측값이다.

**text_prefix + expert 합산: 24.1 MB → 24 MB 한계 초과, 동시 pinning 불가**

---

## 4. L2 Pinning 실험 결과 — 효과가 미미한 이유

L2에 올릴 수 있는 블록(14.7 MB + 9.4 MB)을 고정했을 때 예상 효과를 계산했다.

```
Flow 단계 전체 DRAM 접근량:
  expert 가중치 읽기:  10 steps × 4.56 GB = 45,600 MB  ← 91%
  VLM KV 읽기        : 10 steps × 455 MB  =  4,550 MB  ←  9%
  합계: 50,150 MB

L2 Pinning으로 절약 가능:
  (14.7 + 9.4) MB × ~300회 = 7,230 MB
  전체 50,150 MB 대비 약 0.5%

예상 시간 절감: ~4 ms
```

KV Cache를 빠르게 읽어도, **전체 시간의 91%를 차지하는 가중치(4.56 GB) 읽기 시간은 줄어들지 않는다.**  
→ L2 Pinning 단독으로는 의미 있는 개선이 불가능하다.

## 5. 실제로 효과가 있었던 최적화 — AppendOnlyCache-C

L2 Pinning 대신, **KV Cache를 메모리에 접근하는 방식 자체를 바꿔** 성공했다.

### 기존 방식의 문제 (DynamicCache)

Decode는 매 step마다 KV Cache에 토큰 1개(0.15 MB)를 추가해야 한다.  
기존 `DynamicCache`는 이를 위해 불필요한 복사를 반복했다.

```
[매 Decode step마다 발생하는 낭비]

목표: 455 MB 버퍼에 0.15 MB 추가
실제: 455 MB 전체를 새 공간에 복사 → 0.15 MB 추가 → 원래 버퍼 삭제

1 step당 낭비: 455 MB 읽기 + 455 MB 쓰기 = 910 MB
17 steps 합계: 15,470 MB 낭비 (의미 없는 복사)
```

### AppendOnlyCache-C의 해결

세 가지 변화를 적용했다.

```
① 사전 할당 (torch.cat 제거)
   처음부터 최대 크기 버퍼를 확보해두고
   매 step마다 빈 자리에 in-place로 바로 쓴다
   → 복사 없음

② 연속 메모리 배치 (.contiguous())
   head(8개) 사이에 빈 공간 없이 붙여서 반환
   → GPU prefetcher가 순차 접근 인식 → 효율 최대

③ L2 재사용 (의도치 않은 발견)
   in-place write 직후 바로 read하는 패턴 덕분에
   방금 쓴 데이터(12.6 MB/layer < 32 MB L2)가 L2에 남아있음
   → FlashAttention이 DRAM 대신 L2에서 읽음
```

### 실험 결과

| 구현 방식 | Decode step당 시간 | 전체 추론 시간 |
|-----------|:-----------------:|:-------------:|
| DynamicCache (기존) | 107 ms | 4,838 ms |
| AppendOnlyCache-B (연속 미적용) | 100 ms | — |
| StaticCache | 102 ms | — |
| **AppendOnlyCache-C (최종)** | **79 ms** | **3,620 ms** |

**Decode: -26% / 전체 파이프라인: -25%**  
모델 수정 없음, 양자화 없음 — 순수 메모리 접근 패턴 최적화만으로 달성.

## 6. VE와 Flow에는 왜 같은 방법을 쓸 수 없나

AppendOnlyCache-C는 Decode에만 효과적이었다. 이유를 단계별로 정리한다.

### VE — 구조 자체가 다름

Decode의 LLM은 "앞에서 뒤로" 순서대로 생성하는 단방향 구조다.  
VE는 이미지의 모든 패치가 서로를 동시에 참조하는 **양방향 구조**다.

```
Decode (단방향):  패치_1 → 패치_2 → 패치_3  ← "이전 결과 저장" 가능
VE (양방향)  :  패치_1 ↔ 패치_2 ↔ 패치_3  ← "이전"이라는 개념 없음
```

이미지는 공간적 관계가 중요하기 때문에 양방향이어야 한다.  
→ KV Cache를 누적해서 재사용하는 방식이 원천적으로 적용되지 않는다.

### Flow — 구조는 맞지만 병목이 다른 곳에 있음

Flow의 10 step ODE 구조는 매 step마다 KV를 리셋해서 누적이 없다.  
그리고 설령 KV를 최적화해도, 전체 시간의 91%는 **가중치(4.56 GB) 반복 읽기**가 차지한다.  
KV를 아무리 빠르게 처리해도 가중치 읽기 시간이 줄어들지 않는다.

## 7. 각 단계별 추가 최적화 가능 여부

| 단계 | 현재 시간 | KV 최적화 가능 | 이유 | 남은 방법 |
|------|:--------:|:---:|------|------|
| VE | 728 ms | ❌ | 양방향 ViT | 비동기 오버랩 (Decode와 병렬 실행) |
| Prefill | 895 ms | ✅ | KV Temporal Reuse 미완 | **다음 실험 최우선** |
| Decode | 1,345 ms | ⚠️ 한계 근처 | 79ms ≈ 이론 하한 86ms | 추가 개선 어려움 |
| Flow | 637 ms | ❌ | 가중치 읽기가 91% 병목 | 없음 (한계 도달) |

---

## 8. 다음 실험 방향

AppendOnlyCache-C로 Decode 병목이 해소된 후, **병목이 Prefill(895 ms)로 이동했다.**  
현재 가장 큰 개선 가능성은 **KV Temporal Reuse**다.

```
아이디어:
  연속된 주행 프레임에서 장면이 거의 변하지 않는다 (100ms 간격).
  → 이전 프레임에서 계산한 vision KV(444 MB)를 다음 프레임에 재사용
  → Prefill에서 전체 3,086 토큰 대신 변경된 75 토큰만 처리

예상 효과:
  Prefill 895 ms → 44 ms  (-95%)
  전체  3,620 ms → 2,769 ms  (-24%)
```

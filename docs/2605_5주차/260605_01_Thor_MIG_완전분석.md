## 1. MIG란 무엇인가

### 1-1. 기본 개념

MIG(Multi-Instance GPU)는 GPU를 여러 개의 **완전히 격리된 물리 인스턴스**로 분할하는 NVIDIA 하드웨어 기능이다.

```
일반 GPU (MIG 없음):
  [프로세스 A] ─┐
  [프로세스 B] ─┼── SM Pool (공유) ← GPU 스케줄러가 직렬 처리
  [프로세스 C] ─┘                     → 실질적 병렬화 없음

MIG GPU:
  [프로세스 A] ── [Instance 1: 12 SM, 전용 캐시, 전용 BW] ─┐
                                                              ├── 진짜 동시 실행 ✅
  [프로세스 B] ── [Instance 2:  8 SM, 전용 캐시, 전용 BW] ─┘
```

핵심 차이: **하드웨어 레벨 격리** (소프트웨어 스케줄링이 아님). 각 인스턴스는 SM, L2 캐시 슬라이스, 메모리 대역폭을 독점적으로 점유.

### 1-2. MIG vs CUDA Multi-Stream vs CUDA MPS

| 기술 | 격리 방식 | 메모리 격리 | 동시 실행 | Thor 지원 |
|------|----------|------------|----------|-----------|
| CUDA Multi-Stream | 소프트웨어 (GPU 스케줄러) | ❌ 없음 | ❌ Thor에서 직렬화됨 | — |
| CUDA MPS | 소프트웨어 (SM% 제한) | ❌ 없음 | ⚠️ 부분적 | ✅ 지금 가능 |
| **MIG** | **하드웨어 물리 분할** | **✅ 완전 격리** | **✅ 진짜 병렬** | **✅ JetPack 7.2** |

**중요**: 우리가 이전 실험(260604_cuda_stream_concurrency_test.py)에서 확인한 SM∥SM 직렬화는 CUDA Multi-Stream 레벨의 문제였다. MIG는 하드웨어가 다른 층에서 동작하므로 이 한계를 우회한다.

## 2. Thor MIG 하드웨어 스펙

### 2-1. Thor GPU 전체 구성

```
Jetson AGX Thor — Blackwell GPU (SM 11.0)
  총 SM 수:   20개 (10 TPC × 2 SM/TPC)
  총 CUDA 코어: 2560개 (20 SM × 128 cores/SM)
  DRAM:       131.9 GB unified (CPU∥GPU 공유 LPDDR5X)
  대역폭:     273 GB/s (이론), 실측 231 GB/s
  L2 캐시:    32 MB
```

### 2-2. MIG 파티션 구성 (JetPack 7.2 기준)

Thor MIG는 **고정 2-way 분할**만 지원한다 (A100의 7-way 자유 분할과 다름):

```
┌─────────────────────────────────────────────────────────┐
│                  Thor GPU (20 SM 전체)                  │
├──────────────────────────────┬──────────────────────────┤
│   Instance 1 (AI∥Graphics)  │  Instance 2 (Robotics)   │
│   12 SM / 1536 CUDA cores   │  8 SM / 1024 CUDA cores  │
│   L2 슬라이스 비례 할당      │  L2 슬라이스 비례 할당   │
│   DRAM BW 비례 할당          │  DRAM BW 비례 할당       │
├──────────────────────────────┴──────────────────────────┤
│          물리적으로 격리된 독립 하드웨어 파티션          │
└─────────────────────────────────────────────────────────┘
```

| 인스턴스 | SM | CUDA 코어 | L2 | BW 배분 | 용도 |
|---------|-----|----------|-----|---------|------|
| Instance 1 | **12 SM** | 1,536 cores | 비례 (60%) | ~139 GB/s | AI inference / graphics / 범용 CUDA |
| Instance 2 | **8 SM** | 1,024 cores | 비례 (40%) | ~92 GB/s | robotics / control / 안전 중요 |

**왜 고정인가**: A100처럼 GPC 단위 자유 분할이 아닌 하드웨어 회로 레벨에서 고정 파티션으로 설계됨. iGPU 특성상 유연한 분할보다 결정론적 격리를 우선.

### 2-3. 메모리 격리 방식 — unified memory의 특수성

A100과 Thor의 근본적 차이:

```
A100 (dGPU):
  Instance 1: 20 GB HBM 독점
  Instance 2: 20 GB HBM 독점
  → nvidia-smi에 "20gb" 표시됨

Thor (iGPU):
  CPU와 GPU가 동일한 물리 DRAM 131.9 GB 공유
  MIG 메모리 할당: 가상 주소 공간 분리 + BW 비례 할당
  → nvidia-smi에 "0gb" 또는 가상 용량 표시 가능
  → 실제 물리 메모리는 비격리 (CPU도 공유)
```

이 때문에 `nvidia-smi mig -lgip`에서 프로파일이 `1g.0gb` 형태로 나올 수 있다.

## 3. 실험 이력 — JetPack 7.1 실패에서 7.2 성공까지

### 3-1. JetPack 7.1 (2026-05-15) — 실패 확인

실제 Thor에서 실행한 결과:

```bash
sudo nvidia-smi -mig 1
# ✅ Enabled MIG Mode for GPU 00000000:01:00.0

sudo nvidia-smi mig -lgip
# ❌ Failed to display GPU instance profiles: Unknown Error (RC=255)
# (테이블 헤더만 출력, 실제 프로파일 없음)
```

**실패 원인 (DCE RPC 레이어 차단)**:

```
nvidia-smi mig -lgip 실패 경로:
  nvidia-smi
    → NVML: nvmlDeviceGetGpuInstanceProfiles()
    → RM 드라이버
    → DCE(Display Controller Engine) RPC 레이어  ← 여기서 차단
       NVRM: rpcRmApiControl_dce: Failed RM ctrl call
             cmd:0x731341 result 0xffff (Generic Error)
```

Thor iGPU는 GPU가 SoC에 내장되어 DCE RPC 경로를 경유하는데, JetPack 7.1까지 이 경로에 MIG GI(GPU Instance) 생성 명령이 구현되어 있지 않았다.

| GI 파티셔닝 (`-cgi`, `-lgip`) | ❌ Unknown Error | ✅ 작동 |

NVIDIA 공식 포럼 발언 (1차 소스):

## 4. MIG 활성화 및 사용 방법

### 4-1. MIG 활성화

```bash
# 1. MIG 모드 활성화 (JetPack 7.2 이상 필요)
sudo nvidia-smi -mig 1
sudo reboot  # 재부팅 필요

# 2. 재부팅 후 GI 프로파일 확인 (이제 작동해야 함)
sudo nvidia-smi mig -lgip
# 예상 출력:
# +----------------------------------------------------------------+
# | GPU instance profiles:                                         |
# | GPU  Name       ID  Instances  Memory  P2P  SM  DEC  ENC      |
# |                      Free/Tot  GiB               CE  JPEG OFA |
# |================================================================|
# |  0  MIG 3g.*gb  1    1/1      0      No   12   ...           |
# |  0  MIG 2g.*gb  2    1/1      0      No    8   ...           |
# +----------------------------------------------------------------+

# 3. GPU 인스턴스 생성 (12SM + 8SM 두 개)
sudo nvidia-smi mig -cgi 1,2 -C
# -cgi: Create GPU Instance (프로파일 ID 지정)
# -C:   Compute Instance도 동시 생성

# 4. 생성된 MIG 인스턴스 UUID 확인
sudo nvidia-smi -L
# 예상 출력:
# GPU 0: Jetson Thor (UUID: GPU-xxxx)
#   MIG 3g.*gb  Device 0: (UUID: MIG-yyyy)   ← 12SM instance
#   MIG 2g.*gb  Device 1: (UUID: MIG-zzzz)   ← 8SM instance

nvidia-smi mig -lgi   # 생성된 인스턴스 목록
nvidia-smi mig -lci   # Compute Instance 목록
```

### 4-2. 프로세스를 특정 MIG 인스턴스에 배정

**방법 1: CUDA_VISIBLE_DEVICES (UUID 방식)**

```bash
# 12 SM 인스턴스에서 Alpamayo 추론 실행
CUDA_VISIBLE_DEVICES=MIG-<UUID-12SM> \
  python3 scripts/inference/260604_cpu_preprocess_pipeline_v2.py

# 8 SM 인스턴스에서 다른 작업 실행
CUDA_VISIBLE_DEVICES=MIG-<UUID-8SM> \
  python3 scripts/inference/some_other_task.py
```

**방법 2: 코드 내 device 지정**

```python
import os

# 12SM 인스턴스 사용
os.environ["CUDA_VISIBLE_DEVICES"] = "MIG-<UUID-12SM>"
import torch
device = torch.device("cuda:0")  # → 12SM 인스턴스만 보임

model = Alpamayo1_5.from_pretrained(
    "nvidia/Alpamayo-1.5-10B",
    dtype=torch.bfloat16,
    local_files_only=True,
).to(device).eval()
```

**방법 3: NVIDIA Container Toolkit (Docker)**

```bash
# 컨테이너에 MIG 인스턴스 직접 할당
docker run --gpus '"device=MIG-<UUID-12SM>"' \
  alpamayo-container python3 run_inference.py
```

### 4-3. MIG 비활성화 (원래 상태 복귀)

```bash
# 인스턴스 삭제 후 MIG 비활성화
sudo nvidia-smi mig -dci   # Compute Instance 삭제
sudo nvidia-smi mig -dgi   # GPU Instance 삭제
sudo nvidia-smi -mig 0     # MIG 모드 비활성화
sudo reboot
```

## 5. 우리 연구(Alpamayo)에서의 활용 방안

### 5-1. 이전 실험에서 확인한 한계와 MIG의 관계

`260604_cuda_stream_concurrency_test.py` 실험 결과 (2026-06-04):

```
SM∥SM 병렬화 (CUDA Multi-Stream):
  Small  (prealloc): 0.97×  ← 직렬화 확인
  Medium (prealloc): 0.99×
  Large  (prealloc): 1.00×

결론: Thor 단일 SM Pool → GPU 스케줄러가 stream을 직렬 처리
```

**MIG는 이 한계를 어떻게 해결하는가?**

```
Multi-Stream (MIG 없음):
  SM Pool [20 SM] 공유
  → stream A, B 모두 같은 스케줄러 큐 → 직렬

MIG (하드웨어 분할):
  Instance 1 [12 SM] → 프로세스 A 전용 스케줄러
  Instance 2 [ 8 SM] → 프로세스 B 전용 스케줄러
  → 두 스케줄러가 물리적으로 독립 → 진짜 동시 실행 ✅
```

### 5-2. Alpamayo 연속 추론에 MIG를 적용하는 3가지 시나리오

#### 시나리오 A: 크로스프레임 파이프라인 (권장)

우리가 `260515_mig_pipeline_experiment_plan.md`에서 설계했던 핵심 실험.

```
현재 (직렬, MIG 없음):
F0: [VE 728ms][LM 1423ms][Decode 1345ms][Flow 870ms]
                                                     → F1 시작
F1:                                                  [VE 728ms][LM 1423ms]...

1 cycle = 4,366ms

MIG 크로스프레임 파이프라인:
  Instance 1 (12 SM, 메인 추론):
    F0: [VE 728ms][LM 1423ms][Decode 1345ms][Flow 870ms]
                                ↓
    F1:                         [VE ~1000ms?][LM...][Decode...][Flow...]

  Instance 2 (8 SM, 전처리∥초기 연산):
    F0:              (대기)
    F1: (Decode F0 동안) [VE_F1 ~1090ms?]  ← 12SM보다 느리지만 동시에 실행
```

Decode(F0, 1345ms)와 VE(F1, ~1090ms on 8SM)를 겹치면:
- 이전 불가능했던 VE overlap이 MIG로 가능해짐
- 이론 가속: Decode+Flow(2215ms) 동안 VE(~1090ms)가 숨음
- 효과: VE 시간 1090ms 중 상당 부분 절감

#### 시나리오 B: Alpamayo 2-inference 병렬 (실험 부하 감소)

```
Instance 1 (12 SM): inference(frame k, 카메라 세트 A)
Instance 2 ( 8 SM): inference(frame k, 카메라 세트 B) ← 동시 실행

용도: 실험 시 서로 다른 config 비교를 half-time에 완료
     (기존: A실험 → B실험 순차, 시간 2× 소요)
```

#### 시나리오 C: 실시간 inference + 안전 모니터링 분리

```
Instance 1 (12 SM): Alpamayo 추론 (main policy)
Instance 2 ( 8 SM): 안전 체크 모델 (collision detection 등)
→ 안전 체크가 추론 성능에 영향 없이 독립 실행
```

### 5-3. MIG가 해결하는 것과 해결하지 못하는 것

| 항목 | MIG 전 | MIG 후 |
|------|--------|--------|
| 두 GPU 프로세스 진짜 병렬 실행 | ❌ 직렬 | ✅ 가능 |
| 크로스프레임 VE overlap | ❌ 불가 | ✅ 가능 (8SM 인스턴스에서) |
| 단일 프레임 내 VE∥Decode | ❌ 불가 | ❌ 여전히 불가 (동일 인스턴스 내) |
| 단일 inference 레이턴시 감소 | — | ❌ 없음 (오히려 증가 가능) |
| 메모리 격리 | ❌ 없음 | ✅ 가상 격리 (unified memory 특성 상 물리 분리는 아님) |

**중요 경고 — 단일 프레임 레이턴시는 증가한다**:

```
현재 (전체 GPU, 20 SM):
  VE: 728ms, LM: 1423ms, Decode: 1345ms
  총: 4,366ms

MIG 후 Instance 1 (12 SM = 60%):
  VE: ~1213ms (728 / 0.6), LM: ~2372ms, Decode: ~2242ms
  총: ~7,300ms ← 67% 증가!

→ MIG는 단일 스트림 레이턴시를 희생하고
  두 프로세스 동시 실행을 가능하게 하는 트레이드오프
```

따라서 **단일 연속 inference에 MIG를 적용하면 오히려 느려진다**. MIG의 이점은 두 개의 독립 프로세스가 동시에 실행될 때만 나타난다.

## 6. 크로스프레임 파이프라인 구체 설계

### 6-1. 이론적 성능 예측

```
파이프라인 조건:
  Instance 1 (12 SM): VE+LM+Decode+Flow (메인 추론)
  Instance 2 ( 8 SM): VE_next 사전 계산

Instance 2에서 VE 시간 (8SM = 40% 자원):
  현재 VE = 728ms @ 20SM
  8SM에서: 728ms × (20/8) ≈ 1,820ms (BW-bound 비율 고려)
  실제: compute-bound VE는 SM 비례 → ~1,456ms (최선)
        BW-bound 일부 포함 시 → ~1,000~1,500ms

Instance 1의 Decode+Flow 시간 = 1,345 + 870 = 2,215ms

VE_next(Instance 2) < Decode+Flow(Instance 1):
  최선 케이스: 1,000ms < 2,215ms → VE 완전히 숨음 ✅
  보통 케이스: 1,456ms < 2,215ms → 759ms 절감 가능 ✅
  최악 케이스: 1,820ms < 2,215ms → 395ms 절감 ✅
```

### 6-2. 구현 코드 구조

```python
import os
import threading
import torch

# ─── MIG 인스턴스 설정 ─────────────────────────────
MIG_UUID_MAIN = "MIG-<12SM-UUID>"   # VE+LM+Decode+Flow
MIG_UUID_VE   = "MIG-<8SM-UUID>"    # VE 사전 계산

# ─── 메인 추론 모델 (12SM) ────────────────────────
os.environ["CUDA_VISIBLE_DEVICES"] = MIG_UUID_MAIN
model_main = Alpamayo1_5.from_pretrained("nvidia/Alpamayo-1.5-10B",
    dtype=torch.bfloat16, local_files_only=True).to("cuda").eval()

# ─── VE 사전 계산 모델 (8SM) ──────────────────────
# 별도 프로세스로 실행 (multiprocessing.Process 또는 subprocess)
# CUDA_VISIBLE_DEVICES=MIG-<8SM-UUID> python3 ve_worker.py

# ─── 파이프라인 루프 ──────────────────────────────
def pipeline_loop():
    # Frame 0: cold start (VE_next를 8SM에서 시작)
    ve_next = asyncio.run(run_ve_on_8sm(frame_1_data))

    for k in range(N_FRAMES):
        data_k   = frames[k]
        data_k1  = frames[k+1] if k+1 < N_FRAMES else None

        # ① 8SM: VE(k+1) 시작 (비동기)
        if data_k1:
            ve_thread = threading.Thread(
                target=run_ve_worker,
                args=(data_k1, ve_queue)
            )
            ve_thread.start()

        # ② 12SM: LM Prefill(k) — VE(k)의 결과 이미 있음
        #    → Decode(k) → Flow(k) (2,215ms)
        output_k = run_lm_decode_flow(model_main, ve_result_k, data_k)

        # ③ VE(k+1) 완료 대기 (8SM이 <2,215ms면 wait≈0ms)
        ve_result_k1 = ve_queue.get()

        yield output_k
        ve_result_k = ve_result_k1
```

### 7-2. JetPack 7.2 업데이트 절차

```bash
# SDK Manager (권장) — Windows/Linux host에서:
sdkmanager --product Jetson --version 7.2 --target AGX_Thor

# 또는 Jetson ISO 직접 플래싱:
# UEFI r38.4.x (JetPack 7.1 설치 후) → JetPack 7.2 업그레이드 가능
```

**⚠️ 주의**: JetPack 7.2로 업데이트 후 PyTorch 2.8.0 소스 빌드를 재확인해야 할 수 있음 (드라이버 버전 변경으로 CUDA 헤더 불일치 가능성).

### 7-3. 업데이트 후 즉시 실행할 실험 순서

```bash
# Step 1: MIG GI 프로파일 확인
sudo nvidia-smi mig -lgip
# → 이제 프로파일 목록이 나와야 함 (JetPack 7.1에서는 Unknown Error였음)

# Step 2: 12SM + 8SM 인스턴스 생성
sudo nvidia-smi mig -cgi <12sm_profile_id>,<8sm_profile_id> -C
sudo nvidia-smi -L  # UUID 확인

# Step 3: 단순 병렬 실행 확인 (기준선)
# 터미널 1: 12SM에서 inference
CUDA_VISIBLE_DEVICES=MIG-<12SM> python3 scripts/inference/260604_cpu_preprocess_pipeline_v2.py

# 터미널 2: 8SM에서 inference 동시 실행
CUDA_VISIBLE_DEVICES=MIG-<8SM> python3 scripts/inference/260604_cpu_preprocess_pipeline_v2.py

# 두 프로세스의 총 실행 시간 비교:
# MIG 없음: 순차 → 4,366ms × 2 = 8,732ms
# MIG 있음: 병렬 → max(12SM 시간, 8SM 시간) ← 진짜 동시 실행 검증

# Step 4: 크로스프레임 파이프라인 구현 및 측정
# (260515_mig_pipeline_experiment_plan.md의 EXP-3/EXP-5 실행)
```

## 8. A100 MIG vs Thor MIG 비교

| 항목 | A100 MIG | Thor MIG |
|------|---------|---------|
| 분할 방식 | 7-way 자유 분할 (1g~7g) | **고정 2-way** (12SM + 8SM) |
| 메모리 | HBM 물리 분할 (GB 단위) | unified DRAM 가상 분리 |
| nvidia-smi 표기 | `3g.20gb` 등 | `Xg.0gb` (unified 특성) |
| 목적 | 멀티테넌시 서버 | **혼합 실시간 워크로드** |
| 격리 강도 | 완전 물리 격리 | SM/BW 격리, 물리 메모리 공유 |
| Technology Preview | — | ✅ (JetPack 7.2 기준) |

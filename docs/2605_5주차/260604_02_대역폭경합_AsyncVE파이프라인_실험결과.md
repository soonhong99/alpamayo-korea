## 1. 왜 이 실험을 했는가

### 배경 — Alpamayo의 추론 구조

Alpamayo 1.5가 한 번 추론할 때 내부에서 3단계가 순서대로 실행된다.

```
[Vision Encoder (VE)]  →  [LM Prefill]  →  [Autoregressive Decode]
       724ms                 1,772ms              ~1,200ms
```

- **VE**: 16장의 카메라 이미지를 입력받아 시각적 특징을 계산한다. Qwen3-VL의 Vision Transformer가 실행되며, 2,880개의 visual token을 생성한다.
- **LM Prefill**: 시각 특징 + 텍스트 프롬프트 + 주행 이력을 LLM(Qwen3-8B 계열)에 한꺼번에 입력한다. KV 캐시가 구축된다.
- **Decode**: autoregressive하게 토큰을 하나씩 생성하며, 추론 결과(trajectory)가 나올 때까지 반복한다.

현재 이 3단계는 **순서대로** 실행된다. 따라서 한 번 추론하는 데 약 3,200ms가 걸린다.

### 핵심 질문

Alpamayo는 10Hz(100ms 간격)로 연속 추론해야 한다. 이는 현재 구조로는 추론 하나가 완료되기 전에 다음 추론이 이미 시작되어야 한다는 의미다. 이 상황에서 **"추론 k의 Decode가 실행되는 동안, 추론 k+1의 VE를 미리 실행해 둘 수 있는가?"** 라는 질문이 핵심이다.

```
[현재 순차 구조]
추론 k:   ─── VE ─── Pf ───────── Dec ──────────
추론 k+1:                                         ─── VE ─── Pf ──── Dec ───

[목표: Async VE Pipeline]
추론 k:   ─── VE ─── Pf ───────── Dec ──────────
추론 k+1:              └─── VE ───┘     Pf ─── Dec ───
```

VE(724ms)가 Decode(~1,200ms) 안에 완전히 숨는다면 매 추론마다 VE 시간을 절약할 수 있다.

### iGPU 고유 이슈

이 아이디어는 논리적으로는 맞지만, Thor의 하드웨어 구조 때문에 확인이 필요하다.

| | Thor (iGPU) | RTX PRO 6000 (dGPU, FlashDrive 환경) |
|---|---|---|
| GPU 메모리 | CPU와 공유 DRAM 231 GB/s | 전용 VRAM 1,008 GB/s |
| GPU 코어(SM) | 20개 | 96개 |
| VE와 Decode 동시 실행 시 | DRAM 대역폭과 SM 공유 | VRAM과 SM 각자 독립 |

**의문**: Thor에서 VE와 Decode를 동시에 실행하면 DRAM 대역폭이나 SM 자원을 서로 빼앗는 경합이 발생하는가? 발생한다면 얼마나 느려지는가?

이 경합 인수를 실측해야 Async VE Pipeline의 실제 효과를 계산할 수 있다.

## 2. 핵심 원리 — 왜 VE와 Decode를 동시에 실행할 수 있는가

이 실험 전체의 근거가 되는 원리다. 실험 결과를 이해하기 전에 반드시 이해해야 한다.

### GPU가 일하는 방식 — SM과 DRAM

GPU 안에는 실제로 연산을 수행하는 처리 단위인 **SM(Streaming Multiprocessor)** 이 있다. Thor GPU에는 20개가 있다. SM이 연산을 하려면 먼저 데이터(모델 가중치, KV 캐시 등)를 메인 메모리(DRAM)에서 읽어와야 한다.

```
SM이 연산하는 흐름:

  [DRAM에 가중치 요청] → [DRAM이 가중치 전송] → [SM이 연산] → [다음 가중치 요청] → ...
         ↑                       ↑                    ↑
     즉시 완료             시간이 걸림             매우 빠름
   (명령만 전달)         (데이터가 와야 함)        (데이터만 있으면)
```

SM이 요청을 보낸 후 DRAM에서 데이터가 올 때까지 SM은 **아무것도 하지 못하고 기다린다**.

### Decode는 왜 느린가 — 계산보다 데이터 이동이 병목

Decode 한 step에서 무슨 일이 벌어지는지 수치로 본다.

```
Decode 한 step이 하는 일:

  DRAM에서 읽어야 하는 데이터:  16.4 GB  (LM 가중치 전체)
  그 데이터로 하는 연산:         16.4 GFLOPs

  데이터 이동에 걸리는 시간:  16.4 GB ÷ 196 GB/s ≈ 84ms
  연산에 걸리는 시간:         16.4 GFLOPs ÷ 7,700 GFLOPS ≈ 2ms
```

데이터를 가져오는 데 84ms, 실제 연산은 2ms.

```
Decode 한 step (99ms) 동안 SM의 상태:

│░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░◼◼│
 ←─────────────── 98%: DRAM 응답 대기 중 ──────────→ ←2%→
                                                    SM 연산
```

**이론 계산 기준: Decode 실행 중 SM의 약 98%는 DRAM 응답을 기다리며 아무것도 하지 않고 있다.**

> **측정 한계**: 이 98% 수치는 FLOPs 이론식과 추정 GPU 피크로부터 계산된 값이다. SM 사용률을 직접 측정하려면 NCU 하드웨어 카운터가 필요한데, Thor의 SM 11.0 아키텍처에서는 해당 메트릭이 미지원이다(`ncu dram__bytes_read.sum → (!) n/a`). 다만 실측 decode 시간(99ms)이 이론 메모리 하한(84ms)에 근접한다는 사실, 그리고 Nsight Systems에서 decode 커널의 85%가 "BW 위주"로 분류된다는 사실이 memory-bound라는 결론을 간접적으로 지지한다.

이것이 Decode가 느린 이유다. 계산 자체가 오래 걸리는 게 아니라, 계산에 필요한 가중치를 DRAM에서 가져오는 데 시간이 걸리는 것이다. 계산은 데이터가 도착하면 순식간에 끝난다.

### Prefill은 왜 빠른가 (비교)

```
Prefill이 하는 일:

  DRAM에서 읽어야 하는 데이터:  16.4 GB  (Decode와 동일)
  그 데이터로 하는 연산:         3,086토큰 × 16.4 GFLOPs ≈ 50,000 GFLOPs

  데이터 이동에 걸리는 시간:    84ms
  연산에 걸리는 시간:           50,000 GFLOPs ÷ 7,700 GFLOPS ≈ 6,500ms
```

같은 16.4GB를 읽지만, 3,086개 토큰을 동시에 처리하기 때문에 연산량이 3,086배 많다. 연산이 너무 많아 DRAM 대기가 아니라 SM 연산이 병목이 된다. 이것이 **compute-bound** 상태다.

| | Decode (토큰 1개) | Prefill (토큰 3,086개) |
|---|---|---|
| DRAM 로드 | 16.4 GB | 16.4 GB (동일) |
| 연산량 | 16.4 GFLOPs | 50,600 GFLOPs |
| 병목 | DRAM 대기 | SM 연산 |
| SM 상태 | 98% 대기 중 | 대부분 바쁨 |

### VE와 Decode가 동시에 실행 가능한 이유

VE(Vision Encoder)는 2,880개의 visual token에 대해 attention 연산을 집중적으로 수행한다. Prefill과 비슷하게 **compute-bound** 작업이다.

Decode가 실행되는 동안 SM의 98%가 DRAM 대기 상태로 놀고 있다. VE는 이 유휴 SM 사이클에 연산을 집어넣을 수 있다.

```
Decode + VE 동시 실행 시 SM 활용:

Decode만 실행:
│░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░◼◼│
  ↑ SM 98%가 이 시간 동안 완전히 낭비됨

Decode + VE 동시 실행:
│VE연산VE연산VE연산VE연산VE연산VE연산VE연산VE연산◼◼│
  ↑ 같은 시간 동안 VE가 이 유휴 사이클을 사용함
```

**VE는 Decode가 DRAM을 기다리는 동안 SM에서 연산을 수행할 수 있다. 두 작업이 각자 다른 하드웨어 자원을 쓰기 때문에 서로를 방해하지 않는다.**

Decode: DRAM 대역폭을 소비하며 SM은 대기  
VE: SM 연산 능력을 소비하며 DRAM은 거의 사용 안 함

이것이 Phase 2-B에서 VE가 단독 실행(724ms)보다 빠르게(574ms) 완료된 이유다. Decode의 DRAM 대기 사이클 동안 VE가 SM을 전부 활용할 수 있어서, 사실상 SM 점유율이 단독 실행보다 높아졌다.

## 4. 실험 설계

### 전체 구성

실험은 4개의 Phase로 구성된다.

```
Phase 0: 모델 로드 + JIT 워밍업
Phase 1: 각 단계를 단독으로 실행해 기준선 측정
Phase 2: 두 작업을 동시에 실행해 경합 인수 측정
Phase 3: 실제 연속 추론 파이프라인 실측 및 이론 추정
```

### 타이머 방식 — CUDA Event 기반

CPU의 `time.perf_counter()`는 GPU 작업이 실제로 끝난 시점이 아니라 GPU 작업을 **요청한** 시점을 잰다. GPU 연산 시간을 정확히 재려면 GPU 내부 하드웨어 타이머를 사용해야 한다.

```python
start_event = torch.cuda.Event(enable_timing=True)
end_event   = torch.cuda.Event(enable_timing=True)

start_event.record()     # GPU 타임라인에 시작 시점 기록
(GPU 연산 실행)
end_event.record()       # GPU 타임라인에 종료 시점 기록
torch.cuda.synchronize() # GPU 완료까지 CPU 대기

elapsed_ms = start_event.elapsed_time(end_event)  # GPU 하드웨어 시간 차이
```

이 방식을 쓰면 서로 다른 CUDA Stream에서 기록된 이벤트들 간의 시간 차이도 GPU 하드웨어 타이머 기준으로 계산할 수 있다. Phase 2의 겹침 판정이 이 원리를 사용한다.

### CUDA Stream 이란?

GPU는 작업을 Stream이라는 채널로 구분해서 받는다. 같은 Stream 안의 작업들은 순서대로 실행된다. 서로 다른 Stream의 작업들은 **GPU가 하드웨어 자원이 허락하는 한** 동시에 실행될 수 있다.

```
stream_default: [decode step 0] → [decode step 1] → ...
stream_ve:      [VE]
                   ↕ 두 stream이 실제로 동시에 실행되는가?
```

Phase 2는 이 질문에 직접 답한다.

## 5. Phase 1 — 기준선 측정

5회 JIT 워밍업 후 각 단계를 10회 반복 측정했다.

### 3-A. VE 단독

| 항목 | 수치 |
|------|------|
| 평균 | **724.0ms** |
| 표준편차 | 9.0ms |
| 중앙값 | 722.4ms |
| 출력 텐서 형태 | (2880, 4096) |

출력 텐서 (2880, 4096)의 의미:
- 2880 = 16장 이미지 × 720 patches ÷ 4 (2×2 spatial merge) = 장당 180 visual token
- 4096 = Qwen3-VL의 hidden dimension

JIT 첫 실행(warmup 0)에서 1,086ms → 2번째부터 714ms로 안정화. 371ms가 JIT 컴파일 비용이다.

### 3-B. Decode Step 단독

| 항목 | 수치 |
|------|------|
| 평균 | **99.45ms** |
| 표준편차 | 0.60ms |
| 캐시 종류 | DynamicCache (기본값) |

이 실험에서 Decode는 DynamicCache를 사용한다. 이전 실험(AppendOnlyCache-C)에서는 약 79~80ms가 측정됐었다. 두 수치의 차이는 캐시 구현 방식(torch.cat 반복 vs in-place 쓰기)에서 비롯된다.

참고값: 22GB 모델 가중치 ÷ 231 GB/s DRAM = 95.2ms (DRAM 대역폭 기준 순수 전송 시간)

### 3-C. Full Prefill 단독 (VE + LM 합산)

| 항목 | 수치 |
|------|------|
| 평균 | **2,496.8ms** |
| 표준편차 | 4.2ms |

분리 계산:
- VE 부분: 724ms
- LM Prefill만: 2,497 − 724 = **1,773ms**

이 실험은 DynamicCache 기본값을 사용하기 때문에 2,496ms가 나온다. AppendOnlyCache-C를 적용하면 이전 실험에서 약 1,981ms로 감소했다.

## 6. Phase 2-A — Decode 경합 측정 (VE를 백그라운드로 실행)

### 실험 구성

```
[GPU stream_ve  ]  ───── VE (~724ms) ─────────────────────────────→
[GPU default    ]  [step0][step1][step2]...[step14]
                    ↑
                    VE와 겹치는가?
```

VE를 백그라운드 stream(`stream_ve`)에 올려놓고, default stream에서 즉시 Decode step 15개를 연속 실행했다. GPU가 두 stream을 동시에 처리하면 앞쪽 decode step들이 VE 실행 중에 겹쳐야 한다.

### CUDA Event 기반 겹침 판정 방식

```
ve_start_ev (stream_ve에서 기록)
step_start_ev[k] (default stream에서 기록)

step_start_rel = ve_start_ev.elapsed_time(step_start_ev[k])
  → "VE가 시작된 후 몇 ms 뒤에 step k가 시작됐는가"

판정:
  step_start_rel < VE_총시간  AND  step_end_rel > 0  →  "VE와 겹침"
  step_start_rel ≥ VE_총시간                          →  "VE 이후"
```

서로 다른 stream에 기록된 CUDA Event들도 GPU 하드웨어 타이머 기준으로 시간 차이를 구할 수 있어서, 실제로 겹쳤는지 여부를 GPU 내부 기준으로 판정할 수 있다.

### 측정 결과

| 실행 | VE 시간 | VE 중 겹친 steps | VE 중 step 평균 | VE 후 step 평균 |
|------|---------|-----------------|-----------------|-----------------|
| Run 1 | 723ms | **1개** | 112.5ms | 99.7ms |
| Run 2 | 725ms | **1개** | 112.5ms | 99.2ms |
| Run 3 | 726ms | **1개** | 113.6ms | 99.5ms |
| Run 4 | 722ms | **1개** | 113.2ms | 99.3ms |
| Run 5 | 563ms† | **1개** | 80.3ms | 72.0ms |

†Run 5는 VE와 decode step이 모두 이례적으로 빠르게 측정됨 (GPU 상태 변화 추정, outlier).

**집계 (Runs 1–4 기준):**
- 단독 decode: **99.45ms**
- VE 중 겹친 step: **112.9ms** (13% 느림)
- VE 후 step: **99.4ms** (단독과 동일)
- **경합 인수 c_dec = 1.07**

### 핵심 관찰 — 예상보다 훨씬 적은 겹침

예상: VE = 724ms, step = 99ms → **7.3 steps** 가 겹쳐야 함  
실제: **1 step** 만 겹침

기대했던 것의 1/7 수준이다. 이 결과는 GPU의 SM(Streaming Multiprocessor) 자원 배분 방식에서 비롯된 것으로 보인다.

VE는 ViT의 attention 연산 중심으로 compute-intensive하다. Thor GPU의 SM이 20개인데, VE 커널이 대부분의 SM을 점유한다. Decode를 위한 SM이 거의 남지 않아 Decode가 VE 완료를 기다리는 형태가 된다. 첫 번째 step만 VE 커널이 완전히 SM을 점령하기 직전 짧은 구간에 일부 SM을 얻어 실행된 것으로 보인다.

> **c_dec = 1.07 해석 주의**: 이 수치는 "1개의 step이 겹쳤을 때 7% 느려졌다"는 의미다. "VE와 동시에 실행해도 decode가 7%만 느려진다"는 의미가 아니다. 실제로는 대부분의 decode steps가 VE가 끝날 때까지 기다렸다.

## 7. Phase 2-B — VE 경합 측정 (Decode를 백그라운드로 실행)

### 실험 구성

이번에는 반대 방향으로 실험했다. Decode를 백그라운드 stream(`stream_dec`)에 올려놓고, VE를 default stream에서 실행했다.

```
[GPU stream_dec ]  [step0][step1]...[step14]
[GPU default    ]  ─────── VE ───────
```

### 측정 결과

| 실행 | VE with Decode |
|------|---------------|
| Run 1 | 576.4ms |
| Run 2 | 574.8ms |
| Run 3 | 572.5ms |
| Run 4 | 574.4ms |
| Run 5 | 574.1ms |

| 항목 | 수치 |
|------|------|
| VE 단독 | 724ms |
| VE + Decode 동시 | **574ms** |
| **경합 인수 c_ve = 0.793** | VE가 오히려 21% 빠름 |

### 핵심 관찰 — VE가 더 빨라졌다

두 작업을 동시에 실행했는데 VE가 150ms 빠른 574ms가 나왔다. 단순한 경합을 예상했다면 반직관적인 결과다.

이 현상을 이해하려면 두 작업의 자원 사용 패턴이 다르다는 점을 봐야 한다.

```
Decode = memory-bound
  GPU SM들이 DRAM에 가중치를 요청하고 데이터가 도착하길 기다린다.
  SM이 연산 없이 대기하는 시간이 많다.

VE = compute-bound  
  SM에서 attention matrix 연산이 집중적으로 실행된다.
  DRAM 요청보다 연산 사이클이 많다.
```

Decode가 백그라운드 stream에서 실행되면서 SM들이 DRAM 응답을 기다리는 동안, VE가 default stream에서 그 유휴 SM 사이클을 활용해 연산을 처리한다. 두 작업의 자원 병목이 달라 SM 사이클 관점에서 상호 보완적으로 작동하는 것이다.

또한 default stream은 GPU 스케줄러에서 일반적으로 우선 처리되기 때문에, VE가 우선적으로 SM을 배정받는 효과도 있을 것으로 보인다.

### 수치 요약

### Phase 2 두 실험 비교

| | Phase 2-A | Phase 2-B |
|---|---|---|
| 구성 | VE → 백그라운드 stream<br>Decode → default stream | Decode → 백그라운드 stream<br>VE → default stream |
| 결과 | Decode steps가 VE와 거의 겹치지 않음<br>(7.3개 예상, 1개 실측) | VE가 단독보다 21% 빠름<br>(724ms → 574ms) |
| 해석 | VE가 SM 대부분 점유 → Decode 대기 | Decode의 DRAM 대기 중<br>유휴 SM을 VE가 활용 |

두 실험의 VE/Decode 역할이 바뀌었을 뿐인데 결과가 크게 다르다. GPU 스케줄러에서 default stream이 우선 처리되는 특성과 두 작업의 자원 병목 차이(compute vs memory)가 복합적으로 작용한 결과다.

### Async VE Pipeline 효과 추정 (수동 계산)

Async VE Pipeline을 실제로 구현하려면 현재 하나로 묶인 Full Prefill(VE+LM)을 분리해야 한다:
1. VE만 단독으로 먼저 실행 → `image_embeds` 캐시
2. LM Prefill에 `image_embeds`를 직접 주입 (VE 재실행 없이)
3. 이전 추론의 Decode 실행 중 다음 추론의 VE를 미리 실행

이 구조에서 steady-state per inference 시간:

```
현재 sequential:
  Full Prefill (VE 포함):   2,013ms
  Decode (AOC-C):           1,200ms
  합계:                     3,213ms

Async VE Pipeline (steady-state):
  LM-only Prefill:          2,013 − 724 = 1,289ms
  Decode (VE 겹침, c_dec=1.07 적용):  1,200 × 1.07 = 1,284ms
  합계:                     2,573ms

절감:   3,213 − 2,573 = 640ms/inference
가속비: 3,213 ÷ 2,573 = 1.25×
```

단, 이 추정은 VE가 Decode 안에 완전히 숨는다는 조건 하에 성립한다. Phase 2-B에서 c_ve = 0.793이므로 VE는 574ms에 완료되고, Decode는 1,284ms → VE < Decode 조건 만족. ✅

Phase 2-A에서 관찰한 "1 step만 겹침" 현상은 Phase 2-B 구성(VE가 default stream)을 사용하면 발생하지 않는다. 실제 Async 구현에서는 VE를 default stream에서 실행하는 Phase 2-B 방식을 적용해야 한다.

## 10. 다음 단계

### 즉시 구현 가능한 것

**Async VE Pipeline 구현** (`scripts/inference/260604_async_ve_pipeline.py`)

구현에 필요한 두 가지 핵심 변경:

1. `model.vlm.visual(pixel_values, grid_thw)` 로 VE를 분리 호출해 `image_embeds`를 얻는다.
2. LM Prefill에서 VE를 건너뛰고 `image_embeds`를 직접 주입한다. (Qwen3-VL의 `inputs_embeds` 경로 활용)

이 구현이 완료되면 이론 추정값(2,573ms/inference, 1.25× 가속)과 실측값을 비교할 수 있다.

### 추가 탐색 가치가 있는 방향

- **DLA(Deep Learning Accelerator) 활용**: Jetson AGX Thor에는 GPU와 별도의 DLA 하드웨어가 있다. VE를 DLA에서 실행하면 GPU SM과 DRAM 대역폭을 공유하지 않아 Phase 2의 경합 문제가 원천 차단된다. 다만 TensorRT로 VE를 DLA용 엔진으로 변환하는 추가 작업이 필요하다.
- **L2 Persistent Residency**: KV 캐시의 최근 레이어를 GPU L2(32MB)에 고정 배치해 DRAM 접근 횟수를 줄이는 실험.

# (2a 타당성) 구조축은 우리 CoT에 적용 불가 — 그래서 시간축이 더 넓다

**날짜**: 2026-06-16
**보드 상태**: MIG off, 클럭 1386 MHz 고정 (Thor SM 11.0, UMIC 융합)
**스크립트**: `umic/scripts/260616_cot_structure.py`, 로그 `profiling_results/260616_cot.log`

---

## 0. 한 줄

(2a) 구조축×시간축 합성을 구현하기 전에 **전제부터 측정**했다: FastDriveCoT식 구조 병렬은
*템플릿화된 다중 독립 하위작업* CoT를 가정한다. 그런데 **Alpamayo 1.5가 배포에서 실제로 내는 CoT는
"[행동] since [이유]" 단일 절(~17토큰)** 이라, 독립적으로 병렬 생성할 구조가 없다. 따라서 **구조축은
우리 CoT에 적용 불가**다. 이는 오히려 차별점이다 — 구조축이 *존재하지 않는* 최소 CoT에서도 **시간축
(우리)은 10.8×** 로 작동하므로, 우리 방법이 FastDriveCoT보다 **적용 범위가 넓다.**

시간 안정성 재확인.)

## 2. 왜 구조축이 안 되나

- FastDriveCoT(arXiv 2602.02864)는 CoT를 **"critical objects / traffic rules / decision …" 같은 독립
  하위작업의 의존성 그래프**로 분해해 한 forward에 병렬 생성한다. 전제는 *그런 다중 필드 템플릿*이다.
- 우리 CoT엔 **분해할 독립 필드가 없다.** 조각은 행동/이유 둘뿐이고, **이유가 행동을 설명하는 순차 의존**
  (\"Keep distance … since it is ahead\")이라 병렬화 의미가 없다. 길이도 ~17토큰으로 짧다.
- 즉 **구조축의 중복(독립 하위작업)이 이 모델의 배포 CoT엔 존재하지 않는다.** 합성할 두 축 중 하나가 없다.

## 3. 결론과 재포지셔닝

- **(2a) "구조축×시간축 곱셈 이득"은 이 CoT에선 증명 불가** — 구조축 자체가 성립 안 하기 때문. 구현(3–5일)
  하지 않는 게 옳다. 하루짜리 타당성 측정이 이를 막았다.
- **대신 더 강한 한 줄**: *구조축이 적용조차 안 되는 최소 CoT에서 시간축은 10.8× 가속*한다(260616_02).
  FastDriveCoT는 verbose 템플릿 CoT를 요구하지만, **우리 시간축은 Alpamayo가 실제 배포하는 미니멀
  CoT에서 바로 작동** → 적용 범위가 더 넓다는 게 측정으로 섰다.
- **단서/한계**: 모델을 프롬프트해 더 긴 구조화 CoT를 유도할 여지는 있으나, 그건 *모델/프롬프트 수정*이라
  우리 무수정 원칙 밖이다. 우리는 배포 그대로의 CoT를 가속한다.
